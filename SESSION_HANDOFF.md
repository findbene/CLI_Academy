## Decisions Made This Session

- Lesson-step tutor affordances must tolerate Next.js server rendering even inside a `"use client"` subtree; strict provider hooks remain useful, but leaf lesson UI that can prerender before the runtime bridge is hydrated should use an optional context read and guard side effects until runtime context exists.
- `apps/web/playwright.config.ts` is the authoritative smoke-test entry point for frontend route checks; running `apps/web/e2e/smoke.spec.ts` outside that config context produces invalid-URL artifacts because the spec intentionally uses relative route navigation against the configured `baseURL`.
- Progress reconciliation regressions should be tested deterministically by mocking `/api/progress` responses and, where needed, seeding anonymous local lesson progress directly in browser storage before the authenticated page load.
- The floating tutor should optimize first for visible conversation space on lesson pages; helper affordances stay available, but they belong behind an explicit Tools drawer instead of permanently occupying the chat viewport.
- Learning Mode needs to change visible lesson scaffolding in the step card itself, not just the tutor behavior; render-time mode logic is now the chosen contract.
- Hosted learner progress continues to self-heal by syncing the live curriculum catalog into Supabase instead of trusting stale seed rows.
- Browser-local learner progress is now treated as user-scoped state, not machine-global state, and only legacy anonymous entries are merged for backfill when no competing signed-in scopes exist.
- Local fallback success is now limited to recoverable sync failures (`401`, published-lesson mapping gaps, and transport failures) instead of masking all server-side save errors as successful completion.
- The guided-learning engine should evolve through schema-backed lesson contracts and shared runtime affordances, not through one-off lesson prose rewrites; Paths 01-19 are now explicit and ready for stricter enforcement.
- The guided-step contract is now enforced by default for live content validation; compatibility mode is opt-in and reserved for transitional audits rather than everyday validation.
- Targeted tutor regressions should mock `/api/tutor` when the assertion is about step-context handoff or tutor-mode routing, so the tests stay deterministic even when local tutor runtime or Anthropic connectivity varies.
- Local monorepo development can carry server-only tutor secrets in the repo root `.env`, but browser-exposed frontend keys still need to live in `apps/web/.env.local` to be visible to Next's client-side runtime.
- The floating tutor launcher should stay visible on learning/app surfaces, but it should not render on marketing routes where it blocks mobile hero and editorial content.
- On lesson/app surfaces, the open tutor modal should also own the viewport cleanly on mobile: no duplicate floating launcher while open and no launcher-sized bottom gap that exposes the bottom nav behind the modal.

## Risks To Watch

- Path overview completion on `/learn/[pathSlug]` still reflects hosted state only, so it can briefly lag behind lesson/dashboard UI after a purely local fallback save.
- Shared browsers can still hold legacy anonymous progress that is intentionally not auto-merged once multiple signed-in user scopes exist; that isolation is safer, but it can leave old anonymous progress stranded until manually resolved.
- Password recovery still depends on Supabase email templates and redirect configuration in the active environment.
- The new step schema is now fully explicit in Paths 01-19 and strict validation is live, but broader learner-flow regression coverage still needs to catch up around progress reconciliation and tutor gating.
