# Create a manual kill-switch routine

## Objective

Our goal today is to create a personal "kill-switch" routine for your local development setup. This isn't about being dramatic; it's about being prepared and safe! You'll learn how to identify your running local agents and services (like a local web server, database, or Docker containers) and document the quickest *safe* way to stop them. Having this routine at your fingertips gives you peace of mind and control, ensuring you can quickly shut things down without causing data loss or corruption, should you ever need to.

## Walkthrough

Let's create your personalized, documented stop procedure:

1.  **Identify Your Local Services**: Take a moment to think about what you commonly run locally. This might include:
    *   A development web server (e.g., Node.js, Python Flask, Ruby on Rails)
    *   A local database (e.g., PostgreSQL, MySQL, MongoDB)
    *   Docker containers
    *   Background services started by Homebrew, `systemctl`, or other package managers.

2.  **Find Their Graceful Stop Commands**: For each service, identify the *safest* and *fastest* way to shut it down.
    *   **For Foreground Processes (like a dev server):** If you started a process directly in your terminal and it's running in the foreground, pressing `Ctrl+C` is usually the safest and fastest way to stop it. This sends a `SIGINT` (interrupt) signal, which typically allows the application to perform a graceful shutdown (saving data, cleaning up connections) before exiting.
    *   **For Background Services/Agents:** These usually have dedicated stop commands:
        *   **Docker Containers:** Use `docker stop [container_name_or_id]`. This command sends a `SIGTERM` signal to the container, giving it a grace period to shut down cleanly before eventually sending a `SIGKILL` if it doesn't respond. This is a safe, controlled stop.
        *   **Systemd Services (Linux):** Use `sudo systemctl stop [service_name]`. This sends a stop signal to the system service, allowing it to shut down gracefully.
        *   **Homebrew Services (macOS):** Use `brew services stop [service_name]`. This safely tells Homebrew to stop the background service it manages.
        *   **Specific Application Commands:** Many applications (like a local database or a specific development tool) have their own `stop` commands. Consult their documentation if `Ctrl+C` or a generic service manager command doesn't apply.

3.  **Document Your Kill-Switch Routine**: Create a simple text file, perhaps named `stop_routine.md` or `kill_switch.txt`, in an easily accessible location (like your home directory or a `dev_tools` folder). In this file, list each service and its corresponding stop command.

    **Example Entry:**

    ```markdown
    # My Local Kill-Switch Routine

    ## Development Web Server (e.g., Node.js app)
    -   If running in foreground: Press `Ctrl+C` in the terminal where it's running. (Sends SIGINT for graceful shutdown)

    ## Local PostgreSQL Database (if running in Docker)
    -   `docker stop my-postgres-container` (Allows graceful shutdown before stopping)

    ## Local Redis Cache (if running via Homebrew)
    -   `brew services stop redis` (Safely stops the background Redis service)

    ## Important Note:
    -   Only use `kill -9 [process_id]` as an absolute last resort if a process is unresponsive. This immediately terminates a process without allowing it to clean up or save data, which can lead to data corruption or orphaned files. Prioritize graceful stops!
    ```

Now you have a clear, documented path to safely shut down your local development environment!
