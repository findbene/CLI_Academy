## What You'll See

By the end of this lesson, you will understand every configuration option ZeroClaw supports, what it auto-detects, and which defaults you should change before exposing ZeroClaw to your team.

## Why This Matters

ZeroClaw's configuration surface is intentionally small. The goal is minimal cognitive load. But "minimal" still requires deliberate choices on a handful of settings: session limits, token expiry, which models are allowed, and log rotation. This lesson is the complete reference.

## Core Concepts

### Full Configuration Reference

```yaml
# /etc/zeroclaw/config.yml

server:
  hostname: zeroclaw.yourdomain.com     # Required. Your server's FQDN
  port: 443                              # Default: 443
  tls_cert: /etc/zeroclaw/server.crt    # Required
  tls_key: /etc/zeroclaw/server.key     # Required
  read_timeout_seconds: 30              # Default: 30
  write_timeout_seconds: 120            # Default: 120
  max_connections: 500                  # Default: 500

api:
  anthropic_api_key: ${ANTHROPIC_API_KEY}  # Required (or set as env var)
  default_model: claude-sonnet-4-6         # Default: claude-sonnet-4-6
  allowed_models:                          # Default: all models
    - claude-haiku-4-5-20251001
    - claude-sonnet-4-6
    - claude-opus-4-6
  request_timeout_seconds: 180             # Default: 180 (3 minutes)

auth:
  mode: token                              # Only supported mode in ZeroClaw
  token_expiry_hours: 24                   # Default: 24
  max_tokens: 1000                         # Max tokens stored in memory
  allow_long_lived_tokens: false           # Default: false

sessions:
  max_concurrent: 200                      # Default: 200 active sessions
  idle_timeout_minutes: 60                 # Default: 60 minutes
  # Note: Sessions are in-memory. They are lost on process restart.

rate_limiting:
  enabled: true
  global_requests_per_minute: 600          # Global limit across all sessions
  burst: 50                                # Burst allowance

logging:
  path: /var/log/zeroclaw/zeroclaw.log
  level: info                              # debug | info | warn | error
  rotation: daily
  retention_days: 14                       # Default: 14 (shorter than OpenClaw's 30)

# ZeroClaw does NOT support:
# - auth.mode: oidc
# - admin dashboard configuration
# - per-developer rate limits
# - audit_log configuration (uses async batch logging internally)
# - nvidia / openShell configuration
```

### What ZeroClaw Auto-Detects

ZeroClaw infers these settings and does not require you to configure them:

- **System memory available** — automatically tunes `max_connections` down if the server is low on RAM
- **Number of CPU cores** — tunes worker thread count for optimal throughput
- **TLS protocol versions** — defaults to TLS 1.2 minimum, prefers TLS 1.3
- **OS-level connection limits** — reads `/proc/sys/net/core/somaxconn` and respects it

### Important Defaults to Review Before Going Live

**1. `allowed_models`**

If not set, ZeroClaw allows access to all Anthropic models including Opus. Set an explicit allowlist to control costs:

```yaml
api:
  allowed_models:
    - claude-sonnet-4-6       # Remove opus if you don't need it
```

**2. `allow_long_lived_tokens`**

Defaults to `false`, which is correct. Do not change this unless you have a specific automation use case. Long-lived tokens that are compromised give permanent access.

**3. `global_requests_per_minute`**

The default of 600 allows 10 requests per second across all sessions. For a 20-person team making frequent requests, this may be too low. Estimate your peak load: `expected_developers_active × avg_requests_per_minute` and set accordingly.

**4. `retention_days`**

ZeroClaw's default log retention is 14 days — shorter than OpenClaw's 30 days. If your organization has a 30-day log retention requirement, set this to `30`.

### Environment Variable Overrides

Any configuration value can be overridden with an environment variable using the pattern `ZEROCLAW_SECTION_KEY`:

```bash
# Override log level without editing config.yml
export ZEROCLAW_LOGGING_LEVEL=debug

# Override max connections
export ZEROCLAW_SERVER_MAX_CONNECTIONS=300
```

This is useful for container deployments or when you want different settings in different environments without multiple config files.

### Token Management

ZeroClaw's token commands:

```bash
# Create a token
zeroclaw token create --label "alice" --expiry 30d
# Output: zc_tok_abc123...

# List all tokens
zeroclaw token list

# Revoke a token
zeroclaw token revoke zc_tok_abc123

# Rotate a token (revoke + create in one step)
zeroclaw token rotate zc_tok_abc123
# Output: New token: zc_tok_xyz789...
```

Token labels are for your reference — they appear in logs as the developer identifier. Use meaningful labels: `alice-laptop`, `bob-ci`, `carol-dev`.

### Validating Configuration

```bash
zeroclaw validate --config /etc/zeroclaw/config.yml
# Exit code 0: valid
# Exit code 1: errors printed with line numbers
```

Run this before restarting the service after any config change.

## Key Takeaways

- ZeroClaw's configuration is intentionally minimal — approximately 20 settings versus OpenClaw's 60+.
- No OIDC, no per-developer rate limits, no audit log — these are the intentional feature gaps.
- Review `allowed_models`, `allow_long_lived_tokens`, `global_requests_per_minute`, and `retention_days` before going live.
- Use `zeroclaw validate` before any restart.
