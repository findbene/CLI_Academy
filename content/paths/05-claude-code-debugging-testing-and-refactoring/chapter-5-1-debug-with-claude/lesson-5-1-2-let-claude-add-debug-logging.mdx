# Let Claude add debug logging

## Objective

Ask Claude Code to insert temporary `console.log` or `print()` statements at key points in a script so you can trace what the code is actually doing at runtime. Run the script, read the trace, spot where the values go wrong, and then ask Claude to remove the debug lines once you understand the bug. This teaches you the most common real-world debugging technique — observation before guessing.

## Walkthrough

### Step 1 — Create a script with a subtle data bug

<StepMeta purpose="Create a non-crashing wrong-answer bug so you can practice observing runtime values instead of relying on stack traces." action="Write the grading script exactly as shown, including the incorrect hardcoded divisor that produces a believable but wrong result." expectedResult="You have a script that runs successfully but computes the wrong average and grade." whyItMatters="Many real bugs do not crash; they quietly produce bad data, which means you need visibility into values rather than an exception message." nextStep="Run the script and confirm that the output looks wrong even though nothing crashes." askTutor="I'm on Step 1 of Let Claude add debug logging. Help me understand why this bug is harder than a crash and what kind of wrong output I should expect." />

This script is supposed to calculate the average score and assign a letter grade, but it has a subtle bug — the average calculation is wrong because it divides by the wrong number.

**macOS / Linux**

```bash
mkdir -p ~/projects/debug-practice && cd ~/projects/debug-practice
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force ~\projects\debug-practice | Set-Location
```

**JavaScript version — `grader.js`**

```js
// grader.js — subtle math bug
function calculateAverage(scores) {
  let total = 0;
  for (const s of scores) {
    total += s;
  }
  return total / 4; // BUG: hardcoded 4 instead of scores.length
}

function letterGrade(avg) {
  if (avg >= 90) return "A";
  if (avg >= 80) return "B";
  if (avg >= 70) return "C";
  return "F";
}

const scores = [88, 92, 79];
const avg = calculateAverage(scores);
console.log(`Average: ${avg}, Grade: ${letterGrade(avg)}`);
```

**Python version — `grader.py`**

```python
# grader.py — subtle math bug
def calculate_average(scores):
    total = 0
    for s in scores:
        total += s
    return total / 4  # BUG: hardcoded 4 instead of len(scores)

def letter_grade(avg):
    if avg >= 90:
        return "A"
    if avg >= 80:
        return "B"
    if avg >= 70:
        return "C"
    return "F"

scores = [88, 92, 79]
avg = calculate_average(scores)
print(f"Average: {avg}, Grade: {letter_grade(avg)}")
```

### Step 3 — Ask Claude Code to add debug logging

<StepMeta purpose="Instrument the code so you can observe the internal values that lead to the wrong answer." action="Ask Claude to add temporary log statements around the critical calculations instead of guessing which line is broken." expectedResult="Claude inserts readable debug output that exposes the intermediate values involved in the bug." whyItMatters="Observation before guessing is the core of effective debugging, especially when the script is logically wrong but syntactically valid." nextStep="Run the instrumented script and read the logged values to identify where the numbers go off course." askTutor="I'm on Step 3 of Let Claude add debug logging. Help me decide what values are worth logging so the trace explains the bug without flooding the terminal." />

Open Claude Code in the project directory and prompt:

```
The grader script is outputting the wrong average. Add console.log statements at each step so I can see the total, the divisor, and the result.
```

Claude inserts logging lines throughout the script. For the JavaScript version it might add:

```js
console.log("Scores:", scores);
console.log("Total:", total);
console.log("Dividing by:", 4);
console.log("Average:", avg);
```

Accept the changes and run the script again.

### Step 4 — Read the trace and spot the problem

<StepMeta purpose="Use the runtime trace to identify the exact incorrect assumption or hardcoded value causing the bug." action="Run the logged script and compare the printed totals, divisors, and outputs against the expected math." expectedResult="The trace makes the bug visible, such as a hardcoded divisor that does not match the actual array length." whyItMatters="A good trace shrinks debugging time because you stop scanning code abstractly and start reasoning from concrete runtime facts." nextStep="Now that the bug is obvious, ask Claude to replace the broken logic with the correct calculation." askTutor="I'm on Step 4 of Let Claude add debug logging. Help me interpret this trace and point out which logged value proves where the bug lives." />

The output now shows something like:

```
Scores: [ 88, 92, 79 ]
Total: 259
Dividing by: 4
Average: 64.75, Grade: F
```

You can see it immediately: the total is 259 and there are 3 scores, but the code divides by 4. The hardcoded `4` is the bug. You would have struggled to spot this just by reading the code, but the trace makes it obvious.

### Step 5 — Ask Claude to fix the bug

<StepMeta purpose="Convert the evidence from the trace into one precise code fix." action="Tell Claude exactly what the trace proved and ask it to replace the hardcoded divisor with the real collection length." expectedResult="Claude updates the calculation so the script returns the mathematically correct average and grade." whyItMatters="Debug logs are only useful if you follow them through to a focused fix rather than leaving the root cause implicit." nextStep="After the result is correct, ask Claude to remove the temporary logging it added." askTutor="I'm on Step 5 of Let Claude add debug logging. Help me verify that this fix addresses the true root cause and not just the visible symptom." />

```
The divisor is hardcoded as 4 but there are only 3 scores. Fix it to use the actual array length.
```

Claude changes `return total / 4` to `return total / scores.length` (JavaScript) or `return total / len(scores)` (Python). Accept the fix and run again:

```
Average: 86.33333333333333, Grade: B
```

Correct.

### Step 6 — Ask Claude to remove the debug logging

<StepMeta purpose="Clean the script back to a production-worthy state after the debugging session is done." action="Ask Claude to remove every temporary debug log it added, then run the script one final time." expectedResult="The script keeps the correct behavior but no longer prints internal debug noise." whyItMatters="Temporary instrumentation is useful during diagnosis, but leaving it behind makes future output noisy and can leak internal data in real projects." nextStep="After runtime tracing, practice a more systematic isolation method with binary-search debugging prompts." askTutor="I'm on Step 6 of Let Claude add debug logging. Help me confirm that all temporary logging is gone and only the meaningful program output remains." />

```
The bug is fixed. Remove all the debug console.log lines you added.
```

Claude strips out every temporary logging statement and leaves only the original logic. Accept the changes.

Run the script one last time to confirm it produces clean output:

```
Average: 86.33333333333333, Grade: B
```

No debug noise, correct answer.

> **Real danger — do not commit debug logging.** Debug `console.log` and `print()` statements left in production code clutter output and can accidentally expose internal data. Always remove them before committing. Asking Claude to clean them up is a reliable way to make sure none slip through.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude adds logging inside a loop and you get hundreds of lines | The loop iterates over a large dataset | Ask Claude to log only the first 3 iterations or log a summary after the loop |
| The debug output is in a different order than expected | Asynchronous code (Node.js callbacks/promises) runs out of order | Ask Claude to add `await` or use synchronous logging; or ask Claude to number each log line |
| You accepted the cleanup but one `console.log` remains | Claude missed a line it did not add (it was already in your code) | Run a quick search: `grep -n "console.log" grader.js` on macOS/Linux or `Select-String "console.log" grader.js` on Windows |
| Output looks correct but the test still fails | Floating point rounding — `86.333...` is not exactly `86.33` | Ask Claude about rounding: `Math.round(avg * 100) / 100` (JS) or `round(avg, 2)` (Python) |
| Claude adds logging to the wrong function | You were not specific enough about where the bug is | Tell Claude exactly which function: "Add logging only inside calculateAverage" |
