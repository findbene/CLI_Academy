# Run existing tests and interpret failures

## Objective
You'll learn how to run automated tests in a project from the command line. More importantly, we'll gently walk through how to interpret test failures, helping you understand *what* went wrong and *where* to start looking, without feeling overwhelmed.

## Walkthrough

1.  **Navigate to the project directory:**
    First, let's make sure we're in the right place. Use `cd` to move into your project folder. For example, if your project is named `my-app`:
    ```bash
    cd my-app
    ```
    *(Why this is safe: `cd` only changes your current location in the file system; it doesn't modify any files or data.)*

2.  **Run the project's tests:**
    Most projects use a common command to run tests. We'll use `npm test`, a very common practice in JavaScript projects. If your project uses something different (like `yarn test` or `pytest`), your instructions might vary slightly, but the principles are the same!
    ```bash
    npm test
    ```
    *(Why this is safe: Running tests typically just executes code to check for errors; it doesn't usually make permanent changes to your system or files outside the project scope. It's a "read-only" operation in terms of impact on your overall machine.)*

3.  **Observe the test output:**
    You'll likely see a stream of information in your terminal. For this lesson, we expect to see some failures. Look for keywords like `FAIL`, `Error`, or `Expected`. A common output might look something like this:

    ```
    > my-app@1.0.0 test
    > jest

    FAIL src/utils/calculator.test.js
      ● add function › should correctly add two numbers

        expect(received).toBe(expected) // Object.is equality

        Expected: 5
        Received: 6

          at Object.<anonymous>.test (src/utils/calculator.test.js:10:20)
    ```

4.  **Interpret the failure:**
    Don't worry, this output isn't as scary as it looks! Let's break it down:
    *   `FAIL src/utils/calculator.test.js`: This tells us *which test file* contained the failing test.
    *   `● add function › should correctly add two numbers`: This is the *name of the specific test* that failed.
    *   `Expected: 5 Received: 6`: This is the core of the problem! The test *expected* the result to be `5`, but the code actually returned `6`. This indicates our `add` function isn't working quite right for this specific case.
    *   `at Object.<anonymous>.test (src/utils/calculator.test.js:10:20)`: This is super helpful! It's a "stack trace" that points directly to the *exact line number* (line 10) and character (column 20) within `src/utils/calculator.test.js` where the test assertion failed.

5.  **Create your `test-readout.md` file:**
    Now, based on the output you observed (or the example above), create a file named `test-readout.md` in your current directory. Inside this file, explain in plain English what failed and why it matters, including the likely file to inspect first.

    Here's an example of what your `test-readout.md` could contain, adapting it to your specific output:

    ```markdown
    The tests failed!

    **What failed:**
    The test "should correctly add two numbers" in `src/utils/calculator.test.js` failed. It expected the `add` function to return `5`, but it actually returned `6`.

    **Why it matters:**
    This means our `add` function is not producing the correct sum in at least one scenario. If this function is used elsewhere in the application, it could lead to incorrect calculations and unexpected behavior for users. It's crucial to fix this to ensure the application's reliability.

    **Likely file to inspect first:**
    Based on the error message, the first file I'd look at is `src/utils/calculator.test.js` at line 10, to understand the test's exact expectation. After understanding the test, I would then examine the `add` function itself (likely in a file like `src/utils/calculator.js` or similar) to correct the logic.
