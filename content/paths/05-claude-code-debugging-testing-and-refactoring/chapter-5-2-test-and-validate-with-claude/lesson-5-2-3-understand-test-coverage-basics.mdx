# Understand test coverage basics

## Objective

Run a coverage report on your test suite using `npx jest --coverage` or `pytest --cov`, read the report to find which lines and branches are not tested, and then ask Claude Code to write a test that covers one of those gaps. By the end of this lesson you will understand what the percentages in a coverage report mean and how to improve them intentionally.

## Walkthrough

### Step 1 — Add a function with branches to test

<StepMeta purpose="Create code with multiple paths so a coverage report has meaningful gaps to reveal." action="Add the new clamp function to the existing source module but intentionally do not write tests for it yet." expectedResult="The source file contains uncovered branch logic that should appear as missing lines in the first coverage report." whyItMatters="Coverage only teaches something interesting when there are branches or lines your tests are not already executing." nextStep="Install or confirm the coverage tooling so you can generate a report against the current suite." askTutor="I'm on Step 1 of Understand test coverage basics. Help me see why we are adding an untested function on purpose before generating the first coverage report." />

We need a function with `if/else` paths so that coverage reporting has something interesting to show. Add a `clamp` function to your existing module.

**JavaScript — add to `math-utils.js`**

```js
function clamp(value, min, max) {
  if (value < min) return min;
  if (value > max) return max;
  return value;
}

module.exports = { add, multiply, clamp };
```

**Python — add to `math_utils.py`**

```python
def clamp(value, min_val, max_val):
    if value < min_val:
        return min_val
    if value > max_val:
        return max_val
    return value
```

Do **not** add tests for `clamp` yet. We want the coverage report to show a gap.

### Step 4 — Read the coverage report

<StepMeta purpose="Translate raw coverage percentages and missing-line numbers into an actionable testing plan." action="Look at statements, branch coverage, and missing line numbers and connect them back to the clamp function you just added." expectedResult="You understand that uncovered lines mean tests never executed that function or its branches." whyItMatters="Coverage is only useful if you can read the report and decide what specific test would close the gap." nextStep="Ask Claude to write tests that cover every branch in the uncovered clamp function." askTutor="I'm on Step 4 of Understand test coverage basics. Help me make sense of statement coverage versus branch coverage and what each one tells me here." />

The key columns are:

- **% Stmts** (or **Cover**) — what percentage of lines in the file were executed during tests.
- **% Branch** — what percentage of `if/else` paths were taken. 0% means no branches were tested.
- **Uncovered Line #s** (or **Missing**) — the exact line numbers your tests never reached.

Lines 10–12 (or 9–11 in Python) are the `clamp` function. Since we have no tests for `clamp`, those lines were never executed. That is the coverage gap.

### Step 5 — Ask Claude Code to add a test for the uncovered function

<StepMeta purpose="Turn a specific coverage gap into a targeted new test rather than chasing percentages blindly." action="Tell Claude exactly which function and branches are uncovered and ask it to generate tests for each missing case." expectedResult="Claude creates tests that exercise all three clamp branches: below range, above range, and in range." whyItMatters="The best use of coverage is directing your next test toward a concrete missing path, not trying to improve numbers at random." nextStep="Run coverage again and compare the new report to the original gap." askTutor="I'm on Step 5 of Understand test coverage basics. Help me make this prompt specific enough that Claude covers the uncovered branches instead of generating redundant tests." />

```
My coverage report shows that the clamp function on lines 10-12 of math-utils.js is 0% covered. Write tests that cover all three branches: value below min, value above max, and value in range.
```

Claude generates tests like:

```js
test("clamp returns min when value is below range", () => {
  expect(clamp(1, 5, 10)).toBe(5);
});

test("clamp returns max when value is above range", () => {
  expect(clamp(15, 5, 10)).toBe(10);
});

test("clamp returns value when in range", () => {
  expect(clamp(7, 5, 10)).toBe(7);
});
```

Or for Python:

```python
def test_clamp_below():
    assert clamp(1, 5, 10) == 5

def test_clamp_above():
    assert clamp(15, 5, 10) == 10

def test_clamp_in_range():
    assert clamp(7, 5, 10) == 7
```

Accept the changes.

### Step 7 — Understand what 100% does and does not mean

<StepMeta purpose="Keep coverage in perspective so you use it as a guide rather than mistaking it for proof of correctness." action="Ask Claude whether 100% coverage means the code is bug-free and review the limits of that metric." expectedResult="You understand that coverage means lines ran, not that every important scenario or edge case was validated correctly." whyItMatters="Teams misuse coverage when they optimize for the number instead of the quality of the tests, so this distinction matters early." nextStep="With testing basics covered, move on to refactoring a messy function while using tests as a safety net." askTutor="I'm on Step 7 of Understand test coverage basics. Help me separate the value of coverage from the mistaken idea that 100% means the code is automatically correct." />

Ask Claude Code:

```
Does 100% test coverage mean my code is bug-free?
```

Claude explains that coverage only means every line was executed — it does not mean every possible input was tested or that the logic is actually correct. For example, if `clamp` had `>=` instead of `>` in one comparison, 100% coverage would still pass but boundary values would be wrong. Coverage is a tool for finding gaps, not a guarantee of correctness.

> **Real danger — do not chase 100% coverage as a goal in itself.** In real projects, some code (error handlers, platform-specific branches) is hard to test and the effort is not worth it. Focus on testing the logic that matters most: calculations, data transformations, and business rules.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `npx jest --coverage` shows 0% for all files | Jest is not finding your test file | Make sure the test file ends in `.test.js` and is in the same directory |
| pytest `--cov` says "No data was collected" | You passed the wrong path to `--cov` | Use `--cov=.` to measure coverage for files in the current directory |
| Coverage is 100% but you never added tests for `clamp` | Another test accidentally calls `clamp` through a dependency | Check which test covers the line: `npx jest --coverage --verbose` shows per-test details |
| Branch coverage is lower than line coverage | Some `if` paths are never taken | Add a test that triggers the untaken path — ask Claude which branch is missing |
| The coverage HTML report does not open | Your terminal does not have a default browser | Open the `coverage/lcov-report/index.html` file manually in a browser |
