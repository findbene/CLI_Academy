# Run tests and fix failures with Claude

## Objective

Introduce a bug into the function you tested in the previous lesson, run the test suite, see it fail with a clear error message, paste the failure into Claude Code, and let Claude fix the source code (not the test). Then run the tests again to confirm they pass. This teaches the red-green debugging cycle that professional developers use every day.

## Walkthrough

### Step 1 — Start from the working state

<StepMeta purpose="Establish a known-good baseline before you intentionally break the code and trigger a test failure." action="Return to the project from the previous lesson and rerun the tests to confirm everything is green first." expectedResult="The current suite passes, proving that any later failures come from the deliberate bug you are about to introduce." whyItMatters="A clean baseline makes debugging much easier because you know exactly when and why the failure appeared." nextStep="Introduce one intentional bug into the source code while leaving the tests untouched." askTutor="I'm on Step 1 of Run tests and fix failures with Claude. Help me confirm that this project is in a true green state before I start the red-green cycle." />

**macOS / Linux**

```bash
cd ~/projects/testing-practice
```

**Windows PowerShell**

```powershell
cd ~\projects\testing-practice
```

Run the tests to confirm everything is green:

```bash
npm test
```

```bash

```

All tests should pass.

### Step 2 — Introduce a deliberate bug

<StepMeta purpose="Create a controlled failure so you can see how tests surface the exact behavior that broke." action="Edit the source file to replace the correct add logic with an obviously wrong subtraction bug, without touching the tests." expectedResult="The source code is now broken in one small, known way while the tests still describe the intended correct behavior." whyItMatters="Intentional bugs are a safe way to learn how the test runner reports failures and how Claude should respond to them." nextStep="Run the test suite again and inspect which assertions fail and which ones still pass." askTutor="I'm on Step 2 of Run tests and fix failures with Claude. Help me make sure I'm only changing the source file and not accidentally invalidating the test lesson." />

Open the source file and break the `add` function on purpose. Change the `+` to `-`:

**JavaScript — `math-utils.js`**

```js
function add(a, b) {
  return a - b; // BUG: subtraction instead of addition
}
```

**Python — `math_utils.py`**

```python
def add(a, b):
    return a - b  # BUG: subtraction instead of addition
```

Save the file. Do not change the test file.

### Step 4 — Paste the failure into Claude Code

<StepMeta purpose="Use the failing test output as evidence to ask Claude for a source-code fix instead of a speculative rewrite." action="Paste the test failure into Claude and state clearly that the tests are correct and only the source code should change." expectedResult="Claude identifies the broken operator in the source and proposes a small fix without touching the test file." whyItMatters="The wording of the prompt matters here because otherwise Claude may 'fix' the problem by editing the expectations instead of the implementation." nextStep="Accept the fix and rerun the tests to confirm the suite returns to green." askTutor="I'm on Step 4 of Run tests and fix failures with Claude. Help me phrase this prompt so Claude fixes the implementation and leaves the tests alone." />

Copy the test failure output and paste it into Claude Code:

```
I ran `npm test` and got these test failures. The tests are correct — the bug is in the source code. Fix the source code, not the tests.

FAIL ./math-utils.test.js
  ● add returns the sum of two positive numbers
    Expected: 5
    Received: -1
```

Adding "the tests are correct" is important. Without that hint, Claude might try to change the test expectations instead of fixing the code.

Claude reads the test, reads `math-utils.js`, and immediately spots:

> `return a - b` should be `return a + b`.

Claude proposes the one-character fix. Accept it.

### Step 5 — Run the tests again to confirm green

<StepMeta purpose="Close the red-green loop by proving the source-code fix restored the intended behavior." action="Rerun the same test suite immediately after accepting the patch and verify that every test now passes." expectedResult="The terminal shows a fully green suite, confirming that the failure was resolved without regressions." whyItMatters="A fix is only trustworthy once the same automated checks that caught the bug now pass against the updated code." nextStep="Reflect on why this red-green cycle is more reliable than manual checking alone." askTutor="I'm on Step 5 of Run tests and fix failures with Claude. Help me confirm that this green test run proves the fix and not just a change to the test expectations." />

**macOS / Linux**

```bash
npm test
```

**Windows PowerShell**

```powershell
npm test
```

Output:

```
PASS  ./math-utils.test.js
  ✓ add returns the sum of two positive numbers (2 ms)
  ✓ add handles zero
  ✓ add handles negative numbers
  ✓ multiply returns the product of two numbers
```

All green. The red-green cycle is complete:

1. **Red** — Tests fail, showing exactly what is broken.
2. **Green** — Fix the code, run the tests, everything passes.

### Step 6 — Reflect on what just happened

<StepMeta purpose="Turn the exercise into a reusable development habit instead of a one-off fix." action="Ask Claude to explain the red-green cycle and compare it to manual spot-checking of output." expectedResult="You can describe how failing tests, targeted fixes, and reruns work together as a fast debugging loop." whyItMatters="The lesson is not just 'Claude fixed a bug' but 'tests gave precise evidence and verified the repair in seconds.'" nextStep="After red-green debugging, learn how coverage reports reveal untested code paths and guide the next test you should add." askTutor="I'm on Step 6 of Run tests and fix failures with Claude. Help me connect this exercise to the real red-green workflow developers use every day." />

Ask Claude Code:

```
Explain the red-green cycle we just went through. Why is it better than manually checking the output?
```

Claude explains that tests are repeatable, run in milliseconds, check multiple cases at once, and catch regressions automatically. Manual checking requires you to remember what to look for every time.

> **Real danger — do not fix the test to match the bug.** When a test fails, it is tempting to change the expected value so everything "passes." That hides the bug instead of fixing it. Always fix the source code unless you are genuinely sure the test expectation is wrong.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude changes the test instead of the source code | Your prompt did not specify where the bug is | Explicitly say "the tests are correct, fix the source code" |
| All tests still fail after the fix | You saved the wrong file or Claude edited the test file | Check which file was modified with `git diff` and undo if needed |
| Only some `add` tests fail, others pass | Some test inputs happen to give the same result for `+` and `-` (e.g., `0 + 0 = 0 - 0`) | This is expected — tests with zero inputs may not catch an `+` vs `-` swap. It shows why diverse test inputs matter |
| `npm test` runs but shows no output | Jest is running in watch mode and waiting for input | Press `q` to quit, then run `npx jest --no-watchman` instead |
| pytest says "1 passed, 2 failed" instead of "2 passed, 3 failed" | There are fewer test functions than expected | Check that all test functions in the test file start with `test_` |
