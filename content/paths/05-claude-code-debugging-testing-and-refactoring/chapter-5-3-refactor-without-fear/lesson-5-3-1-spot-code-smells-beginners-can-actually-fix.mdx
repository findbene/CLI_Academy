# Spot code smells beginners can actually fix

## Objective

In this lesson, you will learn to identify common "code smells" in a sample script that beginners can safely and confidently fix. You'll develop a practical checklist and pinpoint three key refactoring opportunities that enhance clarity without altering core functionality.

## Walkthrough

Let's practice spotting code smells together. For this exercise, we'll review a file named `sample_script.sh`.

1.  **View the Sample Script:**
    First, let's look at the script's content.
    ```bash
    cat sample_script.sh
    ```
    The `cat` command is safe; it simply displays the file's contents to your terminal and doesn't make any changes. Take a moment to read through it.

2.  **Understand Beginner-Safe Code Smells:**
    We'll focus on three straightforward code smells perfect for your first refactoring steps:
    *   **Unclear Naming:** Are variable or function names vague (e.g., `x`, `temp`, `data`) and don't immediately tell you their purpose?
    *   **Magic Numbers/Strings:** Are numbers or text values used directly in the code without explanation or being assigned to a named constant? These are often hard to understand or change later.
    *   **Duplicated Code:** Do you see the same or very similar lines of code appearing in multiple places?

3.  **Identify Top 3 Refactors in `sample_script.sh`:**
    Now, let's apply our checklist to `sample_script.sh` and identify beginner-safe refactors. Remember, safe refactors focus on clarity and organization, not complex logic changes.

    *   **Refactor 1: Rename an Unclear Variable.**
        Look for lines where a variable's name doesn't clearly explain its purpose. For example, if you see `val = 100`, and `val` represents a maximum retry count, renaming it to `MAX_RETRIES` makes it much clearer. This is a very safe change as it only affects readability.

    *   **Refactor 2: Introduce a Constant for a Magic Number/String.**
        Find a number or string used directly that could benefit from a descriptive name. For instance, if `sleep 5` appears, and `5` is always the `DEFAULT_DELAY_SECONDS`, defining `DEFAULT_DELAY_SECONDS=5` at the top and using that variable improves understanding and makes future changes easier. This is also very safe.

    *   **Refactor 3: Extract a Small, Duplicated Block into a Function/Helper.**
        Identify a few lines of code that are repeated verbatim, or with minor variations, in two or more places. Encapsulating this logic into a small, clearly named function (e.g., `log_message()`) and calling that function instead is a powerful, yet low-risk, refactor for beginners.
