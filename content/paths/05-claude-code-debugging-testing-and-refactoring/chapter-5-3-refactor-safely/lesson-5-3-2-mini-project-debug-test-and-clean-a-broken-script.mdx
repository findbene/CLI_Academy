# Mini project: debug, test, and clean a broken script

## Objective

Bring together everything from Path 05 in a single end-to-end exercise. You start with a broken, messy script. You debug it with Claude Code, add tests, fix the bugs, refactor for clarity, confirm all tests pass, and commit the result. This is the real workflow professional developers follow every day.

### Step 2 — Create the broken, messy script

<StepMeta purpose="Start from code that is both buggy and poorly structured so the full workflow has something real to improve." action="Write the invoice calculator exactly as shown, including the off-by-one loop bug and the incorrect total calculation." expectedResult="You have a small source file on disk with two distinct bugs and low-quality structure." whyItMatters="This exercise is realistic because production cleanup often requires both fixing behavior and improving readability, not just one or the other." nextStep="Add a tiny runner and execute the script so you can observe the first crash." askTutor="I'm on Step 2 of Mini project: debug, test, and clean a broken script. Help me understand the two intentional bugs and why this is a good candidate for the full workflow." />

This script is a tiny invoice calculator. It has **two bugs** and is written in a messy, hard-to-read style on purpose.

**JavaScript — `invoice.js`**

```js
// invoice.js — broken and messy on purpose
function calcInvoice(items, taxRate) {
  let s = 0;
  let i = 0;
  while (i <= items.length) { // BUG 1: should be < not <=
    let item = items[i];
    let p = item.price;
    let q = item.qty;
    let t = p * q;
    s = s + t;
    i = i + 1;
  }
  let tax = s * taxRate;
  let tot = s + taxRate; // BUG 2: should add tax, not taxRate
  return { subtotal: s, tax: tax, total: tot };
}

module.exports = { calcInvoice };
```

**Python — `invoice.py`**

```python
# invoice.py — broken and messy on purpose
def calc_invoice(items, tax_rate):
    s = 0
    i = 0
    while i <= len(items):  # BUG 1: should be < not <=
        item = items[i]
        p = item["price"]
        q = item["qty"]
        t = p * q
        s = s + t
        i = i + 1
    tax = s * tax_rate
    tot = s + tax_rate  # BUG 2: should add tax, not tax_rate
    return {"subtotal": s, "tax": tax, "total": tot}
```

Create a small runner to call the function:

```js

const items = [
  { name: "Widget", price: 10, qty: 2 },
```

**macOS / Linux**


**Windows PowerShell**

```powershell

```

You get a crash:

```

```

The `while (i <= items.length)` goes one index past the end of the array.

### Step 4 — Paste the error into Claude Code and fix Bug 1

<StepMeta purpose="Use the runtime error to repair the first bug with a focused, evidence-based Claude prompt." action="Paste the crash into Claude, let it identify the off-by-one loop condition, and accept the one-line fix." expectedResult="The script no longer crashes when run, but you may still discover logical errors in the output afterward." whyItMatters="Fixing the first blocking bug often exposes the next layer of issues, which is exactly what happens in real debugging work." nextStep="Rerun the script and inspect the totals to discover the second bug that survives after the crash is fixed." askTutor="I'm on Step 4 of Mini project: debug, test, and clean a broken script. ```

Claude identifies that `i <= items.length` should be `i < items.length` because array indices go from 0 to length - 1. Accept the fix.

```js
const { calcInvoice } = require("./invoice");

test("calculates invoice with multiple items", () => {
  const items = [
    { name: "Widget", price: 10, qty: 2 },
    { name: "Gadget", price: 25, qty: 1 },
  ];
  const result = calcInvoice(items, 0.08);
  expect(result.subtotal).toBe(45);
  expect(result.tax).toBeCloseTo(3.6);
  expect(result.total).toBeCloseTo(48.6);
});

test("calculates invoice with a single item", () => {
  const items = [{ name: "Thing", price: 100, qty: 3 }];
  const result = calcInvoice(items, 0.1);
  expect(result.subtotal).toBe(300);
  expect(result.tax).toBeCloseTo(30);
  expect(result.total).toBeCloseTo(330);
});

test("handles empty items array", () => {
  const result = calcInvoice([], 0.08);
  expect(result.subtotal).toBe(0);
  expect(result.tax).toBe(0);
  expect(result.total).toBe(0);
});
```

### Step 7 — Ask Claude to refactor

<StepMeta purpose="Improve the structure of the now-correct code without altering the invoice results that the tests protect." action="Prompt Claude to replace the manual loop and vague variable names with descriptive names and clearer array-based logic." expectedResult="Claude produces a shorter, clearer implementation that still computes the same subtotal, tax, and total." whyItMatters="This is the full professional loop: debug to correctness first, then refactor to readability once tests are guarding behavior." nextStep="Run the test suite again to prove the refactor preserved the fixed behavior." askTutor="I'm on Step 7 of Mini project: debug, test, and clean a broken script. Help me review this refactor and confirm it improves clarity without changing any invoice math." />

```
The calcInvoice function in invoice.js works but uses single-letter variable names and a manual while loop. Refactor it to use descriptive names and modern array methods. All tests must still pass.
```

Claude rewrites the function:

```js
function calcInvoice(items, taxRate) {
  const subtotal = items.reduce(
    (sum, item) => sum + item.price * item.qty,
    0
  );
  const tax = subtotal * taxRate;
  const total = subtotal + tax;
  return { subtotal, tax, total };
}
```

This is 6 clear lines instead of 13 confusing ones. No single-letter variables, no manual index management, no off-by-one risk.

Accept the refactoring.

> **Real danger — committing before tests pass.** Never commit code that has failing tests. If you commit a broken state, anyone who pulls your code inherits your bugs. Always run `npm test` (or `pytest`) immediately before `git commit`.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Bug 2 is hard to spot because the numbers are close | Tax rate (0.08) and tax amount (3.6) are both small numbers | Use a larger tax rate (like 0.5) in your test runner to make the difference more obvious |
| Claude fixes both bugs at once before you ask about Bug 2 | Claude saw the entire file and fixed everything proactively | This is fine — Claude being thorough is a feature. Just make sure you understand both fixes before accepting |
| `git commit` says "nothing to commit" | You forgot `git add` or the files are gitignored | Run `git status` first; if files show as untracked, run `git add -A` |
| Empty array test fails with `NaN` | The `reduce` call has no initial value | Make sure `reduce` has `, 0` as the second argument: `.reduce((sum, item) => ..., 0)` |
| The refactored code uses `reduce` which you do not understand | Array methods are new to you | Ask Claude: "Explain this reduce call step by step" — learning the idiom is part of the exercise |
| Python `IndexError: list index out of range` instead of `TypeError` | Different language, same off-by-one bug | The fix is the same: change `<=` to `<` in the while condition |
