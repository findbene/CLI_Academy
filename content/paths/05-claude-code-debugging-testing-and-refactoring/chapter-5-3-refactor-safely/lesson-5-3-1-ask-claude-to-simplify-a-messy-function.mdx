# Ask Claude to simplify a messy function

## Objective

Start with a working but ugly function — deeply nested, repetitive, and hard to read. Run the existing tests to prove it works. Ask Claude Code to refactor it for clarity. Run the tests again to prove behavior is preserved. By the end of this lesson you will understand that refactoring means changing how code looks without changing what it does, and that tests are your safety net for proving that.

**JavaScript — `categorize.js`**

```js
// categorize.js — works but is a mess
function categorizeUser(age, isMember) {
  let result;
  if (age !== undefined && age !== null) {
    if (typeof age === "number") {
      if (age >= 0) {
        if (age < 13) {
          if (isMember === true) {
            result = "child-member";
          } else {
            result = "child-guest";
          }
        } else {
          if (age < 18) {
            if (isMember === true) {
              result = "teen-member";
            } else {
              result = "teen-guest";
            }
          } else {
            if (isMember === true) {
              result = "adult-member";
            } else {
              result = "adult-guest";
            }
          }
        }
      } else {
        result = "invalid";
      }
    } else {
      result = "invalid";
    }
  } else {
    result = "invalid";
  }
  return result;
}

module.exports = { categorizeUser };
```

### Step 4 — Ask Claude Code to refactor

<StepMeta purpose="Improve clarity and structure without changing the function's external behavior." action="Prompt Claude to flatten the logic with early returns and clearer category composition while keeping the tests authoritative." expectedResult="Claude rewrites the function into a shorter, easier-to-read form without touching the test suite." whyItMatters="This is the core refactoring skill: change the inside of the function so it is easier to maintain while leaving the outside behavior identical." nextStep="Rerun the same tests after the refactor to prove nothing about the behavior changed." askTutor="I'm on Step 4 of Ask Claude to simplify a messy function. Help me evaluate whether this refactor genuinely improves readability without sneaking in a behavior change." />

```
The categorizeUser function in categorize.js works correctly but is deeply nested and hard to read. Refactor it to be flat and clear. Use early returns for the invalid cases and a simple lookup for the categories. Do not change the behavior — all existing tests must still pass.
```

Claude rewrites the function. A typical result:

```js
function categorizeUser(age, isMember) {
  if (age === undefined || age === null || typeof age !== "number" || age < 0) {
    return "invalid";
  }

  const group = age < 13 ? "child" : age < 18 ? "teen" : "adult";
  const tier = isMember ? "member" : "guest";
  return `${group}-${tier}`;
}
```

This is 6 lines instead of 30. The deep nesting is gone. Early returns handle all the invalid cases upfront, and the category logic is a simple two-part string.

Accept the change.

### Step 6 — Compare before and after

<StepMeta purpose="Extract the lessons from the refactor so you can recognize similar cleanup opportunities later." action="Ask Claude to contrast the messy and refactored versions in terms of nesting, line count, readability, and maintainability." expectedResult="You can explain exactly what got better about the code besides 'it looks cleaner.'" whyItMatters="Refactoring skill grows when you learn to identify specific improvements such as flatter control flow, better naming, and easier change points." nextStep="Bring all of the Path 05 skills together in a mini-project that starts broken and ends debugged, tested, refactored, and committed." askTutor="I'm on Step 6 of Ask Claude to simplify a messy function. Ask Claude Code:

```
Show me a side-by-side comparison of the original categorizeUser and the refactored version. What specifically improved?
```

Claude explains:

- **Nesting depth**: went from 7 levels to 1.
- **Lines of code**: went from 30+ to 6.
- **Readability**: a new developer can understand the refactored version in seconds.
- **Maintainability**: adding a new age group (like "senior") requires one change, not several nested blocks.

> **Real danger — refactoring without tests is gambling.** If you did not have those 10 tests, you would have no way to know if the refactored function still produces the same results. Always write or confirm tests before refactoring. The tests are your safety net, not the refactoring.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Tests fail after refactoring | Claude's refactored logic does not exactly match the original edge cases | Reject the refactoring, ask Claude what case it changed, and refactor in smaller steps |
| Claude rewrites the function and also changes the test file | You did not tell Claude to leave the tests alone | Say explicitly: "Only change categorize.js. Do not modify the test file." |
| The refactored code uses syntax you do not understand (ternary, template literals) | Claude used advanced language features | Ask Claude: "Explain the ternary operator you used on line 6" — understanding the syntax is part of learning |
| Claude refuses to refactor, saying the original is fine | Claude does not see a problem because the code works | Be specific: "The function has 7 levels of nesting. Flatten it using early returns." |
| Python version uses `match/case` which is not available in your Python version | `match` requires Python 3.10+ | Tell Claude your Python version: "I'm using Python 3.9, do not use match/case" |
