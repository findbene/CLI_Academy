# Use virtual environments for Python isolation

## Objective

By the end of this lesson you will create a Python virtual environment, install a package inside it, and prove that the package exists only in the venv — not in your system Python. You will also understand exactly what a venv does and does not protect you from.

## Walkthrough

### Step 1 — Check your system Python's installed packages

<StepMeta purpose="Capture the baseline Python environment before you create a venv." action="List the current system Python packages and print the path to the system interpreter so you have a before-state to compare against later." expectedResult="You see the system Python executable path and a sample of the globally installed packages." whyItMatters="A venv only makes sense when you compare it to the system interpreter it is isolating you from, so this baseline matters for the proof later." nextStep="Create a new virtual environment so you can compare its interpreter and package set to the system baseline." askTutor="I'm on Step 1 of Use virtual environments for Python isolation. Help me understand what baseline details I should pay attention to before creating the venv." />

Before creating a venv, take a snapshot of what your system Python already has installed.

**macOS / Linux**

```bash
python3 -m pip list --format=columns 2>/dev/null | head -20
echo "---"
python3 -c "import sys; print('System Python:', sys.executable)"
```

**Windows PowerShell**

```powershell
python -m pip list --format=columns 2>$null | Select-Object -First 20
Write-Host "---"
python -c "import sys; print('System Python:', sys.executable)"
```

Note the location printed for `sys.executable` — you will compare it to the venv's Python later.

### Step 2 — Create a virtual environment

<StepMeta purpose="Create an isolated Python dependency environment for the experiment." action="Make the venv-lab workspace, then run python -m venv .venv to create the environment directory." expectedResult="A .venv folder appears containing a separate Python interpreter and package area." whyItMatters="The venv directory is the mechanism that separates project dependencies from your system interpreter without changing your global Python installation." nextStep="Activate the environment so your shell starts using the venv Python and pip instead of the system ones." askTutor="I'm on Step 2 of Use virtual environments for Python isolation. Help me understand what this .venv directory contains and what it changes about my Python workflow." />

**macOS / Linux**

```bash
mkdir -p ~/venv-lab && cd ~/venv-lab
python3 -m venv .venv
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\venv-lab"
Set-Location "$HOME\venv-lab"
python -m venv .venv
```

This creates a `.venv` folder containing a copy of the Python interpreter and an empty `site-packages` directory. No packages have been installed yet.

### Step 3 — Activate the virtual environment

<StepMeta purpose="Switch your shell context so subsequent python and pip commands target the venv." action="Run the activation command for your shell and confirm the prompt shows the venv indicator." expectedResult="Your shell is now using the .venv interpreter and package context instead of the system Python." whyItMatters="Activation is what makes the environment convenient to use, and without it you can accidentally install packages to the wrong interpreter." nextStep="Verify the active interpreter path and current package list so you can prove the venv is separate from system Python." askTutor="I'm on Step 3 of Use virtual environments for Python isolation. Help me verify that activation actually switched my shell to the venv instead of just changing the prompt text." />

**macOS / Linux**

```bash
source .venv/bin/activate
```

**Windows PowerShell**

```powershell
.\.venv\Scripts\Activate.ps1
```

Your terminal prompt should now show `(.venv)` at the beginning. This means every `python` and `pip` command now targets the venv, not your system Python.

### Step 4 — Prove the venv has its own Python and empty packages

<StepMeta purpose="Show concretely that the venv uses a different interpreter path and a nearly empty package set." action="Print the active python path, inspect sys.executable, and list installed packages from inside the activated venv." expectedResult="The interpreter path points inside .venv and the package list is limited to the default bootstrap packages." whyItMatters="This is the direct proof that a venv isolates package state even though it does not isolate files or network access." nextStep="Install a harmless package and compare its visibility inside and outside the venv to prove the dependency boundary works." askTutor="I'm on Step 4 of Use virtual environments for Python isolation. Help me compare this venv Python path and package list to the earlier system baseline so I can prove the isolation clearly." />

**macOS / Linux**

```bash
which python
python -c "import sys; print('Venv Python:', sys.executable)"
pip list --format=columns
```

**Windows PowerShell**

```powershell
Get-Command python | Select-Object -ExpandProperty Source
python -c "import sys; print('Venv Python:', sys.executable)"
pip list --format=columns
```

The Python path should point inside `.venv/`, not your system installation. The package list should show only `pip` and `setuptools` — nothing from your system Python.

### Step 5 — Install a package and prove it exists only in the venv

<StepMeta purpose="Demonstrate package isolation in practice by adding a dependency to the venv and checking that system Python cannot import it." action="Install cowsay inside the venv, run it successfully there, deactivate, and then try importing it from system Python." expectedResult="cowsay works inside the venv and fails with ModuleNotFoundError outside it." whyItMatters="This inside-versus-outside comparison is the clearest proof of what a venv is good at: isolating dependencies, not securing execution." nextStep="Pair that proof with a clear statement of what venvs do not protect you from so you do not mistake them for a security sandbox." askTutor="I'm on Step 5 of Use virtual environments for Python isolation. Help me explain why this cowsay test proves dependency isolation but says nothing about file or network safety." />

Install a small, harmless package and verify it is not visible from system Python.

**macOS / Linux**

```bash
pip install cowsay
python -c "import cowsay; cowsay.cow('Hello from the venv')"
echo "---"
# Now check from OUTSIDE the venv:
deactivate
python3 -c "import cowsay" 2>&1 || echo "cowsay is NOT installed in system Python — isolation works"
```

**Windows PowerShell**

```powershell
pip install cowsay
python -c "import cowsay; cowsay.cow('Hello from the venv')"
Write-Host "---"
# Now check from OUTSIDE the venv:
deactivate
python -c "import cowsay" 2>&1
if ($LASTEXITCODE -ne 0) { Write-Host "cowsay is NOT installed in system Python — isolation works" }
```

You should see the cow art from inside the venv, and a `ModuleNotFoundError` from outside it. That is dependency isolation working.

### Step 6 — Understand what a venv does NOT protect

<StepMeta purpose="Prevent a common security mistake by separating dependency isolation from security isolation." action="Read the table of threats venvs do not stop and map each one to the stronger isolation tool you would need instead." expectedResult="You can state clearly that venvs do not block file reads, network calls, shell commands, or malicious install scripts." whyItMatters="Many learners over-trust venvs, and this step prevents you from using the wrong tool when the risk is untrusted code rather than package conflicts." nextStep="Freeze the environment state so the experiment ends with a reproducible dependency snapshot." askTutor="I'm on Step 6 of Use virtual environments for Python isolation. Help me turn this 'what venvs do NOT protect' table into a practical rule for when I should switch to Docker instead." />

A venv isolates installed packages. It does **not** isolate:

| Threat | Venv protection | What to use instead |
|---|---|---|
| Malicious code reading your files | None — venv code has full file access | Docker with no volume mounts |
| Code making network requests | None — venv code has full network access | Docker with `--network none` |
| Code running shell commands | None — `os.system()` works normally | Docker with `--read-only` and no shell |
| Install scripts running during `pip install` | None — install scripts run as your user | `pip install --no-deps` in Docker, or review the package first |

> **Real danger:** `pip install` executes arbitrary code in the package's `setup.py` during installation. A typosquatted package can steal credentials at install time, even if you never import it. A venv does not prevent this. Use Docker when installing packages you have not verified.

### Step 7 — Freeze and clean up

<StepMeta purpose="Capture the environment state in a reproducible file before leaving the lab." action="Reactivate the venv if needed, run pip freeze into requirements.txt, inspect the file, and then deactivate cleanly." expectedResult="You end with a requirements.txt file listing the installed package set from the venv." whyItMatters="Freezing the environment is how you preserve the exact dependency state for later recreation, which is a core venv workflow even outside this lesson." nextStep="Carry this package-isolation knowledge into the sandboxed AI-code workflow, where Docker adds the security boundary that venvs lack." askTutor="I'm on Step 7 of Use virtual environments for Python isolation. Help me understand what this requirements.txt captures and how it relates to recreating the same environment later." />

**macOS / Linux**

```bash
cd ~/venv-lab
source .venv/bin/activate
pip freeze > requirements.txt
cat requirements.txt
deactivate
```

**Windows PowerShell**

```powershell
Set-Location "$HOME\venv-lab"
.\.venv\Scripts\Activate.ps1
pip freeze | Out-File requirements.txt
Get-Content requirements.txt
deactivate
```

The `requirements.txt` file captures exactly what was installed. Anyone can recreate your environment from this file with `pip install -r requirements.txt` in a fresh venv.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `python3 -m venv` fails with "ensurepip is not available" | System Python was installed without the venv module | Install it: `sudo apt install python3-venv` (Debian/Ubuntu) or `brew install python` (macOS) |
| Activate script is blocked on Windows | PowerShell execution policy prevents running `.ps1` files | Run `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned` and try again |
| `pip install` still installs to system Python | You forgot to activate the venv | Check your prompt for `(.venv)` — if missing, run the activate command from Step 3 |
| `deactivate` command not found | You are not currently in an active venv | This is fine — you were already outside the venv |
| `which python` still shows system Python after activation | Shell hash table is stale | Run `hash -r` (bash) or restart your terminal |
| `ModuleNotFoundError` inside the venv for a package you just installed | pip installed to a different Python than the one running | Always use `python -m pip install` instead of bare `pip` to ensure they match |
