# Run untrusted code in a Docker container

## Objective

By the end of this lesson you will build a minimal Docker image designed for sandboxing, run untrusted code inside it with file system, network, and resource limits, and prove that the container cannot affect your host machine.

## Walkthrough

### Step 1 — Create a sandbox project folder

<StepMeta purpose="Set up a dedicated workspace so the Docker sandbox files stay isolated from your main projects." action="Create the sandbox-lab directory and move into it before writing any Docker or test files." expectedResult="You have a clean workspace ready to hold the Dockerfile and the untrusted test script." whyItMatters="Keeping sandbox experiments separate from real projects reduces accidental file mixing and makes it easier to see what the container can and cannot touch." nextStep="Write the minimal Dockerfile that defines the locked-down environment you will use for the test." askTutor="I'm on Step 1 of Run untrusted code in a Docker container.
```bash
mkdir -p ~/sandbox-lab && cd ~/sandbox-lab
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\sandbox-lab"
Set-Location "$HOME\sandbox-lab"
```

### Step 2 — Write a minimal Dockerfile

<StepMeta purpose="Define a deliberately small and restricted container image for sandboxing." action="Create the Dockerfile with the slim Python base, non-root sandbox user, and narrow working directory." expectedResult="You have a minimal image definition that limits privileges and avoids unnecessary extra tools." whyItMatters="The image design is part of the sandbox boundary: fewer packages, no root runtime, and a constrained user reduce the blast radius if the code misbehaves." nextStep="Build the image so you can use it as the execution environment for the untrusted script." askTutor="I'm on Step 2 of Run untrusted code in a Docker container. Help me understand which lines in this Dockerfile actually improve the sandbox boundary and which are just convenience." />

This image includes only Python and nothing else. No shell utilities, no package managers, no extras.

**macOS / Linux**

```bash
cat > Dockerfile << 'EOF'
FROM python:3.12-slim
RUN useradd --create-home --shell /bin/false sandboxuser
USER sandboxuser
WORKDIR /home/sandboxuser
EOF
```

**Windows PowerShell**

```powershell
@"
FROM python:3.12-slim
RUN useradd --create-home --shell /bin/false sandboxuser
USER sandboxuser
WORKDIR /home/sandboxuser
"@ | Out-File -Encoding utf8 Dockerfile
```

Key security choices:
- **Non-root user** — even if code exploits a bug, it runs as `sandboxuser`, not root.
- **No shell for the user** — `--shell /bin/false` prevents interactive shell escape.
- **Slim base** — fewer installed tools means fewer attack surfaces.

### Step 3 — Build the sandbox image

<StepMeta purpose="Turn the Dockerfile into a runnable sandbox image you can reuse during the rest of the lesson." action="Run docker build and wait for the image to complete successfully with the sandbox-runner tag." expectedResult="Docker finishes building the image and tags it as sandbox-runner:latest." whyItMatters="You need a reproducible container image before you can test any of the execution controls, and building it once makes repeated safe runs faster." nextStep="Create a deliberately suspicious script that tries file, write, and network actions so you can test the sandbox limits." askTutor="I'm on Step 3 of Run untrusted code in a Docker container. **macOS / Linux**

```bash
docker build -t sandbox-runner .
```

**Windows PowerShell**

```powershell
docker build -t sandbox-runner .
```

You should see `Successfully tagged sandbox-runner:latest` at the end.

### Step 4 — Write an untrusted script to test

<StepMeta purpose="Create a realistic probe that attempts the kinds of actions a suspicious script might try." action="Write evil_test.py so it attempts to inspect files, write outside its home directory, and reach the network." expectedResult="You have a test script that exercises the key sandbox boundaries you want Docker to enforce." whyItMatters="A sandbox is easier to trust when you test it against realistic misuse patterns instead of a harmless hello-world script." nextStep="Run that script with the full locked-down docker flags so you can observe which actions succeed and which are blocked." askTutor="I'm on Step 4 of Run untrusted code in a Docker container. Help me review this test script and explain why each attempted action is a useful sandbox check." />

Create a script that tries to do three dangerous things: read the host file system, access the internet, and write outside its home directory.

**macOS / Linux**

```bash
cat > evil_test.py << 'PYEOF'
import os, sys

print("=== Attempting dangerous operations ===")

# Attempt 1: Read host files
try:
    contents = os.listdir("/etc")
    print(f"1. Can see /etc: {contents[:3]}... (container's /etc, not host)")
except PermissionError as e:
    print(f"1. BLOCKED from /etc: {e}")

# Attempt 2: Write outside home directory
try:
    with open("/tmp/escape.txt", "w") as f:
        f.write("escaped!")
    print("2. Wrote to /tmp — but it is the container's /tmp, not host")
except PermissionError as e:
    print(f"2. BLOCKED from writing /tmp: {e}")

# Attempt 3: Access the network
try:
    import urllib.request
    urllib.request.urlopen("https://example.com", timeout=3)
    print("3. NETWORK ACCESS SUCCEEDED — sandbox is not locked down")
except Exception as e:
    print(f"3. Network BLOCKED: {e}")

print("\n=== Test complete ===")
PYEOF
```

### Step 5 — Run the untrusted script in the locked-down container

<StepMeta purpose="Exercise the full Docker sandbox with file, network, and resource restrictions enabled." action="Run evil_test.py inside sandbox-runner using the locked-down docker flags and inspect the output for blocked network access and container-local writes." expectedResult="The script runs inside the container, can only affect container-local state, and fails to reach the network." whyItMatters="This is the core proof that your sandbox configuration is doing real work rather than just wrapping the script in Docker cosmetically." nextStep="Check the host afterward to confirm no file escaped from the container into your real machine." askTutor="I'm on Step 5 of Run untrusted code in a Docker container. Help me interpret this sandbox run and decide whether the combination of flags actually gives me the protections I expect." />

Use every sandbox flag: read-only file system, no network, drop all Linux capabilities, restrict memory.

**macOS / Linux**

```bash
docker run \
  --rm \
  --network none \
  --read-only \
  --tmpfs /tmp:size=10m \
  --memory=64m \
  --cpus=0.5 \
  -v "$(pwd)/evil_test.py:/home/sandboxuser/evil_test.py:ro" \
  sandbox-runner \
  python /home/sandboxuser/evil_test.py
```

**Windows PowerShell**

```powershell
docker run `
  --rm `
  --network none `
  --read-only `
  --tmpfs /tmp:size=10m `
  --memory=64m `
  --cpus=0.5 `
  -v "${PWD}/evil_test.py:/home/sandboxuser/evil_test.py:ro" `
  sandbox-runner `
  python /home/sandboxuser/evil_test.py
```

Flag breakdown:

| Flag | What it does |
|---|---|
| `--rm` | Delete the container and its file system when the process exits |
| `--network none` | No internet, no DNS, no outbound connections |
| `--read-only` | The container's root file system is mounted read-only |
| `--tmpfs /tmp:size=10m` | Allow writes only to `/tmp`, capped at 10 MB |
| `--memory=64m` | Kill the container if it uses more than 64 MB RAM |
| `--cpus=0.5` | Limit to half a CPU core so it cannot starve your machine |
| `-v ... :ro` | Mount only the script file, read-only — no other host files are visible |

Expected output: the script can see the container's own `/etc` (not yours), can write to the container's `/tmp` (not yours), and the network attempt is blocked.

### Step 6 — Confirm nothing leaked to your host

<StepMeta purpose="Verify from the host side that the container run left no unexpected artifact behind." action="Check your host temp location for escape.txt and confirm the file does not exist after the container exits." expectedResult="The host reports that no leaked file exists, confirming the write stayed inside the container." whyItMatters="Host-side verification is the last proof that the sandbox boundary held and that the containerized script did not leave artifacts outside the controlled output path." nextStep="Use the same isolation concepts in the venv lesson, where you will see package isolation without security isolation." askTutor="I'm on Step 6 of Run untrusted code in a Docker container. Help me verify from the host side that nothing escaped from the sandbox run." />

**macOS / Linux**

```bash
ls /tmp/escape.txt 2>/dev/null && echo "LEAK DETECTED" || echo "Host is clean — nothing escaped"
```

**Windows PowerShell**

```powershell
if (Test-Path "$env:TEMP\escape.txt") { Write-Host "LEAK DETECTED" } else { Write-Host "Host is clean — nothing escaped" }
```

> **Real danger:** If you use `-v $HOME:/host` to mount your home directory into a container, all sandboxing is worthless. The `:ro` (read-only) flag on volume mounts is the minimum when you must share files. Never mount broad directories into untrusted containers.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `docker build` fails with `permission denied` | Docker daemon not running or user not in docker group | Start Docker Desktop, or run `sudo usermod -aG docker $USER` and log out/in |
| `--read-only` causes Python import errors | Python tries to write `.pyc` bytecode cache files | Add `--tmpfs /home/sandboxuser/__pycache__:size=5m` or set `PYTHONDONTWRITEBYTECODE=1` via `-e` flag |
| Script says `NETWORK ACCESS SUCCEEDED` | You forgot `--network none` | Add the flag and rerun — without it the container has full internet |
| `OCI runtime error` on Windows | Docker Desktop has WSL 2 backend issue | Open Docker Desktop → Settings → General → confirm "Use the WSL 2 based engine" is checked and restart |
| Volume mount path error on Windows | PowerShell `$PWD` resolves differently | Use `${PWD}` or provide the absolute path explicitly |
| Container exits instantly with code 137 | Out-of-memory kill from `--memory=64m` | Increase to `--memory=128m` if the script needs more RAM |
