# Mini-project: Build a safe execution pipeline

## What you will build
A working safe-pipeline directory containing sandbox-run.sh (or .ps1), at least one PASSED execution log, output files from the sandbox run, and one FAILED log from the deliberately broken script.

## Why this matters
Custom runtimes are how agentic developers move from using Claude to building with Claude. Understanding the sandbox model and runtime foundations is the prerequisite for everything in the advanced Claw ecosystem. This knowledge is increasingly sought after in AI engineering roles.

## Before you start
By the end of this mini-project you will have a working pipeline script that implements the full safe execution flow: Claude Code writes a script, the script is saved to a staging area, it runs inside a locked-down Docker container, the exit code is checked, and only if execution succeeds are the outputs promoted to your real project directory. You will build and test every piece.


## Walkthrough

### Step 1 — Create the pipeline project structure

**macOS / Linux**

```bash
mkdir -p ~/safe-pipeline/{staging,approved,output,logs}
cd ~/safe-pipeline
ls -la
```

**Windows PowerShell**

```powershell
$dirs = "staging", "approved", "output", "logs"
foreach ($d in $dirs) {
    New-Item -ItemType Directory -Force -Path "$HOME\safe-pipeline\$d"
}
Set-Location "$HOME\safe-pipeline"
Get-ChildItem
```

| Directory | Purpose |
|---|---|
| `staging/` | Where AI-generated scripts land before review |
| `approved/` | Scripts you have reviewed and approved |
| `output/` | Files produced by sandbox runs |
| `logs/` | Execution logs with timestamps and exit codes |

### Step 2 — Write the sandbox runner script

This is the core of the pipeline — a shell script that runs any Python file in a Docker sandbox and captures the result.

**macOS / Linux**

```bash
cat > ~/safe-pipeline/sandbox-run.sh << 'SCRIPT'
#!/usr/bin/env bash
set -euo pipefail

# Usage: ./sandbox-run.sh <script.py>
SCRIPT_PATH="$1"
SCRIPT_NAME="$(basename "$SCRIPT_PATH")"
TIMESTAMP="$(date +%Y%m%d-%H%M%S)"
LOG_FILE="logs/run-${SCRIPT_NAME}-${TIMESTAMP}.log"

if [ ! -f "$SCRIPT_PATH" ]; then
  echo "ERROR: File not found: $SCRIPT_PATH" | tee "$LOG_FILE"
  exit 1
fi

echo "=== Sandbox Run: $SCRIPT_NAME ===" | tee "$LOG_FILE"
echo "Timestamp: $TIMESTAMP" | tee -a "$LOG_FILE"
echo "Script: $SCRIPT_PATH" | tee -a "$LOG_FILE"
echo "---" | tee -a "$LOG_FILE"

# Run in locked-down Docker container
docker run \
  --rm \
  --network none \
  --read-only \
  --tmpfs /tmp:size=20m \
  --memory=128m \
  --cpus=0.5 \
  --user "$(id -u):$(id -g)" \
  -v "$(realpath "$SCRIPT_PATH"):/app/script.py:ro" \
  -v "$(realpath output):/output" \
  -w /tmp \
  -e PYTHONDONTWRITEBYTECODE=1 \
  python:3.12-slim \
  python /app/script.py 2>&1 | tee -a "$LOG_FILE"

EXIT_CODE=${PIPESTATUS[0]}

echo "---" | tee -a "$LOG_FILE"
echo "Exit code: $EXIT_CODE" | tee -a "$LOG_FILE"

if [ "$EXIT_CODE" -eq 0 ]; then
  echo "STATUS: PASSED — output files are in output/" | tee -a "$LOG_FILE"
else
  echo "STATUS: FAILED — check log: $LOG_FILE" | tee -a "$LOG_FILE"
fi

exit "$EXIT_CODE"
SCRIPT

chmod +x ~/safe-pipeline/sandbox-run.sh
cat ~/safe-pipeline/sandbox-run.sh
```

**Windows PowerShell**

```powershell
@"
# sandbox-run.ps1 — Run a Python script in a locked-down Docker container
param(
    [Parameter(Mandatory=`$true)]
    [string]`$ScriptPath
)

`$ErrorActionPreference = 'Stop'
`$scriptName = Split-Path `$ScriptPath -Leaf
`$timestamp = (Get-Date).ToString('yyyyMMdd-HHmmss')
`$logFile = "logs\run-`$scriptName-`$timestamp.log"

if (-not (Test-Path `$ScriptPath)) {
    "ERROR: File not found: `$ScriptPath" | Tee-Object -FilePath `$logFile
    exit 1
}

"=== Sandbox Run: `$scriptName ===" | Tee-Object -FilePath `$logFile
"Timestamp: `$timestamp" | Tee-Object -FilePath `$logFile -Append
"Script: `$ScriptPath" | Tee-Object -FilePath `$logFile -Append
"---" | Tee-Object -FilePath `$logFile -Append

`$fullScriptPath = (Resolve-Path `$ScriptPath).Path
`$fullOutputPath = (Resolve-Path "output").Path

docker run ``
  --rm ``
  --network none ``
  --read-only ``
  --tmpfs /tmp:size=20m ``
  --memory=128m ``
  --cpus=0.5 ``
  -v "`${fullScriptPath}:/app/script.py:ro" ``
  -v "`${fullOutputPath}:/output" ``
  -w /tmp ``
  -e PYTHONDONTWRITEBYTECODE=1 ``
  python:3.12-slim ``
  python /app/script.py 2>&1 | Tee-Object -FilePath `$logFile -Append

`$exitCode = `$LASTEXITCODE

"---" | Tee-Object -FilePath `$logFile -Append
"Exit code: `$exitCode" | Tee-Object -FilePath `$logFile -Append

if (`$exitCode -eq 0) {
    "STATUS: PASSED — output files are in output/" | Tee-Object -FilePath `$logFile -Append
} else {
    "STATUS: FAILED — check log: `$logFile" | Tee-Object -FilePath `$logFile -Append
}

exit `$exitCode
"@ | Out-File -Encoding utf8 "$HOME\safe-pipeline\sandbox-run.ps1"
Get-Content "$HOME\safe-pipeline\sandbox-run.ps1"
```

### Step 3 — Ask Claude Code to generate a test script

Open Claude Code in the `safe-pipeline` directory and give it a bounded task:

```
Write a Python script called staging/analyze_numbers.py that:
1. Generates a list of 100 random integers between 1 and 1000
2. Calculates the mean, median, and standard deviation
3. Writes a summary to /output/stats_report.txt
4. Prints the summary to stdout

Use only the Python standard library. Do NOT run the script.
```

Claude will write the file to `staging/analyze_numbers.py`.

### Step 4 — Review the generated script

**macOS / Linux**

```bash
cat staging/analyze_numbers.py
```

**Windows PowerShell**

```powershell
Get-Content staging\analyze_numbers.py
```

Check for:
- No imports beyond the standard library (`random`, `statistics`, `pathlib` are okay)
- Output goes only to `/output/` and `stdout`
- No file reads outside the script's own logic
- No network access, no `os.system()`, no `eval()`

If the script looks clean, move it to the approved directory.

**macOS / Linux**

```bash
cp staging/analyze_numbers.py approved/
```

**Windows PowerShell**

```powershell
Copy-Item staging\analyze_numbers.py approved\
```

### Step 5 — Run the approved script through the sandbox

**macOS / Linux**

```bash
./sandbox-run.sh approved/analyze_numbers.py
```

**Windows PowerShell**

```powershell
.\sandbox-run.ps1 -ScriptPath approved\analyze_numbers.py
```

If the exit code is 0, you will see `STATUS: PASSED` and the output files will be in the `output/` directory.

### Step 6 — Inspect the output and the log

**macOS / Linux**

```bash
echo "=== Output files ==="
ls -la output/
echo ""
echo "=== Stats report ==="
cat output/stats_report.txt
echo ""
echo "=== Execution log ==="
cat logs/run-analyze_numbers.py-*.log
```

**Windows PowerShell**

```powershell
Write-Host "=== Output files ==="
Get-ChildItem output\
Write-Host ""
Write-Host "=== Stats report ==="
Get-Content output\stats_report.txt
Write-Host ""
Write-Host "=== Execution log ==="
Get-Content logs\run-analyze_numbers.py-*.log
```

The log file is your audit trail — it records exactly what ran, when, and whether it passed.

### Step 7 — Test the pipeline with a deliberately failing script

Create a script that will fail to verify the pipeline catches errors correctly.

**macOS / Linux**

```bash
cat > staging/bad_script.py << 'EOF'
# This script tries to do something it should not
import os
# Try to read a sensitive file
with open("/etc/shadow", "r") as f:
    print(f.read())
EOF

cp staging/bad_script.py approved/
./sandbox-run.sh approved/bad_script.py
echo "Exit code: $?"
```

**Windows PowerShell**

```powershell
@"
# This script tries to do something it should not
import os
# Try to read a sensitive file
with open("/etc/shadow", "r") as f:
    print(f.read())
"@ | Out-File -Encoding utf8 staging\bad_script.py

Copy-Item staging\bad_script.py approved\
.\sandbox-run.ps1 -ScriptPath approved\bad_script.py
Write-Host "Exit code: $LASTEXITCODE"
```

The script will fail with a `PermissionError` because the container runs as a non-root user (on the base image) and `/etc/shadow` is not readable. The log will show `STATUS: FAILED`.

### Step 8 — Promote safe outputs to your project

Only after a `PASSED` run should you copy outputs into your real project.

**macOS / Linux**

```bash
# Only for PASSED runs:
cp output/stats_report.txt ~/my-project/reports/
cd ~/my-project
git add reports/stats_report.txt
git commit -m "feat(reports): add stats report (sandbox-verified, AI-generated)"
```

**Windows PowerShell**

```powershell
# Only for PASSED runs:
Copy-Item output\stats_report.txt "$HOME\my-project\reports\"
Set-Location "$HOME\my-project"
git add reports\stats_report.txt
git commit -m "feat(reports): add stats report (sandbox-verified, AI-generated)"
```

> **Real danger:** Never copy outputs from a FAILED sandbox run into your project. A script might partially succeed and write corrupt or incomplete data before crashing. Only PASSED runs with exit code 0 should be trusted.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `sandbox-run.sh: Permission denied` | Script is not executable | Run `chmod +x sandbox-run.sh` |
| Docker says `no such file or directory` for the script | Path resolution failed with relative paths | Use `realpath` or provide absolute paths; on Windows use `Resolve-Path` |
| Output directory is empty after a PASSED run | Script wrote to `/tmp` instead of `/output` | Edit the script to write to `/output/` — that is the only mounted writable directory |
| Log file shows exit code 137 | Container was killed by the OOM (out-of-memory) killer | Increase `--memory=256m` in the sandbox runner |
| Script passes in sandbox but fails in real project | Different Python version or missing data files | Pin your Docker image to the same Python version you use locally |
| Files in `output/` owned by root (Linux only) | Docker creates files as root by default | Add `--user $(id -u):$(id -g)` to the `docker run` command (already in the script) |
| PowerShell script fails with backtick errors | Copy-paste introduced invisible characters | Retype the script or save it fresh from Claude Code |
