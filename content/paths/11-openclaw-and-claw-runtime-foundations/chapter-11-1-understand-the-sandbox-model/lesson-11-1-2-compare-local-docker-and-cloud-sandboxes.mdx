# Compare local, Docker, and cloud sandboxes

## Objective

By the end of this lesson you will understand four levels of code isolation — bare metal, language-level virtual environments, OS-level Docker containers, and cloud sandboxes — and be able to choose the right one for a given situation. You will generate a comparison snapshot from your own machine to anchor the theory.

## Walkthrough

### Step 1 — See what bare metal execution looks like

<StepMeta purpose="Set the zero-isolation baseline so the tradeoffs of stronger sandboxing are easier to judge." action="Run the bare-metal Python command and observe that it can read your username, list home-directory contents, and resolve DNS normally." expectedResult="You see direct evidence that a normal local process has unrestricted file and network access by default." whyItMatters="Comparisons only make sense when you start from the real baseline: without isolation, code runs with your full permissions and network reach." nextStep="Use that baseline to compare the four isolation levels and what each one actually protects." askTutor="I'm on Step 1 of Compare local, Docker, and cloud sandboxes. Help me interpret this bare-metal command so I can use it as the baseline for the rest of the comparison." />

When you run code directly on your machine with no isolation, the process has the same permissions you do.

**macOS / Linux**

```bash
python3 -c "
import os, socket
print('User:', os.getenv('USER'))
print('Home contents:', os.listdir(os.path.expanduser('~'))[:3])
print('Can resolve DNS:', socket.getaddrinfo('example.com', 80)[0][4])
"
```

**Windows PowerShell**

```powershell
python -c "import os, socket; print('User:', os.getenv('USERNAME')); print('Home contents:', os.listdir(os.path.expanduser('~'))[:3]); print('Can resolve DNS:', socket.getaddrinfo('example.com', 80)[0][4])"
```

The script can read your username, list your files, and reach the internet. This is the baseline — zero isolation.

> **Real danger:** Running `pip install some-package && python -c "import some_package"` on bare metal means that package's install script and runtime code have full access to your machine. Typosquatting attacks exploit this by publishing packages with names one letter off from popular libraries.

### Step 2 — Understand the four isolation levels

<StepMeta purpose="Create a decision framework for choosing the lightest isolation level that still matches the risk." action="Read the comparison table and map bare metal, venvs, Docker, and cloud sandboxes to their file, network, dependency, cost, and use-case tradeoffs." expectedResult="You can explain how the four isolation levels differ and what each one is best suited for." whyItMatters="People often overestimate what a venv protects or underestimate when Docker or a cloud sandbox is justified, so this table anchors those decisions." nextStep="Test one of the key distinctions directly by proving a venv isolates packages but not file access." askTutor="I'm on Step 2 of Compare local, Docker, and cloud sandboxes. Help me turn this comparison table into a practical rule for choosing the right isolation level." />

| Level | Tool | File isolation | Network isolation | Dependency isolation | Cost | Best for |
|---|---|---|---|---|---|---|
| **Bare metal** | None | None — full access | None — full access | None — system Python | Free | Trusted code you wrote yourself |
| **Language-level** | `python -m venv` | None — full access | None — full access | Yes — separate `site-packages` | Free | Isolating project dependencies |
| **OS-level** | Docker | Full — separate file tree | Optional via `--network none` | Full — separate OS | Free (local) | Running untrusted code safely |
| **Cloud sandbox** | E2B, Replit, GitHub Codespaces | Full — remote machine | Full — your machine untouched | Full — remote OS | Paid / free tier | Ephemeral execution, no local Docker needed |

Key insight: each level adds a wall, but also adds complexity. Use the lightest isolation that matches your threat.

### Step 3 — Prove that a virtual environment does NOT isolate files

<StepMeta purpose="Show that Python virtual environments are for dependency separation, not security isolation." action="Create a temporary venv, run Python from inside it, and verify it can still list files from your actual home directory." expectedResult="The venv Python can still see real home-directory contents even though it uses a separate package environment." whyItMatters="This is the most common conceptual mistake with venvs: they prevent dependency clashes but do nothing to stop file reads, shell access, or network calls." nextStep="Contrast that behavior with Docker, where the process sees a separate file system instead of your host home directory." askTutor="I'm on Step 3 of Compare local, Docker, and cloud sandboxes. Help me explain why this venv test proves package isolation without proving any security isolation." />

Create a venv and show that code inside it can still read your home directory.

**macOS / Linux**

```bash
python3 -m venv /tmp/test-venv
/tmp/test-venv/bin/python -c "import os; print('Home visible:', os.listdir(os.path.expanduser('~'))[:3])"
rm -rf /tmp/test-venv
```

**Windows PowerShell**

```powershell
python -m venv "$env:TEMP\test-venv"
& "$env:TEMP\test-venv\Scripts\python.exe" -c "import os; print('Home visible:', os.listdir(os.path.expanduser('~'))[:3])"
Remove-Item -Recurse -Force "$env:TEMP\test-venv"
```

Your home directory is fully visible. A venv only isolates installed packages — not file access.

### Step 4 — Prove that Docker DOES isolate files

<StepMeta purpose="Contrast Docker with venvs by proving Docker changes the file view available to the process." action="Run the same visibility check inside a Docker container and compare the /root listing to what the venv could see." expectedResult="Docker shows only container-owned files, not your real home directory contents." whyItMatters="This direct contrast is what makes the file-isolation difference memorable: venvs share your host file system, Docker does not unless you mount it." nextStep="Translate the comparison into a practical selection rule for real projects and threat levels." askTutor="I'm on Step 4 of Compare local, Docker, and cloud sandboxes. Help me compare this Docker file view to the earlier venv test so I can explain the difference clearly." />

Run the same check inside a Docker container.

**macOS / Linux**

```bash
docker run --rm python:3.12-slim python -c "import os; print('Home visible:', os.listdir('/root'))"
```

**Windows PowerShell**

```powershell
docker run --rm python:3.12-slim python -c "import os; print('Home visible:', os.listdir('/root'))"
```

The container sees only its own empty `/root` — your real home directory is invisible.

### Step 5 — Understand when to use each level

<StepMeta purpose="Convert the isolation theory into a fast decision flow you can apply during real work." action="Use the three scenario questions to decide whether your current task belongs on bare metal, in a venv, inside Docker, or in a cloud sandbox." expectedResult="You can choose an isolation level based on trust, dependency-conflict risk, and local tooling availability." whyItMatters="A sandbox comparison is only useful if it helps you choose a safe default quickly when code or tooling shows up in real projects." nextStep="Record what isolation options are actually available on your machine so your future decisions are grounded in your current setup." askTutor="I'm on Step 5 of Compare local, Docker, and cloud sandboxes. Help me apply these three questions to a real scenario I have in mind so I pick the right level of isolation." />

Answer these three questions for your scenario to pick the right level.

| Question | If yes → | If no → |
|---|---|---|
| Did I write this code or review every line? | Bare metal is okay | Go to next question |
| Am I only worried about dependency conflicts, not security? | Use a venv | Go to next question |
| Do I have Docker installed locally? | Use Docker with `--rm --network none` | Use a cloud sandbox like E2B or Codespaces |

### Step 6 — Create your own comparison snapshot

<StepMeta purpose="Capture the isolation tools you actually have available so your future choices are based on your machine, not abstractions." action="Generate sandbox-comparison.md listing the date and the detected versions or absence of venv, Docker, and similar tools." expectedResult="You create a local comparison snapshot file documenting which isolation tools are currently available on your machine." whyItMatters="A tool availability snapshot turns the lesson into a reusable personal reference and prevents decision advice from drifting away from your actual environment." nextStep="Use the same evidence-based mindset in the Docker lesson where you build a locked-down container for untrusted code." askTutor="I'm on Step 6 of Compare local, Docker, and cloud sandboxes. Help me turn this snapshot file into a practical reminder of which isolation tools I can actually use right now." />

Save a record of what isolation tools are available on your machine.

**macOS / Linux**

```bash
echo "=== Isolation Tools Snapshot ===" > ~/sandbox-comparison.md
echo "Date: $(date +%Y-%m-%d)" >> ~/sandbox-comparison.md
echo "" >> ~/sandbox-comparison.md
echo "Python venv: $(python3 -m venv --help 2>&1 | head -1)" >> ~/sandbox-comparison.md
echo "Docker: $(docker --version 2>/dev/null || echo 'NOT INSTALLED')" >> ~/sandbox-comparison.md
echo "Podman: $(podman --version 2>/dev/null || echo 'NOT INSTALLED')" >> ~/sandbox-comparison.md
cat ~/sandbox-comparison.md
```

**Windows PowerShell**

```powershell
$snapshot = @"
=== Isolation Tools Snapshot ===
Date: $((Get-Date).ToString('yyyy-MM-dd'))

Python venv: $(python -m venv --help 2>&1 | Select-Object -First 1)
Docker: $(try { docker --version } catch { 'NOT INSTALLED' })
"@
$snapshot | Out-File "$HOME\sandbox-comparison.md"
Get-Content "$HOME\sandbox-comparison.md"
```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `python3: command not found` | Python not installed or aliased differently | On Windows use `python` instead of `python3` |
| Docker command hangs | Docker Desktop is installed but the engine is not running | Open Docker Desktop and wait for the engine status to show "Running" |
| venv creation fails with `ensurepip` error | System Python was installed without pip | Install the `python3-venv` package: `sudo apt install python3-venv` (Debian/Ubuntu) |
| Confused about when a venv is "enough" | venvs do not provide security isolation | Use Docker when running code you did not write or fully review |
| Cloud sandbox requires a credit card | Free tier limits on E2B or Codespaces | Docker locally is free and provides equivalent isolation for learning |
