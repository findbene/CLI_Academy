# Rename files and update imports

## Objective

Use Claude Code to rename a source file in a real project, update every import or require statement that references the old name, and verify with a search that zero stale references remain. By the end you will have a cleanly renamed file with no broken imports and passing tests.

## Walkthrough

Continue in your cloned project from earlier lessons.

### Step 1 — Pick a file to rename

<StepMeta purpose="Choose a practice target that is real enough to exercise import updates but not risky enough to break startup." action="Ask Claude for a non-entry-point file that is referenced in multiple places and is safe to rename." expectedResult="You identify a specific source file plus the rough number of files that currently reference it." whyItMatters="A good target lets you practice the rename workflow without accidentally disrupting the project's main execution path." nextStep="Before renaming anything, capture the current reference count so you have a baseline to compare later." askTutor="I'm on Step 1 of Rename files and update imports. Help me choose a safe file to rename and explain why the entry point is a bad practice target." />

Ask Claude which file would be safe to practice on:

```
Which source file in this project would be safe to rename for practice? I want one that is imported by other files but is NOT the main entry point. Tell me how many files reference it.
```

Claude suggests a utility or helper file and tells you its import count. For json-server it might pick something like `src/server/utils.js` or a similar module.

### Step 2 — Record the current reference count

<StepMeta purpose="Create a baseline measurement you can use to verify that every old import was updated." action="Search the codebase for the old file name or import path and count how many matches exist before the rename." expectedResult="You know how many references should disappear for the old name and roughly how many should reappear for the new one." whyItMatters="Without a before-count, it is much harder to prove that the rename was complete instead of almost complete." nextStep="Now instruct Claude to perform the rename and update every import that points to the old path." askTutor="I'm on Step 2 of Rename files and update imports. Help me build the right search pattern so my before-count reflects real import references instead of unrelated matches." />

Before renaming, count every file that references the old name:

**macOS / Linux**

```bash
grep -rn "require.*utils\|from.*utils" src/
```

**Windows PowerShell**

```powershell
Get-ChildItem -Recurse -Filter *.js src\ | Select-String -Pattern "require.*utils|from.*utils"
```

Count the results. Write this number down — this is your "before" count.

### Step 3 — Ask Claude to rename and update

<StepMeta purpose="Perform the rename as one coordinated change across the whole repo instead of manually editing scattered import lines." action="Give Claude the exact old path and new path and require it to list every file it changes." expectedResult="The file is renamed on disk and the repo now points at the new path in every updated import or require statement." whyItMatters="Precise instructions reduce the chance of partial renames or accidental edits to files you did not intend to touch." nextStep="Search for the old file name again and confirm that zero stale references remain." askTutor="I'm on Step 3 of Rename files and update imports. Help me phrase this rename instruction so Claude changes only the intended file and references." />

Give Claude an explicit instruction:

```
Rename src/server/utils.js to src/server/helpers.js and update every import or require statement across the entire project that references the old path. Show me every file you change.
```

Claude will:
1. Rename the file on disk
2. Update every `require('./utils')` or `import ... from './utils'` to `'./helpers'`
3. List every file it modified

Review the list. Make sure it only changed the files you expected.

### Step 4 — Verify zero stale references remain

<StepMeta purpose="Prove that the old filename has been fully removed from live import paths." action="Run the same search you used before and confirm the old path now produces no matches." expectedResult="The old file name or import path returns zero relevant results across the codebase." whyItMatters="This is the key correctness check for a rename; any leftover old import is a future broken-module error waiting to happen." nextStep="After the old name disappears, search for the new name and compare that count to the baseline." askTutor="I'm on Step 4 of Rename files and update imports. Help me interpret these leftover matches and tell me whether they are real stale imports or harmless strings." />

This is the critical check. Search for the OLD name:

**macOS / Linux**

```bash
grep -rn "require.*utils\|from.*utils" src/
```

**Windows PowerShell**

```powershell
Get-ChildItem -Recurse -Filter *.js src\ | Select-String -Pattern "require.*utils|from.*utils"
```

The result must be empty — zero matches for the old filename. If any results appear, those are missed references. Tell Claude: "You missed these files: [list]. Update them too."

### Step 5 — Verify the new name is referenced correctly

<StepMeta purpose="Confirm that the rename not only removed stale imports but also replaced them with the new path everywhere needed." action="Search for the new file name and compare the result count to your original before-count." expectedResult="The new import path appears in the expected places, matching the project state you intended to create." whyItMatters="A rename is only complete when the new path is wired in correctly, not merely when the old path stops appearing." nextStep="Run the test suite so runtime behavior validates the rename beyond string matching." askTutor="I'm on Step 5 of Rename files and update imports. Help me compare my before-count and after-count and decide whether the rename coverage is complete." />

Search for the NEW name:

**macOS / Linux**

```bash
grep -rn "require.*helpers\|from.*helpers" src/
```

**Windows PowerShell**

```powershell
Get-ChildItem -Recurse -Filter *.js src\ | Select-String -Pattern "require.*helpers|from.*helpers"
```

The match count should equal your "before" count from Step 2.

### Step 6 — Run the tests

<StepMeta purpose="Catch missed paths or broken module resolution that a simple string search might not reveal." action="Run the project's tests and use any failure message to identify unresolved or incorrect import paths." expectedResult="Tests pass cleanly or pinpoint the exact import path that still needs attention." whyItMatters="Runtime verification protects you from subtle rename bugs such as wrong relative depths or environment-specific path issues." nextStep="If verification passes, commit the rename as one clean refactor change." askTutor="I'm on Step 6 of Rename files and update imports. Help me use this test output to figure out whether the rename missed a reference or broke a path depth." />

```bash
npm test
```

If tests pass, the rename is clean. If they fail with "Cannot find module," ask Claude:

```
Tests failed after renaming utils.js to helpers.js. The error says "Cannot find module './utils'". What reference did we miss?
```

### Step 7 — Commit the rename

<StepMeta purpose="Record the verified rename as a single refactor unit in git history." action="Stage the deletion, new file, and updated imports together, then commit with a message that names the rename clearly." expectedResult="Git stores the rename and related import edits in one focused commit." whyItMatters="Refactor commits are easiest to review and revert when the entire rename lives in one isolated history entry." nextStep="After a safe rename, move on to adding a new feature file and wiring it into the existing project." askTutor="I'm on Step 7 of Rename files and update imports. Help me make sure this commit includes the full rename and nothing unrelated." />

```bash
git add -A
git commit -m "refactor: rename utils.js to helpers.js and update all imports"
```

> **Why `git add -A`?** A file rename involves both deleting the old file and creating the new one. Using `-A` stages both the deletion and creation. Git detects this as a rename in the commit history.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `grep` still finds the old name after the rename | Claude missed a dynamic `require()` call, a string in a test fixture, or a comment | Fix remaining references manually or ask Claude: "Search for the literal string 'utils' in all files, not just import statements" |
| Tests fail with "Cannot find module" | An import path has a typo or a wrong relative path depth | Check the exact error path and compare it to where the new file actually lives |
| Git shows the change as delete + unrelated add instead of a rename | Git only detects renames when content similarity is high enough | This is cosmetic — the code is correct. Run `git diff --find-renames` to see it displayed as a rename |
| Windows path separators cause an import mismatch | Windows uses `\` but Node.js and Python imports require `/` | Always use forward slashes in import paths, even on Windows |
| Claude renames a file you did not intend | Your instruction was ambiguous | Be specific: give the exact current full path and the exact new full path |
| Python imports break differently than expected | Python uses dot-separated module paths, not file paths | Ask Claude: "Update the Python import statements using dot notation, not file paths" |
