# Update dependencies and lock files safely

## Objective

Use Claude Code to check for outdated dependencies, pick the safest one to update, perform the update, and verify the project still works. By the end you will have updated one package with tests passing and a clean commit.

## Walkthrough

You need a project with installed dependencies for this lesson. If you have been following along with json-server, install its dependencies first. If you prefer a Python project, substitute pip commands where noted.

> **Real danger — `npm install` executes arbitrary code.** The `postinstall`, `preinstall`, and `prepare` scripts in `package.json` run automatically during install. Only install dependencies in repos you trust. For unfamiliar repos, read the `"scripts"` block in `package.json` before running install.

### Step 4 — Update the single package

<StepMeta purpose="Make one tightly scoped dependency change that you can reason about and test." action="Update only the selected package using the package manager command appropriate for your stack." expectedResult="Your manifest and lock file update for a single dependency and the working tree reflects a bounded change." whyItMatters="One-package updates are much easier to verify and revert than sweeping version changes across the whole dependency graph." nextStep="Run the tests immediately to confirm that the update did not break the project." askTutor="I'm on Step 4 of Update dependencies and lock files safely. Help me understand what changed in my manifest and lock file after this single-package update." />

Follow Claude's recommendation. For example, if Claude suggests updating `lodash`:

**Node.js**

```bash
npm update lodash
```

**Python (example: updating `requests`)**

```bash
pip install --upgrade requests
```

> **Real danger — major version bumps can break your code.** If npm outdated shows a package at version 3.x and latest is 5.x, do NOT run `npm install package@latest` without understanding what changed. Ask Claude: "What are the breaking changes between [package] v3 and v5?" and only update after you understand the impact.

<StepMeta purpose="Test whether the dependency update preserved existing behavior before you commit anything." action="Run the project's test suite, inspect any failures, and decide whether the update should stay or be rolled back." expectedResult="Tests either pass cleanly or produce specific failures that tell you what broke." whyItMatters="Dependency updates are only safe when the project still behaves correctly afterward; passing tests are your main evidence." nextStep="If the verification looks good, stage the dependency files and commit the update with a clear message." askTutor="I'm on Step 5 of Update dependencies and lock files safely. Run the test suite:

**Node.js**

```bash
npm test
```

**Python**

```bash
pytest
```

If tests pass, the update is safe. If tests fail, ask Claude:

```
Tests failed after updating lodash. Here is the error:
[paste error output]
What broke and should I roll back?
```

To roll back a Node.js update:

```bash
git checkout -- package.json package-lock.json
npm install
```

To roll back a Python update:

```bash
pip install requests==2.28.0    # whatever the old version was
```


For Python:

```bash

```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `npm outdated` shows nothing | All packages are already current | No action needed — your dependencies are up to date |
| `npm test` fails after the update | The new version has breaking changes or removed APIs | Roll back: `git checkout -- package.json package-lock.json && npm install`, then investigate before retrying |
| `pip list --outdated` is very slow | pip is checking remote indexes over a slow connection | Add `--timeout 15` or try `pip list --outdated --local` to skip globally installed packages |
| Lock file diff is enormous | Updating one package triggered cascading sub-dependency updates | This is normal — review the lock file diff briefly but focus on the direct dependency you changed |
| `npm install` runs a suspicious script you did not expect | A package has a `postinstall` hook | Read the `"scripts"` block in `package.json` first; use `npm install --ignore-scripts` for untrusted projects |
| Python venv is not activated | You forgot to activate before `pip install` | Run `source .venv/bin/activate` (macOS/Linux) or `.\.venv\Scripts\Activate.ps1` (Windows) |
