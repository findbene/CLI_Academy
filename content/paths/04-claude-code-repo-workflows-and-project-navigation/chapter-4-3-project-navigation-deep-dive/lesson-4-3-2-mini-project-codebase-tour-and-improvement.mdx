# Mini-project: codebase tour and improvement

## Objective

Complete a full codebase tour on a project you have never worked with before. You will clone it, map its structure with Claude, trace a feature, make one real improvement, verify it works, and commit everything. This combines every skill from Path 04 into a single end-to-end workflow.

## Walkthrough

Pick a NEW open-source project for this mini-project — do not reuse the one from earlier lessons. Good choices include:

- **Node.js:** `https://github.com/expressjs/morgan` (HTTP request logger, small and focused)
- **Python:** `https://github.com/psf/requests` (HTTP library, well-organized)
- **Any language you know:** pick something under 5,000 lines of source code with a clear purpose

### Step 1 — Clone and enter the project

<StepMeta purpose="Set up a fresh practice repo so you can run the full tour workflow on code you have not seen before." action="Clone a new small open-source project, move into the repo directory, and keep the setup in read-only mode until you understand it." expectedResult="You have a second unfamiliar codebase available locally and are positioned inside it without having executed install scripts yet." whyItMatters="A new repo prevents you from leaning on memory from earlier lessons and makes the mini-project a real test of your navigation process." nextStep="With the new repo cloned, ask Claude for a full orientation covering purpose, structure, entry point, config, and stack." askTutor="I'm on Step 1 of Mini-project: codebase tour and improvement. Help me choose a new small repo and explain how to vet it before I run any install commands." />

**macOS / Linux**

```bash
cd ~/projects
git clone https://github.com/expressjs/morgan.git
cd morgan
```

**Windows PowerShell**

```powershell
cd ~\projects
git clone https://github.com/expressjs/morgan.git
cd morgan
```

> **Reminder — do not install dependencies from untrusted repos blindly.** Read the `"scripts"` section in `package.json` (or the `Makefile` / `setup.py` for Python) before running any install commands.

### Step 2 — Map the structure with Claude

<StepMeta purpose="Create a full working map of the new codebase before you pick a feature or change anything." action="Ask Claude for a structured orientation covering what the project does, how the directories are organized, where the entry point lives, and which configs matter." expectedResult="You get a concise but complete tour of the repo's purpose, structure, key files, and stack." whyItMatters="The quality of the rest of the mini-project depends on this orientation being good enough to guide your later trace and improvement work." nextStep="Pick one core feature and ask Claude to trace it through the repo end to end." askTutor="I'm on Step 2 of Mini-project: codebase tour and improvement. Help me evaluate whether Claude's orientation is complete enough to start tracing features." />

Start Claude Code in the project directory and ask for a full orientation:

```
I just cloned this project and have never seen it before. Give me a full orientation:
1. What does this project do?
2. What is the folder structure and what does each directory contain?
3. Where is the main entry point?
4. What are the key config files?
5. What is the tech stack and key dependencies?
```

Read Claude's answer carefully. This is your map of the codebase. Note the entry point and the main source directory.

### Step 3 — Trace one feature

<StepMeta purpose="Move from broad orientation into one concrete behavior so you can understand how the repo actually works." action="Ask Claude to trace a core feature from input to output and manually verify at least one hop in the resulting flow." expectedResult="You produce a believable feature trace and confirm at least one file or line with your own search." whyItMatters="Feature tracing is where repo understanding becomes operational, because you stop naming directories and start following behavior." nextStep="After understanding one core flow, ask Claude for a safe, bounded improvement you could make to the project." askTutor="I'm on Step 3 of Mini-project: codebase tour and improvement. Help me choose a trace target that is central enough to matter but small enough to verify." />

Pick a core feature and ask Claude to trace it end-to-end:

```
Trace how the "combined" log format works from the moment a request comes in to the moment the log line is written. Name every file and function involved.
```

Verify at least one hop in the trace yourself:

**macOS / Linux**

```bash
grep -n "combined" index.js
```

**Windows PowerShell**

```powershell
Select-String -Path index.js -Pattern "combined"
```

Confirm the file and line match Claude's trace.

### Step 4 — Identify an improvement

<StepMeta purpose="Choose a real change that improves the repo without turning the exercise into an open-ended feature project." action="Ask Claude for one safe internal improvement that does not change the public API and select the best bounded option." expectedResult="You pick a small improvement such as docs cleanup, a clearer error message, or a tiny internal refactor." whyItMatters="A bounded improvement lets you practice making real changes in an unfamiliar repo while keeping risk and review scope low." nextStep="Once the improvement is chosen, have Claude implement it and show the exact edits." askTutor="I'm on Step 4 of Mini-project: codebase tour and improvement. Help me reject risky suggestions and choose an improvement that is genuinely small and safe." />

Ask Claude for a safe, small improvement:

```
What is one small, safe improvement I could make to this project? I want something real — a missing JSDoc comment, a clearer error message, a typo in documentation, or a small code cleanup. Do NOT suggest anything that changes the public API.
```

Claude might suggest:
- Adding a JSDoc comment to an undocumented function
- Improving a confusing error message
- Fixing a typo in the README
- Adding a missing edge-case test

Pick whichever suggestion feels right.

### Step 5 — Make the improvement

<StepMeta purpose="Turn the chosen idea into an actual repo change with a clear explanation of what changed and why." action="Ask Claude to implement the bounded improvement and review the resulting diff carefully." expectedResult="The repo now contains one small focused change that matches the improvement you selected." whyItMatters="Implementation is where you test whether you can guide Claude precisely enough to change unfamiliar code without drifting scope." nextStep="Verify the change with tests or a tight diff review before you commit anything." askTutor="I'm on Step 5 of Mini-project: codebase tour and improvement. Help me review this proposed change and confirm it stays within the safe scope I asked for." />

Ask Claude to implement it:

```
Go ahead and make that improvement. Show me exactly what you changed and why.
```

Review the change. Confirm it is small, correct, and does not affect the public API.

### Step 6 — Verify the improvement

<StepMeta purpose="Confirm that the change is correct and limited before you record it in git history." action="Run the relevant tests if available or inspect the diff line by line when the change is documentation-only or otherwise hard to execute." expectedResult="You either see passing verification or gain a precise view of the exact lines changed and their effect." whyItMatters="Verification is what separates a good-looking edit from a trustworthy improvement in a repo you barely know." nextStep="Once the change is verified, commit it with a conventional message that matches the kind of improvement you made." askTutor="I'm on Step 6 of Mini-project: codebase tour and improvement. Help me decide whether this change is sufficiently verified by tests or if I need a stricter manual diff review." />

Install dependencies (if you trust the project) and run the test suite:

```bash
npm install
npm test
```

If the project does not have tests, or your change was documentation-only, verify the diff manually:

**macOS / Linux and Windows PowerShell**

```bash
git diff
```

Review every line. The change should be exactly what you expected — nothing more, nothing less.

### Step 7 — Commit the improvement

<StepMeta purpose="Preserve the verified improvement as a clean milestone in the repo's local history." action="Stage only the intended changes and commit with the conventional commit prefix that fits the improvement type." expectedResult="Git records one clear commit that describes the improvement accurately and contains no unrelated noise." whyItMatters="A clean commit proves that you can finish a real repo task end to end, not just make edits in a dirty working tree." nextStep="After the commit is recorded, ask Claude to summarize the session so you leave behind a reusable artifact." askTutor="I'm on Step 7 of Mini-project: codebase tour and improvement. Help me pick the right conventional commit prefix and confirm I am not staging anything unrelated." />

```bash
git add -A
git commit -m "docs: add JSDoc comment to compile function"
```

Use the appropriate conventional commit type:
- `docs:` for documentation changes
- `fix:` for bug fixes
- `refactor:` for code cleanups that do not change behavior
- `test:` for adding or fixing tests

<StepMeta purpose="Capture the full repo tour and improvement work in a short artifact you can reuse or share later." action="Ask Claude for a markdown summary of the project, the feature trace, the improvement, and the outcome, then save it to a file." expectedResult="You produce a session-summary document that records what repo you explored and what you changed." whyItMatters="A written summary reinforces what you learned and gives you proof of completion for later review or portfolio use." nextStep="After this mini-project, you have completed the full Path 04 repo-tour workflow and can reuse the same pattern on real repos." askTutor="I'm on Step 8 of Mini-project: codebase tour and improvement. Ask Claude to summarize the tour:

```
Write a short session summary of what we did today: which project we explored, what we learned about its structure, the feature we traced, and the improvement we made. Format it as markdown.
```

Save the summary:

**macOS / Linux**

```bash

```

**Windows PowerShell**

```powershell

# Paste Claude's output and save
```

This summary is proof you completed the tour and a useful reference for the future.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `npm test` fails before you even change anything | The project has pre-existing test failures or needs a specific Node version | Check the project README for setup instructions; try `nvm use` if a `.nvmrc` exists |
| Claude suggests a change that modifies the public API | The improvement was too ambitious for a practice exercise | Ask: "That changes the public API. Give me a safer, internal-only improvement instead" |
| `git diff` shows untracked files you did not create | `npm install` created `node_modules/` or build artifacts | Check that `node_modules/` is in `.gitignore`; only stage your actual improvement files |
| You accidentally cloned a huge repo | The project has more source code than expected | Check line count: `find . -name "*.js" -not -path "*/node_modules/*" \| xargs wc -l` (macOS/Linux) or `(Get-ChildItem -Recurse -Filter *.js -Exclude node_modules | Get-Content \| Measure-Object -Line).Lines` (Windows). Pick something under 5,000 lines |
| Claude's improvement does not apply because the code has changed | The source file differs from what Claude expects | Ask Claude: "Re-read index.js and adapt your suggestion to what the file actually says right now" |
| You cannot push the commit | You do not have write access to the upstream repo | This is expected — you are working on a local clone for practice. The commit exists in your local git history, which is the goal |
