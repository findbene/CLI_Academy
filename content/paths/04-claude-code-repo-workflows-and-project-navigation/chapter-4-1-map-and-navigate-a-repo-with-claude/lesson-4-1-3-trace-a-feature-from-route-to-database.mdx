# Trace a feature from route to database

## Objective

Use Claude Code to trace how a single feature works end-to-end — from the incoming HTTP request through the route handler, any middleware or service layers, and down to the data store. By the end you will have a numbered step-by-step trace naming every file and function involved in handling one request.

## Walkthrough

Continue in the same cloned project. If you are using json-server, you will trace how a GET request to `/posts` is handled. If you are using a different project, pick any route that reads data.

### Step 1 — Ask Claude to trace the route

<StepMeta purpose="Get an end-to-end hypothesis for how one request moves through the system before you inspect files manually." action="Ask Claude to trace a specific route from the incoming request through handlers, helpers, and the data layer in order." expectedResult="Claude returns an ordered list of files and functions involved in serving the request." whyItMatters="A full trace gives you a navigation plan for understanding behavior across multiple files instead of reading modules with no sense of sequence." nextStep="Open the first file in the trace and confirm that the routing setup is really there." askTutor="I'm on Step 1 of Trace a feature from route to database. Help me choose a good route to trace and tell me what makes a useful end-to-end answer from Claude." />

```
How does a GET request to /posts work in this project? Trace the flow from the incoming HTTP request through every file and function it touches until the response is sent back. Name each file and function in order.
```

Claude walks through something like:
1. `src/server/router/index.js` — sets up the Express router
2. `src/server/router/plural.js` — handles GET for collection resources
3. `src/server/mixins/...` — applies filtering and pagination
4. Reads from the lowdb database instance, returns JSON response

The exact files depend on the project. The important thing is you get an ordered list of hops.

### Step 2 — Verify the first file in the trace

<StepMeta purpose="Check that the trace begins in a real routing file instead of trusting Claude's first hop blindly." action="Open the first file Claude named and look for the routing setup or handoff it described." expectedResult="You see evidence that the request really enters this part of the code, such as route registration or router mounting." whyItMatters="Validating the first hop tells you whether the rest of the trace is likely grounded in the current repo or needs correction." nextStep="After the first hop checks out, inspect the next file in the chain and confirm the handler logic." askTutor="I'm on Step 2 of Trace a feature from route to database. Help me read this routing file and explain which line proves the request actually passes through here." />

Open the first file Claude named and look for the routing setup:

**macOS / Linux**

```bash
cat src/server/router/index.js
```

**Windows PowerShell**

```powershell
Get-Content src\server\router\index.js
```

Look for the line that maps GET requests to a handler function. You should see something like `router.get('/:resource', ...)` or `app.use(router)`.

### Step 3 — Verify the next hop

<StepMeta purpose="Confirm that the second file in Claude's route trace actually contains the handler or export it claimed." action="Open the next file, search for the handler definition, and match it to the route flow you are tracing." expectedResult="You find a function or export that receives the request and moves it closer to the data layer." whyItMatters="A trace only becomes trustworthy when each hop is independently supported by real code, not just by a plausible explanation." nextStep="Once the handler layer is verified, ask Claude to unpack the database or storage object it eventually calls." askTutor="I'm on Step 3 of Trace a feature from route to database. Help me identify whether this file is the real request handler or just a thin wrapper around another module." />

Open the second file and find the handler:

**macOS / Linux**

```bash
grep -n "function\|module.exports\|export" src/server/router/plural.js | head -10
```

**Windows PowerShell**

```powershell
Select-String -Path src\server\router\plural.js -Pattern "function|module.exports|export" | Select-Object -First 10
```

You should find a function that receives the request and calls the data layer. This confirms the second hop in Claude's trace.

### Step 4 — Ask Claude to explain the data layer

<StepMeta purpose="Clarify what object or service actually provides the data at the bottom of the route flow." action="Ask Claude where the database-like object comes from, how it is initialized, and how data gets loaded into it." expectedResult="You get a concrete explanation of the storage mechanism and the file or startup code that creates it." whyItMatters="Many beginner traces break at the data layer because they stop at a method call without understanding what object is behind it." nextStep="Search the repo yourself for the database initialization code Claude described." askTutor="I'm on Step 4 of Trace a feature from route to database. Help me understand what this db object is and how to distinguish data access code from routing code." />

When you reach the bottom of the trace, ask Claude directly:

```
In src/server/router/plural.js, I see a call to db.get(name).value(). What is `db` here, where is it created, and how does data get loaded?
```

Claude explains that `db` is a lowdb (or similar) instance created during server startup and loaded from a JSON file. This is the "database" in json-server's case — a flat JSON file on disk.

### Step 5 — Confirm the data layer with a search

<StepMeta purpose="Verify that Claude pointed you to the real storage setup instead of an approximate explanation." action="Search the repo for the library call or initialization pattern Claude mentioned and compare the results to its answer." expectedResult="You locate the actual file where the database or storage layer is created and can reconcile any mismatch." whyItMatters="Searching for initialization code is a repeatable way to validate claims about hidden app state and startup wiring." nextStep="After the data layer is confirmed, ask Claude to rewrite the whole route flow as a clean numbered trace." askTutor="I'm on Step 5 of Trace a feature from route to database. Help me interpret these search results and tell me which one is the real database initialization point." />

Verify Claude's explanation by searching for where the database is initialized:

**macOS / Linux**

```bash
grep -rn "lowdb\|new Low\|JSONFile\|readFileSync.*\.json" src/
```

**Windows PowerShell**

```powershell
Get-ChildItem -Recurse -Filter *.js src\ | Select-String -Pattern "lowdb|new Low|JSONFile|readFileSync.*\.json"
```

The results should point to the file Claude described. If you see a different file, ask Claude: "I found the database setup in [file]. Is this what you meant?"

Save this trace — it is your deliverable and the foundation for understanding how changes ripple through the project.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude's trace skips middleware | Middleware runs before the route handler but Claude focused on routing only | Ask: "Are there any middleware functions that execute before this route handler?" |
| `grep` returns too many results | The search term is too generic (e.g., "get") | Narrow your search: use a more specific function name or file path |
| A file in the trace does not exist at the path Claude gave | The project reorganized since the version Claude knows about | Ask Claude: "List all files in src/server/router/ so I can find the right one" |
| You cannot find any database layer | The project stores data in memory only | Ask Claude: "Where is data stored — disk, a database, or just in memory?" — some projects do not persist data |
| PowerShell `Select-String` misses nested files | The glob pattern does not recurse | Pipe `Get-ChildItem -Recurse -Filter *.js` into `Select-String` instead of using a flat path pattern |
| Trace has 10+ steps and feels overwhelming | The project has many abstraction layers | Ask Claude: "Simplify the trace to only the five most important files and skip any thin wrappers" |
