# Find the entry point and config files

## Objective

Use Claude Code to pinpoint the main entry point file, locate every configuration file, and identify the tech stack of a real project. By the end you will have a short reference card listing these items with their exact file paths.

## Walkthrough

Continue working in the project you cloned in Lesson 4.1.1 (json-server or your substitute).

### Step 1 — Ask Claude to find the entry point

<StepMeta purpose="Locate the file that actually launches the application so you know where execution begins." action="Ask Claude for the main entry point and verify it against the manifest or script configuration yourself." expectedResult="You identify a specific startup file and confirm that the repo metadata points to the same place." whyItMatters="Entry points tell you where runtime behavior starts, which makes the rest of the repo much easier to reason about." nextStep="After the entry point is verified, inventory the project's configuration files and what each one controls." askTutor="I'm on Step 1 of Find the entry point and config files. Help me understand how to tell whether a main, bin, or script field really points to the true startup file." />

```
Where is the main entry point of this application — the file that runs when a user starts the project? Show me the file path and the first few lines.
```

For a Node.js project, Claude identifies the `"main"` or `"bin"` field in `package.json`. For a Python project it finds `__main__.py`, a console-script entry in `pyproject.toml`, or a `main.py` file.

Confirm by checking the manifest yourself:

**macOS / Linux (Node.js)**

```bash
grep -E '"main"|"bin"' package.json
```

**Windows PowerShell (Node.js)**

```powershell
Select-String -Path package.json -Pattern '"main"|"bin"'
```

**macOS / Linux (Python)**

```bash
grep -E 'entry_points|console_scripts|\[project\.scripts\]' pyproject.toml setup.cfg setup.py 2>/dev/null
```

**Windows PowerShell (Python)**

```powershell
Select-String -Path pyproject.toml,setup.cfg,setup.py -Pattern 'entry_points|console_scripts' -ErrorAction SilentlyContinue
```

The output should match the file Claude named.

### Step 2 — Ask Claude to list all config files

<StepMeta purpose="Surface the files that control tooling, formatting, CI, builds, and project behavior beyond the main source code." action="Ask Claude for a full config-file inventory, then scan the repo root yourself for hidden dot-files it may have missed." expectedResult="You produce a more complete list of config files with one-line descriptions of what each one does." whyItMatters="Config files explain how the project is built, linted, tested, and deployed, which is often as important as understanding the source files." nextStep="Once the config landscape is clear, ask Claude to summarize the languages, frameworks, and libraries in use." askTutor="I'm on Step 2 of Find the entry point and config files. Help me recognize which hidden files count as configuration and which ones are just metadata." />

```
List every configuration file in this project — package.json, tsconfig, eslint, prettier, CI configs, Docker, etc. Give me the file path and a one-line description of what each one controls.
```

Claude returns a list like:
- `package.json` — dependencies, scripts, and project metadata
- `.eslintrc.json` — code style and lint rules
- `.github/workflows/ci.yml` — continuous integration pipeline
- `.gitignore` — files excluded from version control

Scan the project root to see if Claude missed any dot-files:

**macOS / Linux**

```bash
ls -a | head -30
```

**Windows PowerShell**

```powershell
Get-ChildItem -Force | Select-Object -First 30 Name
```

If you spot a config file Claude did not mention, ask: "What is `.editorconfig` for? You missed it in your list."

### Step 3 — Ask Claude to identify the tech stack

<StepMeta purpose="Translate package and dependency files into a readable description of the technologies the project depends on." action="Ask Claude to list the languages, frameworks, and key libraries with versions, then compare the answer to the actual dependency file." expectedResult="You end up with an accurate stack summary grounded in real version numbers from the repo." whyItMatters="The stack tells you what concepts and docs matter for this project, and it helps you judge whether Claude's later explanations fit the tools actually in use." nextStep="Pull the verified entry point, config, and stack details into a reusable reference card." askTutor="I'm on Step 3 of Find the entry point and config files. Help me turn this dependency file into a clean tech-stack summary without overcounting minor packages." />

```
What programming languages, frameworks, and key libraries does this project use? List them with version numbers from the config files.
```

Claude reads `package.json` dependencies (or `requirements.txt` / `pyproject.toml` for Python) and returns a clear stack list. Compare its answer to the actual dependency file:

**macOS / Linux (Node.js)**

```bash
cat package.json | grep -A 30 '"dependencies"'
```

**Windows PowerShell (Node.js)**

```powershell
Get-Content package.json | Select-String -Pattern '"dependencies"' -Context 0,30
```

**macOS / Linux (Python)**

```bash
cat requirements.txt
```

**Windows PowerShell (Python)**

```powershell
Get-Content requirements.txt
```

<StepMeta purpose="Capture the important orientation details in one reusable note instead of rediscovering them every time you reopen the repo." action="Ask Claude to format a short markdown reference card with the verified entry point, config files, and tech stack, then save it locally." expectedResult="You have a small markdown file that names the startup file, major config files, and core technologies with exact paths." whyItMatters="A reference card shortens future onboarding and proves that you verified the repo structure rather than only reading Claude's answers once." nextStep="With orientation basics in place, move on to tracing how one real feature flows through the codebase." askTutor="I'm on Step 4 of Find the entry point and config files. Ask Claude to consolidate everything:

```
Create a short markdown reference card with three sections: Entry Point, Config Files, and Tech Stack. Use the information you found in this project.
```

Review the output, then save it to a file:

**macOS / Linux**

```bash

```

**Windows PowerShell**

```powershell
# Open your editor and paste Claude's output

```

This reference card is your deliverable. Whenever you return to this project later, you can read it instead of re-exploring from scratch.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude names a file that does not exist | The repo structure changed after Claude's training data | Ask Claude to list the actual files: "Run `ls src/` and tell me what you see" |
| `grep` returns nothing for `"main"` in package.json | The project uses `"bin"` or `"exports"` instead | Search for those fields: `grep -E '"bin"|"exports"' package.json` |
| Python project has no `pyproject.toml` | Older project using `setup.py` or `setup.cfg` only | Ask Claude: "This project has no pyproject.toml. Where are config and entry points defined?" |
| Config file list seems incomplete | Dot-files are hidden by default on both OS platforms | Use `ls -a` (macOS/Linux) or `Get-ChildItem -Force` (Windows) to reveal them |
| Claude lists configs from a different version of the project | Open-source repos evolve quickly | Rely on your own `ls -a` output as the source of truth and ask Claude about any files you do not recognize |
