# Share session briefs and prompt libraries

## Objective

By the end of this lesson you will create a git-tracked `prompts/` directory containing reusable session briefs and prompt templates that your whole team can use. When someone needs Claude to generate a standup summary, write a runbook, or review code, they pull from a shared library instead of writing prompts from scratch.

## Walkthrough

### Step 1 — Create the prompts directory in your project

<StepMeta purpose="Create a shared repo location where reusable prompts and session briefs can live." action="Make the prompts directory in the project, move into it, and treat it as the canonical home for team-shared Claude prompts." expectedResult="You have a git-trackable prompts folder ready for templates, concrete prompts, and documentation." whyItMatters="Shared prompt libraries only become reusable when they live in a stable, discoverable location inside the repo rather than in personal notes or chat history." nextStep="Write a general session brief template so future prompt files start from a consistent structure." askTutor="I'm on Step 1 of Share session briefs and prompt libraries. Help me set up this prompts directory so it can become the canonical team library instead of a personal scratch folder." />

**macOS / Linux**

```bash
mkdir -p ~/my-project/prompts
cd ~/my-project/prompts
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\my-project\prompts"
Set-Location "$HOME\my-project\prompts"
```

### Step 2 — Write a session brief template

<StepMeta purpose="Define the standard structure every shared Cowork prompt should follow." action="Create the session-brief-template.md file with sections for context, instructions, and output location." expectedResult="You have a reusable template teammates can copy whenever they need a new session brief." whyItMatters="Prompt quality improves when teams start from a known structure instead of rewriting the same scaffolding from scratch every time." nextStep="Use that template to create concrete prompt files for real team workflows like standups and code review." askTutor="I'm on Step 2 of Share session briefs and prompt libraries. Help me make this session brief template structured enough that teammates can reuse it without confusion." />

A session brief tells Claude what role to play, what files to read, and what output format to follow. Create a reusable template:

**macOS / Linux**

```bash
cat > session-brief-template.md << 'EOF'
# Session Brief: [Task Name]

## Context
- Project: [project name]
- Goal: [one sentence describing what you need]
- Files to read: [list specific files]

## Instructions
1. Read the files listed above.
2. [Specific task — e.g., "Summarize changes since last Monday"]
3. [Output format — e.g., "Use bullet points, max 15 words each"]
4. [Constraints — e.g., "Do not invent information not in the files"]

## Output
Save the result to: [output file path]
EOF
```

**Windows PowerShell**

```powershell
@"
# Session Brief: [Task Name]

## Context
- Project: [project name]
- Goal: [one sentence describing what you need]
- Files to read: [list specific files]

## Instructions
1. Read the files listed above.
2. [Specific task - e.g., "Summarize changes since last Monday"]
3. [Output format - e.g., "Use bullet points, max 15 words each"]
4. [Constraints - e.g., "Do not invent information not in the files"]

## Output
Save the result to: [output file path]
"@ | Out-File "session-brief-template.md"
```

### Step 3 — Create concrete prompt files from the template

<StepMeta purpose="Turn the generic template into prompts the team can use immediately on common workflows." action="Create real prompt files such as standup-summary.md and code-review-checklist.md based on the template." expectedResult="You have at least two concrete prompts that encode repeatable team tasks with explicit instructions and output expectations." whyItMatters="A prompt library becomes valuable when it contains tested, task-specific prompts that solve real recurring problems instead of just abstract templates." nextStep="Document the library with a README so teammates know what exists and how to contribute new prompts." askTutor="I'm on Step 3 of Share session briefs and prompt libraries. Help me decide whether these prompt files are concrete enough to be genuinely useful for teammates." />

Build a few real prompts your team will actually use:

**macOS / Linux**

```bash
cat > standup-summary.md << 'EOF'
# Session Brief: Daily Standup Summary

## Context
- Project: this repository
- Goal: Generate a standup update from yesterday's git activity
- Files to read: none (use git log output)

## Instructions
1. Run: git log --since="yesterday" --author="$(git config user.name)" --oneline
2. Format the output as:
   - **What I did yesterday** (group by theme)
   - **What I'm doing today** (say "Continuing [last theme]" if unsure)
   - **Blockers** (say "None" unless something looks stalled)
3. Keep each bullet under 15 words.
4. Do not invent work not shown in the git log.

## Output
Save to: standup-summaries/standup-YYYY-MM-DD.md
EOF
```

```bash
cat > code-review-checklist.md << 'EOF'
# Session Brief: Code Review Checklist

## Context
- Project: this repository
- Goal: Review a pull request diff for common issues
- Files to read: the diff output (provided by the user)

## Instructions
1. Check for: security issues, missing error handling, unclear naming, dead code.
2. For each issue found, state: file, line, what is wrong, and suggested fix.
3. End with a summary: "Approve", "Approve with minor fixes", or "Request changes".
4. Do not suggest style changes unless they affect readability.

## Output
Print to screen (do not save to file).
EOF
```

### Step 4 — Add a README to the prompts directory

<StepMeta purpose="Make the prompt library discoverable and maintainable for people who did not create it." action="Write a README that explains usage, lists available prompts, and describes how to add or improve entries." expectedResult="The prompts directory includes a README that tells teammates how to use the library and extend it safely." whyItMatters="Without local documentation, prompt libraries become opaque collections of files that teammates ignore or misuse." nextStep="Commit the library into the project so it becomes a shared artifact rather than a local experiment." askTutor="I'm on Step 4 of Share session briefs and prompt libraries. Help me make this README clear enough that a teammate can find and use the right prompt without asking me first." />

**macOS / Linux**

```bash
cat > README.md << 'EOF'
# Prompt Library

Reusable session briefs and prompt templates for Claude.

## Usage
1. Copy the relevant `.md` file content.
2. Paste it into Claude Code as your opening prompt.
3. Follow any setup instructions in the brief.

## Available Prompts
| File | Purpose |
|------|---------|
| `session-brief-template.md` | Blank template for new prompts |
| `standup-summary.md` | Generate daily standup from git log |
| `code-review-checklist.md` | Review a PR diff for common issues |

## Contributing
1. Copy `session-brief-template.md` as your starting point.
2. Fill in all sections — do not leave placeholders.
3. Test the prompt yourself before committing.
4. Add your new prompt to the table above.
EOF
```

**Windows PowerShell**


### Step 6 — Verify a teammate can use a prompt

<StepMeta purpose="Test whether the library is actually reusable by someone other than the person who wrote it." action="Have a teammate try one prompt from the library without modifications and note any confusion or missing assumptions." expectedResult="You learn whether the prompt file naming, instructions, and output format are clear enough for another person to use directly." whyItMatters="Prompt libraries fail when they only work for their author, so this external usability check is the fastest way to improve reuse." nextStep="Use the same team-scale mindset in the playbook lesson, where you codify when and how the team should use Cowork at all." askTutor="I'm on Step 6 of Share session briefs and prompt libraries. Help me evaluate whether a teammate can use this prompt library without hidden assumptions or local setup knowledge." />

The best test: have someone else on your team open `prompts/standup-summary.md`, paste it into Claude Code, and check that the output matches the expected format. If they need to modify the prompt to get it working, update the shared version.

> **Real danger:** A prompt library nobody uses is wasted effort. The prompts must be tested, clearly named, and easy to find. If teammates keep writing their own prompts from scratch, the library needs better discoverability or the prompts need improvement.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Teammates ignore the prompt library | They do not know it exists or it is hard to find | Add a link to the prompts directory in the project README |
| Prompt works for you but not for a teammate | Prompt assumes your local setup (e.g., specific branch name) | Replace hardcoded values with placeholders and add a "Before you start" checklist |
| Prompts produce inconsistent results across sessions | Claude interprets vague instructions differently each time | Add stricter output format rules and explicit constraints to the prompt |
| Library grows large and unorganized | No naming convention | Use the pattern `[category]-[action].md` (e.g., `review-pr-diff.md`, `report-weekly.md`) |
| Sensitive context leaks into shared prompts | Someone included API keys or internal URLs in a prompt | Review prompts during PR review; add a note in the README: "Never include secrets in prompts" |
