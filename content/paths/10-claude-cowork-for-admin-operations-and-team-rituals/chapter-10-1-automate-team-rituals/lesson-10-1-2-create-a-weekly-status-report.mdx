# Create a weekly status report

## Objective

By the end of this lesson you will produce a structured weekly status report in Progress / Plans / Problems format by combining real git history with your own context. The report will be saved as a markdown file you can share with a manager, a team channel, or your own records.

## Walkthrough

### Step 1 — Set up a reports directory

<StepMeta purpose="Create a stable workspace where weekly report inputs and outputs can be stored together." action="Make the weekly-reports directory, move into it, and use it as the working folder for source files and dated status reports." expectedResult="You have a dedicated reporting workspace ready to hold shortlogs, diff stats, context notes, and the final report." whyItMatters="A repeatable reporting routine works better when the inputs and outputs live in one predictable place instead of scattered across ad hoc terminals and chat history." nextStep="Gather a week of commit activity from your repo so the Progress section starts from real evidence." askTutor="I'm on Step 1 of Create a weekly status report. Help me set up this reporting workspace so the weekly inputs and final reports stay organized." />

**macOS / Linux**

```bash
mkdir -p ~/weekly-reports
cd ~/weekly-reports
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\weekly-reports"
Set-Location "$HOME\weekly-reports"
```

### Step 2 — Gather the week's commit activity

<StepMeta purpose="Collect the main evidence for the Progress section from real repository history." action="Run git shortlog for the last seven days, save the results to week-shortlog.txt, and inspect the file before moving on." expectedResult="You have a readable shortlog summarizing the recent commit activity grouped by author for the reporting window." whyItMatters="The Progress section should come from observable work, and shortlog provides a compact summary that Claude can interpret without relying on memory." nextStep="Add diff statistics next so the report also has a rough sense of which parts of the codebase changed." askTutor="I'm on Step 2 of Create a weekly status report. Help me tune this git shortlog step so the report captures the right week of work." />

Navigate to your project repository and generate a shortlog for the past seven days.

**macOS / Linux**

```bash
cd ~/my-project
git shortlog --since="7 days ago" --no-merges -n > ~/weekly-reports/week-shortlog.txt
cat ~/weekly-reports/week-shortlog.txt
```

**Windows PowerShell**

```powershell
Set-Location "$HOME\my-project"
git shortlog --since="7 days ago" --no-merges -n | Out-File "$HOME\weekly-reports\week-shortlog.txt"
Get-Content "$HOME\weekly-reports\week-shortlog.txt"
```

`git shortlog` groups commits by author and counts them. The `--no-merges` flag strips merge commits that add noise.

> **Safety note:** `git shortlog` is read-only. It will not change any files or branches.

### Step 3 — Collect file-level change stats

<StepMeta purpose="Give Claude a second view of the week's work so the report can describe themes, not just commit counts." action="Generate diff stats for roughly the same reporting window and save them to week-diffstat.txt for later summarization." expectedResult="You have a file-level summary of the changes that complements the week-shortlog source." whyItMatters="Diff stats help Claude identify where the work happened, which often produces better grouped progress bullets than commit titles alone." nextStep="Write the Plans and Problems context yourself so Claude has explicit human-provided forward-looking information." askTutor="I'm on Step 3 of Create a weekly status report. Help me decide whether this diffstat window is a good match for the same week covered by the shortlog." />

This gives you a sense of which parts of the codebase moved.

**macOS / Linux**

```bash
git diff --stat HEAD~20 HEAD > ~/weekly-reports/week-diffstat.txt
cat ~/weekly-reports/week-diffstat.txt
```

**Windows PowerShell**

```powershell
git diff --stat HEAD~20 HEAD | Out-File "$HOME\weekly-reports\week-diffstat.txt"
Get-Content "$HOME\weekly-reports\week-diffstat.txt"
```

If `HEAD~20` goes further back than one week, adjust the number. The goal is to capture roughly one week of change.

### Step 4 — Write a brief Plans and Problems note by hand

<StepMeta purpose="Add the human context that git history cannot provide, especially future work and current blockers." action="Create context.txt with the real plans and problems you want in the report, replacing the sample items with your own notes." expectedResult="You have a small context file that explicitly states next steps and blockers in your own words." whyItMatters="Claude should not infer plans or problems from commit history, so this file gives the report a trustworthy source for the forward-looking sections." nextStep="Ask Claude to combine the git-derived evidence with your context.txt notes into the final weekly report." askTutor="I'm on Step 4 of Create a weekly status report. Help me write a context.txt that gives Claude enough real Plans and Problems without letting it guess the rest." />

Create a small text file with what you know that git does not:

**macOS / Linux**

```bash
cat > ~/weekly-reports/context.txt << 'EOF'
Plans for next week:
- Finish the API rate-limit middleware
- Write integration tests for the auth flow

Problems:
- Staging environment DNS is misconfigured, blocking QA
- Waiting on design review for the settings page
EOF
```

**Windows PowerShell**

```powershell
@"
Plans for next week:
- Finish the API rate-limit middleware
- Write integration tests for the auth flow

Problems:
- Staging environment DNS is misconfigured, blocking QA
- Waiting on design review for the settings page
"@ | Out-File "$HOME\weekly-reports\context.txt"
```

Replace the example lines with your real plans and blockers. This human-written context is critical — Claude cannot invent your future plans accurately.

### Step 5 — Ask Claude to compile the report

<StepMeta purpose="Synthesize the week's evidence and your own context into a report that is useful to teammates or managers." action="Prompt Claude to read week-shortlog.txt, week-diffstat.txt, and context.txt, then write the report in the exact Progress / Plans / Problems structure." expectedResult="Claude produces a concise weekly report whose Progress bullets come from repo history and whose Plans and Problems come from your context file." whyItMatters="The prompt structure is what keeps the report honest by tying each section to the correct source instead of letting Claude improvise missing details." nextStep="Save the report to a dated file and review every section against its source inputs before sharing it." askTutor="I'm on Step 5 of Create a weekly status report. Help me shape this prompt so Claude uses the git files for Progress and only context.txt for Plans and Problems." />

Open Claude Code in `~/weekly-reports` and prompt:

```
Read week-shortlog.txt, week-diffstat.txt, and context.txt.

Produce a weekly status report in this format:

## Weekly Status Report — [date range]

### Progress (what got done)
- (bullets grouped by theme, referencing real commits)

### Plans (what's next)
- (pulled from context.txt, not invented)

### Problems (blockers and risks)
- (pulled from context.txt, not invented)

Use plain English. Each bullet under 20 words. Do not invent plans or problems that are not in context.txt.
```

### Step 6 — Save and verify the report

<StepMeta purpose="Validate the final report before it leaves your workspace so it reflects real progress without invented risks or plans." action="Save the report using today's date, open it from the shell, and compare each section to week-shortlog.txt and context.txt." expectedResult="You confirm that every Progress bullet is traceable to git history and every Plan or Problem is traceable to context.txt." whyItMatters="Status reports are high-trust summaries, and this final audit prevents subtle fabrication from reaching a manager or team channel." nextStep="Carry the same verify-before-share habit into the runbook and knowledge-base lessons that document repeatable team processes." askTutor="I'm on Step 6 of Create a weekly status report. Help me check that this report only says what the source files actually support before I share it." />

Ask Claude to save the output:

```
Save that report to report-YYYY-MM-DD.md using today's date.
```

Then verify:

**macOS / Linux**

```bash
cat ~/weekly-reports/report-$(date +%Y-%m-%d).md
```

**Windows PowerShell**

```powershell
Get-Content "$HOME\weekly-reports\report-$((Get-Date).ToString('yyyy-MM-dd')).md"
```

Read every line. Confirm:
1. Each Progress bullet maps to real commits in `week-shortlog.txt`.
2. Plans and Problems match `context.txt` exactly — nothing added.
3. The date range is correct.

> **Real danger:** A status report that overstates progress or invents blockers damages trust with your team. Always verify the Progress section against actual commits and never let Claude fabricate Plans or Problems.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `git shortlog` is empty | No commits in the last 7 days, or author name mismatch | Run `git log --oneline -10` to check recent history; verify `git config user.name` |
| Progress section lists work you did not do | Claude inferred from file names or commit messages | Cross-check each bullet against `week-shortlog.txt` and delete anything unverifiable |
| Plans section contains items you never wrote | Claude invented likely next steps | Add to your prompt: "Only use Plans from context.txt — do not guess" |
| `HEAD~20` error on a new repo | Fewer than 20 commits exist | Use `git diff --stat $(git rev-list --max-parents=0 HEAD) HEAD` to diff from the first commit |
| Report tone is too corporate or jargon-heavy | Claude defaults to formal business writing | Add "Write like a teammate giving a casual update, not a consultant" to your prompt |
