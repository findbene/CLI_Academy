# Build a team knowledge base entry

## Objective

By the end of this lesson you will create a knowledge base article that documents a real team process — "How to set up the dev environment" — complete with verified commands, troubleshooting notes, and a new-hire-friendly tone. The article will live in your project's `docs/` directory under version control.

## Walkthrough

### Step 1 — Create the knowledge base directory

<StepMeta purpose="Create the permanent docs location where team process articles can live under version control." action="Make the knowledge-base directory in the project docs tree and move into it before drafting the article inputs." expectedResult="You have a docs/knowledge-base folder ready to hold a reusable setup article for the team." whyItMatters="A knowledge base only helps if its articles live somewhere consistent and discoverable for new hires and existing teammates." nextStep="Write down the raw setup steps from your own experience so Claude has a truthful source to expand." askTutor="I'm on Step 1 of Build a team knowledge base entry. Help me place this article in the right docs location so new teammates will actually find it." />

**macOS / Linux**

```bash
mkdir -p ~/my-project/docs/knowledge-base
cd ~/my-project/docs/knowledge-base
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\my-project\docs\knowledge-base"
Set-Location "$HOME\my-project\docs\knowledge-base"
```

### Step 2 — Capture the raw setup steps from your own experience

<StepMeta purpose="Create the source-of-truth notes that describe how the project is really set up today." action="Write raw-setup-notes.txt from your own lived setup experience, replacing the sample commands with the actual steps for your project." expectedResult="You have a raw notes file containing the real clone, install, env, migration, and startup steps the article should document." whyItMatters="Claude can polish and structure a process, but the human operator needs to supply the accurate procedural truth first." nextStep="Ask Claude to turn those raw notes into a polished article with prerequisites, numbered steps, verification, and troubleshooting." askTutor="I'm on Step 2 of Build a team knowledge base entry. Help me turn my actual setup experience into raw notes that are accurate enough for Claude to expand safely." />

Before involving Claude, write down the steps you actually follow when setting up this project from scratch. This is your source of truth.

**macOS / Linux**

```bash
cat > raw-setup-notes.txt << 'EOF'
1. Clone the repo: git clone git@github.com:team/my-project.git
2. cd my-project
3. Install Node 20 (I use nvm: nvm install 20)
4. npm install
5. Copy .env.example to .env and fill in the DATABASE_URL
6. Run migrations: npm run db:migrate
7. Start dev server: npm run dev
8. Open http://localhost:3000 and check the home page loads
EOF
```

**Windows PowerShell**

```powershell
@"
1. Clone the repo: git clone git@github.com:team/my-project.git
2. cd my-project
3. Install Node 20 (I use nvm-windows: nvm install 20)
4. npm install
5. Copy .env.example to .env and fill in the DATABASE_URL
6. Run migrations: npm run db:migrate
7. Start dev server: npm run dev
8. Open http://localhost:3000 and check the home page loads
"@ | Out-File "raw-setup-notes.txt"
```

Replace these with your actual steps. Accuracy here is more important than polish.

### Step 3 — Ask Claude to expand the notes into a knowledge base article

<StepMeta purpose="Transform the raw setup notes into a beginner-friendly article without losing command accuracy." action="Open Claude in the knowledge-base directory and ask it to create dev-environment-setup.md with prerequisites, numbered steps, verification details, troubleshooting, and platform-specific commands." expectedResult="Claude produces a structured setup article aimed at someone who has never seen the project before." whyItMatters="The article becomes genuinely useful only when the steps are both beginner-readable and still anchored in the raw notes you supplied." nextStep="Open a fresh terminal and test the article's critical commands as if you were a new hire following it for the first time." askTutor="I'm on Step 3 of Build a team knowledge base entry. Help me phrase this prompt so Claude expands my notes into a clear article without inventing extra setup steps." />

Open Claude Code in the knowledge-base directory:

```
Read raw-setup-notes.txt. Turn it into a polished knowledge base article called "Dev Environment Setup" with:

1. A "Prerequisites" section listing required tools and versions.
2. Each step as a numbered section with the exact command, what it does, and how to verify it worked.
3. A "Troubleshooting" section at the bottom with common failure modes.
4. Include both macOS/Linux and Windows PowerShell commands where they differ.
5. Write for someone who has never seen this project before.
6. Save the result as dev-environment-setup.md.
```

### Step 4 — Verify every command in the article

<StepMeta purpose="Test whether the generated article actually works as onboarding documentation instead of just reading well." action="Run the critical commands from the article in a fresh terminal and compare the results to what the article promises." expectedResult="You know which setup instructions and verification steps are accurate and which ones need correction before the article is shared." whyItMatters="New-hire docs are high leverage, and every broken command compounds confusion and trust loss on day one." nextStep="Once the article is accurate, add verification metadata at the top so future readers know when and where it was last tested." askTutor="I'm on Step 4 of Build a team knowledge base entry. Help me test this article like a new hire and identify which commands or assumptions still need correction." />

This is non-negotiable. Open a fresh terminal and follow the article step by step as if you were a new hire.

**macOS / Linux**

```bash
# Spot-check critical commands from the generated article:
node --version          # Should show v20.x
npm --version           # Should show 10.x or similar
cat .env.example        # Should exist in the repo
```

**Windows PowerShell**

```powershell
# Spot-check critical commands from the generated article:
node --version           # Should show v20.x
npm --version            # Should show 10.x or similar
Get-Content .env.example # Should exist in the repo
```

If any command fails or produces unexpected output, update the article. Do not commit a knowledge base entry with untested instructions.

> **Real danger:** A new team member will follow this document on their first day. If Step 4 fails silently and Step 5 throws a cryptic error, they will lose hours and trust the docs less going forward. Every command must work.

### Step 5 — Add a metadata header to the article

<StepMeta purpose="Mark the article with verification context so future readers know whether to trust it as-is or re-test it." action="Edit dev-environment-setup.md so it starts with the title plus Last verified, Verified on, and Author metadata." expectedResult="The final article clearly states when it was last tested and on which operating systems." whyItMatters="Operational docs decay quickly, and visible verification metadata is a simple signal that helps teams keep them fresh." nextStep="Stage the article from the project root and commit it so the knowledge base entry becomes part of the repo." askTutor="I'm on Step 5 of Build a team knowledge base entry. Help me choose the right metadata header so future readers know how fresh and trustworthy this article is." />

Open the generated `dev-environment-setup.md` and ensure it starts with a clear header:

```markdown
# Dev Environment Setup

**Last verified:** 2025-04-09
**Verified on:** macOS 14.3 / Windows 11 23H2
**Author:** [Your name]

---
```

The "Last verified" date tells future readers whether to trust the instructions or re-test them.

### Step 6 — Commit to version control

<StepMeta purpose="Turn the verified article into a shared team artifact by adding it to version control." action="Stage docs/knowledge-base/dev-environment-setup.md from the project root and commit the knowledge base entry." expectedResult="You create a repo commit containing a tested, metadata-stamped setup article for the team." whyItMatters="Once the article is committed, it becomes discoverable, reviewable, and maintainable as part of the team's operating documentation." nextStep="Apply the same reuse-first mindset in the next lesson by building a shared prompts library the whole team can start from." askTutor="I'm on Step 6 of Build a team knowledge base entry. Help me do a final review so I only commit setup instructions that are tested and newcomer-friendly." />

**macOS / Linux**

```bash
cd ~/my-project
git add docs/knowledge-base/dev-environment-setup.md
git commit -m "docs: add dev environment setup knowledge base entry"
```

**Windows PowerShell**

```powershell
Set-Location "$HOME\my-project"
git add docs/knowledge-base/dev-environment-setup.md
git commit -m "docs: add dev environment setup knowledge base entry"
```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude adds steps you did not write (e.g., "Install Docker") | Claude assumed a common dev setup | Remove any step not in your raw notes; only add steps you have personally tested |
| Commands work on your machine but fail for a teammate | Missing a machine-specific dependency or env var | Have a second person follow the doc on a clean machine and note failures |
| `.env.example` is missing from the repo | Someone never committed it | Create one with placeholder values and add it to the article as Step 0 |
| Article is too long and overwhelming | Claude over-explained each step | Ask Claude to "trim each step to command + one-sentence explanation + verify" |
| Knowledge base entry goes stale after a dependency upgrade | No review cadence | Add a calendar reminder to re-verify the article quarterly; update the "Last verified" date |
