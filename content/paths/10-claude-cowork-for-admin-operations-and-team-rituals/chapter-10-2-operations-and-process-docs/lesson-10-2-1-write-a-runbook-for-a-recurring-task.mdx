# Write a runbook for a recurring task

## Objective

By the end of this lesson you will have a complete deployment runbook — a step-by-step document that anyone on your team can follow to deploy your application safely. Each step includes the exact command to run and a verification command to confirm it worked before moving on.

## Walkthrough

### Step 1 — Create a runbooks directory in your project

<StepMeta purpose="Create the permanent home where operational runbooks for the project will live." action="Make the runbooks directory inside your docs tree and move into it before creating the deployment runbook." expectedResult="You have a docs/runbooks location ready to hold versioned operational procedures." whyItMatters="Runbooks are only useful when they are easy to find during normal work and during incidents, so placing them in a stable repo location matters." nextStep="Draft a human-written skeleton with the core deploy flow before asking Claude to improve it." askTutor="I'm on Step 1 of Write a runbook for a recurring task. Help me place this runbook in the right repo location so teammates will actually find it." />

**macOS / Linux**

```bash
mkdir -p ~/my-project/docs/runbooks
cd ~/my-project/docs/runbooks
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\my-project\docs\runbooks"
Set-Location "$HOME\my-project\docs\runbooks"
```

### Step 2 — Draft the runbook skeleton by hand

<StepMeta purpose="Establish the deploy flow and rollback guardrails yourself before Claude starts polishing wording and structure." action="Write deploy-runbook.md with the pre-deploy checklist, main steps, rollback procedure, and post-deploy checks as a first draft." expectedResult="You have a rough but concrete runbook skeleton that already contains the real commands and verification checkpoints you expect to use." whyItMatters="Humans should define the operational truth first, because Claude is better at refining a real procedure than inventing one safely from scratch." nextStep="Ask Claude to improve the structure and completeness of the runbook without removing the rollback safeguards." askTutor="I'm on Step 2 of Write a runbook for a recurring task. Help me make this first draft specific enough that Claude improves it instead of inventing a different deployment process." />

Start with a known structure so Claude has clear guardrails. Create the file:

**macOS / Linux**

```bash
cat > deploy-runbook.md << 'EOF'
# Deployment Runbook

## Pre-deploy checklist
- [ ] All tests pass on main
- [ ] No unmerged hotfixes
- [ ] Database migrations reviewed

## Steps

### Deployment Example 1 — Pull latest main
**Command:**
```
git checkout main && git pull origin main
```
**Verify:**
```
git log --oneline -3
```
**Expected:** You see the latest merged PR at the top.

### Deployment Example 2 — Run tests
**Command:**
```
npm test
```
**Verify:**
```
echo $?
```
**Expected:** Exit code 0.

### Deployment Example 3 — Build the application
**Command:**
```
npm run build
```
**Verify:**
```
ls -la dist/ | head -5
```
**Expected:** dist/ directory contains fresh build artifacts.

### Deployment Example 4 — Deploy
**Command:**
```
npm run deploy
```
**Verify:**
```
curl -s -o /dev/null -w "%{http_code}" https://your-app.example.com/health
```
**Expected:** HTTP 200.

## Rollback procedure
If any step fails:
1. Do NOT continue to the next step.
2. Run: `git revert HEAD && npm run deploy`
3. Notify the team channel.

## Post-deploy
- [ ] Smoke test critical user paths manually
- [ ] Monitor error rate for 15 minutes
EOF
```

### Step 3 — Ask Claude to review and improve the runbook

<StepMeta purpose="Use Claude to strengthen the draft while keeping the human-defined deploy and rollback logic intact." action="Open Claude in the runbooks directory and ask it to add prerequisites, timing, contacts, and consistent Command/Verify/Expected structure without deleting the rollback section." expectedResult="Claude produces an improved runbook that is clearer, more complete, and better structured than the first draft." whyItMatters="Claude adds value here by improving usability and completeness, but the prompt boundaries keep it from stripping out the safety-critical rollback guidance." nextStep="Run the verification commands in the actual project so the runbook is tested, not merely polished." askTutor="I'm on Step 3 of Write a runbook for a recurring task. Open Claude Code in the runbooks directory:

```
Read deploy-runbook.md. Improve it:
1. Add a "Prerequisites" section listing required tools and access.
2. Add estimated time for each step.
3. Add a "Who to call" section with placeholder roles (on-call engineer, team lead).
4. Make sure every Step has a Command, Verify, and Expected subsection.
5. Do not remove the Rollback section.
```

### Step 4 — Test every verification command

<StepMeta purpose="Prove the runbook works in reality before anyone relies on it during a deploy or incident." action="Execute each Verify command in the actual project directory and compare the real output to the Expected text in the runbook." expectedResult="You know whether every verification step is correct or which parts of the runbook still need correction." whyItMatters="Untested runbooks create false confidence, and this test pass is what turns the document from plausible documentation into an operational tool." nextStep="Once every command has been tested and corrected, commit the runbook into version control for the team to use." askTutor="I'm on Step 4 of Write a runbook for a recurring task. Help me test these verification commands and decide what to change if the documented expectations do not match reality." />

This is the most important step. Run each **Verify** command from the runbook in your actual project directory and confirm it produces the **Expected** result.

**macOS / Linux**

```bash
cd ~/my-project
git log --oneline -3        # Step 1 verify
npm test                    # Step 2 command
echo $?                     # Step 2 verify
```

**Windows PowerShell**

```powershell
Set-Location "$HOME\my-project"
git log --oneline -3          # Step 1 verify
npm test                      # Step 2 command
$LASTEXITCODE                 # Step 2 verify
```

If any verify command does not match the expected result, update the runbook. A runbook with untested steps is worse than no runbook — someone will follow it during an outage and get stuck.

> **Real danger:** An untested runbook creates false confidence. The person following it during an incident will discover the broken step at the worst possible moment. Test every command before committing.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Verify command passes in dev but fails in CI | Environment differences (different Node version, missing env vars) | Add a Prerequisites section that pins exact tool versions and required env vars |
| Runbook mentions commands that do not exist in this project | Claude assumed a standard toolchain (e.g., `npm run deploy` when you use a different deployer) | Replace every command with the actual command you use, then re-test |
| Rollback section is vague ("undo changes") | Claude generated a generic rollback | Write the exact rollback commands yourself and test them on a branch |
| Teammate follows the runbook and gets stuck at Step 3 | A step assumes context you have but they do not (e.g., VPN access) | Have someone unfamiliar with the process follow the runbook and note where they get confused |
| Runbook drifts out of date after infra changes | No review process | Add a "Last verified" date at the top and schedule a quarterly review |
