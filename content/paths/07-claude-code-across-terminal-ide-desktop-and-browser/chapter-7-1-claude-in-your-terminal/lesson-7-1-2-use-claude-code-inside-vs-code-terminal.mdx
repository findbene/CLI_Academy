# Use Claude Code inside VS Code terminal

## Objective

By the end of this lesson you will launch Claude Code from inside the VS Code integrated terminal, prove it can read and edit files in the open workspace, and understand the practical differences between using Claude Code here versus in a standalone terminal window.

## Walkthrough

### Step 1 — Open the VS Code integrated terminal

<StepMeta purpose="Start from the terminal that already shares context with the open editor workspace." action="Open the VS Code integrated terminal and confirm its working directory matches the folder open in the Explorer sidebar." expectedResult="The terminal opens at the workspace root and the printed path matches the project shown in VS Code." whyItMatters="Shared working directory is what makes Claude, shell commands, and editor navigation line up without extra path juggling." nextStep="Launch Claude Code inside this terminal so it operates directly inside the open workspace." askTutor="I'm on Step 1 of Use Claude Code inside VS Code terminal. Help me confirm that the integrated terminal is really starting in the correct workspace folder." />

Launch VS Code and open a project folder. Then open the terminal:

**Keyboard shortcut (all platforms):**

```
Ctrl + `   (backtick — the key above Tab)
```

Or use the menu: **Terminal → New Terminal**.

The terminal panel opens at the bottom of VS Code. Its working directory is automatically set to the root of the open workspace. You do not need to `cd` anywhere — you are already in the right place.

Confirm the path:

**macOS / Linux:**

```bash
pwd
```

**Windows PowerShell:**

```powershell
Get-Location
```

The output should match the folder shown in the VS Code Explorer sidebar.

### Step 2 — Start Claude Code inside the integrated terminal

<StepMeta purpose="Run Claude Code inside VS Code so the terminal prompt and the editor stay in the same workspace." action="Start Claude or your alias from the integrated terminal and wait for the interactive Claude prompt to appear." expectedResult="Claude Code opens successfully inside the terminal panel and is ready to accept requests." whyItMatters="This setup combines terminal-native Claude control with immediate visual feedback from the editor in the same window." nextStep="Ask Claude to read an existing project file so you confirm workspace file access is working." askTutor="I'm on Step 2 of Use Claude Code inside VS Code terminal. Help me start Claude successfully inside the integrated terminal and explain what should appear when it is ready." />

Run Claude Code exactly as you would in any terminal:

```bash
claude
```

Or if you set up the alias from Lesson 7.1.1:

```bash
cc
```

Claude Code starts and its interactive prompt appears right inside VS Code. You can type natural-language requests just like in a standalone terminal.

> **Key insight:** Claude Code running here can see every file that VS Code has open. It reads from and writes to the same workspace. This means Claude Code edits and VS Code's editor stay in sync — when Claude writes a file, VS Code immediately shows the change.

### Step 3 — Ask Claude Code to read a file in the workspace

<StepMeta purpose="Verify that Claude can inspect real files from the current workspace without manual copy-paste." action="Prompt Claude to read a known file such as package.json or README.md and summarize a specific detail from it." expectedResult="Claude replies using the contents of a real workspace file that it opened from disk." whyItMatters="Reliable file reading is the baseline capability that makes Claude useful for real project work inside VS Code." nextStep="Create a disposable file and ask Claude to edit it so you can watch the editor update live." askTutor="I'm on Step 3 of Use Claude Code inside VS Code terminal. Help me choose a safe file to test with and confirm Claude is reading the right one." />

Try a simple request to prove it has access:

```
> Read package.json and tell me the project name and version.
```

Claude Code will open `package.json` from the workspace root, parse it, and respond. No file paths needed — it defaults to the workspace directory.

If your project does not have a `package.json`, pick any file visible in the sidebar:

```
> Read README.md and summarize it in one sentence.
```

### Step 4 — Ask Claude Code to edit a file and watch VS Code update

<StepMeta purpose="See the live-edit loop that makes the integrated terminal especially useful during coding sessions." action="Create claude-test.txt, ask Claude to add the second line, and then open the file in the editor to confirm the change." expectedResult="The file contains both lines and VS Code shows Claude's edit immediately in the workspace." whyItMatters="Instant visibility of Claude's file writes is the main ergonomic advantage over using a separate terminal window." nextStep="Keep Claude running while you use surrounding VS Code tools like Source Control and Problems." askTutor="I'm on Step 4 of Use Claude Code inside VS Code terminal. Help me verify whether Claude really edited the file on disk or only described the change in chat." />

Create a tiny test file so this step is risk-free:

**macOS / Linux:**

```bash
echo "Hello from the test file" > claude-test.txt
```

**Windows PowerShell:**

```powershell
"Hello from the test file" | Out-File -Encoding utf8 claude-test.txt
```

Now ask Claude Code to modify it:

```
> In claude-test.txt, add a second line that says "Edited by Claude Code inside VS Code."
```

After Claude Code confirms the edit:

1. Click on `claude-test.txt` in the VS Code Explorer sidebar.
2. You should see both lines in the editor — the original and the new one Claude added.

This instant sync is the main advantage of running Claude Code inside VS Code: you see edits the moment they happen, without switching windows.

### Step 5 — Use VS Code features alongside Claude Code

<StepMeta purpose="Practice combining Claude with VS Code features instead of treating them as separate workflows." action="Try at least one supporting feature such as split terminal, Problems, Source Control, or Go to Definition while Claude stays open." expectedResult="You can use editor diagnostics, Git UI, or navigation tools alongside Claude without leaving the window." whyItMatters="The real productivity gain comes from pairing Claude with VS Code's built-in feedback loops, not just from running Claude in a smaller terminal." nextStep="Compare this integrated workflow against a standalone terminal so you know when each surface is better." askTutor="I'm on Step 5 of Use Claude Code inside VS Code terminal. Help me understand which VS Code features pair best with Claude during a normal coding session." />

While Claude Code is running in the terminal panel, you still have full access to VS Code features:

- **Split the terminal:** Click the split icon in the terminal toolbar (or press `Ctrl+Shift+5`) to open a second terminal side by side. Run normal shell commands in one pane while Claude Code runs in the other.
- **Jump to problems:** If Claude Code changes code that introduces an error, VS Code's Problems panel (`Ctrl+Shift+M`) flags it immediately.
- **Source control:** The Source Control panel (Ctrl+Shift+G) shows changed files in real time. You can stage and commit Claude Code's edits without leaving VS Code.
- **Peek definitions:** Click on a function name in the editor while Claude Code is explaining it — use `F12` to jump to the definition.

### Step 6 — Compare: standalone terminal vs VS Code terminal

<StepMeta purpose="Turn the comparison into a concrete decision rule instead of a vague preference." action="Read the comparison table and decide which environment you would choose for terminal-heavy work versus editor-heavy work." expectedResult="You can name at least one situation where the integrated terminal is better and one where a standalone terminal is better." whyItMatters="Choosing the right surface deliberately helps you avoid cramped layouts when you need full terminal space and avoid unnecessary window switching when you need editor feedback." nextStep="Clean up the temporary exercise file so the workspace stays tidy." askTutor="I'm on Step 6 of Use Claude Code inside VS Code terminal. Help me build a simple rule for when to use the integrated terminal versus a standalone terminal." />

Here is a practical comparison:

| Feature | Standalone terminal | VS Code integrated terminal |
|---|---|---|
| Claude Code access to files | Same directory and below | Same directory and below |
| See file edits live | Must open files manually | Editor updates instantly |
| Run other shell commands | Open a new tab | Split the terminal panel |
| Git status at a glance | Use `git status` | Source Control sidebar |
| Code error detection | None | Problems panel highlights errors |
| Window switching | Alt-Tab between windows | Everything in one window |
| Terminal size | Full screen available | Shares screen with editor |

The standalone terminal wins when you need maximum terminal space (long logs, large diffs). The VS Code terminal wins when you are actively editing code alongside Claude Code.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `claude: command not found` in VS Code terminal | Claude Code is not on PATH, or VS Code loaded an older PATH | Close and reopen VS Code so it picks up PATH changes; or open a new terminal with `Terminal → New Terminal` |
| Claude Code edits are not visible in the editor | The file tab was already open with unsaved changes | Close the tab without saving and reopen the file |
| Terminal shows the wrong directory | You opened VS Code without a folder | Use **File → Open Folder** to open your project, then open a new terminal |
| Split terminal is too small to read | Default split gives 50/50 space | Drag the divider or use `Ctrl+`` to maximize the terminal panel |
| PowerShell execution policy blocks profile | Policy is set to Restricted | Run `Set-ExecutionPolicy RemoteSigned -Scope CurrentUser` once |
| `cc` alias not recognized | Profile was not loaded in this VS Code terminal instance | Run `. $PROFILE` (PowerShell) or `source ~/.bashrc` (Bash) |
