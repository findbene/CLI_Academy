# Build: shared weekly brief or intake assistant

## Objective
Build a team-shared assistant that creates either:
- a weekly engineering brief from multiple contributors' git logs, or
- an intake assistant that reads `requests.md` and categorizes incoming work.

## Walkthrough
### Step 1: Create shared inputs

<StepMeta purpose="Create the bounded shared inputs that a team-facing assistant can consume without guessing at who or what it should process." action="Create `team/members.txt` and `requests.md` with the sample data so the assistant has explicit team names or intake items to operate on." expectedResult="The workspace now contains the source files for either the weekly brief mode or the intake-triage mode." whyItMatters="Team assistants become untrustworthy quickly when they infer inputs from broad repo scans instead of reading agreed shared files." nextStep="Define the team-safe contract in `CLAUDE.md` next so the assistant has one clear output path and bounded permissions." askTutor="Review these shared inputs and tell me whether they are explicit enough for a team assistant or whether I still need more source structure before automation is safe." />

```bash
mkdir -p team
cat > team/members.txt <<'EOF'
alice
bob
EOF
cat > requests.md <<'EOF'
- Need onboarding checklist for new contractors
- API timeout bug reported by support
- Request for weekly KPI dashboard export
EOF
```

```powershell
New-Item -ItemType Directory -Path team -Force | Out-Null
@"
alice
bob
"@ | Set-Content team/members.txt
@"
- Need onboarding checklist for new contractors
- API timeout bug reported by support
- Request for weekly KPI dashboard export
"@ | Set-Content requests.md
```

### Step 2: Define team-safe contract in CLAUDE.md

<StepMeta purpose="Turn the team workflow into a narrow operating contract with one output file and explicit safety rules." action="Write `CLAUDE.md` for either the weekly brief or intake assistant, preserving the read-only model except for one output artifact and stop-on-missing-input behavior." expectedResult="You have a team-helper contract that is understandable, mode-specific, and bounded to one generated file." whyItMatters="Shared automation fails fast when nobody can tell what it is allowed to read, what it is allowed to write, or who owns the result." nextStep="Run one assistant mode next so you can see whether the contract produces a useful artifact another teammate could actually consume." askTutor="Audit this team-helper contract and tell me whether it creates enough safety and ownership clarity for shared use." />

```md
# Team Helper Contract

## Option A: Weekly brief
- Read team/members.txt
- For each member, gather last-week git highlights from agreed repos
- Produce team-weekly-brief-YYYY-MM-DD.md

## Option B: Intake assistant
- Read requests.md
- Categorize each item as Bug, Ops, Content, or Feature
- Produce intake-triage-YYYY-MM-DD.md

## Safety
- Read-only access except writing one output file
- No deletions, no commits, no pushes
- If source files missing, stop and report
```

### Step 3: Run one assistant mode

<StepMeta purpose="Exercise one concrete team workflow so the contract produces a real artifact instead of remaining a design sketch." action="Use either the weekly-brief or intake prompt and generate the corresponding dated markdown output from the shared input files." expectedResult="The assistant creates a single team artifact aligned with the chosen mode." whyItMatters="Shared assistants only become useful when their output is readable and operationally relevant to someone besides the person who built them." nextStep="Validate the generated artifact next by opening it and checking that teammates could understand and use it without extra context." askTutor="Help me evaluate this team-helper run and tell me whether the generated artifact is specific enough for teammates to trust and use." />
Weekly brief prompt:

```text
Read CLAUDE.md and run Option A. Create one combined weekly brief with per-member highlights and a short risks section.
```

Intake prompt:

```text
Read CLAUDE.md and run Option B. Categorize each request and output a triage table with owner suggestions.
```

### Step 4: Validate team artifact

<StepMeta purpose="Check the generated output from the perspective of another teammate who did not run the assistant." action="List the output file, open it, and verify that the structure, headings, and content make sense without hidden run context." expectedResult="You confirm the assistant produced a team-usable weekly brief or intake-triage file." whyItMatters="A team assistant is only valuable when its artifact stands on its own as a readable, actionable document." nextStep="Add owner and cadence metadata next so the automation has clear accountability and an operational rhythm." askTutor="Review this team artifact with me and tell me whether a teammate could use it immediately or whether the format still assumes too much background knowledge." />

```bash
ls *-YYYY-*.md 2>/dev/null || ls team-weekly-brief-*.md intake-triage-*.md
```

```powershell
Get-ChildItem team-weekly-brief-*.md, intake-triage-*.md -ErrorAction SilentlyContinue
```

Open the generated file and confirm it is understandable by teammates who did not run the assistant.

### Step 5: Add owner and cadence metadata

<StepMeta purpose="Add the ownership and scheduling metadata that keeps shared automation from becoming a nobody-owns-this artifact." action="Insert owner, run date, next scheduled run, and escalation contact at the top of the generated file." expectedResult="The output file now names who owns it, when it ran, when it should run again, and who to contact if it fails." whyItMatters="Clear ownership is one of the simplest ways to keep team assistants sustainable once they move beyond personal experiments." nextStep="Run a short pilot in the next lesson so feedback from real users can improve the contract instead of leaving it fixed by assumption." askTutor="Check this metadata block and tell me whether it gives the team enough operational clarity to trust and adopt the assistant." />
At the top of the output file, include:
- Owner
- Run date
- Next scheduled run
- Escalation contact

This prevents "shared automation with no owner," which fails quickly in real teams.

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| Team artifact has inconsistent format each run | No output schema in contract | Define fixed sections and required headings |
| Intake categories are random | Categories not constrained | Limit categories to Bug/Ops/Content/Feature |
| Assistant fails for one missing source file | Missing stop/continue policy | Define which missing inputs stop the run vs skip with warning |
| Team distrusts output | No owner or traceability | Add owner, date, and source references in artifact |
| Permission risk in shared repo | Assistant can write broadly | Restrict to one output filename pattern |
