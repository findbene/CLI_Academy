# Add evidence, logs, and stop rules

## Objective
By the end of this lesson, you will have created an "evidence and stop-rule pack" for your helper. This pack will tell your helper what clues to look for ("evidence") to confirm progress or success, and when to safely stop its actions ("stop rules"), making your automated tasks more reliable and predictable.

## Walkthrough

It's super important to build safety into your automated helpers right from the start! Evidence and stop rules are like giving your helper a checklist and an emergency brake.

1.  **Create your rule pack file:**
    We'll start by creating a new file to hold our rules. Let's call it `helper_rules.yaml`. You can use the `touch` command, which safely creates an empty file without altering anything else on your system.
    ```bash
    touch helper_rules.yaml
    ```
    This command is safe because it simply creates an empty file in your current directory; it doesn't run any programs or make system-wide changes.

2.  **Add evidence rules:**
    Now, let's open `helper_rules.yaml` using your favorite text editor (like `nano` or VS Code). Inside, we'll define some "evidence" your helper should look for. Evidence could be a specific log message, a file appearing, or even the absence of an error.
    ```yaml
    # helper_rules.yaml
    evidence:
      - name: "Deployment Success Log"
        type: "log_pattern"
        pattern: "Deployment complete successfully"
        path: "/var/log/my_app/deploy.log"
      - name: "Configuration File Created"
        type: "file_exists"
        path: "/etc/my_app/config.conf"
    ```
    You're just describing patterns and file paths here, not executing anything. This is a safe step, defining what "success" or "expected behavior" looks like *before* any action is taken.

3.  **Add stop rules:**
    Below your `evidence` section in the same `helper_rules.yaml` file, let's add some `stop_rules`. These rules are crucial for preventing unwanted actions.
    ```yaml
    # helper_rules.yaml (continued)
    stop_rules:
      - name: "Critical Error Logged"
        condition: "log_pattern_match"
        pattern: "ERROR: Critical failure"
        path: "/var/log/my_app/deploy.log"
        action: "stop_immediately"
      - name: "Disk Space Low"
        condition: "system_metric"
        metric: "disk_usage_percent"
        threshold: 90 # Stop if disk usage is 90% or higher
        action: "stop_and_notify"
    ```
    These rules are your helper's safety net. They tell it, "If you see X, stop immediately!" This makes your helper *safer* to use by preventing it from continuing under unsafe conditions.

4.  **Save your `helper_rules.yaml` file.**

You've now successfully created an evidence and stop-rule pack! This file is your blueprint for a more controlled and safer helper.
