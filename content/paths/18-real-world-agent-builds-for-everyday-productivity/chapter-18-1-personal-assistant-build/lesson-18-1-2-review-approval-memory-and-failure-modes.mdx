# Review: approval, memory, and failure modes

## Objective
Harden your daily brief assistant with human approval gates, brief memory, explicit failure handling, and stop conditions so the workflow is useful and safe to run every morning.

## Walkthrough
### Step 1: Add review gate and memory policy to CLAUDE.md

<StepMeta purpose="Upgrade the first assistant from a one-shot generator into a safer workflow with explicit human checkpoints and bounded memory." action="Edit `CLAUDE.md` to add the review gate, seven-entry memory limit, and clear failure-handling rules exactly as shown." expectedResult="The contract now requires approval before follow-up actions, records brief run memory, and defines safe behavior for missing inputs or empty git history." whyItMatters="Daily-use assistants become risky when they keep acting after drafting output or accumulate uncontrolled memory over time." nextStep="Create the supporting memory and failure-playbook files next so the new rules have real artifacts behind them." askTutor="Review my hardened `CLAUDE.md` and tell me whether the approval gate, memory retention, and failure handling are explicit enough for recurring use." />
Open your existing `CLAUDE.md` and add this section:

```md
## Review and approval gate
- Generate the brief draft first.
- Ask for explicit human approval before any follow-up action.
- If no approval is provided, stop.

## Memory policy
- Append a one-line summary of each run to brief-memory.md.
- Keep only the latest 7 entries.

## Failure handling
- If TODO.md missing: report and stop.
- If git log command errors: report stderr and stop.
- If git log is empty: still create brief with "No commits yesterday".
```

### Step 2: Create a memory file and a failure-playbook file

<StepMeta purpose="Back the contract up with concrete files that hold lightweight run memory and deterministic recovery guidance." action="Create `brief-memory.md` and `failure-playbook.md`, then populate the playbook with the three recovery cases from the lesson." expectedResult="The assistant workspace now includes persistent run memory plus a small runbook for predictable failure recovery." whyItMatters="Operational safety improves when error handling lives in an explicit playbook instead of relying on improvisation during failures." nextStep="Run the assistant with approval behavior next so you can test whether the updated contract really changes the interaction pattern." askTutor="Check these support files and tell me whether they are practical for real daily use or still too vague to help during a failure." />

```bash
touch brief-memory.md
cat > failure-playbook.md <<'EOF'
# Failure Playbook

- Missing TODO.md -> create TODO.md from template, rerun.
- git log error -> verify you are in a git repo and rerun.
- empty git log -> continue with explicit "No commits yesterday" note.
EOF
```

```powershell
New-Item brief-memory.md -ItemType File -Force | Out-Null
@"
# Failure Playbook

- Missing TODO.md -> create TODO.md from template, rerun.
- git log error -> verify you are in a git repo and rerun.
- empty git log -> continue with explicit "No commits yesterday" note.
"@ | Set-Content failure-playbook.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| Assistant keeps acting without approval | Missing explicit review gate text | Add "ask for approval, then stop" language in CLAUDE.md |
| Memory file grows indefinitely | No retention rule | Enforce "keep latest 7 entries" in contract |
| Agent retries forever on failure | No stop condition on repeated errors | Add max attempts (for example, 2 retries) then stop |
| Sensitive data ends up in memory | Logging raw terminal output | Store only concise, non-secret run summaries |
| Unexpected cost increase | Re-running broad prompts during failures | Keep prompts narrow and failure handling deterministic |
