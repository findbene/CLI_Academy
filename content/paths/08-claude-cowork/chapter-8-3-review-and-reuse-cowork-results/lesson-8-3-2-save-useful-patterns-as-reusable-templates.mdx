# Save useful patterns as reusable templates

## Objective

By the end of this lesson you will have a `cowork-patterns/` folder containing reusable templates extracted from your successful Cowork sessions. These templates capture what worked — the prompt structures, the brief format, and the review checklists — so you never have to reinvent them.

## Walkthrough

### Step 1 — Identify what is worth templating

<StepMeta purpose="Filter for reusable session assets instead of saving every interesting sentence as a template." action="Review the session and pick only the structures that would help on a similar future task, such as a brief format, prompt flow, or review checklist." expectedResult="You can name which session artifacts are worth templating and which ones were too specific to reuse." whyItMatters="Reusable patterns save time only when they are general enough to apply again; otherwise the patterns folder turns into archived clutter." nextStep="Create a dedicated patterns folder so the reusable pieces have one obvious home." askTutor="I'm on Step 1 of Save useful patterns as reusable templates. Help me decide which parts of this Cowork session are actually worth saving as reusable patterns." />

Not every Cowork session produces a reusable pattern. Ask yourself:

- **Will I do this kind of task again?** If yes, template it.
- **Did the prompt structure lead to a good first draft?** If yes, save the prompt.
- **Did I develop a review checklist during the session?** If yes, extract it.

From the previous lessons, you already have two reusable artifacts:
1. The **session brief format** (Goal, Context, Success criteria, Boundaries).
2. The **accuracy checklist** (factual, complete, boundary-compliant, tone, platform coverage).

<StepMeta purpose="Give reusable Cowork assets a stable location instead of scattering them across practice folders." action="Create a cowork-patterns directory in the project using the command for your operating system." expectedResult="A dedicated cowork-patterns folder exists and is ready to hold generalized templates and checklists." whyItMatters="One concern, one home: reusable Cowork assets are easier to find and maintain when they live in a single predictable directory." nextStep="Populate the folder with the first generalized template: the session brief structure." askTutor="I'm on Step 2 of Save useful patterns as reusable templates. ### Step 4 — Save a prompt pattern

<StepMeta purpose="Preserve the exact prompt sequence that created a clean outline-first Cowork workflow." action="Write the README outline prompt pattern into its own markdown file, including when to use it, the prompts themselves, and why they work." expectedResult="cowork-patterns contains a prompt pattern file that another session could reuse with minimal changes." whyItMatters="Good prompt flows are reusable process assets, and documenting the sequence prevents you from rebuilding a successful opening from memory later." nextStep="Extract the post-session review checklist so successful sessions also end with the same discipline they started with." askTutor="I'm on Step 4 of Save useful patterns as reusable templates. Help me document this prompt pattern in a way that makes it reusable instead of tied only to the lintr example." />

A prompt pattern is the exact wording you used to get a good result. Save the opening prompt from Lesson 8.2.1 as a reusable pattern:

**macOS / Linux:**

```bash
cat > cowork-patterns/prompt-readme-outline.md << 'EOF'
# Prompt Pattern: README Outline Request

## When to use
When starting a Cowork session to generate project documentation.

## The prompt
```
Read cowork-session-brief.md and confirm you understand the goal, context,
success criteria, and boundaries. Summarize them back to me before we start.
```

Then after confirmation:

```
Start with an outline for the README. Include section headings and a one-line
description of what each section will cover. Do not write the full content yet.
```

## Why it works
- Forces Claude to confirm understanding before writing.
- Requests an outline first, giving you a review checkpoint before full content.
- Explicit "do not write full content yet" prevents a wall of text you cannot easily redirect.

## Variations
- For API documentation: replace "README" with "API reference" and adjust headings.
- For onboarding guides: replace with "Getting Started guide" and specify audience.
EOF
```

Extract the accuracy checklist from Lesson 8.3.1 into its own file:

**macOS / Linux:**

```bash
cat > cowork-patterns/review-checklist.md << 'EOF'
# Post-Session Review Checklist

Use this after every Cowork session before saving deliverables.

## Accuracy checks
- [ ] Every shell command runs successfully on the target OS
- [ ] File paths, flag names, and version numbers are correct
- [ ] All sections from the success criteria are present
- [ ] Nothing from the boundaries list was included
- [ ] Tone matches the intended audience
- [ ] Both macOS/Linux and Windows commands are present (if required)

## Save checks
- [ ] Final version saved (not an intermediate draft)
- [ ] File encoding is UTF-8
- [ ] Session logs saved separately from deliverables
- [ ] Only deliverables are staged for Git commit
EOF

### Step 7 — Use a pattern in a future session

<StepMeta purpose="Connect the saved templates back to future action so the patterns folder becomes part of your routine instead of a static archive." action="Read the three-step reuse workflow and imagine or note how you would apply it the next time you start a documentation-focused Cowork session." expectedResult="You can describe how to use the brief template, prompt pattern, and review checklist together in a future session." whyItMatters="Patterns only create leverage when they re-enter live work, so this step turns saved assets into a repeatable operating system." nextStep="Put all of these ideas together in the mini project by running a full end-to-end Cowork session." askTutor="I'm on Step 7 of Save useful patterns as reusable templates. Next time you need to write documentation with Claude, your workflow becomes:

1. Copy `brief-template.md` and fill it in for your new task.
2. Open a Cowork session and use the prompt from `prompt-readme-outline.md`.
3. After the session, run through `review-checklist.md` before saving.

Over time, add more patterns to this folder: prompt patterns for code review, for refactoring, for writing tests, for editing prose. Each pattern captures what worked so you start every future session with an advantage.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| You save a pattern but never use it again | The pattern was too specific to one session and not generalized | When saving, strip out project-specific details and replace them with placeholders like `[PROJECT_NAME]` |
| Patterns folder gets cluttered with dozens of files | You saved everything instead of only what was reusable | Review the folder monthly. Delete patterns you have not used in the last 30 days. |
| You reuse a prompt pattern but get worse results than the original session | The context or constraints were different | The prompt pattern is only the starting point. Always pair it with a fresh session brief that describes the current task. |
| Another team member cannot use your patterns because they reference your local paths | Patterns contained hardcoded paths like `/Users/yourname/project` | Use relative paths and placeholders in all templates |
