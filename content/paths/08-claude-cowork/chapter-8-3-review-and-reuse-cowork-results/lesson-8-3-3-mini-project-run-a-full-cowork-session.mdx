# Mini project: run a full Cowork session

## Objective

In this mini project you will run a complete Cowork session from start to finish: write a brief, start the session, iterate through multiple turns, review the output, save the deliverables, and template the pattern. The task is to produce a project README and a CONTRIBUTING guide for a real or practice project.

## Walkthrough

### Step 1 — Set up the project folder

<StepMeta purpose="Create a small realistic project so the Cowork session has concrete files and metadata to work from." action="Make the cowork-mini-project directory, add package.json plus placeholder source and test files, and confirm the practice project exists." expectedResult="You have a local taskflow practice project with a minimal Node CLI structure ready for documentation work." whyItMatters="A realistic project makes the Cowork session feel like real product work rather than a purely abstract writing exercise." nextStep="Write the session brief that tells Claude exactly what documentation this project needs." askTutor="I'm on Step 1 of Mini project: run a full Cowork session. Help me set up the practice project correctly and explain which files Claude will likely use as context." />

Create a practice project to work with. This simulates having an existing codebase that needs documentation.

**macOS / Linux:**

```bash
mkdir -p cowork-mini-project/src
mkdir -p cowork-mini-project/tests
cd cowork-mini-project

cat > package.json << 'EOF'
{
  "name": "taskflow",
  "version": "0.2.0",
  "description": "A lightweight CLI task runner for small teams",
  "type": "module",
  "bin": { "taskflow": "./src/cli.js" },
  "scripts": {
    "test": "node --test tests/",
    "lint": "eslint src/"
  },
  "license": "MIT"
}
EOF

echo "// CLI entry point" > src/cli.js
echo "// Test placeholder" > tests/cli.test.js
```


### Step 2 — Write the session brief

<StepMeta purpose="Define the task contract so the end-to-end Cowork session begins with a precise deliverable and review standard." action="Create cowork-session-brief.md for taskflow using the provided goal, context, success criteria, and boundaries." expectedResult="The mini project includes a complete session brief that scopes the README and CONTRIBUTING work clearly." whyItMatters="The brief is what turns a general collaboration into a bounded workflow with a known definition of done." nextStep="Start Claude and use the brief plus package metadata as the grounding context for the session." askTutor="I'm on Step 2 of Mini project: run a full Cowork session. Help me review this taskflow brief and point out anything unclear before I start the session." />

Use the template from your `cowork-patterns/` folder (Lesson 8.3.2) or create a fresh one:

**macOS / Linux:**

```bash
cat > cowork-session-brief.md << 'EOF'
# Cowork Session Brief

## Goal
Produce two files: README.md and CONTRIBUTING.md for the `taskflow` project.

## Context
- `taskflow` is a Node.js CLI tool for running predefined tasks (like a
  simpler Make or just).
- Commands: `taskflow run <task>`, `taskflow list`, `taskflow init`.
- Repository layout: `src/`, `tests/`, `package.json`.
- Target audience: developers who use npm and are comfortable with a terminal.
- License: MIT.

## Success criteria
- README includes: description, installation (npm), usage with examples,
  configuration, and license section.
- CONTRIBUTING includes: how to fork and clone, how to run tests, code style
  expectations, and how to submit a PR.
- Every shell command block includes both bash (macOS/Linux) and PowerShell
  (Windows) variants.
- Tone: clear, direct, no emoji, no marketing language. Write for developers.

## Boundaries
- Do not generate application source code.
- Do not write a changelog.
- Do not add badges or shields to the README.
EOF
```

**Windows PowerShell:**

```powershell
@"
# Cowork Session Brief

## Goal
Produce two files: README.md and CONTRIBUTING.md for the ``taskflow`` project.

## Context
```

### Step 3 — Start the Cowork session

<StepMeta purpose="Launch the real collaboration loop and confirm Claude is grounded in the brief before any drafting begins." action="Open Claude in the project directory, ask it to read cowork-session-brief.md and package.json, and verify the summary matches the brief." expectedResult="Claude summarizes the taskflow goals, constraints, and deliverables accurately before drafting." whyItMatters="Misunderstandings are cheapest to fix at the summary stage; after drafting begins, corrections become more expensive and disruptive." nextStep="Use outline-first drafting and section-by-section review so the session stays steerable." askTutor="I'm on Step 3 of Mini project: run a full Cowork session. Help me phrase the startup prompt so Claude confirms the brief and package metadata before it writes anything." />

Open Claude Code in the project directory (or create a claude.ai Project and upload the brief plus `package.json`).

**macOS / Linux:**

```bash
claude
```

**Windows PowerShell:**

```powershell
claude
```

Send your opening prompt:

```
> Read cowork-session-brief.md and package.json. Confirm you understand the
> goal, context, success criteria, and boundaries. Summarize them back to me.
```

Wait for Claude's summary. Verify it matches your brief. If anything is wrong, correct it now.

### Step 4 — Iterate: outline → sections → review

<StepMeta purpose="Run the full Cowork rhythm end to end: structure first, then content, then targeted revision." action="Request the README outline, steer it, draft sections one at a time, then repeat the same review-first flow for CONTRIBUTING.md." expectedResult="Claude produces reviewed drafts for both README and CONTRIBUTING through multiple guided turns rather than one uncontrolled dump." whyItMatters="This is the core operational pattern of Cowork, and practicing it across two files makes the habit durable." nextStep="Check both deliverables against the review checklist before you decide they are done." askTutor="I'm on Step 4 of Mini project: run a full Cowork session. Help me keep the README and CONTRIBUTING workflow reviewable instead of letting it turn into one giant uncontrolled conversation." />

**Turn 1 — Request the README outline:**

```
> Give me an outline for README.md. Section headings with a one-line
> description of each. Do not write full content yet.
```

Review the outline. Steer if needed:

```
> Add a "Troubleshooting" section after "Configuration". Remove "Acknowledgments"
> — it is not in the brief.
```

**Turn 2 — Write the README section by section:**

```
> Write the "Installation" section. Include both bash and PowerShell commands.
```

Review. Continue for each section:

```
> Write the "Usage" section. Show three examples: `taskflow run`, `taskflow
> list`, and `taskflow init`. Each example should have the command and expected
> output.
```

**Turn 3 — Write CONTRIBUTING.md:**

```
> Now write CONTRIBUTING.md. Follow the success criteria from the brief. Start
> with an outline, then write the full document.
```

Review and steer:

```
> The "Running tests" section uses `npm test` but doesn't explain what testing
> framework is used. Add that it uses Node.js built-in test runner. Also add
> the Windows PowerShell equivalent for every command.
```

Fix any issues by giving Claude specific feedback or editing the files yourself.

### Step 7 — Template the pattern

<StepMeta purpose="Preserve the working session flow as a reusable playbook instead of treating the mini project as a one-off success." action="Create a prompt-project-docs pattern file that records when to use the workflow, the session steps, key prompts, and expected turn count." expectedResult="The project now contains a generalized Cowork pattern that could guide a similar README plus CONTRIBUTING session later." whyItMatters="Mini projects become durable skills when you extract the process that worked, not just the artifacts it produced." nextStep="Commit the project, session brief, and optionally the patterns folder so the full exercise is preserved in history." askTutor="I'm on Step 7 of Mini project: run a full Cowork session. Help me capture the successful workflow as a reusable pattern without leaving it tied only to taskflow." />

This session worked. Save the pattern for future use:

**macOS / Linux:**

```bash
mkdir -p cowork-patterns

cat > cowork-patterns/prompt-project-docs.md << 'EOF'
# Prompt Pattern: Project Documentation (README + CONTRIBUTING)

## When to use
When a project needs a README and CONTRIBUTING guide generated from a session brief.

## Session flow
1. Share brief + package.json (or equivalent project metadata).
2. Ask Claude to confirm understanding.
3. Request README outline first → review → approve.
4. Write README section by section → review each.
5. Request CONTRIBUTING outline → review → write full.
6. Run review checklist.
7. Save deliverables.

## Key prompts
- "Read cowork-session-brief.md and package.json. Confirm understanding."
- "Give me an outline. Do not write full content yet."
- "Write the [section] section. Include both bash and PowerShell commands."

## Estimated turns
8-12 turns for both documents.
EOF
```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude writes both files in one giant message and you lose the ability to steer | You asked for both files at once instead of sequentially | Always request one deliverable at a time. Outline first, then content, then the next file. |
| README and CONTRIBUTING contradict each other (e.g., different test commands) | Claude forgot the README content when writing CONTRIBUTING | Point out the contradiction: "The README says `npm test` but CONTRIBUTING says `node tests/`. These should match. Use `npm test` in both." |
| The session takes 20+ turns and context starts degrading | Too much back-and-forth on minor wording | Approve structure early, polish wording after the session by editing the files directly |
| You cannot find the final version in a long conversation | Multiple revisions with no clear "this is the final" marker | Before ending the session, ask Claude: "Save the final version of README.md and CONTRIBUTING.md to my project directory." |
| Git commit includes files you did not intend to track | You used `git add .` which staged everything | Use explicit file names with `git add`. Add `session-logs/` to `.gitignore`. |
