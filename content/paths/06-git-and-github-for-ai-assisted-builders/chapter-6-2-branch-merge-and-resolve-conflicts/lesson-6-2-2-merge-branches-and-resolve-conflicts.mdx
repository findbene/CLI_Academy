# Merge branches and resolve conflicts

## Objective

By the end of this lesson you will merge a feature branch into `main`, deliberately create a merge conflict by editing the same line in two branches, read the conflict markers Git places in the file, resolve the conflict manually, and complete the merge. You will also learn how to ask Claude Code for help resolving conflicts.

## Walkthrough

### Step 1 — Set up a repository with a clean main branch

<StepMeta purpose="Create a simple repo where you can practice merging without risking any real work." action="Initialize the merge-lab repo, add config.txt, and commit the baseline welcome message on main." expectedResult="You have a stable main branch with one tracked config file ready for branch experiments." whyItMatters="Conflict practice is safest in a tiny sandbox where every branch and edit is easy to inspect." nextStep="Create a simple feature branch and perform a clean merge first so you can see the non-conflict case." askTutor="I'm on Step 1 of Merge branches and resolve conflicts. Help me understand why we start with a tiny config file instead of a more realistic project." />

**macOS / Linux:**

```bash
mkdir merge-lab && cd merge-lab
git init
echo "welcome = Hello World" > config.txt
git add config.txt
git commit -m "Add config with welcome message"
```

**Windows PowerShell:**

```powershell
mkdir merge-lab; cd merge-lab
git init
"welcome = Hello World" | Out-File -Encoding utf8 config.txt
git add config.txt
git commit -m "Add config with welcome message"
```

### Step 2 — Do a clean merge first (no conflict)

<StepMeta purpose="See the easy merge case before introducing conflicts, so you know what normal looks like." action="Create feature-footer, add a new file on that branch, switch back to main, and merge it cleanly." expectedResult="Git performs a fast-forward merge with no conflict markers or manual steps required." whyItMatters="Understanding a clean merge makes the later conflict case less mysterious because you know when Git can combine histories automatically." nextStep="Create two branches that edit the same line differently so Git can no longer merge by itself." askTutor="I'm on Step 2 of Merge branches and resolve conflicts. Help me understand why this first merge is a fast-forward and what conditions made it conflict-free." />

Create a feature branch, add a new file, and merge it back.

```bash
git switch -c feature-footer
```

**macOS / Linux:**

```bash
echo "Footer text goes here." > footer.txt
git add footer.txt
git commit -m "Add footer file"
```

**Windows PowerShell:**

```powershell
"Footer text goes here." | Out-File -Encoding utf8 footer.txt
git add footer.txt
git commit -m "Add footer file"
```

Now merge into main:

```bash
git switch main
git merge feature-footer
```

Git performs a **fast-forward merge** because main had no new commits. No conflict. Check the log:

```bash
git log --oneline --all --graph
```

Both `main` and `feature-footer` point to the same commit.

### Step 3 — Create a deliberate conflict

<StepMeta purpose="Set up the exact situation that causes a real content conflict: two branches editing the same line differently." action="Create feature-spanish, change config.txt there, then switch back to main and change the same line to a different value." expectedResult="Both branches now contain incompatible edits to the same line in config.txt." whyItMatters="Conflicts happen when Git cannot automatically decide which competing edit should win, and same-line edits are the clearest example." nextStep="Attempt the merge so Git can stop and show you the conflict state explicitly." askTutor="I'm on Step 3 of Merge branches and resolve conflicts. Help me see why these two edits are guaranteed to conflict and how Git detects that." />

Now you will edit the **same line** in two different branches — the classic cause of a merge conflict.

First, create a branch and change the welcome message:

```bash
git switch -c feature-spanish
```

**macOS / Linux:**

```bash
echo "welcome = Hola Mundo" > config.txt
git add config.txt
git commit -m "Change welcome to Spanish"
```

**Windows PowerShell:**

```powershell
"welcome = Hola Mundo" | Out-File -Encoding utf8 config.txt
git add config.txt
git commit -m "Change welcome to Spanish"
```

Now switch back to `main` and make a different change to the same line:

```bash
git switch main
```

**macOS / Linux:**

```bash
echo "welcome = Bonjour le Monde" > config.txt
git add config.txt
git commit -m "Change welcome to French"
```

**Windows PowerShell:**

```powershell
"welcome = Bonjour le Monde" | Out-File -Encoding utf8 config.txt
git add config.txt
git commit -m "Change welcome to French"
```

### Step 4 — Attempt the merge and see the conflict

<StepMeta purpose="Observe the exact signals Git gives when it cannot merge automatically." action="Run git merge feature-spanish, then inspect git status to see the unmerged path state." expectedResult="Git reports a content conflict in config.txt and marks the repo as needing manual resolution." whyItMatters="Knowing the standard conflict messages helps you stay calm and methodical instead of panicking when a merge stops midway." nextStep="Open the conflicted file and read the conflict markers Git inserted." askTutor="I'm on Step 4 of Merge branches and resolve conflicts. Help me interpret this merge error output and explain what git status is telling me now." />

```bash
git merge feature-spanish
```

Git output:

```
Auto-merging config.txt
CONFLICT (content): Merge conflict in config.txt
Automatic merge failed; fix conflicts and then commit the result.
```

Check status:

```bash
git status
```

`config.txt` appears under **Unmerged paths**. Git is waiting for you to resolve the conflict.

### Step 5 — Read the conflict markers

<StepMeta purpose="Decode the special marker lines Git inserts so you can compare the competing changes correctly." action="Open config.txt and identify which side belongs to HEAD and which side belongs to feature-spanish." expectedResult="You can point to the current-branch version, the incoming version, and the separator lines with confidence." whyItMatters="Conflict resolution starts with understanding the marker format; otherwise you are editing a file full of symbols without context." nextStep="Choose the final content you want and replace the conflict-marked block with one clean resolved line." askTutor="I'm on Step 5 of Merge branches and resolve conflicts. Help me read these conflict markers and decide which side came from which branch." />

Open the file:

**macOS / Linux:**

```bash
cat config.txt
```

**Windows PowerShell:**

```powershell
Get-Content config.txt
```

You will see:

```
<<<<<<< HEAD
welcome = Bonjour le Monde
=======
welcome = Hola Mundo
>>>>>>> feature-spanish
```

Here is what each section means:

- `<<<<<<< HEAD` — the version on your current branch (`main`)
- `=======` — the divider between the two versions
- `>>>>>>> feature-spanish` — the version from the branch you are merging

### Step 6 — Resolve the conflict manually

<StepMeta purpose="Turn the conflicted file back into valid content by making an explicit human decision." action="Edit config.txt to the final desired line and remove every conflict marker from the file." expectedResult="The file contains one clean resolved value and no <<<<<<<, =======, or >>>>>>> lines remain." whyItMatters="Git can detect the conflict, but only a human can choose the correct final meaning for the file." nextStep="Stage the resolved file and complete the merge commit." askTutor="I'm on Step 6 of Merge branches and resolve conflicts. Help me check whether this resolution is minimal, correct, and completely free of leftover markers." />

Edit `config.txt` to keep the version you want. Remove all three marker lines and leave one clean line.

For example, to keep both languages:

**macOS / Linux:**

```bash
cat > config.txt << 'EOF'
welcome = Bonjour le Monde / Hola Mundo
EOF
```

**Windows PowerShell:**

```powershell
"welcome = Bonjour le Monde / Hola Mundo" | Out-File -Encoding utf8 config.txt
```

There must be no `<<<<<<<`, `=======`, or `>>>>>>>` lines remaining.

### Step 7 — Complete the merge

<StepMeta purpose="Record the manual resolution as the official merge result in history." action="Stage config.txt, create the merge commit, and inspect the graph to see the branches converge." expectedResult="Git history now shows a merge commit connecting main and feature-spanish with the resolved file content on main." whyItMatters="A conflict is not finished when the file looks fixed; it is finished when the merge commit captures that decision in history." nextStep="Learn how Claude can help explain or suggest resolutions when a real conflict is harder to reason about." askTutor="I'm on Step 7 of Merge branches and resolve conflicts. Help me read this merge commit in the graph and explain what it means for the two branch histories." />

```bash
git add config.txt
git commit -m "Merge feature-spanish: combine French and Spanish greetings"
```

Check the graph:

```bash
git log --oneline --all --graph
```

You will see both branches converging into a single merge commit on `main`.

### Step 8 — Ask Claude Code for help (when conflicts are complex)

<StepMeta purpose="Practice using Claude as a conflict-reading assistant without giving up final human review." action="Prompt Claude to explain the conflict markers or propose a resolution strategy for the file, then review the result yourself." expectedResult="Claude can summarize each side's intent or generate a cleaned file, but you still understand the final choice before committing." whyItMatters="In real repos, conflict context can be larger than one line, so a second reader is helpful as long as you keep the final decision in human hands." nextStep="After local branching and conflict resolution feel solid, move on to pushing feature branches to GitHub and opening pull requests." askTutor="I'm on Step 8 of Merge branches and resolve conflicts. Help me use Claude to understand this conflict without blindly accepting a resolution I have not reviewed." />

If you hit a conflict you do not understand, you can ask Claude Code:

```
Show me the conflict in config.txt and explain what each side changed.
```

Or:

```
Resolve the merge conflict in config.txt by keeping the French version.
```

Claude Code can read the conflict markers and produce the resolved file. You still review and commit the result.

> **Safety note:** Always read the resolved file before committing. Automated resolution can miss context that only you know about.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| `Already up to date` when merging | The branch has no new commits relative to main | Check you committed on the correct branch with `git log --oneline --all` |
| Conflict markers still in file after editing | You left a `<<<<<<<` or `>>>>>>>` line | Open the file and delete all marker lines |
| `error: you need to resolve your current index first` | Previous merge conflict was never completed | Run `git merge --abort` to cancel, then start over |
| `git merge --abort` not working | You already committed the merge | Use `git revert HEAD` to undo the merge commit |
| Merge commit opens a text editor | Git wants a merge message | Save and close the editor, or use `git commit -m "message"` |
| File encoding issues on Windows | PowerShell added BOM to the file | Use `Out-File -Encoding utf8` consistently |
