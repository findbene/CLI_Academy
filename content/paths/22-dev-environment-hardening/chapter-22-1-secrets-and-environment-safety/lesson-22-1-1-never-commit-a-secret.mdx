# Never Commit a Secret: .env, .gitignore, and Secret Scanning

## What you will build

A project directory with a properly configured `.gitignore` that blocks `.env` files from being tracked by Git. You will verify with `git status` that secret files are invisible to Git, and optionally run `gitleaks` to confirm no sensitive patterns exist in your repo.

## Why this matters

Git history is permanent. When you push a commit containing an API key, that key is now in a public record — even if you delete the file in the very next commit. The old commit still exists, and automated bots scrape GitHub around the clock hunting for exactly this. Anthropic, AWS, and GitHub all have secret scanning that will flag and sometimes auto-revoke keys found in public repos. But by that point, the damage may already be done: unexpected API charges, data breaches, or a revoked key that breaks production at 2am.

The defense is simple: never let secrets enter Git in the first place. A correct `.gitignore` is the first line of that defense.

## Before you start

- Git installed and `git --version` returns a version number.
- A terminal open at a directory you can experiment in.
- Optional: Python 3 installed (for `gitleaks` via pip, or you can download the binary directly).
- You do not need an actual API key — we will use fake placeholder values throughout.

## The walkthrough

### Step 1 — What counts as a secret

A secret is any value that grants access to a system and must be kept private. Common examples:

- **API keys**: `sk-ant-api03-...`, `sk-proj-...`, `AKIA...` (AWS)
- **Access tokens**: OAuth tokens, GitHub personal access tokens (`ghp_...`)
- **Passwords**: Database passwords, service account passwords
- **Private keys**: SSH private keys, TLS private keys
- **Webhook secrets**: Values used to verify incoming webhook payloads

These values should never appear in source code files, commit messages, or filenames that Git tracks. The standard home for them is a `.env` file — which must be excluded from Git.

**macOS / Linux:**
```bash
cat > .env << 'EOF'
ANTHROPIC_API_KEY=sk-ant-api03-FAKE-KEY-DO-NOT-USE
DATABASE_URL=postgres://admin:hunter2@localhost:5432/myapp
GITHUB_TOKEN=ghp_FAKEFAKEFAKEFAKEFAKEFAKEFAKEFAKE

```

**Windows PowerShell:**
```powershell
@"
ANTHROPIC_API_KEY=sk-ant-api03-FAKE-KEY-DO-NOT-USE

"@ | Set-Content .env
```

Now check what Git sees:

**macOS / Linux:**
```bash
git status
```

**Windows PowerShell:**
```powershell
git status
```

Expected output — Git can see your secret file:
```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .env

nothing added to commit but untracked files present
```

This is dangerous. If you ran `git add -A` and `git commit` right now, your fake API keys would be in history forever.

### Step 4 — Create a .gitignore that blocks secrets

**macOS / Linux:**
```bash
cat > .gitignore << 'EOF'
# Secrets and environment files — NEVER commit these
.env
.env.*
.env.local
.env.production
.env.staging
.env.development

# Backup files that might contain secrets
*.env.bak
*.env.backup

# OS and editor noise
.DS_Store
Thumbs.db
.idea/
.vscode/
*.swp
EOF
```

**Windows PowerShell:**
```powershell
@"
# Secrets and environment files — NEVER commit these
.env
.env.*
.env.local
.env.production
.env.staging
.env.development

# Backup files that might contain secrets
*.env.bak
*.env.backup

# OS and editor noise
.DS_Store
Thumbs.db
.idea/
.vscode/
*.swp
"@ | Set-Content .gitignore
```

### Step 5 — Verify the secret is now hidden from Git

**macOS / Linux:**
```bash
git status
```

**Windows PowerShell:**
```powershell
git status
```

Expected output — `.env` is gone from Git's view:
```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore

nothing added to commit but untracked files present
```

Only `.gitignore` appears. The `.env` file exists on disk but Git cannot see it. This is the correct state.

### Step 6 — Commit the .gitignore

**macOS / Linux:**
```bash
git add .gitignore
git commit -m "chore: add .gitignore to block secrets from being committed"
```

**Windows PowerShell:**
```powershell
git add .gitignore
git commit -m "chore: add .gitignore to block secrets from being committed"
```

### Step 7 — Check whether a secret was already staged (the safety check)

Before any commit, run this to confirm you are not about to commit a secret:

**macOS / Linux:**
```bash
git diff --cached --name-only
```

**Windows PowerShell:**
```powershell
git diff --cached --name-only
```

If you see `.env`, `.key`, `*_secret*`, or similar, run `git reset HEAD <filename>` to unstage the file before committing.

### Step 8 — Install and run gitleaks for automated scanning

`gitleaks` scans your repository for over 150 patterns of common secrets. It is the tool security teams use in CI pipelines.

**macOS / Linux (Homebrew):**
```bash
brew install gitleaks
```

**macOS / Linux (direct binary):**
```bash
curl -sSfL https://raw.githubusercontent.com/gitleaks/gitleaks/main/scripts/install.sh | sh -s -- -b /usr/local/bin v8.18.4
```

**Windows PowerShell (Scoop):**
```powershell
scoop install gitleaks
```

**Windows PowerShell (direct binary):**
```powershell
# Download from https://github.com/gitleaks/gitleaks/releases and add to PATH
Invoke-WebRequest -Uri "https://github.com/gitleaks/gitleaks/releases/download/v8.18.4/gitleaks_8.18.4_windows_x64.zip" -OutFile gitleaks.zip
Expand-Archive gitleaks.zip -DestinationPath "$HOME\bin"
```

Once installed, scan your repo:

**macOS / Linux:**
```bash
gitleaks detect --source . --no-git
```

**Windows PowerShell:**
```powershell
gitleaks detect --source . --no-git
```

Because `.env` is ignored by Git but still exists on disk, `gitleaks` will find the fake keys in it. That is expected — this confirms the scanner works. In a clean project with no untracked secrets, the output will be:

```
    ○
    │╲
    │ ○
    ○ ░
    ░    gitleaks

5:10PM INF 1 commits scanned.
5:10PM INF scan completed in 84ms
5:10PM WRN leaks found: 0
```

## What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `.env` still shows in `git status` after adding `.gitignore` | The `.gitignore` file was not saved, or the pattern does not match the filename | Check the `.gitignore` content with `cat .gitignore`; make sure `.env` appears on its own line with no extra spaces |
| `.env` was already tracked before `.gitignore` was created | Git does not stop tracking files it has already committed | Run `git rm --cached .env` to untrack it, then commit; the file stays on disk but leaves Git's index |
| `gitleaks` not found after install | Binary not on PATH | On macOS/Linux run `which gitleaks`; on Windows check that your `$HOME\bin` or Scoop bin folder is in `$env:PATH` |
| `gitleaks` reports findings in `.gitignore` itself | You accidentally pasted a real-looking value while writing the ignore file | Replace with clearly fake values; `.gitignore` is committed and public |
| Teammate pushes a secret you thought was blocked | They do not have the same `.gitignore`, or they used `git add --force` | Add a pre-commit hook (see below) and set up GitHub secret scanning on the repository |

**Adding a pre-commit hook for extra safety:**

**macOS / Linux:**
```bash
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
if git diff --cached --name-only | grep -qE '\.env$|\.env\.'; then
  echo "ERROR: Attempting to commit a .env file. Aborting."
  exit 1
fi
EOF
chmod +x .git/hooks/pre-commit
```

**Windows PowerShell:**
```powershell
@"
#!/bin/sh
if git diff --cached --name-only | grep -qE '\.env$|\.env\.'; then
  echo "ERROR: Attempting to commit a .env file. Aborting."
  exit 1
fi
"@ | Set-Content .git/hooks/pre-commit
```

This hook runs before every `git commit` and aborts if a `.env` file is staged.
