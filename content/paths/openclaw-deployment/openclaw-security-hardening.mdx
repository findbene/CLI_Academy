## What You'll See

By the end of this lesson, you will have applied the production security hardening checklist to your OpenClaw deployment: tightened network access, secured secrets storage, configured TLS correctly, enabled audit logging, and applied developer isolation policies.

## Why This Matters

An unconfigured OpenClaw deployment is functional but not secure. The default installation is designed to get you to a working state quickly — production security requires additional steps. A poorly secured OpenClaw server exposes your Anthropic API key (which can generate significant billing), your developers' code context, and potentially your internal network to anyone who can reach port 443. This lesson closes those gaps.

## Core Concepts

### Layer 1: Network Hardening

The first line of defense is restricting what can reach your server.

**Firewall rules (UFW on Ubuntu):**

```bash
# Reset to clean state
sudo ufw --force reset

# Allow SSH only from known admin IPs (replace with your IP range)
sudo ufw allow from 10.0.0.0/8 to any port 22 proto tcp

# Allow HTTPS from anywhere (developer connections)
sudo ufw allow 443/tcp

# Allow admin dashboard only from internal network
sudo ufw allow from 10.0.0.0/8 to any port 8080 proto tcp

# Deny everything else
sudo ufw default deny incoming
sudo ufw default allow outgoing

sudo ufw enable
sudo ufw status verbose
```

**iptables equivalent for production servers without UFW:**

```bash
# Drop inbound by default
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Allow established connections
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow SSH from admin subnet
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/8 -j ACCEPT

# Allow HTTPS
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow admin dashboard from internal only
iptables -A INPUT -p tcp --dport 8080 -s 10.0.0.0/8 -j ACCEPT

# Save rules
iptables-save > /etc/iptables/rules.v4
```

### Layer 2: TLS Configuration

OpenClaw uses TLS for all developer connections. These settings ensure you are using modern, secure cipher suites.

In `openclaw.yml`:

```yaml
server:
  tls: true
  tls_min_version: "1.2"           # Minimum TLS 1.2; prefer 1.3 only if possible
  tls_cipher_suites:               # Explicit cipher suite allowlist
    - TLS_AES_256_GCM_SHA384
    - TLS_CHACHA20_POLY1305_SHA256
    - TLS_AES_128_GCM_SHA256
    - TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
  hsts_enabled: true               # HTTP Strict Transport Security
  hsts_max_age_seconds: 63072000   # 2 years
```

**Verify your TLS configuration:**

```bash
# Test from another machine (replace with your hostname)
openssl s_client -connect openclaw.yourdomain.com:443 -brief 2>/dev/null | head -20

# Check certificate expiry
openssl s_client -connect openclaw.yourdomain.com:443 2>/dev/null \
  | openssl x509 -noout -dates
```

### Layer 3: Secrets Management

Your Anthropic API key is the highest-value secret in your OpenClaw deployment. These rules protect it:

**Rule 1: Never store the API key in a file that could be accidentally committed.**

Use a secrets manager if your organization has one:

```bash
# HashiCorp Vault example
vault kv put secret/openclaw anthropic_api_key="sk-ant-..."

# Retrieve at startup via vault agent or in the systemd service:
# EnvironmentFile uses a file generated by vault-agent, not the vault binary directly
```

**Rule 2: Restrict who can read the `.env` file.**

```bash
# Verify permissions — only the openclaw user should be able to read
ls -la /opt/openclaw/config/.env
# Expected: -rw------- 1 openclaw openclaw

# If permissions are wrong:
sudo chown openclaw:openclaw /opt/openclaw/config/.env
sudo chmod 600 /opt/openclaw/config/.env
```

**Rule 3: Rotate the API key on any suspected exposure.**

If your server is compromised, rotated, or if the `.env` file is accidentally committed:
1. Immediately generate a new API key in the Anthropic console
2. Update the `.env` file with the new key
3. Restart OpenClaw
4. Revoke the old key in the Anthropic console

### Layer 4: Audit Logging

Enable a separate audit log that tracks authentication events, policy violations, and admin actions:

```yaml
logging:
  audit_log_path: /logs/audit.log
  audit_events:
    - auth_success
    - auth_failure
    - token_created
    - token_revoked
    - policy_violation
    - admin_login
    - config_change
    - server_start
    - server_stop
```

The audit log is append-only and should not be writable by the `openclaw` user in production. Route it to your SIEM if you have one:

```bash
# For bare-metal: rotate but never truncate
sudo logrotate -d /etc/logrotate.d/openclaw-audit

# For environments with a SIEM (Splunk, Datadog, etc.):
# Configure rsyslog or filebeat to tail /logs/audit.log and forward to your collector
```

### Layer 5: Developer Isolation

In a shared OpenClaw deployment, one developer's runaway agent job should not starve other developers of capacity. Configure per-developer resource limits:

```yaml
policies:
  # Hard limits per developer session
  default_daily_token_limit: 500000
  default_monthly_token_limit: 10000000

  # Per-session concurrency limits
  max_concurrent_requests_per_session: 3

  # Rate limiting
  rate_limit:
    requests_per_minute: 60
    burst: 10

  # Dangerous mode restrictions
  allow_shell_execution: true
  require_project_dir_for_shell: true   # Shell execution only inside project directories
  shell_blocklist:                       # Commands never allowed in shell mode
    - "rm -rf /"
    - "dd if=/dev/zero"
    - ":(){ :|:& };:"                   # Fork bomb

  # File access restrictions
  allow_file_uploads: true
  max_upload_size_mb: 50
  blocked_file_patterns:               # Never upload these file types
    - "*.env"
    - "*.pem"
    - "*.key"
    - "id_rsa*"
```

### Layer 6: Admin Dashboard Hardening

The admin dashboard is the highest-privilege interface in OpenClaw. Additional controls:

```yaml
admin:
  listen_port: 8080
  require_auth: true
  session_timeout_minutes: 30      # Shorter timeout for admin sessions
  max_failed_login_attempts: 5     # Lock after 5 failures
  lockout_duration_minutes: 15
  allowed_ip_ranges:
    - "10.0.0.0/8"                 # Internal only — no public access ever
  require_2fa: false               # Enable if your admin secret method supports it
```

**Change the admin secret periodically:**

```bash
# Generate a new admin secret
NEW_SECRET=$(openssl rand -base64 32)

# Update .env (Docker)
sed -i "s/ADMIN_SECRET=.*/ADMIN_SECRET=${NEW_SECRET}/" /opt/openclaw/.env

# Restart
docker compose restart openclaw

# Update your password manager with the new secret
```

### Security Hardening Checklist

Work through this before exposing OpenClaw to your team:

**Network:**
- [ ] UFW / iptables blocks all ports except 22 (SSH), 443 (HTTPS), 8080 (internal only)
- [ ] Port 8080 is restricted to internal IP ranges
- [ ] SSH is restricted to known admin IP ranges or uses key-only auth

**TLS:**
- [ ] TLS minimum version set to 1.2
- [ ] Certificate expiry is monitored (Let's Encrypt auto-renews; self-signed needs a reminder)
- [ ] HSTS is enabled

**Secrets:**
- [ ] `.env` file is mode 600 and owned by the openclaw user (or equivalent)
- [ ] API key is not in version control
- [ ] Admin secret is strong (32+ random characters) and stored in a password manager

**Audit:**
- [ ] Audit logging is enabled and writing to a separate file
- [ ] Audit log rotation is configured
- [ ] Log retention meets your compliance requirements

**Developer policies:**
- [ ] Daily and monthly token limits are set
- [ ] Rate limiting is configured
- [ ] Shell execution is scoped to project directories
- [ ] Dangerous file types are blocked from upload

**Admin:**
- [ ] Admin dashboard is not accessible from the public internet
- [ ] Login lockout is enabled
- [ ] Admin sessions time out after 30 minutes

## Key Takeaways

- Security happens in layers: network, TLS, secrets, audit, developer isolation, admin controls.
- The Anthropic API key deserves the same protection as a database root password.
- Developer isolation prevents runaway jobs from affecting the whole team.
- Audit logging is the difference between detecting an incident and finding out about it in a billing email.
- Run the full hardening checklist before your first production developer connection.
