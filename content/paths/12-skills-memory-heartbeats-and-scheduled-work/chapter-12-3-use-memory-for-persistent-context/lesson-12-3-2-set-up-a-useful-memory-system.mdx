# Set up a useful memory system

## Objective

By the end of this lesson you will have a structured memory system with topic-based files — `preferences.md`, `patterns.md`, and `debugging.md` — containing real entries. You will verify that Claude Code loads these files and draws on them during a new session.

## Walkthrough

### Step 1 — Create the user memory directory if it does not exist

<StepMeta purpose="Make sure the base user-memory location exists before you start organizing files inside it." action="Create ~/.claude/memory if needed so you have a stable directory for topic-based memory files." expectedResult="The user memory directory exists and is ready to hold preferences, patterns, and debugging notes." whyItMatters="A structured memory system only works if all of the topic files live in the path Claude auto-loads at session start." nextStep="Create a preferences file that captures how you want Claude to work with you across projects." askTutor="Confirm the correct user-memory path for my OS and explain how to tell whether I am creating the directory in the wrong place." />

**macOS / Linux**

```bash
mkdir -p ~/.claude/memory
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\.claude\memory"
```

### Step 2 — Create preferences.md

<StepMeta purpose="Store your broad, cross-project working preferences in one file Claude can load automatically." action="Create preferences.md and add concise entries about answer style, language choices, git habits, editor, shell, and comment preferences." expectedResult="A user memory file exists that summarizes how you generally like Claude to behave across many projects." whyItMatters="Preferences belong in memory, not CLAUDE.md, because they are soft defaults that should travel with you from project to project." nextStep="Create a separate patterns file for conventions you want Claude to reuse consistently in implementation work." askTutor="Review my preferences.md entries and tell me whether any of them are too rigid and should probably move into CLAUDE.md instead." />

This file captures how you like to work. Claude reads it at session start and adjusts its behavior.

**macOS / Linux**

```bash
cat > ~/.claude/memory/preferences.md << 'EOF'
# Preferences

- I prefer concise answers — skip unnecessary introductions.
- Use TypeScript over JavaScript when both are options.
- For git commits, follow conventional commit format.
- I use VS Code as my editor.
- On Windows I use PowerShell. On macOS I use zsh.
- Do not add comments to code unless the logic is non-obvious.
EOF
```

**Windows PowerShell**

```powershell
@"
# Preferences

- I prefer concise answers — skip unnecessary introductions.
- Use TypeScript over JavaScript when both are options.
- For git commits, follow conventional commit format.
- I use VS Code as my editor.
- On Windows I use PowerShell. On macOS I use zsh.
- Do not add comments to code unless the logic is non-obvious.
"@ | Set-Content -Path "$HOME\.claude\memory\preferences.md" -Encoding UTF8
```

### Step 3 — Create patterns.md

<StepMeta purpose="Separate durable coding conventions from personal style preferences so Claude can apply both cleanly." action="Create patterns.md and record repeatable implementation patterns such as naming, validation, test placement, and control-flow preferences." expectedResult="You have a dedicated patterns file that captures conventions Claude can reuse during actual coding tasks." whyItMatters="Grouping patterns separately keeps memory easier to audit and stops style preferences from mixing with codebase-level habits." nextStep="Create a debugging notes file so recurring troubleshooting lessons are easy for Claude to recall later." askTutor="Help me rewrite any vague patterns into short, reusable rules that Claude can apply consistently during code changes." />

This file records patterns and conventions you want Claude to follow consistently. Add entries as you discover them during real work.

**macOS / Linux**

```bash
cat > ~/.claude/memory/patterns.md << 'EOF'
# Patterns

- When creating React components, use function declarations, not arrow functions.
- API routes should validate inputs with Zod at the boundary.
- File names use kebab-case, not camelCase or snake_case.
- Test files go next to the source file, named `<source>.test.ts`.
- Prefer early returns over deep nesting in functions.
EOF
```

**Windows PowerShell**

```powershell
@"
# Patterns

- When creating React components, use function declarations, not arrow functions.
- API routes should validate inputs with Zod at the boundary.
- File names use kebab-case, not camelCase or snake_case.
- Test files go next to the source file, named ``<source>.test.ts``.
- Prefer early returns over deep nesting in functions.
"@ | Set-Content -Path "$HOME\.claude\memory\patterns.md" -Encoding UTF8
```

### Step 4 — Create debugging.md

<StepMeta purpose="Preserve hard-won troubleshooting lessons so you do not re-discover the same fixes every session." action="Create debugging.md and add brief notes about known failure modes and first checks, such as BOM-damaged .gitignore files or stale Next.js caches." expectedResult="A debugging memory file exists with practical checks Claude can reuse when similar issues appear again." whyItMatters="Debugging notes are high-leverage memory because they turn prior mistakes into faster diagnosis the next time a similar problem happens." nextStep="List the memory directory contents to confirm all three topic files are present." askTutor="Suggest a few debugging notes from my recent work that are specific enough to be useful but short enough to keep memory focused." />

This file captures lessons from past debugging sessions so Claude does not repeat the same investigation steps.

**macOS / Linux**

```bash
cat > ~/.claude/memory/debugging.md << 'EOF'
# Debugging Notes

- On Windows, if .gitignore does not work, check for UTF-16 BOM encoding.
  Fix: recreate the file with UTF-8 encoding using Set-Content -Encoding UTF8.
- If Next.js dev server shows stale content, delete .next/ and restart.
- If Python imports fail after installing a package, check that the
  virtual environment is activated: `which python` should point to .venv/.
- If git push is rejected, check for a pre-push hook that blocks dirty trees.
EOF
```

**Windows PowerShell**

```powershell
@"
# Debugging Notes

- On Windows, if .gitignore does not work, check for UTF-16 BOM encoding.
  Fix: recreate the file with UTF-8 encoding using Set-Content -Encoding UTF8.
- If Next.js dev server shows stale content, delete .next/ and restart.
- If Python imports fail after installing a package, check that the
  virtual environment is activated: ``which python`` should point to .venv/.
- If git push is rejected, check for a pre-push hook that blocks dirty trees.
"@ | Set-Content -Path "$HOME\.claude\memory\debugging.md" -Encoding UTF8
```

### Step 5 — Verify all three files exist

<StepMeta purpose="Confirm the memory system is physically present before testing whether Claude uses it." action="List ~/.claude/memory and open at least one of the files so you can confirm preferences.md, patterns.md, and debugging.md were saved correctly." expectedResult="The three topic files appear in the directory listing and their contents look intact." whyItMatters="If Claude later ignores a memory entry, you need to know whether the problem is loading behavior or a file that never saved correctly." nextStep="Start a new Claude Code session and ask questions that should pull from multiple memory files." askTutor="Check my memory directory listing and tell me whether the file names and organization are set up for reliable auto-loading." />

**macOS / Linux**

```bash
ls -la ~/.claude/memory/
cat ~/.claude/memory/preferences.md
```

**Windows PowerShell**

```powershell
Get-ChildItem "$HOME\.claude\memory"
Get-Content "$HOME\.claude\memory\preferences.md"
```

You should see `preferences.md`, `patterns.md`, and `debugging.md`.

### Step 6 — Test that Claude Code loads your memory

<StepMeta purpose="Prove that the organized files are not just stored correctly but also influence Claude's reasoning in a fresh session." action="Restart Claude Code and ask one question that depends on preferences and patterns plus a second question that depends on debugging notes." expectedResult="Claude recommends my-component.ts based on your preferences and patterns, and mentions BOM encoding when asked about .gitignore on Windows." whyItMatters="A memory system is only useful if Claude can combine entries from different topic files into practical advice during new sessions." nextStep="Keep refining these files over time so the memory stays short, current, and genuinely helpful." askTutor="Evaluate Claude's answers against my memory files and tell me whether it actually used the stored preferences, patterns, and debugging notes." />

Start a new Claude Code session:

```
claude
```

Ask a question that your memory files should inform:

```
I have a file called myComponent.js. According to my preferences and patterns, 
what changes should I make to the filename?
```

Claude should recommend:
1. Rename from `.js` to `.ts` (preferences say TypeScript over JavaScript).
2. Rename from camelCase to kebab-case: `my-component.ts` (patterns say kebab-case).

This proves Claude loaded both `preferences.md` and `patterns.md`.

Then test debugging memory:

```
My .gitignore file is not ignoring anything on Windows. What should I check first?
```

Claude should mention checking for UTF-16 BOM encoding, because that is in your `debugging.md`.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude does not reference your preferences | Memory files exceed 200 lines (only the first 200 lines of user memory auto-load) | Keep each file short and focused. Move low-priority entries to the bottom |
| Claude follows old preferences you have since changed | You updated a preference but the old version still exists in the file | Open the memory file and delete or overwrite the outdated line. There should only be one entry per topic |
| Memory files are not loading | Files are not in `~/.claude/memory/` exactly | Double-check the directory path. On Windows, `~` expands to `$HOME` which is usually `C:\Users\<username>` |
| Claude mixes up project memory and user memory | You put project-specific notes in user memory | Move project-specific entries to `<project>/.claude/memory/` so they only load in that project |
| Claude treats memory entries as absolute commands | You wrote entries as rigid rules instead of preferences | Phrase entries as preferences: "I prefer X" not "Always do X." Rigid rules belong in `CLAUDE.md`, not memory |
