# Mini project: customize Claude for your workflow

## Objective

By the end of this mini project you will have a personal project set up with all three customization layers working together: a project `CLAUDE.md`, a custom skill, and organized memory files. You will start a fresh Claude Code session and verify that Claude reads all three layers by running a single end-to-end test.

## Walkthrough

### Step 1 — Create a new project

<StepMeta purpose="Start from a clean repository so you can assemble all customization layers in one controlled place." action="Create the project directory, move into it, and run git init so the repo has a clear root for local instructions and memory." expectedResult="You have a fresh project repository ready for project-level CLAUDE.md, rules, and repo memory files." whyItMatters="The end-to-end test only proves anything if all three customization layers are anchored to a real project root." nextStep="Write a project CLAUDE.md that defines the mission, boundaries, and working rules for this repo." askTutor="Help me choose a realistic practice project and confirm that I initialized the repo in the correct directory." />

Pick a real project idea — a personal website, a utility script collection, or a side project. If you do not have one in mind, create a practice project.

**macOS / Linux**

```bash
mkdir -p ~/my-side-project
cd ~/my-side-project
git init
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\my-side-project"
Set-Location "$HOME\my-side-project"
git init
```

### Step 2 — Write a project CLAUDE.md

<StepMeta purpose="Define the repo's core behavior so Claude has project-specific mission, rules, and loading boundaries from the start." action="Create CLAUDE.md at the repo root and fill it with the mission, read order, hard rules, and context loading guidance shown in the lesson." expectedResult="The project now has a root CLAUDE.md that tells Claude how to work safely and efficiently inside this repo." whyItMatters="Project instructions are the backbone of local customization because they shape how Claude approaches every task in this repository." nextStep="Add a narrower rules file for code-style conventions that should override broader defaults when needed." askTutor="Review my project CLAUDE.md and tell me whether the rules are specific enough to influence real behavior during coding tasks." />

Create a `CLAUDE.md` that describes your project. Tailor the rules to your actual needs.

**macOS / Linux**

```bash
cat > CLAUDE.md << 'EOF'
# CLAUDE.md

## Mission
A personal side project for building [describe your project here].
Optimize for working code over perfect code. Ship small increments.

## Read order
1. CLAUDE.md
2. README.md
3. Any file directly relevant to the current task

## Hard rules
1. Do not create files outside this project directory.
2. Do not install dependencies without listing them first and asking for approval.
3. Run the linter before declaring any task complete.
4. Write commit messages in conventional commit format.

## Context loading
- Read the minimum files needed for the task.
- If unsure which files to read, ask rather than scanning everything.
- Never read node_modules/, .git/, or build output directories.
EOF
```

**Windows PowerShell**

```powershell
@"
# CLAUDE.md

## Mission
A personal side project for building [describe your project here].
Optimize for working code over perfect code. Ship small increments.

## Read order
1. CLAUDE.md
2. README.md
3. Any file directly relevant to the current task

## Hard rules
1. Do not create files outside this project directory.
2. Do not install dependencies without listing them first and asking for approval.
3. Run the linter before declaring any task complete.
4. Write commit messages in conventional commit format.

## Context loading
- Read the minimum files needed for the task.
- If unsure which files to read, ask rather than scanning everything.
- Never read node_modules/, .git/, or build output directories.
"@ | Set-Content -Path "CLAUDE.md" -Encoding UTF8
```

### Step 3 — Add a .claude/rules file for project-specific conventions

<StepMeta purpose="Split detailed code-style conventions into a more specific instruction layer." action="Create .claude/rules/code-style.md and add the indentation, quoting, and export rules for this project." expectedResult="A project rules file exists that Claude can use for narrow code-style questions without cluttering the main CLAUDE.md." whyItMatters="Using .claude/rules keeps the instruction stack maintainable and demonstrates how the most specific layer can answer targeted questions later." nextStep="Verify that your custom commit-message skill is installed so the skill layer is ready for the end-to-end test." askTutor="Check whether these code-style entries belong in the rules file or whether any of them would be better placed in the main project CLAUDE.md." />

**macOS / Linux**

```bash
mkdir -p .claude/rules
cat > .claude/rules/code-style.md << 'EOF'
# Code style rules
- Use 2-space indentation for TypeScript and JSON files.
- Use double quotes for strings, not single quotes.
- Export functions as named exports, not default exports.
EOF
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path ".claude\rules"
@"
# Code style rules
- Use 2-space indentation for TypeScript and JSON files.
- Use double quotes for strings, not single quotes.
- Export functions as named exports, not default exports.
"@ | Set-Content -Path ".claude\rules\code-style.md" -Encoding UTF8
```

### Step 4 — Confirm your custom skill is installed

<StepMeta purpose="Make sure the reusable skill layer is available before you try the final integrated workflow." action="Inspect the commit-message skill file in ~/.claude/skills and confirm the folder and SKILL.md exist." expectedResult="You verify that the commit-message skill is present and ready to trigger during the end-to-end test." whyItMatters="If the skill layer is missing, the final verification will fail for the wrong reason and hide whether the local project setup works." nextStep="Add repo memory so Claude also knows project-specific facts that should persist across future sessions." askTutor="Help me verify that my commit-message skill is installed correctly and remind me what parts of the file make it trigger." />

You should already have the `commit-message` skill from lesson 12.2.2. Verify it exists:

**macOS / Linux**

```bash
cat ~/.claude/skills/commit-message/SKILL.md | head -5
```

**Windows PowerShell**

```powershell
Get-Content "$HOME\.claude\skills\commit-message\SKILL.md" | Select-Object -First 5
```

If it does not exist, go back to lesson 12.2.2 and create it.

### Step 5 — Add repo-level memory for this specific project

<StepMeta purpose="Teach Claude durable facts about this repo without polluting your cross-project user memory." action="Create .claude/memory/project-context.md and record the stack, test framework, CI system, and deploy target for this project." expectedResult="The repo has a project-context memory file that Claude can use to answer project-specific questions in future sessions." whyItMatters="Repo memory is the right home for durable project facts because it persists across sessions but stays local to this one repository." nextStep="Confirm that your broader user-memory files still exist so the end-to-end test can exercise both personal and project-specific memory." askTutor="Review my repo memory entries and tell me whether any of them belong in user memory or CLAUDE.md instead of project-context.md." />

**macOS / Linux**

```bash
mkdir -p .claude/memory
cat > .claude/memory/project-context.md << 'EOF'
# Project Context

- This project uses Node.js 20 and TypeScript 5.
- The database is SQLite for local development.
- Tests use Vitest, not Jest.
- The CI pipeline runs on GitHub Actions.
- The deploy target is Vercel.
EOF
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path ".claude\memory"
@"
# Project Context

- This project uses Node.js 20 and TypeScript 5.
- The database is SQLite for local development.
- Tests use Vitest, not Jest.
- The CI pipeline runs on GitHub Actions.
- The deploy target is Vercel.
"@ | Set-Content -Path ".claude\memory\project-context.md" -Encoding UTF8
```

### Step 6 — Verify user-level memory files exist

<StepMeta purpose="Ensure the user-memory layer is available so the final test covers both global preferences and repo-specific context." action="List ~/.claude/memory and confirm that preferences.md, patterns.md, and debugging.md are present from the previous lesson." expectedResult="You can see the user-memory files that should influence Claude across all projects." whyItMatters="The final workflow only proves full customization if global memory, repo memory, skills, and project instructions are all active together." nextStep="Start a fresh Claude Code session and run the five-part verification sequence." askTutor="Help me check whether my user-memory files are complete enough to show up clearly in the final integrated test." />

These should be in place from lesson 12.3.2. Confirm:

**macOS / Linux**

```bash
ls ~/.claude/memory/
```

**Windows PowerShell**

```powershell
Get-ChildItem "$HOME\.claude\memory"
```

You should see `preferences.md`, `patterns.md`, and `debugging.md`.

### Step 7 — Run the end-to-end verification test

<StepMeta purpose="Validate in one session that project instructions, rules files, repo memory, user memory, and skills all work together." action="Restart Claude Code in the project and run the five tests in order, checking project rules, code style, language preference, test framework, and staged-diff commit message behavior." expectedResult="Claude answers each question from the correct customization source and generates a conventional commit message for the staged file." whyItMatters="An end-to-end test is the fastest way to prove that your customization stack is complete and that each layer is being read from the right place." nextStep="Tighten any weak layer by editing the specific file that produced a wrong or missing answer, then rerun the same tests." askTutor="Walk through my five test results with me and identify which customization layer succeeded or failed on each one." />

Start a fresh Claude Code session in your project:

```
claude
```

Run this sequence of tests. Each one checks a different layer.

**Test 1 — Project CLAUDE.md** Ask:

```
What are my project's hard rules?
```

Claude should list the four rules from your `CLAUDE.md` (no files outside the project, no silent installs, run linter, conventional commits).

**Test 2 — Project rules file** Ask:

```
What indentation should I use for TypeScript files?
```

Claude should say 2-space indentation, from `.claude/rules/code-style.md`.

**Test 3 — User memory** Ask:

```
What language should I use: JavaScript or TypeScript?
```

Claude should say TypeScript, from your `~/.claude/memory/preferences.md`.

**Test 4 — Repo memory** Ask:

```
What test framework does this project use?
```

Claude should say Vitest, from `.claude/memory/project-context.md`.

**Test 5 — Custom skill** Create a test file and stage it:

**macOS / Linux**

```bash
echo "console.log('hello');" > index.ts
git add index.ts
```

**Windows PowerShell**

```powershell
"console.log('hello');" | Set-Content -Path "index.ts" -Encoding UTF8
git add index.ts
```

Then ask Claude:

```
Write a commit message for my staged changes.
```

Claude should use your `commit-message` skill and produce something like:

```
feat: add initial entry point
```

If all five tests pass, your full customization system is working.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude does not know the project rules | `CLAUDE.md` is not at the repo root, or `git init` was not run | Verify with `ls CLAUDE.md` and `git status` from the project root |
| Claude uses Jest instead of Vitest | Repo memory file is not loading | Check that `.claude/memory/project-context.md` exists inside the project directory, not in your home directory |
| Claude ignores coding style rules | Rules file is not in `.claude/rules/` at the project root | Verify the path: `ls .claude/rules/` (macOS/Linux) or `Get-ChildItem .claude\rules` (PowerShell) |
| User preferences are not applied | `~/.claude/memory/` files are missing or empty | Re-create them following lesson 12.3.2 |
| Commit message skill does not trigger | Skill directory is misnamed or `SKILL.md` has a typo | Check: `ls ~/.claude/skills/commit-message/SKILL.md` (macOS/Linux) or `Test-Path "$HOME\.claude\skills\commit-message\SKILL.md"` (PowerShell) |
| Claude seems to load everything but gives contradictory answers | Conflicting entries across layers (e.g., user memory says "tabs" but project rules say "2 spaces") | Project-level instructions win over user memory. Remove or update the conflicting entry in user memory |
| Changes to any customization file are not picked up | Claude Code was already running when you edited the file | Restart Claude Code. Customization files are read at session start |
