# Understand how Claude Code memory works

## Objective

By the end of this lesson you will understand the three scopes of Claude Code memory — user memory, session memory, and repo memory — and know what gets saved automatically versus what you save manually. You will inspect each memory location on disk and confirm which scope persists across sessions.

## Walkthrough

### Step 1 — Understand the three memory scopes

<StepMeta purpose="Build a clear model of what each memory scope is for before you start saving anything." action="Read the table and classify which kinds of facts belong in user memory, session memory, and repo memory." expectedResult="You can explain the location and persistence behavior of each memory scope without mixing them up." whyItMatters="Most memory mistakes come from saving the right fact to the wrong scope, which causes either lost context or unwanted bleed across projects." nextStep="Inspect the user memory path on disk so the abstract concept becomes a real file location." askTutor="Give me a few example facts and quiz me on which memory scope each one belongs in." />

Claude Code has three places it stores information to carry context forward:

| Scope | Location | Persists across sessions? | Persists across projects? |
|---|---|---|---|
| **User memory** | `~/.claude/memory/` | Yes | Yes — applies to all projects |
| **Session memory** | Managed by Claude Code runtime | No — cleared when the session ends | No |
| **Repo memory** | `<project>/.claude/memory/` | Yes | No — scoped to one project |

**User memory** is for preferences and insights that apply everywhere: your coding style, patterns you like, tools you prefer. Claude loads the first 200 lines of user memory automatically at session start.

**Session memory** is temporary scratch space for the current conversation. Use it for task-specific notes like tracking which files you have already reviewed. It disappears when the session ends.

**Repo memory** is for project-specific facts that should survive across sessions: build commands, environment setup notes, architecture decisions. It lives inside the project repository.

### Step 2 — Find your user memory directory

<StepMeta purpose="Verify where cross-project persistent memory lives on your machine." action="List the ~/.claude/memory directory and note whether it already contains Markdown files or needs to be created later." expectedResult="You either see existing user memory files or confirm that the directory has not been populated yet." whyItMatters="Knowing the actual disk location makes it easier to audit, edit, or troubleshoot memory entries instead of treating memory as invisible magic." nextStep="Inspect a project's repo-scoped memory location so you can compare the two persistent scopes." askTutor="Help me interpret what I am seeing in my user memory directory and what it implies about previous Claude sessions." />

**macOS / Linux**

```bash
ls -la ~/.claude/memory/ 2>/dev/null || echo "No user memory directory yet."
```

**Windows PowerShell**

```powershell
if (Test-Path "$HOME\.claude\memory") {
    Get-ChildItem "$HOME\.claude\memory"
} else {
    Write-Host "No user memory directory yet."
}
```

If the directory does not exist, Claude Code has not written any user memory yet. That is normal for a fresh installation.

### Step 3 — Find your repo memory directory

<StepMeta purpose="Compare user-level persistence to project-scoped persistence in a real repository." action="Move into a project and check whether .claude/memory exists there, then inspect any repo memory files if present." expectedResult="You can see whether the current repo already has project-specific memory notes stored under its own .claude/memory directory." whyItMatters="Repo memory is how project facts survive across sessions without leaking into unrelated work, so its location matters." nextStep="Ask Claude to save a user memory entry so you can watch persistence happen in practice." askTutor="Explain the difference between what should live in repo memory versus user memory using this project as the example." />

Navigate to any project that has been used with Claude Code:

**macOS / Linux**

```bash
cd ~/claude-md-lab
ls -la .claude/memory/ 2>/dev/null || echo "No repo memory directory yet."
```

**Windows PowerShell**

```powershell
Set-Location "$HOME\claude-md-lab"
if (Test-Path ".claude\memory") {
    Get-ChildItem ".claude\memory"
} else {
    Write-Host "No repo memory directory yet."
}
```

### Step 4 — Ask Claude Code to save a user memory

<StepMeta purpose="Create a visible test entry so you can distinguish persistent user memory from temporary session notes." action="Tell Claude to remember a personal preference that should apply across sessions, then inspect the user memory Markdown files afterward." expectedResult="A new or updated user memory note records your preference on disk under ~/.claude/memory." whyItMatters="Seeing the saved note makes memory concrete and proves that user memory is editable local data, not an opaque hidden feature." nextStep="Start a brand new Claude Code session and test whether the saved preference is recalled automatically." askTutor="Help me phrase a memory request so Claude stores it as cross-session user memory rather than temporary session context." />

Start Claude Code in any project:

```
claude
```

Tell Claude to remember something:

```
Remember that I prefer tabs over spaces for indentation.
```

Claude will save this to user memory. After it confirms, exit the session and check the file:

**macOS / Linux**

```bash
cat ~/.claude/memory/*.md 2>/dev/null
```

**Windows PowerShell**

```powershell
Get-ChildItem "$HOME\.claude\memory\*.md" | ForEach-Object { Get-Content $_.FullName }
```

You should see your preference written as a note in a Markdown file.

### Step 5 — Confirm user memory persists into a new session

<StepMeta purpose="Validate that the user memory entry survives the end of the current session." action="Restart Claude Code in a fresh session and ask about the preference you just saved without restating it." expectedResult="Claude recalls the preference from user memory and answers correctly in the new session." whyItMatters="Persistence across sessions is the defining difference between user memory and scratchpad-style session context." nextStep="Review the different ways memory can be created so you know when Claude saves something automatically versus when you should do it explicitly." askTutor="If Claude fails to remember my preference in the new session, walk me through the most likely causes in order." />

Start a brand new Claude Code session:

```
claude
```

Ask:

```
What are my indentation preferences?
```

Claude should recall that you prefer tabs over spaces, even though this is a fresh session. This proves user memory persists across sessions.

### Step 6 — Understand what is saved automatically versus manually

<StepMeta purpose="Decide when to rely on Claude's judgment versus when to manage memory deliberately yourself." action="Read the table and sort the examples into automatic saves, explicit save requests, and direct file edits." expectedResult="You understand the three save paths and can choose the right one for preferences, project facts, or one-off notes." whyItMatters="Intentional memory hygiene keeps Claude helpful without letting outdated or misplaced notes pile up across sessions." nextStep="Use the same reasoning in the next lesson to design a small, topic-based memory system that stays easy to review." askTutor="Give me a few new examples and ask whether they should be saved automatically, requested explicitly, or written directly to a memory file." />

| How it is saved | What triggers it | Example |
|---|---|---|
| **Automatically** | Claude Code decides something is worth remembering based on your interaction | You correct Claude repeatedly about naming style — Claude saves a note about your preference |
| **Manually (you ask)** | You explicitly say "Remember this" or "Save this to memory" | "Remember that this project uses PostgreSQL, not MySQL" |
| **Manually (you write)** | You create or edit a `.md` file directly in the memory directory | Adding a `debugging.md` file to `~/.claude/memory/` with your debugging checklist |

> **Safety note:** Memory files are plain Markdown on your local disk. Claude Code does not send your memory files to any server. They stay on your machine and are read locally at session start.

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude does not remember anything from the last session | Memory was stored as session memory (temporary) instead of user memory | Explicitly say "Save this to user memory" or "Remember this across sessions" |
| Memory directory is empty even after asking Claude to remember | Claude Code stored it in a different file path than expected | Search for it: `find ~/.claude -name "*.md"` (macOS/Linux) or `Get-ChildItem -Recurse "$HOME\.claude" -Filter "*.md"` (PowerShell) |
| Claude remembers something wrong | An outdated or incorrect memory entry exists | Read the memory files, find the wrong entry, and either delete it or ask Claude: "Forget that I prefer tabs — I actually prefer spaces now" |
| Repo memory shows up in the wrong project | You copied `.claude/` from one project to another | Remove the `.claude/memory/` directory from the wrong project |
| Claude loads too much memory and gets confused | Memory files are too long or unfocused | Keep memory files short and organized by topic. See the next lesson for how to structure them |
