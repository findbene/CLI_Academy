# Layer global and project instructions

## Objective

By the end of this lesson you will understand the three layers of Claude Code instructions — global `~/.claude/CLAUDE.md`, project `CLAUDE.md`, and `.claude/rules/*.md` — and prove which layer wins when they conflict. You will set up all three layers and run a conflict test so you can see the priority order firsthand.

## Walkthrough

### Step 1 — Create a global CLAUDE.md for your user account

<StepMeta purpose="Set a baseline instruction layer that follows you into every project." action="Create ~/.claude/CLAUDE.md with a few clear global preferences about tone, format, and answer length." expectedResult="A global CLAUDE.md exists in your home directory and contains default preferences Claude can apply everywhere." whyItMatters="You need a global baseline before you can observe how project-level instructions override it." nextStep="Create a project CLAUDE.md that intentionally conflicts with one of these global preferences." askTutor="Review my global CLAUDE.md and tell me whether these entries read like reusable preferences instead of project-specific rules." />

The global file lives at `~/.claude/CLAUDE.md`. It applies to every project you open with Claude Code.

**macOS / Linux**

```bash
mkdir -p ~/.claude
cat > ~/.claude/CLAUDE.md << 'EOF'
# Global CLAUDE.md

## Preferences
- Always respond in plain English.
- Use bullet lists for multi-item answers.
- Default to concise answers (under 200 words).
EOF
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\.claude"
@"
# Global CLAUDE.md

## Preferences
- Always respond in plain English.
- Use bullet lists for multi-item answers.
- Default to concise answers (under 200 words).
"@ | Set-Content -Path "$HOME\.claude\CLAUDE.md" -Encoding UTF8
```

### Step 2 — Create a project CLAUDE.md with a conflicting rule

<StepMeta purpose="Introduce a deliberate conflict so the instruction precedence is visible instead of theoretical." action="Create a practice repo and add a project CLAUDE.md whose response-format rule contradicts the global preference you just wrote." expectedResult="You have a project-level instruction file that clearly conflicts with the global rule about using bullet lists." whyItMatters="Without a real conflict, you cannot tell which layer won because both layers could be satisfied at once." nextStep="Add a separate rules file so you can test all three layers, not just global versus project." askTutor="Help me choose a cleaner conflict example if my global and project rules are still too similar to prove precedence." />

Navigate to your practice project (or create one) and write a project-level file that deliberately contradicts the global preference.

**macOS / Linux**

```bash
mkdir -p ~/layering-lab && cd ~/layering-lab && git init
cat > CLAUDE.md << 'EOF'
# CLAUDE.md

## Mission
A test project for instruction layering.

## Hard rules
1. Always respond in numbered paragraphs, not bullet lists.
2. Do not modify files outside this project.
EOF
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\layering-lab"
Set-Location "$HOME\layering-lab"
git init
@"
# CLAUDE.md

## Mission
A test project for instruction layering.

## Hard rules
1. Always respond in numbered paragraphs, not bullet lists.
2. Do not modify files outside this project.
"@ | Set-Content -Path "CLAUDE.md" -Encoding UTF8
```

Notice the conflict: global says "Use bullet lists," but the project says "Use numbered paragraphs."

### Step 3 — Add a rules file for fine-grained project rules

<StepMeta purpose="Add the most specific project instruction layer so you can test the full stacking model." action="Create .claude/rules/naming.md at the project root and place one or two narrowly scoped naming conventions inside it." expectedResult="The project now has a dedicated rules file that Claude can cite separately from the main project CLAUDE.md." whyItMatters="Keeping specific conventions in rule files makes large projects easier to maintain and lets the most specific rule win when needed." nextStep="Start Claude Code in the repo and ask questions that exercise both the formatting conflict and the naming rule." askTutor="Check whether my rules file is in the correct location and whether its conventions are specific enough to test separately from CLAUDE.md." />

Claude Code also reads any `.md` files inside `.claude/rules/`. This is where you put rules you want to keep separate from the main `CLAUDE.md`.

**macOS / Linux**

```bash
mkdir -p .claude/rules
cat > .claude/rules/naming.md << 'EOF'
# Naming conventions
- Use kebab-case for all filenames.
- Prefix test files with `test-`.
EOF
```

**Windows PowerShell**

```powershell
New-Item -ItemType Directory -Force -Path ".claude\rules"
@"
# Naming conventions
- Use kebab-case for all filenames.
- Prefix test files with ``test-``.
"@ | Set-Content -Path ".claude\rules\naming.md" -Encoding UTF8
```

### Step 4 — Test the layering by asking Claude Code

<StepMeta purpose="Observe which instruction source Claude actually follows when multiple layers are active." action="Launch Claude Code from the layering-lab repo, ask for a multi-item answer, then ask about naming conventions and inspect which formatting and rules Claude uses." expectedResult="Claude formats the first answer according to the project rule and cites the naming rule from .claude/rules/ when asked directly." whyItMatters="A live conflict test is the fastest way to prove that the project layer overrides global preferences while the rules folder is also active." nextStep="Write down the precedence order that matches the behavior you just observed." askTutor="Help me interpret Claude's responses and map each part of the answer back to the instruction layer that produced it." />

Start Claude Code in the `layering-lab` directory:

```
claude
```

Ask this question:

```
List three benefits of version control.
```

Watch the format of Claude's response. If the project rule is working, Claude should use numbered paragraphs instead of bullet points — because project-level instructions override global preferences when they conflict.

Then ask:

```
What are my naming conventions?
```

Claude should cite the kebab-case rule from `.claude/rules/naming.md`. This proves all three layers are loading.

### Step 5 — Understand the priority order

<StepMeta purpose="Turn the test results into a mental model you can reuse when designing future instruction stacks." action="Compare the observed behavior to the documented order and restate the precedence as rules files over project CLAUDE.md over global CLAUDE.md." expectedResult="You can explain which layer wins in a conflict and why Claude behaved the way it did in the earlier test." whyItMatters="Once you understand the ordering, you can put defaults globally, project-wide rules in CLAUDE.md, and narrow conventions in .claude/rules without surprises." nextStep="Use this model in your own projects to decide what belongs in global preferences versus project-local instructions." askTutor="Quiz me on a few conflict scenarios so I can prove I understand which instruction layer should win." />

The layering works like this (highest priority first):

1. **`.claude/rules/*.md`** — Project-specific rule files. Most specific wins.
2. **`CLAUDE.md` at project root** — Project-wide instructions.
3. **`~/.claude/CLAUDE.md`** — Global user preferences. Lowest priority.

When a project rule contradicts a global preference, the project rule wins. When a rules file contradicts the project `CLAUDE.md`, the rules file wins.

This is useful because you can set comfortable defaults globally (like "be concise" or "use plain English") and override them per project when needed (like "use formal language" for a work project).

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Claude uses bullets despite the project rule saying numbered paragraphs | Project `CLAUDE.md` is not at the repo root | Run `ls CLAUDE.md` (macOS/Linux) or `Test-Path CLAUDE.md` (PowerShell) from the repo root to confirm the file exists there |
| Claude does not know about naming conventions | `.claude/rules/` folder is in the wrong location | The folder must be at the project root: `<repo>/.claude/rules/`, not `~/.claude/rules/` |
| Global preferences do not apply in a new project | `~/.claude/CLAUDE.md` does not exist or has a typo in the path | Run `cat ~/.claude/CLAUDE.md` (macOS/Linux) or `Get-Content "$HOME\.claude\CLAUDE.md"` (PowerShell) to confirm it exists |
| Rules file is ignored | File does not have a `.md` extension | Claude Code only reads Markdown files from `.claude/rules/`. Rename any `.txt` files to `.md` |
| Changes to CLAUDE.md do not take effect | Claude Code session was already running | Restart Claude Code. It reads instruction files at session start, not dynamically during a session |
