# Create a contribution guide or onboarding doc

## Objective

By the end of this lesson you will use Claude Code to generate a `CONTRIBUTING.md` for a real project that covers environment setup, pull request process, and coding standards — all verified against the actual project structure and tooling.

## Walkthrough

### Step 1 — Prepare a project with real tooling to describe

<StepMeta purpose="Create a small but believable repo so the contribution guide can describe real setup steps, tests, and code conventions." action="Build the sample project, add tests and ignore rules, then commit the baseline so Claude has actual tooling and structure to inspect." expectedResult="You end with a contrib-lab repo containing app.py, tests/test_app.py, and .gitignore committed to Git." whyItMatters="A contribution guide only works when it mirrors the repo contributors will actually clone, so the underlying project needs real commands and files first." nextStep="Ask Claude to draft CONTRIBUTING.md from the full repo and explicitly forbid invented workflows or tools." askTutor="I'm on Step 1 of Create a contribution guide or onboarding doc. Help me make sure this practice repo has enough real structure for a truthful CONTRIBUTING.md." />

You need a project that has enough structure for a meaningful contribution guide. Reuse the `readme-lab` project from lesson 9.1.1, or create a fresh one.

**macOS / Linux:**

```bash
mkdir contrib-lab && cd contrib-lab
git init

cat > app.py << 'EOF'
"""Bookmark manager — save and list URLs."""
import json, os

DB = "bookmarks.json"

def load():
    if os.path.exists(DB):
        with open(DB) as f:
            return json.load(f)
    return []

def save(data):
    with open(DB, "w") as f:
        json.dump(data, f, indent=2)

def add(url, tag="general"):
    bm = load()
    bm.append({"url": url, "tag": tag})
    save(bm)

if __name__ == "__main__":
    import sys
    if len(sys.argv) >= 2:
        add(sys.argv[1], sys.argv[2] if len(sys.argv) > 2 else "general")
    else:
        for b in load():
            print(f"  [{b['tag']}] {b['url']}")
EOF

echo "bookmarks.json" > .gitignore

mkdir tests
cat > tests/test_app.py << 'EOF'
import app, os, json

def test_add_and_load():
    app.DB = "test_bookmarks.json"
    app.add("https://example.com", "test")
    data = app.load()
    assert len(data) == 1
    assert data[0]["url"] == "https://example.com"
    os.remove("test_bookmarks.json")
EOF

git add -A
git commit -m "Initial bookmark manager with tests"
```

### Step 2 — Ask Claude to draft a CONTRIBUTING.md from the real project

<StepMeta purpose="Generate a first-pass contribution guide that is grounded in the repo instead of generic open source advice." action="Open Claude in the project, ask it to read the files, and request a CONTRIBUTING.md covering prerequisites, setup, tests, PR process, and coding standards." expectedResult="Claude produces a draft contribution guide tailored to the actual files and commands present in the repo." whyItMatters="The strongest contribution docs are operationally specific, and this prompt sets the expectation that Claude must describe the project as it exists today." nextStep="Follow the guide yourself as if you were a new contributor and note any instructions that fail or reference missing files." askTutor="I'm on Step 2 of Create a contribution guide or onboarding doc. Help me write a prompt that makes Claude draft CONTRIBUTING.md from the real repo instead of best-practice boilerplate." />

Open Claude Code in the `contrib-lab` directory and use this prompt:

```
Read all files in this project. Write a CONTRIBUTING.md that includes:
1. Prerequisites (Python version, any tools)
2. Environment setup (clone, install, run)
3. How to run tests
4. Pull request process (branch naming, commit messages, review)
5. Coding standards (file naming, import style, docstrings)

Base every instruction on what actually exists in this repo — do not invent tools or workflows that are not present.
```

Claude reads `app.py`, `tests/test_app.py`, and `.gitignore`, then produces a draft.

### Step 3 — Walk through the setup steps yourself

<StepMeta purpose="Test the guide from a contributor's point of view before trusting that the setup instructions are usable." action="Run the setup and test commands Claude documented and compare each step to what the repo actually supports." expectedResult="You discover whether the draft's environment setup and test instructions work literally when followed." whyItMatters="Contributor docs fail most often in setup, and manually following the guide is the fastest way to catch invented dependencies or broken commands." nextStep="After the setup instructions are accurate, inspect the workflow section so branch and review guidance also matches the repo." askTutor="I'm on Step 3 of Create a contribution guide or onboarding doc. Help me verify these setup steps like a brand-new contributor and spot instructions that don't really work." />

The most common error in contribution guides is setup instructions that do not work. Follow every step Claude wrote as if you were a new contributor.

**macOS / Linux:**

```bash
# Does the test command Claude listed actually work?
python -m pytest tests/
```

**Windows PowerShell:**

```powershell
# Does the test command Claude listed actually work?
python -m pytest tests/
```

If Claude wrote `pip install -r requirements.txt` but no `requirements.txt` exists (this project has no external dependencies), that instruction is wrong. Tell Claude:

```
There is no requirements.txt in this project. Remove the pip install step
and note that the project uses only Python standard library modules.
```

### Step 4 — Verify the PR workflow section

<StepMeta purpose="Make sure the workflow guidance matches the repo's real branch and review reality rather than a default GitHub template." action="Check the current branch state and compare it to the PR and branching process described in CONTRIBUTING.md." expectedResult="You confirm whether the workflow section uses the correct branch names and only references processes that actually exist." whyItMatters="Process guidance becomes misleading quickly when it assumes CI, branch names, or review gates that the repo does not use." nextStep="Compare the coding standards section to the codebase so style guidance is just as honest as the workflow guidance." askTutor="I'm on Step 4 of Create a contribution guide or onboarding doc. Help me review the PR workflow section and correct any branch or process assumptions Claude made." />

Check that the branching and commit message conventions Claude described are realistic for a solo or small-team project. Common mistakes:

- Describing a CI pipeline that does not exist
- Referencing a linter config that is not in the repo
- Naming a default branch `main` when `git branch` shows it is `master`

Verify:

```bash
git branch
```

If the output shows `master` but CONTRIBUTING.md says `main`, ask Claude to correct it.

### Step 5 — Check the coding standards section against real code

<StepMeta purpose="Align the documented coding standards with what contributors will actually see in the codebase." action="Open app.py, compare the style guidance in CONTRIBUTING.md to the current code, and decide whether to soften the guide or improve the code." expectedResult="The coding standards section accurately reflects the repo's present conventions or an explicitly chosen standard you are ready to enforce." whyItMatters="Standards docs lose credibility when they describe an idealized codebase instead of the one contributors will open on day one." nextStep="When the guide is truthful about setup, workflow, and style, stage and commit the finished CONTRIBUTING.md." askTutor="I'm on Step 5 of Create a contribution guide or onboarding doc. Help me compare the coding standards section to the actual code and decide what should change." />

Open `app.py` and compare what Claude wrote about coding standards to what the code actually does.

```bash
head -20 app.py
```

If Claude says "all functions must have docstrings" but `add()` and `load()` have none, either:
- Ask Claude to soften the standard to match reality, or
- Add the docstrings first, then keep the standard

Pick whichever is honest. The contribution guide must not describe a codebase that does not exist.

### Step 6 — Commit the verified CONTRIBUTING.md

<StepMeta purpose="Record the contribution guide only after every instruction has been checked against the repo." action="Stage CONTRIBUTING.md, inspect the staged diff, and commit the verified contributor documentation." expectedResult="You create a commit containing a contribution guide that has already been tested against the repo's real setup and workflow." whyItMatters="Treating onboarding docs like production artifacts builds the habit that contributor guidance should be validated, not merely drafted." nextStep="Use the same Cowork pattern for research-heavy tasks in the next lesson, where the output is a factual report instead of repo documentation." askTutor="I'm on Step 6 of Create a contribution guide or onboarding doc. Help me do a final review of CONTRIBUTING.md before I commit it." />

```bash
git add CONTRIBUTING.md
git diff --staged
git commit -m "Add CONTRIBUTING.md generated with Claude"
```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Setup instructions fail for a new clone | Claude assumed dependencies or config files that do not exist | Follow every step on a clean directory and feed errors back to Claude |
| PR section references CI checks that are not configured | Claude added generic GitHub Actions advice | Remove CI references or tell Claude to mark them as "planned, not yet configured" |
| Coding standards contradict actual code | Claude described best practices instead of your real conventions | Point Claude at specific files and ask it to describe what is there, not what should be |
| Test command fails | Claude guessed the test runner or path | Run the test command, share the error output, and ask Claude to correct |
| Default branch name is wrong | Claude assumed `main` but repo uses `master` or vice versa | Run `git branch` and share the output with Claude |
