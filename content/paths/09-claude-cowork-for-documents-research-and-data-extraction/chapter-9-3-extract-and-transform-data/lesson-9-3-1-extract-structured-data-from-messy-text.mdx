# Extract structured data from messy text

## Objective

By the end of this lesson you will take messy, unstructured text (meeting notes), ask Claude Code to extract names, dates, and action items, output the results as structured Markdown and JSON, and verify the extraction by comparing it against the original text.

## Walkthrough

### Step 2 — Ask Claude to extract structured data

<StepMeta purpose="Turn unstructured notes into repeatable outputs by giving Claude a strict target format for both human and machine readers." action="Prompt Claude to read meeting-notes.txt and create extracted.md plus extracted.json using the exact sections and JSON schema provided." expectedResult="Claude generates one Markdown summary and one JSON file that attempt to capture attendees, dates, actions, decisions, and the next meeting." whyItMatters="Explicit output contracts reduce ambiguity and make it much easier to verify whether Claude followed your extraction and normalization instructions." nextStep="Review the Markdown output first so you can compare each extracted section against the original notes." askTutor="I'm on Step 2 of Extract structured data from messy text. Help me tighten this extraction prompt so Claude outputs clean Markdown and valid JSON with the fields I need." />

Open Claude Code in the `extract-lab` directory:

```
Read meeting-notes.txt and extract two outputs:

1. A file called extracted.md with these sections:
   ## Attendees (full name, attendance mode)
   ## Dates mentioned (date, context)
   ## Action items (owner, task, deadline, blockers)
   ## Decisions made
   ## Next meeting

2. A file called extracted.json with the same data in JSON format using
   this structure:
   {
     "meeting_date": "...",
     "attendees": [{"name": "...", "mode": "in-person|remote"}],
     "action_items": [{"owner": "...", "task": "...", "deadline": "...", "blockers": "..."}],
     "decisions": ["..."],
     "next_meeting": "..."
   }

Normalize all dates to YYYY-MM-DD format. Expand abbreviations
(e.g., "weds" → Wednesday, "tmrw" → tomorrow). Use full names where
the notes use nicknames or initials.
```

### Step 3 — Verify the Markdown extraction

<StepMeta purpose="Check the human-readable extraction for omissions, invented details, and lost context before trusting the structured data." action="Open extracted.md and compare the attendees, dates, action items, blockers, and next-meeting details against the original notes." expectedResult="You confirm whether the Markdown extraction captured the right people, dates, and exactly five action items without inventing missing details." whyItMatters="The Markdown view makes semantic mistakes easy to spot, which helps you catch bad assumptions before you move on to JSON validation." nextStep="Once the Markdown content looks right, validate the JSON syntax and field formatting so the machine-readable output is usable too." askTutor="I'm on Step 3 of Extract structured data from messy text. Help me audit extracted.md and identify anything Claude missed or invented from the meeting notes." />

```bash
cat extracted.md
```

Check each section against the original notes:

1. **Attendees:** The notes say "J. Chen" — did Claude expand this to a full name it guessed, or keep "J. Chen"? It should keep what is in the text, not invent a first name.
2. **Dates:** The notes mention Oct 14, 15, 16, 18, and 21. Count them in the extraction. If Claude missed one or added one that is not in the text, that is an error.
3. **Action items:** Count them. The notes list 5 action items. The extraction should have exactly 5.
4. **Blockers:** Marcus's API task is blocked on Sarah's design specs. If Claude missed this dependency, ask for a correction.

### Step 4 — Verify the JSON extraction

<StepMeta purpose="Confirm that the machine-readable output is both syntactically valid and normalized the way you requested." action="Open extracted.json, parse it with Python, and inspect whether the dates, fields, and structure match the schema and formatting rules." expectedResult="You know whether the JSON is valid, dates are normalized, and the extracted objects match the requested structure." whyItMatters="Structured data is only useful if downstream tools can parse it, so syntax and schema validation are just as important as content accuracy." nextStep="After the JSON parses, do a final hallucination check by matching each extracted value back to the source text." askTutor="I'm on Step 4 of Extract structured data from messy text. Help me validate this JSON output and figure out whether any fields still violate the requested schema." />

**macOS / Linux:**

```bash
cat extracted.json
python3 -c "import json; d=json.load(open('extracted.json')); print(json.dumps(d, indent=2))"
```

**Windows PowerShell:**

```powershell
Get-Content extracted.json
python -c "import json; d=json.load(open('extracted.json')); print(json.dumps(d, indent=2))"
```

The Python command validates that the JSON is syntactically correct. If it throws a `json.JSONDecodeError`, Claude produced malformed JSON. Share the error with Claude:

```
extracted.json has a JSON syntax error. Here is the error:
[paste the error]
Fix the JSON and save it again.
```

Also check that dates are in `YYYY-MM-DD` format as requested. If any date is still in the original format (e.g., "oct 15"), ask Claude to normalize it.

### Step 5 — Cross-check for hallucinated data

<StepMeta purpose="Make sure every extracted value traces back to the source instead of Claude filling in gaps with plausible guesses." action="Search the source notes for each extracted name, date, and task, then remove any value that cannot be directly justified from the original text." expectedResult="The final extracted files contain only data points supported by the meeting notes and no invented expansions or tasks." whyItMatters="Hallucinated fields are especially dangerous in extraction workflows because they look structured and authoritative even when they are false." nextStep="When every field is grounded in the source, stage the Markdown and JSON outputs and commit the verified extraction." askTutor="I'm on Step 5 of Extract structured data from messy text. Help me check this extraction for hallucinated names, dates, or tasks that don't actually appear in the source notes." />

This is the most important verification step. Go through `extracted.json` field by field and confirm every value appears in the original `meeting-notes.txt`.

**macOS / Linux:**

```bash
# Search for any name Claude added that is not in the original
grep -i "sarah\|marcus\|chen\|priya" meeting-notes.txt
```

**Windows PowerShell:**

```powershell
Select-String -Pattern "sarah|marcus|chen|priya" -Path meeting-notes.txt
```

If Claude added a name, date, or action item that does not appear in the source text, that is a hallucination. Remove it:

```
The extracted data includes [invented item] but that does not appear
in meeting-notes.txt. Remove it.
```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| JSON fails to parse | Claude produced invalid JSON (trailing commas, unquoted keys, etc.) | Run the Python validation command and share the exact error with Claude |
| Claude invented a full name for "J. Chen" | LLMs fill in gaps with plausible data | Tell Claude to use only names exactly as they appear in the source, or mark expanded names with "[assumed]" |
| An action item is missing from the extraction | Claude missed an item buried in prose (not in the bullet list) | Point Claude to the specific line in the source text and ask it to add the missing item |
| Dates are not normalized to YYYY-MM-DD | Claude skipped the normalization instruction | Re-prompt: "Normalize all dates to YYYY-MM-DD. Oct 15 2025 → 2025-10-15" |
| Extra action items appear that are not in the source | Claude inferred tasks from discussion context | Ask Claude to only extract items explicitly stated as tasks or action items, not implied ones |
