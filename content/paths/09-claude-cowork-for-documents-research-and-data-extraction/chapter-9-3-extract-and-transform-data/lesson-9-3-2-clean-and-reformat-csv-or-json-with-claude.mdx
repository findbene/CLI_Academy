# Clean and reformat CSV or JSON with Claude

## Objective

By the end of this lesson you will take a messy CSV file with inconsistent dates, duplicate rows, and mixed formatting, ask Claude Code to clean it, and manually verify the output by comparing row counts and spot-checking specific fields.

## Walkthrough

### Step 2 — Count rows before cleaning

### Step 3 — Ask Claude to clean the CSV

<StepMeta purpose="Turn the raw export into a controlled cleaning task with explicit transformation rules and an audit trail." action="Prompt Claude to create contacts-clean.csv and cleaning-log.md while applying the listed normalization, deduplication, and missing-value rules." expectedResult="Claude produces a cleaned CSV plus a log describing the changes it claims to have made." whyItMatters="Clear cleaning rules help you evaluate Claude's output objectively because you can check each rule one by one instead of guessing what 'clean' should mean." nextStep="Inspect the cleaned file and confirm the deduplicated row count matches the number of unique contacts you expect." askTutor="I'm on Step 3 of Clean and reformat CSV or JSON with Claude. Help me write a prompt that makes Claude clean the CSV systematically and produce a useful change log." />

Open Claude Code in the `csv-clean-lab` directory:

```
Read contacts.csv and produce a cleaned version called contacts-clean.csv.

Apply these cleaning rules:
1. Normalize all names to Title Case (e.g., "jane doe" → "Jane Doe")
2. Lowercase all email addresses and trim whitespace
3. Normalize all dates to YYYY-MM-DD format
4. Normalize city names to Title Case
5. Remove exact duplicate rows (same person appearing twice)
6. Remove near-duplicate rows (same name+email, different formatting)
7. Format phone numbers as 555-XXXX (with hyphen)
8. Mark empty fields with "MISSING" instead of leaving them blank

Also create a cleaning-log.md that lists every change you made, grouped
by type (duplicates removed, dates normalized, etc.) with row numbers.
```

### Step 4 — Verify row count after deduplication

<StepMeta purpose="Check whether Claude removed duplicates correctly before you spend time on field-level formatting review." action="Open contacts-clean.csv, count the lines again, and compare the deduplicated total to the six unique contacts you expect to keep." expectedResult="You can tell whether the cleaned file has the correct seven total lines: one header and six unique data rows." whyItMatters="Row-count verification catches major merge and deduplication errors fast, especially when near-duplicates could be removed incorrectly." nextStep="After the row count matches expectations, spot-check specific rows to confirm dates, emails, cities, and missing fields were normalized correctly." askTutor="I'm on Step 4 of Clean and reformat CSV or JSON with Claude. ```bash


Count the data rows. The original had 10 data rows with these duplicates:
- "John Smith" appears twice (exact duplicate on rows 2 and 4)
- "jane doe" appears twice (rows 2 and 7 — near-duplicate with different formatting)
- "SARAH CONNOR" / "sarah connor" appear twice (rows 3 and 10 — near-duplicate)
- "Marcus Johnson" appears twice (rows 5 and 9 — near-duplicate)

After deduplication you should have 5 unique contacts: John Smith, Jane Doe, Sarah Connor, Marcus Johnson, Priya Patel, and Wei Zhang — 6 people total.

**Windows PowerShell:**

### Step 5 — Spot-check specific fields

<StepMeta purpose="Verify that the cleaned rows actually follow the formatting rules, not just the expected row count." action="Search for specific contacts and missing values, then confirm dates, email casing, city formatting, and MISSING markers match the requested rules." expectedResult="You confirm that representative rows like Sarah Connor, Marcus Johnson, and Wei Zhang were normalized correctly." whyItMatters="A correct row count can still hide bad field-level decisions, so spot checks make sure the retained rows are also clean and usable." nextStep="Review the change log next so you can see whether Claude documented the transformations with enough detail to audit them." askTutor="I'm on Step 5 of Clean and reformat CSV or JSON with Claude. Verify the cleaning rules were applied correctly for at least 3 rows.

**macOS / Linux:**

```bash

```

**Windows PowerShell:**

```powershell

```

The date for Sarah Connor should be `2024-04-01` (normalized from both `2024-04-01` and `04-01-2024`). The email should be lowercase: `sarah.connor@example.com`. The city should be `Los Angeles` in title case.

Check the fields Claude marked as MISSING:

**macOS / Linux:**

```bash
grep "MISSING" contacts-clean.csv
```

**Windows PowerShell:**

```powershell
Select-String -Pattern "MISSING" contacts-clean.csv
```

Marcus Johnson should have `MISSING` in the Phone column. Wei Zhang should have `MISSING` in the City column.

### Step 6 — Review the cleaning log

<StepMeta purpose="Make the cleaning process auditable by checking whether Claude documented what changed, where, and why." action="Open cleaning-log.md and inspect whether each transformation includes the original value, the new value, and the original row reference." expectedResult="You know whether the cleaning log is specific enough to justify the cleaned output and explain every deduplication or normalization decision." whyItMatters="A cleaning log turns the result from a black box into a reviewable process, which matters whenever data cleaning decisions could remove or alter information." nextStep="When the output and audit log are both solid, stage them together and review the final diff before committing." askTutor="I'm on Step 6 of Clean and reformat CSV or JSON with Claude. Help me judge whether this cleaning log is specific enough to audit every change Claude made." />

```bash
cat cleaning-log.md
```

The log should list specific changes, not vague summaries. Good: "Row 2: changed date from `03/15/2024` to `2024-03-15`." Bad: "Normalized some dates."

If the log is vague, ask Claude to be specific:

```
The cleaning log is too vague. For each change, list the original value
and the new value with the row number from the original file.
```

### What can go wrong

| Symptom | Cause | Fix |
|---|---|---|
| Row count after cleaning is wrong | Claude removed too many or too few duplicates | Ask Claude to list every row it removed and why, then verify each removal decision |
| Dates are still in mixed formats | Claude normalized some dates but missed others | Search for non-YYYY-MM-DD patterns with `grep -v "2024-"` and share the results with Claude |
| Claude merged data from near-duplicate rows incorrectly | When two rows have conflicting field values, Claude chose arbitrarily | Ask Claude which value it kept for each conflict and add a decision rule (e.g., "keep the more complete row") |
| Missing fields are blank instead of marked MISSING | Claude overlooked the MISSING instruction | Search for empty fields with `grep ",,"` and ask Claude to fill them |
| Phone numbers still have inconsistent formats | Claude missed the normalization rule | Point out specific rows and ask Claude to re-format |
| The cleaning log is generic ("cleaned data") | Claude summarized instead of logging specifics | Ask for per-row change tracking with original and new values |
