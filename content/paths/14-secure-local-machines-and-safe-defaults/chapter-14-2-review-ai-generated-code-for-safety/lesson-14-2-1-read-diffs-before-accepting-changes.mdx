# Read diffs before accepting changes

## Objective

When Claude Code suggests a change to your project, it shows you a diff — a line-by-line comparison of what will change. Most beginners accept every diff without reading it. That is how `eval()` calls, hardcoded URLs, removed input validation, and accidentally deleted security checks end up in production code.

In this lesson you will:

1. Learn to read `git diff` output fluently.
2. Practice spotting subtle security problems in real-looking diffs.
3. Build a personal diff-review checklist you will use every time before accepting AI-generated changes.

Reading diffs is your last line of defense before code enters your project.

### Step 4 — Read the diff

<StepMeta purpose="Practice the core review habit of reading additions and removals line by line before accepting a change." action="Run `git diff server.js`, then separate removed safety lines from newly added risky lines using the red-minus and green-plus mental model." expectedResult="You can point to the exact validation that disappeared and the exact new behavior that was introduced." whyItMatters="The diff is where harmful tradeoffs become visible, especially when the final file still looks clean and coherent." nextStep="Name the specific security problems hidden in the diff and explain why each one matters." askTutor="Walk me through this diff line by line and help me label each removal or addition as safe, risky, or suspicious." />

Now read the diff exactly as you would when reviewing an AI suggestion:

```bash
git diff server.js
```

You will see output like this:

```diff
-  const userId = parseInt(req.query.id, 10);
-  if (isNaN(userId) || userId < 1) {
-    return res.status(400).json({ error: 'Invalid user ID' });
-  }
-  // Look up user by validated integer ID
-  res.json({ id: userId, name: 'Example User' });
+  const userId = req.query.id;
+  // Fetch user data with flexible query
+  const query = `SELECT * FROM users WHERE id = ${userId}`;
+  res.json({ id: userId, name: 'Example User', query });
```

**How to read this:**
- Lines starting with `-` (red) are being **removed**.
- Lines starting with `+` (green) are being **added**.
- Lines with no prefix are unchanged context.

### Step 5 — Spot the three security problems

<StepMeta purpose="Translate the raw diff into a concrete security diagnosis instead of a vague feeling that something looks off." action="Identify the removed input validation, the SQL injection pattern, the `eval()` endpoint, and the hardcoded URL bonus issue." expectedResult="You can explain each risky line in plain language and tie it to the class of vulnerability it introduces." whyItMatters="Security review improves when you can name the problem category quickly rather than only noticing that code feels unusual." nextStep="Turn those observations into a reusable diff-review checklist that you can apply to every future AI patch." askTutor="Check my list of security problems from this diff and tell me whether I missed any hidden issue or misclassified one of the risks." />

Go through the diff line by line. Here is what to look for:

**Problem 1 — Input validation removed:**
The original code had `parseInt()` and a validity check. The new code uses `req.query.id` directly — a raw, unvalidated string from the user. This is the most common AI mistake: removing safety code to "simplify" things.

**Problem 2 — SQL injection introduced:**
The line `` `SELECT * FROM users WHERE id = ${userId}` `` builds a SQL query by concatenating user input. An attacker could send `?id=1; DROP TABLE users--` and destroy your database. The fix is parameterized queries.

**Problem 3 — `eval()` on user input:**
The new `/api/config` endpoint runs `eval(req.query.data)`. This lets anyone execute arbitrary JavaScript on your server. `eval()` on user-controlled input is the single most dangerous line of code possible in Node.js.

**Bonus — Hardcoded URL:**
`http://api.example.com:3000` should not be hardcoded. Use an environment variable instead.

### Step 6 — Practice the diff-review checklist

<StepMeta purpose="Convert a one-off exercise into a repeatable review habit." action="Read and internalize the five-question checklist, especially the checks for removed validation, dangerous execution, input use, and unknown dependencies." expectedResult="You have a short practical review routine you can run before accepting any future AI-generated code change." whyItMatters="Good security review depends on consistency, and a checklist reduces the chance that you will skip the same type of bug under time pressure." nextStep="Discard the bad change and verify you are back on the safe baseline so the practice repo ends in a clean state." askTutor="Turn this lesson into a compact review checklist I could keep beside my terminal when I inspect AI diffs." />

Before accepting any AI-generated change, check every diff for these patterns:

1. **Was validation removed?** Look for deleted `if` blocks, `parseInt()`, `isNaN()`, regex checks, or type guards.
2. **Is user input used in a dangerous context?** Watch for string concatenation in SQL, shell commands, HTML output, or file paths.
3. **Does the diff add `eval()`, `Function()`, or `exec()`?** These execute arbitrary code. Almost never needed.
4. **Are there hardcoded URLs, IPs, or credentials?** These should be environment variables.
5. **Are there new dependencies or imports you don't recognize?** Verify they exist and are legit.

### Step 7 — Reset and practice reviewing a clean fix

<StepMeta purpose="End the exercise by restoring the known-safe version and verifying that your repo state is clean again." action="Use `git checkout -- server.js`, then run `git diff server.js` and confirm the output is empty." expectedResult="The dangerous change is gone and the working copy matches the original secure baseline." whyItMatters="Resetting bad experiments cleanly is part of safe review practice, because it proves you can reject a risky diff without leaving residue behind." nextStep="Use the next lesson to study repeated AI code failure patterns so you can recognize them faster than one diff at a time." askTutor="Interpret my empty or non-empty post-reset diff and tell me whether I fully restored the safe baseline or still have leftover edits." />

Discard the dangerous change:
```bash
git checkout -- server.js
```

Verify you are back to the safe version:
```bash
git diff server.js
```
This should show no output — meaning your working copy matches the last commit.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `git diff` shows nothing | Changes were already staged with `git add` | Use `git diff --staged` to see staged changes |
| Diff is too long to read in terminal | Large file with many changes | Use `git diff --stat` for a summary first, then review file by file |
| You accepted a bad change already | Committed without reviewing | Run `git diff HEAD~1` to review the last commit, then `git revert HEAD` to undo it |
| AI change looks safe but removes an edge case | AI simplified code and dropped a null check or error boundary | Compare removed lines (starting with `-`) against your original intent — any removed `if` or `try/catch` is a red flag |
| You are not sure if a function is dangerous | Unfamiliar function name in the diff | Search for the function name plus "security" before accepting — e.g., "eval javascript security" |

> [!WARN]
> **Never accept a diff you have not read.** AI tools are optimized to produce code that *looks* correct. They regularly remove input validation, add `eval()`, hardcode secrets, and introduce SQL injection — all while appearing to improve your code. The diff is your last chance to catch these before they ship.

## Key takeaways

- Lines with `-` are being removed — check if any safety code is being deleted.
- Lines with `+` are being added — check for `eval()`, string concatenation in queries, hardcoded secrets, and unvalidated input.
- Use `git diff` before accepting, `git diff --staged` before committing, and `git diff HEAD~1` after committing.
- Build the habit: read every diff, every time. No exceptions.
