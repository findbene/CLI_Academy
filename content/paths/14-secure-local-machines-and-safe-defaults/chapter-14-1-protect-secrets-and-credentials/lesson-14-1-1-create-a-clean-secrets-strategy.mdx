# Create a clean secrets strategy

## Objective

Accidentally committing a secret — an API key, a database password, a token — is one of the most common and most dangerous mistakes in software. Once a secret hits a public Git repository, automated bots can find it within minutes.

In this lesson you will build a three-layer secrets defense:

1. A `.gitignore` that prevents secret files from ever being committed.
2. A `.env.example` template that documents required variables without exposing values.
3. A `secrets-playbook.md` reference document for your team (or your future self).

By the end, your project will be protected before you write a single line of application code.

## Why .gitignore comes first

Many tutorials teach `.env` files but forget the most important step: **telling Git to ignore them**. Without a `.gitignore` entry, running `git add .` will stage your `.env` file — and your secrets go public with the next push.

The rule is simple: **`.gitignore` before `.env`**. Always.

## Walkthrough

### Step 1 — Create a project directory

**macOS / Linux:**
```bash
mkdir -p ~/cli-academy/projects/secrets-lab && cd ~/cli-academy/projects/secrets-lab
git init
```

**Windows (PowerShell):**
```powershell
New-Item -ItemType Directory -Path "$HOME\cli-academy\projects\secrets-lab" -Force
Set-Location "$HOME\cli-academy\projects\secrets-lab"
git init
```

### Step 2 — Create .gitignore FIRST

This is the most important file in your secrets strategy. Create it before anything else:

**macOS / Linux:**
```bash
cat > .gitignore << 'EOF'
# Secrets and environment files — NEVER commit these
.env
.env.local
.env.production
.env.*.local

# API keys and credentials
*.pem
*.key
credentials.json
service-account.json

# OS and editor artifacts
.DS_Store
Thumbs.db
*.swp
*~
EOF
```

**Windows (PowerShell):**
```powershell
@"
# Secrets and environment files — NEVER commit these
.env
.env.local
.env.production
.env.*.local

# API keys and credentials
*.pem
*.key
credentials.json
service-account.json

# OS and editor artifacts
Thumbs.db
*.swp
"@ | Set-Content .gitignore
```

Verify it works:

**macOS / Linux:**
```bash
cat .gitignore
```

**Windows (PowerShell):**
```powershell
Get-Content .gitignore
```

You should see every pattern listed. Git will now refuse to track any file matching these patterns.

### Step 3 — Create .env.example (the public template)

This file documents what environment variables your project needs — but contains **zero real values**:

**macOS / Linux:**
```bash
cat > .env.example << 'EOF'
# Copy this file to .env and fill in real values
# NEVER commit .env — it is in .gitignore

ANTHROPIC_API_KEY=
DATABASE_URL=
APP_SECRET=
EOF
```

**Windows (PowerShell):**
```powershell
@"
# Copy this file to .env and fill in real values
# NEVER commit .env — it is in .gitignore

ANTHROPIC_API_KEY=
DATABASE_URL=
APP_SECRET=
"@ | Set-Content .env.example
```

### Step 4 — Create the actual .env (private, ignored)

Now create the real `.env` file that will hold actual secrets. Because `.gitignore` is already in place, Git will never see it:

**macOS / Linux:**
```bash
cp .env.example .env
echo "ANTHROPIC_API_KEY=sk-ant-test-fake-key-for-demo" >> .env
```

**Windows (PowerShell):**
```powershell
Copy-Item .env.example .env
Add-Content .env "ANTHROPIC_API_KEY=sk-ant-test-fake-key-for-demo"
```

### Step 5 — Verify Git ignores the secrets

This is the critical safety check. Run:

```bash
git status
```

You should see `.gitignore` and `.env.example` as untracked files — but `.env` should **not appear at all**. If `.env` shows up in `git status`, your `.gitignore` is not working. Stop and fix it before continuing.

### Step 6 — Create your secrets playbook

**macOS / Linux:**
```bash
cat > secrets-playbook.md << 'EOF'
# Secrets Playbook

## Rules
1. `.gitignore` is created BEFORE any `.env` file
2. Never commit actual secrets to version control
3. Use `.env.example` to document required variables (no real values)
4. Rotate any secret that was ever exposed — even briefly
5. Use environment variables at runtime, never hardcoded strings

## If a secret is accidentally committed
1. Rotate the secret immediately (generate a new key/password)
2. Remove the secret from Git history: `git filter-branch` or BFG Repo-Cleaner
3. Force-push the cleaned history
4. Verify the old secret no longer works
5. Document the incident
EOF
```

**Windows (PowerShell):**
```powershell
@"
# Secrets Playbook

## Rules
1. ``.gitignore`` is created BEFORE any ``.env`` file
2. Never commit actual secrets to version control
3. Use ``.env.example`` to document required variables (no real values)
4. Rotate any secret that was ever exposed — even briefly
5. Use environment variables at runtime, never hardcoded strings

## If a secret is accidentally committed
1. Rotate the secret immediately (generate a new key/password)
2. Remove the secret from Git history: ``git filter-branch`` or BFG Repo-Cleaner
3. Force-push the cleaned history
4. Verify the old secret no longer works
5. Document the incident
"@ | Set-Content secrets-playbook.md
```

### Step 7 — Commit the safe files

```bash
git add .gitignore .env.example secrets-playbook.md
git commit -m "feat: add three-layer secrets strategy"
```

Check your commit does not contain `.env`:
```bash
git show --stat HEAD
```

You should see exactly three files: `.gitignore`, `.env.example`, and `secrets-playbook.md`. No secrets.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `.env` appears in `git status` | `.gitignore` was created after `.env` was already tracked | Run `git rm --cached .env` then commit |
| Teammate commits secrets | They didn't have `.gitignore` | Add a pre-commit hook that scans for secret patterns |
| Old secret still in Git history | `git rm` only removes from current commit | Use `git filter-branch` or BFG Repo-Cleaner to rewrite history |
| `.env.example` has real values | Someone edited the wrong file | Review `.env.example` in every PR — it should never have real values |

> [!WARN]
> **The #1 rule of secrets:** Once a secret has been exposed — even in a private repo, even for one commit — consider it compromised. Rotate immediately. Bots scan GitHub continuously for leaked keys.

## Key takeaways

- `.gitignore` is your first and most important line of defense against secret leaks.
- Always create `.gitignore` before creating `.env`.
- `.env.example` documents what variables exist without exposing values.
- `git status` is your verification step — if `.env` shows up, something is wrong.
- A secrets playbook gives your team (or future you) a clear protocol.
