# Create a clean secrets strategy

## Objective

Accidentally committing a secret — an API key, a database password, a token — is one of the most common and most dangerous mistakes in software. Once a secret hits a public Git repository, automated bots can find it within minutes.

In this lesson you will build a three-layer secrets defense:

1. A `.gitignore` that prevents secret files from ever being committed.
2. A `.env.example` template that documents required variables without exposing values.
3. A `secrets-playbook.md` reference document for your team (or your future self).

By the end, your project will be protected before you write a single line of application code.

## Why .gitignore comes first

Many tutorials teach `.env` files but forget the most important step: **telling Git to ignore them**. Without a `.gitignore` entry, running `git add .` will stage your `.env` file — and your secrets go public with the next push.

The rule is simple: **`.gitignore` before `.env`**. Always.

### Step 2 — Create .gitignore FIRST

<StepMeta purpose="Put the first real guardrail in place before any secret file can be staged by accident." action="Create the `.gitignore` file exactly as shown and then print it back out to verify every rule landed." expectedResult="Your repo has ignore rules for env files, key files, credential JSON files, and common editor artifacts." whyItMatters="This is the single most important step in the lesson because a correct `.gitignore` is what stops `git add .` from turning into a secret leak." nextStep="After the ignore rules exist, create a public `.env.example` template that documents variables without exposing values." askTutor="Review this `.gitignore` and tell me if any secret-bearing file type or local artifact pattern is still missing for a beginner project." />

This is the most important file in your secrets strategy. Create it before anything else:

**macOS / Linux:**
```bash
cat > .gitignore << 'EOF'
# Secrets and environment files — NEVER commit these
.env
.env.local
.env.production
.env.*.local

# API keys and credentials
*.pem
*.key
credentials.json
service-account.json

# OS and editor artifacts
.DS_Store
Thumbs.db
*.swp
*~
EOF
```

**Windows (PowerShell):**
```powershell
@"
# Secrets and environment files — NEVER commit these
.env
.env.local
.env.production
.env.*.local

# API keys and credentials
*.pem
*.key
credentials.json
service-account.json

# OS and editor artifacts
Thumbs.db
*.swp
"@ | Set-Content .gitignore
```

Verify it works:

**macOS / Linux:**
```bash
cat .gitignore
```

**Windows (PowerShell):**
```powershell
Get-Content .gitignore
```

You should see every pattern listed. Git will now refuse to track any file matching these patterns.

### Step 3 — Create .env.example (the public template)

<StepMeta purpose="Separate secret names from secret values so collaborators know what to configure without seeing credentials." action="Create `.env.example` with empty placeholders for each required variable and no real values at all." expectedResult="You have a version-safe template that documents required environment variables while keeping the actual secrets blank." whyItMatters="Teams need a shared contract for runtime configuration, and `.env.example` provides that contract without turning documentation into a leak." nextStep="Copy the template into a real `.env` file only after the ignore rules are already active." askTutor="Look at my `.env.example` and tell me whether anything in it still looks like a real secret instead of a placeholder." />

This file documents what environment variables your project needs — but contains **zero real values**:

**macOS / Linux:**
```bash
cat > .env.example << 'EOF'
# Copy this file to .env and fill in real values
# NEVER commit .env — it is in .gitignore

ANTHROPIC_API_KEY=
DATABASE_URL=
APP_SECRET=
EOF
```

**Windows (PowerShell):**
```powershell
@"
# Copy this file to .env and fill in real values
# NEVER commit .env — it is in .gitignore

ANTHROPIC_API_KEY=
DATABASE_URL=
APP_SECRET=
"@ | Set-Content .env.example
```

### Step 4 — Create the actual .env (private, ignored)

<StepMeta purpose="Create the real local secret file in a way that proves the ignore rules are doing their job." action="Copy `.env.example` to `.env` and add the fake demo key only to the ignored private file." expectedResult="A local `.env` exists with a fake secret value, but it should remain invisible to Git because of the earlier ignore setup." whyItMatters="You need to practice the normal developer workflow of having local secrets without teaching Git to track them." nextStep="Run `git status` and confirm the secret file stays out of the output before you continue." askTutor="If `.env` still appears in my repo status after this step, help me diagnose whether the ignore rules or file history are wrong." />

Now create the real `.env` file that will hold actual secrets. Because `.gitignore` is already in place, Git will never see it:

**macOS / Linux:**
```bash
cp .env.example .env
echo "ANTHROPIC_API_KEY=sk-ant-test-fake-key-for-demo" >> .env
```

**Windows (PowerShell):**
```powershell
Copy-Item .env.example .env
Add-Content .env "ANTHROPIC_API_KEY=sk-ant-test-fake-key-for-demo"
```

### Step 5 — Verify Git ignores the secrets

<StepMeta purpose="Test the most important assumption in the whole lesson instead of trusting the setup blindly." action="Run `git status` and inspect the output carefully to confirm `.gitignore` and `.env.example` appear but `.env` does not." expectedResult="Git shows only the safe files as untracked and completely omits the private `.env` file." whyItMatters="Verification is what turns a best practice into a real safeguard, and this one check tells you whether your secret file is still at risk." nextStep="Write a short playbook that explains the rules and the incident response if a secret ever leaks." askTutor="Interpret my `git status` output and tell me whether it proves the secret file is protected or whether I still have a tracking problem." />

This is the critical safety check. Run:

```bash
git status
```

You should see `.gitignore` and `.env.example` as untracked files — but `.env` should **not appear at all**. If `.env` shows up in `git status`, your `.gitignore` is not working. Stop and fix it before continuing.

### Step 6 — Create your secrets playbook

<StepMeta purpose="Capture the rules and the leak-response sequence in a reusable document instead of relying on memory." action="Create `secrets-playbook.md` with the prevention rules and the emergency response steps exactly or in your own words." expectedResult="You have a small operational guide that covers prevention, rotation, cleanup, verification, and documentation." whyItMatters="When a secret leak happens, speed matters, and a written playbook helps you act in the right order instead of improvising." nextStep="Commit only the safe files so your first repo history already models the right secret-handling habit." askTutor="Review this secrets playbook and tell me whether the incident response order is correct and complete for a beginner team." />

**macOS / Linux:**
```bash
cat > secrets-playbook.md << 'EOF'
# Secrets Playbook

## Rules
1. `.gitignore` is created BEFORE any `.env` file
2. Never commit actual secrets to version control
3. Use `.env.example` to document required variables (no real values)
4. Rotate any secret that was ever exposed — even briefly
5. Use environment variables at runtime, never hardcoded strings

## If a secret is accidentally committed
1. Rotate the secret immediately (generate a new key/password)
2. Remove the secret from Git history: `git filter-branch` or BFG Repo-Cleaner
3. Force-push the cleaned history
4. Verify the old secret no longer works
5. Document the incident
EOF
```

**Windows (PowerShell):**
```powershell
@"
```

### Step 7 — Commit the safe files

<StepMeta purpose="Finish the setup by creating a clean history entry that contains only the documentation and guardrails, not the secret file." action="Stage `.gitignore`, `.env.example`, and `secrets-playbook.md`, commit them, and inspect the commit with `git show --stat HEAD`." expectedResult="The commit contains exactly the three safe files and still excludes `.env` entirely." whyItMatters="Safe secret handling is not just about current files - it is about making sure your Git history never receives the sensitive file in the first place." nextStep="Use the next lesson to practice the harder case: when a secret has already been committed and must be found and cleaned from history." askTutor="Check this commit summary and tell me whether any unsafe file slipped into history or whether the setup is ready for real use." />

```bash

git commit -m "feat: add three-layer secrets strategy"
```

Check your commit does not contain `.env`:
```bash
git show --stat HEAD
```

You should see exactly three files: `.gitignore`, `.env.example`, and `secrets-playbook.md`. No secrets.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `.env` appears in `git status` | `.gitignore` was created after `.env` was already tracked | Run `git rm --cached .env` then commit |
| Teammate commits secrets | They didn't have `.gitignore` | Add a pre-commit hook that scans for secret patterns |
| Old secret still in Git history | `git rm` only removes from current commit | Use `git filter-branch` or BFG Repo-Cleaner to rewrite history |
| `.env.example` has real values | Someone edited the wrong file | Review `.env.example` in every PR — it should never have real values |

> [!WARN]
> **The #1 rule of secrets:** Once a secret has been exposed — even in a private repo, even for one commit — consider it compromised. Rotate immediately. Bots scan GitHub continuously for leaked keys.

## Key takeaways

- `.gitignore` is your first and most important line of defense against secret leaks.
- Always create `.gitignore` before creating `.env`.
- `.env.example` documents what variables exist without exposing values.
- `git status` is your verification step — if `.env` shows up, something is wrong.
- A secrets playbook gives your team (or future you) a clear protocol.
