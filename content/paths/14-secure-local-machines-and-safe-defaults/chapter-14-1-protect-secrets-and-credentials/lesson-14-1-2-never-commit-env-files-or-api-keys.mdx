# Never commit .env files or API keys

## Objective

In Lesson 14.1.1 you built a secrets strategy that prevents `.env` files from being committed. But what happens when secrets have **already** been committed — maybe by you, maybe by a teammate, maybe months ago?

In this lesson you will learn to:

1. Detect secrets that are already buried in your Git history.
2. Remove them from the entire repository history using `git filter-repo` or BFG Repo-Cleaner.
3. Respond to a real secret exposure incident — rotate, clean, verify.

This is the lesson you wish you had read *before* your API key showed up on a GitHub leak scanner.

### Step 2 — Simulate an accidental secret commit

<StepMeta purpose="Make the exact mistake you are learning to detect and recover from." action="Create a fake `.env`, commit it, then delete it and commit the deletion plus `.gitignore` so the leak still exists in history." expectedResult="The working tree looks cleaned up, but the secret still exists in past commits where Git can still reveal it." whyItMatters="Most beginners believe deleting the file fixes the problem, and this exercise proves why that assumption is dangerously wrong." nextStep="Search the full repo history for the leaked pattern so you can see the secret survive after deletion." askTutor="Explain why the secret is still compromised after the file was removed from the working directory and added to `.gitignore`." />

Create a file with a fake API key and commit it — this is the mistake you need to learn to detect and fix.

**macOS / Linux:**
```bash
echo 'OPENAI_API_KEY=sk-proj-FAKE1234567890abcdef' > .env
git add .env
git commit -m "add config"
```

**Windows (PowerShell):**
```powershell
"OPENAI_API_KEY=sk-proj-FAKE1234567890abcdef" | Set-Content .env
git add .env
git commit -m "add config"
```

Now "fix" it the way most beginners do — delete the file and commit again:

```bash
git rm .env
echo ".env" >> .gitignore
git add .gitignore
git commit -m "remove .env and add gitignore"
```

The file is gone from the working directory. But **the secret is still in Git history**.

### Step 3 — Hunt for secrets in Git history

<StepMeta purpose="Learn the fastest commands for proving a secret exists somewhere in repo history." action="Run the `git log --all` searches for `.env` additions and secret-like string patterns, then verify the fake key appears in the output." expectedResult="You can point to the exact commit and diff where the leaked secret first entered history." whyItMatters="Incident response starts with evidence, and these searches are how you confirm both scope and exposure before cleanup." nextStep="Use `git filter-repo` to remove the secret-bearing file from every commit in the repository history." askTutor="Help me choose the right `git log` search for a suspected secret leak if I only know part of the key pattern or file name." />

These commands reveal secrets hiding in past commits. Run them in any real project you want to check.

**Search for files that were ever committed:**
```bash
git log --all --diff-filter=A -- '*.env' '*.env.*'
```
This shows every commit that added a file matching `.env` or `.env.*`.

**Search for specific secret patterns in file content:**
```bash
git log --all -p -S "API_KEY"
```
The `-S` flag (called the "pickaxe") searches through every commit diff for the string `API_KEY`. It will show you the exact commit and the exact line where the secret appeared.

**Search for common key prefixes:**
```bash
git log --all -p -S "sk-proj-"
git log --all -p -S "sk-ant-"
git log --all -p -S "ghp_"
git log --all -p -S "AKIA"
```
These catch OpenAI keys (`sk-proj-`), Anthropic keys (`sk-ant-`), GitHub tokens (`ghp_`), and AWS access keys (`AKIA`).

Run these now against your practice repo:
```bash
git log --all -p -S "sk-proj-"
```

You should see the full diff including `OPENAI_API_KEY=sk-proj-FAKE1234567890abcdef`.

### Step 4 — Remove secrets from history with git filter-repo

<StepMeta purpose="Practice the modern cleanup workflow for removing a leaked file from the full commit graph." action="Install `git-filter-repo`, run the `--invert-paths --path .env --force` command, and then rerun the secret search to confirm the leak is gone." expectedResult="The `.env` file disappears from repository history and the `git log -S` search for the fake key returns no results." whyItMatters="Cleanup commands are only trustworthy after you verify them against the same pattern that proved the leak existed." nextStep="Review the BFG alternative so you know when to replace specific strings instead of removing whole files." askTutor="Review this cleanup approach and tell me when `git filter-repo` is the better choice than BFG for a real secret incident." />

`git filter-repo` is the modern, safe replacement for `git filter-branch`. It rewrites your entire history.

> **Warning:** History rewriting changes every commit hash after the affected commit. Coordinate with your team before running this on a shared repo.

**Install git filter-repo:**

**macOS / Linux:**
```bash
pip install git-filter-repo
# or: brew install git-filter-repo
```

**Windows (PowerShell):**
```powershell
pip install git-filter-repo
```

**Remove the .env file from all history:**
```bash
git filter-repo --invert-paths --path .env --force
```

Verify the secret is gone:
```bash
git log --all -p -S "sk-proj-"
```

This should return **nothing**. The secret no longer exists in any commit.

### Step 5 — Alternative: BFG Repo-Cleaner for string replacement

<StepMeta purpose="Learn the faster cleanup option for cases where you need to replace a leaked string rather than delete a whole path." action="Read through the BFG workflow, including the secrets list file, the Java command, and the reflog and GC cleanup steps." expectedResult="You understand how BFG replaces matching secret strings with `***REMOVED***` across history and how its cleanup differs from file-based removal." whyItMatters="Real incidents vary, and string replacement can be a better fit when the leak appears in multiple files or mixed content." nextStep="Finish with the human incident-response sequence so you know what must happen before and after history rewriting." askTutor="Compare BFG and `git filter-repo` for me and recommend which one fits a repo where a token was copied into multiple files." />

BFG is faster for large repos and can replace specific strings rather than removing whole files.

**macOS / Linux:**
```bash
# Create a file listing secrets to remove (one per line)
echo "sk-proj-FAKE1234567890abcdef" > ../secrets-to-remove.txt

# Run BFG (requires Java)
java -jar bfg.jar --replace-text ../secrets-to-remove.txt

# Clean up
git reflog expire --expire=now --all
git gc --prune=now --aggressive
```

**Windows (PowerShell):**
```powershell
# Create a file listing secrets to remove
"sk-proj-FAKE1234567890abcdef" | Set-Content ..\secrets-to-remove.txt

# Run BFG (requires Java)
java -jar bfg.jar --replace-text ..\secrets-to-remove.txt

# Clean up
git reflog expire --expire=now --all
git gc --prune=now --aggressive
```

After BFG runs, the secret text is replaced with `***REMOVED***` in every commit.

### Step 6 — Incident response: what to do when a real secret leaks

<StepMeta purpose="Memorize the response order that limits damage when a leak is real instead of simulated." action="Read the six-step incident flow closely and make sure you can restate it as rotate, clean, force-push, notify, verify, and inspect damage." expectedResult="You can explain the full leak-response sequence in the correct order without mixing cleanup before rotation." whyItMatters="In a real incident, the order matters because a still-active key is dangerous even if the Git history cleanup has not finished yet." nextStep="Carry this leak mindset into the code-review lessons, where you practice catching risky AI changes before they ever get committed." askTutor="Quiz me on the incident response order and tell me what the most dangerous mistake would be if I handled a real key leak under time pressure." />

If you discover a real secret in your repo (or get an alert from GitHub Secret Scanning), follow this exact sequence:

1. **Rotate immediately.** Go to the service dashboard (OpenAI, AWS, Anthropic, etc.) and generate a new key. Revoke the old one. Do this *before* cleaning Git history — the old key is already compromised.

2. **Clean the history.** Use `git filter-repo` or BFG as shown above.

3. **Force-push the cleaned history:**
   ```bash
   git push --force --all
   git push --force --tags
   ```

4. **Notify your team.** Anyone who has cloned the repo needs to re-clone or run `git fetch --all && git reset --hard origin/main`.

5. **Verify the old key is dead.** Try using it. It should fail. If it still works, your rotation did not succeed.

6. **Check for damage.** Review API usage logs for the compromised key. Look for unexpected charges or data access.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `git log -S` finds nothing but you know a secret was committed | The secret might be in a different branch | Add `--all` flag: `git log --all -p -S "secret"` |
| `git filter-repo` refuses to run | It requires a fresh clone (no remote configured) | Clone the repo to a new directory, run filter-repo there, then push back |
| Secret is gone from history but still on GitHub | GitHub caches old commits | Contact GitHub support to purge the cache, or delete and recreate the repo |
| BFG misses some occurrences | It does not process the latest commit (HEAD) | Clean the latest commit manually first, then run BFG |
| Teammate still has the old history | They cloned before you force-pushed | They need to run `git fetch --all && git reset --hard origin/main` |
| API key still works after rotation | You rotated the wrong key, or the service has a propagation delay | Double-check the key ID in your service dashboard; wait a few minutes and test again |

> [!WARN]
> **GitHub, GitLab, and Bitbucket all cache commits.** Removing a secret from your local history does not remove it from the remote's cache. For public repos, assume the secret was seen by bots within seconds of the push. Always rotate first, clean second.

## Key takeaways

- Deleting a file and committing does **not** remove it from Git history. The secret is still there.
- `git log --all -p -S "pattern"` is the fastest way to find secrets buried in history.
- `git filter-repo` is the modern tool for rewriting history to remove files or content.
- The first step in any secret leak is **rotation** — not cleanup. Assume the key is already compromised.
- Force-push after history rewriting, and make sure every teammate re-syncs.
