# Set up pre-commit hooks for safety

## Objective

A pre-commit hook runs automatically every time you type `git commit`. It checks your code *before* the commit is created — and blocks the commit if something is wrong. This means secrets, merge conflicts, and broken formatting get caught before they ever enter your Git history.

In this lesson you will:

1. Install the `pre-commit` framework.
2. Configure four essential hooks: secret detection, no-commit-to-branch, merge conflict markers, and trailing whitespace.
3. Test that each hook actually blocks a bad commit.

After this lesson, your repo will automatically protect you from the most common mistakes — even when you forget to check manually.

## Walkthrough

### Step 1 — Create a practice project

**macOS / Linux:**
```bash
mkdir -p ~/cli-academy/projects/hooks-lab && cd ~/cli-academy/projects/hooks-lab
git init
```

**Windows (PowerShell):**
```powershell
New-Item -ItemType Directory -Path "$HOME\cli-academy\projects\hooks-lab" -Force
Set-Location "$HOME\cli-academy\projects\hooks-lab"
git init
```

### Step 2 — Install the pre-commit framework

`pre-commit` is a Python-based tool that manages Git hooks. It works with any project regardless of language.

**macOS / Linux:**
```bash
pip install pre-commit
pre-commit --version
```

**Windows (PowerShell):**
```powershell
pip install pre-commit
pre-commit --version
```

You should see a version number like `pre-commit 3.x.x`. If `pip` is not available, use `pip3` or install via `brew install pre-commit` (macOS) or `pipx install pre-commit`.

### Step 3 — Create the .pre-commit-config.yaml file

This file tells `pre-commit` which hooks to run. Create it in the root of your project:

**macOS / Linux:**
```bash
cat > .pre-commit-config.yaml << 'EOF'
repos:
  # Standard hooks from the pre-commit project
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      # Block commits that contain merge conflict markers
      - id: check-merge-conflict

      # Remove trailing whitespace from all files
      - id: trailing-whitespace

      # Prevent accidental commits directly to main or master
      - id: no-commit-to-branch
        args: ['--branch', 'main', '--branch', 'master']

      # Ensure files end with a newline
      - id: end-of-file-fixer

      # Catch accidentally added large files
      - id: check-added-large-files
        args: ['--maxkb', '500']

  # Secret detection — catches API keys, passwords, tokens
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.5.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
EOF
```

**Windows (PowerShell):**
```powershell
@"
repos:
  # Standard hooks from the pre-commit project
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      # Block commits that contain merge conflict markers
      - id: check-merge-conflict

      # Remove trailing whitespace from all files
      - id: trailing-whitespace

      # Prevent accidental commits directly to main or master
      - id: no-commit-to-branch
        args: ['--branch', 'main', '--branch', 'master']

      # Ensure files end with a newline
      - id: end-of-file-fixer

      # Catch accidentally added large files
      - id: check-added-large-files
        args: ['--maxkb', '500']

  # Secret detection — catches API keys, passwords, tokens
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.5.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
"@ | Set-Content .pre-commit-config.yaml
```

### Step 4 — Install the hooks into your Git repo

```bash
pre-commit install
```

This creates a `.git/hooks/pre-commit` file that runs your configured checks on every commit. You should see:

```
pre-commit installed at .git/hooks/pre-commit
```

### Step 5 — Generate the secrets baseline

The `detect-secrets` hook needs a baseline file — a record of known, intentional "secret-like" strings in your project (so it does not flag false positives on every commit):

```bash
detect-secrets scan > .secrets.baseline
```

If `detect-secrets` is not installed as a standalone CLI:
```bash
pip install detect-secrets
detect-secrets scan > .secrets.baseline
```

### Step 6 — Commit the configuration

```bash
git add .pre-commit-config.yaml .secrets.baseline
git commit -m "feat: add pre-commit hooks for safety"
```

The hooks will run on this commit too. You should see output like:

```
Check for merge conflicts...................................Passed
Trim trailing whitespace...................................Passed
No commit to branch........................................Passed
Fix end of files...........................................Passed
Check for added large files................................Passed
Detect secrets.............................................Passed
```

All green. Your repo is now protected.

### Step 7 — Test: secret detection

Create a file with a fake API key and try to commit it:

**macOS / Linux:**
```bash
echo 'API_KEY="AKIAIOSFODNN7EXAMPLE"' > config.py
git add config.py
git commit -m "add config"
```

**Windows (PowerShell):**
```powershell
'API_KEY="AKIAIOSFODNN7EXAMPLE"' | Set-Content config.py
git add config.py
git commit -m "add config"
```

The `detect-secrets` hook should **block the commit** with a message about a detected secret. This is exactly what you want.

Clean up:
```bash
git reset HEAD config.py
rm config.py
```

On Windows:
```powershell
git reset HEAD config.py
Remove-Item config.py
```

### Step 8 — Test: merge conflict markers

Create a file with a conflict marker and try to commit:

**macOS / Linux:**
```bash
printf "line 1\n<<<<<<< HEAD\nmy change\n=======\ntheir change\n>>>>>>> branch\n" > conflict.txt
git add conflict.txt
git commit -m "add notes"
```

**Windows (PowerShell):**
```powershell
"line 1`n<<<<<<< HEAD`nmy change`n=======`ntheir change`n>>>>>>> branch" | Set-Content conflict.txt
git add conflict.txt
git commit -m "add notes"
```

The `check-merge-conflict` hook should **block the commit**. You accidentally left conflict markers in the file.

Clean up:
```bash
git reset HEAD conflict.txt
rm conflict.txt
```

On Windows:
```powershell
git reset HEAD conflict.txt
Remove-Item conflict.txt
```

### Step 9 — Test: no-commit-to-branch

If you are on the `main` branch, the `no-commit-to-branch` hook will block direct commits. This forces you to work on feature branches:

```bash
git branch
```

If you see `* main`, create a feature branch for your work:

```bash
git checkout -b feat/hooks-practice
```

This is a good habit. Direct commits to `main` bypass code review and are a common source of mistakes.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `pre-commit install` says "command not found" | `pre-commit` is not installed or not on your PATH | Run `pip install pre-commit` again, or use `python -m pre_commit install` |
| Hooks do not run on commit | Hooks were not installed for this repo | Run `pre-commit install` from the repo root |
| `detect-secrets` flags a false positive | A string in your code looks like a key but is not | Run `detect-secrets audit .secrets.baseline` to mark it as a false positive, then re-commit |
| Hooks are slow on first run | `pre-commit` downloads hook environments on first execution | This is normal — subsequent runs use cached environments and are fast |
| You need to bypass hooks for an emergency commit | Rare situation where hooks must be skipped | Use `git commit --no-verify` — but document why you did this and run hooks manually afterward |
| `no-commit-to-branch` blocks your commit | You are on `main` or `master` | Create a feature branch: `git checkout -b feat/your-feature` |

> [!WARN]
> **`git commit --no-verify` disables all hooks.** Use it only in genuine emergencies. If you find yourself using it regularly, your hooks need tuning — not bypassing. Every skipped hook is a missed safety check.

## Key takeaways

- Pre-commit hooks run automatically before every commit — you cannot forget to check.
- The `.pre-commit-config.yaml` file is version-controlled, so every contributor gets the same protections.
- `detect-secrets` catches API keys, passwords, and tokens before they enter Git history.
- `no-commit-to-branch` prevents accidental direct commits to `main`.
- `check-merge-conflict` catches forgotten conflict markers.
- Test your hooks by deliberately triggering them — if they do not block a bad commit, they are not working.
