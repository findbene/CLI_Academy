# Practice the review-test-commit cycle

## Objective

Most security mistakes happen when developers commit without thinking. They write code, `git add .`, `git commit`, and push — skipping the steps that catch problems.

The **review → test → commit** cycle is a three-step habit:

1. **Review** — Read the diff. Check for removed validation, `eval()`, hardcoded secrets, and injection patterns.
2. **Test** — Run your test suite. Verify the code actually works as intended.
3. **Commit** — Only after review and tests pass, create a clean, atomic commit.

In this lesson you will practice the full cycle on a real code change, building muscle memory that will protect every project you work on.

### Step 3 — REVIEW: Read the diff

<StepMeta purpose="Make code review the first reflex instead of jumping straight from editing to committing." action="Run `git diff calculator.py`, identify the removed type-checking block, and write down the safety regression you found." expectedResult="You can point to the exact lines where validation disappeared and explain why the feature change is not safe yet." whyItMatters="Review comes first because tests do not tell you intent - the diff reveals what protections were lost while the feature was added." nextStep="Run the tests next and confirm whether your test suite catches the regression you just noticed in the diff." askTutor="Walk me through this calculator diff and tell me whether I found the most important removed safeguard or if I missed another regression." />

This is the first step in the cycle. **Do not skip it.**

```bash

```

Read the output carefully. You should spot:


### Step 6 — Check for secrets before committing

<StepMeta purpose="Add one final pre-commit inspection so even a good feature change does not accidentally stage unrelated sensitive data." action="Use `git add -p` to stage deliberately, then scan the staged diff for secret-like strings and confirm the search returns nothing." expectedResult="Only the intended calculator and test changes are staged, and the staged diff contains no credentials or other obvious leak patterns." whyItMatters="A clean implementation can still become a risky commit if you stage too broadly, so staging discipline is part of the safety cycle too." nextStep="Create the final atomic commit only after review, testing, and the staged secret scan are all clean." askTutor="Help me interpret my staged diff and tell me whether I staged only the intended feature work or picked up extra changes by accident." />

Scan your staged changes for anything that looks like a secret:

```bash
git add -p
```

The `-p` (patch) flag shows you each change interactively and asks you to confirm. This is one more chance to review.

Then scan:

**macOS / Linux:**
```bash
git diff --staged | grep -iE '(key|password|secret|token|sk-|ghp_|AKIA)\s*='
```

**Windows (PowerShell):**
```powershell
git diff --staged | Select-String -Pattern '(key|password|secret|token|sk-|ghp_|AKIA)\s*='
```

This should return nothing — your code has no secrets.

<StepMeta purpose="Finish the workflow with one intentional commit that contains only the reviewed and tested change." action="Commit the repaired feature, then inspect `git log --oneline` and `git show --stat HEAD` to confirm the history is clean and focused." expectedResult="You end with a single atomic feature commit that includes only `calculator.py` and `test_calculator.py` after the full cycle succeeded." whyItMatters="The point of the cycle is not just safer code - it is safer history, where each commit tells a clear story about a reviewed and verified change." nextStep="Use the final lesson in the path to apply these habits to a broader security audit instead of one single code change." askTutor="Review this final commit summary and tell me whether it qualifies as a clean atomic commit or whether the scope still feels too broad." />

Only now — after reviewing, testing, and scanning — commit:

```bash
git commit -m "feat: add multiply function with input validation"
```

```bash

```

The commit should contain exactly `calculator.py` and `test_calculator.py`.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| You forgot to run tests and committed a broken change | Old habit — went straight from code to commit | Put a sticky note on your monitor: "diff → test → commit" until the habit sticks |
| Tests pass but a security problem exists | No test covers the security case | Write tests for edge cases: empty input, special characters, very large numbers, wrong types |
| `git add .` stages files you did not intend to commit | Unrelated files in the working directory | Use `git add -p` or `git add <specific-file>` instead of `git add .` |
| The diff is too long to review effectively | AI made too many changes at once | Ask Claude Code to make smaller, focused changes. Review one function at a time |
| Tests pass locally but fail in CI | Different Python version or missing dependency | Pin versions in `requirements.txt` and test with the same Python version CI uses |
| You notice a problem after committing | Missed it in review | Run `git revert HEAD` to undo the commit, fix the problem, and commit again |

> [!WARN]
> **`git add .` is the enemy of careful commits.** It stages every changed file — including files you did not mean to include. Use `git add -p` (interactive patch mode) or name specific files. This forces you to see every change before it is staged.

## Key takeaways

- The review → test → commit cycle is three steps. Never skip one.
- `git diff` is step one. Read every removed line — AI loves to "simplify" by deleting safety checks.
- `python -m pytest` (or your language's test runner) is step two. If a test fails, do not commit.
- `git diff --staged` plus a secrets scan is the final check before committing.
- Use `git add -p` instead of `git add .` to force yourself to review every change before staging.
- This cycle takes 60 seconds and prevents hours of debugging. Build the habit.
