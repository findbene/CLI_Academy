# Create a clean secrets strategy

## Objective

Our goal in this lesson is to establish a foundational "secrets strategy" that keeps your sensitive information safe from accidental exposure. You will create two essential files: a `secrets-playbook.md` to document your approach to handling secrets and a `.env.example` file to clearly define the environment variables your project expects, *without* exposing actual secret values. This practice is crucial for protecting your projects and collaborating securely.

## Walkthrough

Let's carefully set up your initial secrets strategy.

1.  **Create your `secrets-playbook.md`**
    This file will be your personal guide and reminder for best practices regarding sensitive data.
    *   First, we'll create the file:
        ```bash
        touch secrets-playbook.md
        ```
        `touch` safely creates a new, empty file if it doesn't already exist. It won't modify any existing files, so it's a very safe command for file creation.
    *   Now, let's add some initial guidance to it:
        ```bash
        echo "# Secrets Playbook

- Never commit actual secrets to version control (e.g., Git).
- Store sensitive data using environment variables.
- Use a .env.example file to document required secret placeholders.
- Always keep real secrets out of shared code and documentation." > secrets-playbook.md
        ```
        The `echo` command prints the text, and `>` redirects that text into `secrets-playbook.md`. Since we just `touch`ed it, we know it's a new file, making this a safe way to add content. This ensures everyone understands the rules for handling secrets.

2.  **Create your `.env.example` file**
    This file serves as a template, showing what environment variables your project needs, but without any actual secret values. It's a placeholder for others (and your future self!) to fill in.
    *   Let's create this important template file:
        ```bash
        touch .env.example
        ```
        Again, `touch` safely creates an empty file.
    *   Now, let's add some common placeholders. Remember, these are *empty* and just show the *names* of the variables needed:
        ```bash
        echo "API_KEY=
DATABASE_URL=
APP_SECRET=" > .env.example
        ```
        Similar to before, `echo` safely writes content to your newly created `.env.example` file. These lines indicate that your project needs an `API_KEY`, `DATABASE_URL`, and `APP_SECRET`, but without providing any real values. This is how you guide others on setup without risking exposure.

You've now set up a clean, safe foundation for handling secrets in your projects!
