# Use Playwright for browser automation

## Objective

By the end of this lesson you will have Playwright installed, written a test that navigates to a real web page, taken a screenshot from the automated browser, and run the test from the command line. You will understand the basic Playwright API well enough to extend the test on your own.

## Walkthrough

Playwright is an open-source browser automation library maintained by Microsoft. It can drive Chromium, Firefox, and WebKit from a single API. Unlike computer use, Playwright interacts with the DOM (the page's structure) rather than pixel coordinates, making it far more reliable for repeatable browser tasks.

### Step 1 — Create a project and install Playwright

<StepMeta purpose="Create a clean browser-automation workspace and let the Playwright bootstrap lay down the standard test structure." action="Make the `browser-automation` folder, run `npm init -y`, then run `npm init playwright@latest -- --yes --quiet` and wait for package and browser downloads to complete." expectedResult="You have a working Playwright project with config, test scaffolding, and browser binaries installed." whyItMatters="A standard project scaffold reduces setup drift and gives you a predictable foundation for every later test and debugging step." nextStep="Write a minimal first test that navigates to a real site, asserts the title, and saves a screenshot." askTutor="Review this Playwright setup with me and tell me what files and directories should exist if the install completed successfully." />

**macOS / Linux:**

```bash
mkdir -p ~/cli-academy-16/browser-automation && cd ~/cli-academy-16/browser-automation
npm init -y
npm init playwright@latest -- --yes --quiet
```

**Windows (PowerShell):**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\cli-academy-16\browser-automation" | Out-Null
Set-Location "$HOME\cli-academy-16\browser-automation"
npm init -y
npm init playwright@latest -- --yes --quiet
```

The `npm init playwright` command does three things:

1. Installs the `@playwright/test` package.
2. Downloads browser binaries (Chromium, Firefox, WebKit) into a cache folder.
3. Creates a `playwright.config.ts` and an example `tests/` folder.

This can take a minute or two — the browser downloads are large (~200 MB each).

### Step 2 — Write your first test

<StepMeta purpose="Build a small but complete Playwright test that demonstrates navigation, assertion, and artifact capture." action="Create `tests/first-page.spec.ts` with the homepage navigation, title assertion, and screenshot step, then create the `screenshots` directory." expectedResult="Your test file expresses a full browser automation flow and is ready to run against Chromium." whyItMatters="A single small test that both asserts content and captures output teaches the core API faster than a large example with many moving parts." nextStep="Run the test from the command line and confirm Playwright reports a clean pass." askTutor="Walk through this first Playwright test with me and explain what each line does, especially `goto`, `toHaveTitle`, and `screenshot`." />

Create a file called `tests/first-page.spec.ts` with the following content:

**macOS / Linux:**

```bash
cat > tests/first-page.spec.ts << 'TESTEOF'
import { test, expect } from '@playwright/test';

test('load the Playwright homepage and take a screenshot', async ({ page }) => {
  // Navigate to the Playwright docs
  await page.goto('https://playwright.dev/');

  // Verify the page title contains "Playwright"
  await expect(page).toHaveTitle(/Playwright/);

  // Take a screenshot and save it to disk
  await page.screenshot({ path: 'screenshots/playwright-home.png', fullPage: true });
});
TESTEOF
mkdir -p screenshots
```

**Windows (PowerShell):**

```powershell
@"
import { test, expect } from '@playwright/test';

test('load the Playwright homepage and take a screenshot', async ({ page }) => {
  // Navigate to the Playwright docs
  await page.goto('https://playwright.dev/');

  // Verify the page title contains "Playwright"
  await expect(page).toHaveTitle(/Playwright/);

  // Take a screenshot and save it to disk
  await page.screenshot({ path: 'screenshots/playwright-home.png', fullPage: true });
});
"@ | Set-Content tests/first-page.spec.ts
New-Item -ItemType Directory -Force -Path screenshots | Out-Null
```

Let's break down what this test does:

- `page.goto()` navigates the headless browser to a URL.
- `expect(page).toHaveTitle()` asserts the page title matches a pattern — if it does not, the test fails.
- `page.screenshot()` captures the rendered page as a PNG file.

### Step 3 — Run the test

<StepMeta purpose="Verify that the installed tooling, downloaded browser, and written test all work together from the command line." action="Run `npx playwright test tests/first-page.spec.ts --project=chromium` and inspect the pass or failure output carefully." expectedResult="Playwright executes one Chromium test and reports `1 passed` if the setup and test are both healthy." whyItMatters="Running the test end to end is what turns your files into a working automation environment rather than just a plausible-looking scaffold." nextStep="Inspect the screenshot file so you confirm the test did the visible browser work you expected." askTutor="Help me debug this Playwright run if it fails and tell me whether the problem looks like setup, network, browser install, or the test code itself." />

**macOS / Linux / Windows (PowerShell):**

```bash
npx playwright test tests/first-page.spec.ts --project=chromium
```

Expected output:

```
Running 1 test using 1 worker

  ✓  tests/first-page.spec.ts:3:5 › load the Playwright homepage and take a screenshot (Xs)

  1 passed (Xs)
```

The number after the arrow is the duration in seconds. If you see `1 passed`, everything worked.

### Step 4 — Verify the screenshot

<StepMeta purpose="Check the concrete artifact produced by the test instead of trusting the terminal output alone." action="List the screenshot file, confirm it exists and has non-zero size, and open it if you want to visually inspect the result." expectedResult="You can prove the automated browser rendered the target page and saved an image artifact to disk." whyItMatters="Artifacts are useful evidence in browser automation because they show not just that the test passed, but what the browser actually saw." nextStep="Optionally open the HTML report so you can inspect richer test metadata and understand Playwright's debugging surface." askTutor="Review the screenshot artifact expectations with me and tell me what a missing, tiny, or blank file would suggest about the run." />

Check that the screenshot file exists:

**macOS / Linux:**

```bash
ls -lh screenshots/playwright-home.png
```

**Windows (PowerShell):**

```powershell
Get-Item screenshots\playwright-home.png | Select-Object Name, Length, LastWriteTime
```

You should see a PNG file that is at least a few hundred kilobytes. Open it in an image viewer to confirm it shows the Playwright homepage.

### Step 5 — View the HTML report (optional)

<StepMeta purpose="Introduce the richer debugging and inspection tools Playwright provides beyond plain terminal output." action="Run `npx playwright show-report` and inspect the generated HTML report if you want timing, test status, and richer debugging context." expectedResult="You know how to open and use Playwright's report UI when a test needs deeper inspection than the CLI summary provides." whyItMatters="As tests grow, the report becomes a faster way to understand failures and artifacts than re-reading raw terminal output every time." nextStep="Use the next lesson to apply the same Playwright setup to practical form filling and data extraction tasks." askTutor="Explain when the Playwright HTML report is worth opening and what kinds of failure details it helps me understand fastest." />

Playwright generates a rich HTML report after each run. Open it:

```bash
npx playwright show-report
```

This launches a local web server and opens the report in your default browser. You can see pass/fail status, timing, and even attach traces for deeper debugging.

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `npm init playwright` fails with a network error | Corporate proxy or firewall blocking the browser download | Set `HTTPS_PROXY` env var or download browsers manually with `npx playwright install --with-deps` |
| `Error: browserType.launch: Executable doesn't exist` | Browser binaries not downloaded | Run `npx playwright install chromium` |
| Test fails with `Timeout exceeded` | Slow network or the target site is down | Increase the timeout in the test: `test.setTimeout(30_000)` or check your connection |
| `ERR_CERT_AUTHORITY_INVALID` in the test | Corporate TLS inspection proxy | Set `ignoreHTTPSErrors: true` in `playwright.config.ts` (dev/test only — never in production) |
| Screenshot file is 0 bytes | Test crashed before the screenshot step | Check the test output for earlier errors; fix those first |
| `npx` is not recognized | Node.js is not installed or not in PATH | Install Node.js from https://nodejs.org (LTS version) and restart your terminal |
