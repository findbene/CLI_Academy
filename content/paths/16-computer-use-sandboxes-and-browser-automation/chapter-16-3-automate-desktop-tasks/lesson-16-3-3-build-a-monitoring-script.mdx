# Build a monitoring script

## Objective

By the end of this lesson you will have a working monitoring script that checks whether a website or service is reachable, logs the result with a timestamp, and prints an alert when the service is down. You will write both a Bash version (macOS / Linux) and a PowerShell version (Windows), and you will test the alert path by pointing the script at a URL that does not exist.

## Walkthrough

Monitoring means checking something on a regular basis and getting notified when it breaks. Professional teams use tools like Uptime Robot, Datadog, or Prometheus — but the core idea is simple enough to build yourself in a few lines of shell script.

### Step 1 — Write the monitoring script

<StepMeta purpose="Create a minimal but functional health-check script that distinguishes success from failure and logs every run." action="Write the monitoring script for your platform, review how it derives HTTP status or errors, and note where it records output in `monitor.log`." expectedResult="You have a script that checks one URL, prints UP or DOWN, and appends a timestamped record to a log file." whyItMatters="Monitoring is just automated verification with evidence, and the log file is what makes the check auditable over time." nextStep="Test the happy path first so you know the normal success output before investigating the alert branch." askTutor="Review this monitoring script and tell me whether the success and failure conditions are broad enough to catch real outages but narrow enough to avoid noise." />

**macOS / Linux (`monitor.sh`):**

```bash
mkdir -p ~/cli-academy-16/monitor && cd ~/cli-academy-16/monitor

cat > monitor.sh << 'SCRIPTEOF'
#!/usr/bin/env bash
set -euo pipefail

# --- Configuration ---
URL="${1:-https://httpstat.us/200}"
LOG_FILE="monitor.log"
TIMEOUT=10

# --- Check the URL ---
timestamp=$(date '+%Y-%m-%d %H:%M:%S')
http_code=$(curl -s -o /dev/null -w "%{http_code}" --max-time "$TIMEOUT" "$URL" 2>/dev/null || echo "000")

if [ "$http_code" -ge 200 ] && [ "$http_code" -lt 400 ]; then
  status="UP"
  echo "[$timestamp] $status — $URL returned HTTP $http_code" >> "$LOG_FILE"
  echo "✓ $URL is UP (HTTP $http_code)"
else
  status="DOWN"
  echo "[$timestamp] $status — $URL returned HTTP $http_code" >> "$LOG_FILE"
  echo "✗ ALERT: $URL is DOWN (HTTP $http_code)"
  echo "  Check the log: $(pwd)/$LOG_FILE"
fi
SCRIPTEOF
chmod +x monitor.sh
```

**Windows (PowerShell — `monitor.ps1`):**

```powershell
New-Item -ItemType Directory -Force -Path "$HOME\cli-academy-16\monitor" | Out-Null
Set-Location "$HOME\cli-academy-16\monitor"

@'
param(
    [string]$Url = "https://httpstat.us/200",
    [int]$TimeoutSec = 10
)

$logFile = "monitor.log"
$timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss"

try {
    $response = Invoke-WebRequest -Uri $Url -TimeoutSec $TimeoutSec -UseBasicParsing -ErrorAction Stop
    $code = $response.StatusCode

    if ($code -ge 200 -and $code -lt 400) {
        $status = "UP"
        Add-Content -Path $logFile -Value "[$timestamp] $status - $Url returned HTTP $code"
        Write-Host "[OK] $Url is UP (HTTP $code)" -ForegroundColor Green
    } else {
        $status = "DOWN"
        Add-Content -Path $logFile -Value "[$timestamp] $status - $Url returned HTTP $code"
        Write-Host "[ALERT] $Url is DOWN (HTTP $code)" -ForegroundColor Red
    }
} catch {
    $status = "DOWN"
    $errorMsg = $_.Exception.Message
    Add-Content -Path $logFile -Value "[$timestamp] $status - $Url failed: $errorMsg"
    Write-Host "[ALERT] $Url is DOWN ($errorMsg)" -ForegroundColor Red
    Write-Host "  Check the log: $((Get-Location).Path)\$logFile" -ForegroundColor Yellow
}
'@ | Set-Content monitor.ps1
```

**How it works:**

- The script sends an HTTP request to the target URL.
- If the response code is in the 2xx–3xx range, the service is UP.
- Anything else — 4xx, 5xx, timeout, DNS failure — counts as DOWN.
- Every check is logged with a timestamp so you can review history later.

### Step 4 — Monitor multiple URLs in one pass

<StepMeta purpose="Turn the single-endpoint monitor into a reusable batch workflow that can check several targets in one run." action="Create `urls.txt` plus the multi-URL wrapper script, then run it and observe the combined output across healthy and failing endpoints." expectedResult="One command now checks multiple URLs sequentially and records each result through the single-URL monitor logic." whyItMatters="Batch execution is the natural next step for real monitoring because most useful workflows watch a set of services, not just one URL." nextStep="Optionally schedule the monitor so it runs unattended at a fixed interval instead of only on demand." askTutor="Review this multi-URL monitor wrapper and tell me whether it reuses the single-URL script cleanly or if there is a better batching pattern." />

Extend the script to check a list of URLs from a file.

**macOS / Linux (`monitor-all.sh`):**

```bash
cat > urls.txt << 'EOF'
https://httpstat.us/200
https://httpstat.us/500
https://playwright.dev
EOF

cat > monitor-all.sh << 'SCRIPTEOF'
#!/usr/bin/env bash
set -uo pipefail

URL_FILE="${1:-urls.txt}"
echo "=== Monitor run: $(date '+%Y-%m-%d %H:%M:%S') ==="

while IFS= read -r url; do
  [ -z "$url" ] && continue
  bash monitor.sh "$url"
done < "$URL_FILE"

echo "=== Run complete ==="
SCRIPTEOF
chmod +x monitor-all.sh
bash monitor-all.sh
```

**Windows (PowerShell — `monitor-all.ps1`):**

```powershell
@"
https://httpstat.us/200
https://httpstat.us/500
https://playwright.dev
"@ | Set-Content urls.txt

@'
param(
    [string]$UrlFile = "urls.txt"
)

Write-Host "=== Monitor run: $(Get-Date -Format 'yyyy-MM-dd HH:mm:ss') ==="

Get-Content $UrlFile | Where-Object { $_.Trim() -ne "" } | ForEach-Object {
    & "$PSScriptRoot\monitor.ps1" -Url $_
}

Write-Host "=== Run complete ==="
'@ | Set-Content monitor-all.ps1
.\monitor-all.ps1
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `curl: command not found` | curl is not installed (rare on modern systems) | Install it: `sudo apt install curl` (Linux), `brew install curl` (macOS), or use `Invoke-WebRequest` on Windows |
| Script says UP but the site shows an error page | The server returns HTTP 200 for custom error pages | Check the response body, not just the status code — add a content check if needed |
| Monitor log grows forever | No log rotation in place | Add a size check at the top of the script: if the log exceeds a threshold, rename it and start fresh |
| `Invoke-WebRequest` throws a certificate error | The site has an expired or self-signed cert | Add `-SkipCertificateCheck` (PowerShell 7+) for testing only — do **not** use in production |
| All URLs report DOWN simultaneously | Your own network is down | Add a canary check (e.g., `https://1.1.1.1`) at the start — if that fails, the problem is your connection, not the targets |
| The 500 test URL returns 200 | `httpstat.us` occasionally has issues or caching | Try `https://httpbin.org/status/500` as an alternative |
