# Build a two-agent research-and-write pipeline

## Objective

In Lesson 13.2.1 you created agent contracts with defined roles, boundaries, and stop conditions. Now you will put those contracts to work. You will build a complete two-agent pipeline — a Researcher that gathers findings into a file and a Writer that turns those findings into a polished document — run it end-to-end on a real topic, and verify the handoff.

By the end you will have a working `research-write-pipeline/` directory with contracts, a research output, and a finished document you can read.

## Walkthrough

### Step 1 — Create the pipeline directory

**macOS / Linux:**
```bash
mkdir -p research-write-pipeline && cd research-write-pipeline
```

**Windows PowerShell:**
```powershell
New-Item -ItemType Directory -Path research-write-pipeline -Force; Set-Location research-write-pipeline
```

### Step 2 — Write the Researcher contract

**macOS / Linux:**
```bash
cat > researcher-CLAUDE.md << 'EOF'
# Researcher Agent Contract

## Role
You are a factual researcher. Gather information on the assigned topic.

## Topic
"What are the most common mistakes beginners make when learning the command line?"

## Outputs
- A file called `research-findings.md` containing:
  - 5–8 bullet points of findings
  - Each bullet includes a brief explanation
  - A confidence note at the bottom: High / Medium / Low

## Boundaries
- Do NOT write prose, conclusions, or recommendations
- Do NOT create any file except `research-findings.md`
- Do NOT continue after writing the file

## Stop Condition
Stop immediately after writing `research-findings.md`. One pass. No iteration.
EOF
```

**Windows PowerShell:**
```powershell
@"
# Researcher Agent Contract

## Role
You are a factual researcher. Gather information on the assigned topic.

## Topic
"What are the most common mistakes beginners make when learning the command line?"

## Outputs
- A file called ``research-findings.md`` containing:
  - 5-8 bullet points of findings
  - Each bullet includes a brief explanation
  - A confidence note at the bottom: High / Medium / Low

## Boundaries
- Do NOT write prose, conclusions, or recommendations
- Do NOT create any file except ``research-findings.md``
- Do NOT continue after writing the file

## Stop Condition
Stop immediately after writing ``research-findings.md``. One pass. No iteration.
"@ | Set-Content researcher-CLAUDE.md
```

### Step 3 — Write the Writer contract

**macOS / Linux:**
```bash
cat > writer-CLAUDE.md << 'EOF'
# Writer Agent Contract

## Role
You are a technical writer. Turn raw findings into a clear, beginner-friendly article.

## Inputs
- `research-findings.md` (produced by the Researcher agent)

## Outputs
- A file called `article-draft.md` containing:
  - A title
  - An introduction (2–3 sentences)
  - A section for each finding, rewritten in plain English
  - A short conclusion

## Boundaries
- Do NOT invent facts that are not in `research-findings.md`
- Do NOT modify `research-findings.md`
- Do NOT create any file except `article-draft.md`
- If `research-findings.md` does not exist or is empty, STOP and report the error

## Stop Condition
Stop immediately after writing `article-draft.md`. Do not iterate.
EOF
```

**Windows PowerShell:**
```powershell
@"
# Writer Agent Contract

## Role
You are a technical writer. Turn raw findings into a clear, beginner-friendly article.

## Inputs
- ``research-findings.md`` (produced by the Researcher agent)

## Outputs
- A file called ``article-draft.md`` containing:
  - A title
  - An introduction (2-3 sentences)
  - A section for each finding, rewritten in plain English
  - A short conclusion

## Boundaries
- Do NOT invent facts that are not in ``research-findings.md``
- Do NOT modify ``research-findings.md``
- Do NOT create any file except ``article-draft.md``
- If ``research-findings.md`` does not exist or is empty, STOP and report the error

## Stop Condition
Stop immediately after writing ``article-draft.md``. Do not iterate.
"@ | Set-Content writer-CLAUDE.md
```

### Step 4 — Run the Researcher

Open Claude Code in the `research-write-pipeline/` directory. Give it this prompt:

```
Follow the instructions in researcher-CLAUDE.md exactly. Do not deviate from the contract.
```

When it finishes, verify the handoff file exists and has real content:

**macOS / Linux:**
```bash
test -s research-findings.md && echo "Handoff file OK" || echo "ERROR: file missing or empty"
cat research-findings.md
```

**Windows PowerShell:**
```powershell
if (Test-Path research-findings.md) { Write-Host "Handoff file OK"; Get-Content research-findings.md } else { Write-Host "ERROR: file missing or empty" }
```

If the file is missing or empty, re-run the Researcher before continuing. The Writer cannot operate without valid input.

### Step 5 — Run the Writer

In the same directory, start a new Claude Code session (or a new prompt in the same session) and say:

```
Follow the instructions in writer-CLAUDE.md exactly. Do not deviate from the contract.
```

Verify the final output:

**macOS / Linux:**
```bash
test -s article-draft.md && echo "Article created" || echo "ERROR: article missing"
cat article-draft.md
```

**Windows PowerShell:**
```powershell
if (Test-Path article-draft.md) { Write-Host "Article created"; Get-Content article-draft.md } else { Write-Host "ERROR: article missing" }
```

### Step 6 — Audit the handoff

Check that the Writer did not invent facts:

1. Open `research-findings.md` and `article-draft.md` side-by-side.
2. For every claim in the article, confirm it traces back to a bullet in the findings.
3. If you find a claim with no source, the Writer violated its contract.

This manual audit is how real multi-agent systems are validated. Automated checks are possible (and you will build them in Lesson 13.3.2), but the human eye catches subtle drift that scripts miss.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `research-findings.md` is empty | Researcher wrote nothing or misunderstood the topic | Check the contract's Topic section is clear and specific |
| Writer invents facts not in the findings | Boundary rule ignored or too vague | Strengthen the contract: "Every claim MUST trace to a bullet in research-findings.md" |
| Writer modifies `research-findings.md` | Contract boundary not enforced | Add "NEVER open research-findings.md for writing" to the contract |
| Researcher writes an article instead of bullet points | Role description ambiguous | Add "Do NOT write prose, paragraphs, or conclusions" explicitly |
| Pipeline takes many minutes and racks up cost | Agents iterating beyond stop condition | Verify each contract has "Stop immediately after writing X. One pass." |

> **Cost warning:** Each agent session consumes API tokens proportional to context size plus output. A two-agent pipeline costs roughly twice what a single session costs. Run on small topics first to validate contracts before scaling to larger tasks.
