# When to use subagents vs a single session

## Objective

Not every task needs multiple agents. In this lesson you will learn a practical decision framework for choosing between one long Claude Code session and a multi-agent split. You will apply the framework to three realistic scenarios and write a one-page decision record explaining your reasoning.

By the end you will be able to look at any task and say confidently: "One session is enough" or "This needs separate agents, and here is how I would split it."

## Walkthrough

### Step 1 — Learn the three decision criteria

<StepMeta purpose="Give yourself a decision rule so you do not split tasks into agents by instinct alone." action="Read the size, independence, and specialization questions and decide how each one changes the case for one session versus multiple agents." expectedResult="You can apply the three-criteria test to a real task without hand-waving about complexity or scale." whyItMatters="Most bad multi-agent designs start with an unjustified split, so the decision framework is the real foundation of the path." nextStep="Create a small decision-record workspace where you can score example scenarios concretely." askTutor="Walk through a real task I have and help me score it honestly on size, independence, and specialization." />

Use these three questions to decide:

1. **Size** — Will the task exceed what one session can hold in context? If your project has dozens of files and thousands of lines, a single session may lose track. Splitting lets each agent focus on a small slice.

2. **Independence** — Can the subtasks run without constantly referencing each other? If task A needs output from task B, and B needs output from A, splitting creates more problems than it solves. If A finishes completely before B starts, splitting is clean.

3. **Specialization** — Does each subtask benefit from a different set of instructions? A researcher agent told "only gather facts, never analyze" will produce cleaner findings than a general-purpose agent that tries to do everything.

If you answer "yes" to at least two of these three, splitting is likely worth it. If the task is small, tightly coupled, or uniform, keep it in one session.

### Step 3 — Evaluate three scenarios

<StepMeta purpose="Practice the framework on concrete cases before you trust yourself to use it in live work." action="Write out the scenario analyses exactly or in your own words, making sure each one names all three criteria and ends with a clear decision." expectedResult="Your decision record contains grounded reasoning for several scenarios rather than vague 'it depends' conclusions." whyItMatters="The scoring only becomes useful when it produces a defendable yes-or-no decision about splitting." nextStep="Read the scenarios back and pressure-test whether the criteria really support the final choices." askTutor="Review my scenario writeups and tell me where the logic between the criteria and the decision is still weak." />

Create a decision record that analyzes three scenarios. Write it as a Markdown file:

**macOS / Linux:**
```bash
cat > decision-record.md << 'EOF'
# Agent Split Decision Record

## Scenario A: Rename 200 files using a consistent pattern
- Size: Small per-file work, but many files → moderate
- Independence: Each rename is independent → high
- Specialization: Same operation every time → low
- **Decision: Single session.** One agent with a loop is simpler than a manager-worker split for uniform tasks.

## Scenario B: Research a topic, then write a report from the findings
- Size: Research can produce pages of notes; writing is a separate creative task → moderate to high
- Independence: Writer needs Researcher output, but Researcher does not need Writer → sequential, clean split
- Specialization: Researcher should not editorialize; Writer should not invent facts → high
- **Decision: Two agents.** Researcher writes findings to a file, Writer reads that file and produces the report.

## Scenario C: Fix a bug by reading logs, identifying the cause, and writing a patch
- Size: Usually fits in one context window → small
- Independence: Each step depends heavily on the previous → tightly coupled
- Specialization: Same skill set (debugging) throughout → low
- **Decision: Single session.** Splitting debugging across agents would create more handoff friction than value.
EOF
```

### Step 4 — Review your reasoning

<StepMeta purpose="Catch sloppy logic before the framework hardens into a bad habit." action="Read the finished scenarios and check whether each one evaluated every criterion and whether the decision actually follows from those inputs." expectedResult="You either confirm the record is coherent or identify the places where your scoring and conclusion do not line up." whyItMatters="A review pass keeps the framework honest and teaches you to justify splits with evidence instead of preference." nextStep="Add one scenario from your own work so the framework leaves the lesson and enters your real workflow." askTutor="Pressure-test one of my decisions and argue the opposite side so I can see whether my reasoning still holds up." />

Read back your decision record:

**macOS / Linux:**
```bash
cat decision-record.md
```

**Windows PowerShell:**
```powershell
Get-Content decision-record.md
```

For each scenario, check:
- Did you evaluate all three criteria (size, independence, specialization)?
- Does the decision follow logically from the scores?
- Could you defend the choice to someone who disagrees?

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| You split a tiny task into agents and it takes longer | Over-engineering; the handoff overhead exceeds the task itself | Apply the three-question framework first — if the task is small and uniform, use one session |
| Single session loses context halfway through a big task | Context window exceeded for a complex, multi-phase job | Split into agents with file-based handoffs so each agent starts fresh |
| Agents duplicate work | Roles overlap because specialization was not defined | Write explicit boundary rules: "Do NOT do X" in each contract |
| You cannot decide whether to split | Criteria scores are mixed (e.g., 1 yes, 1 no, 1 maybe) | Default to single session. You can always split later if the single session struggles |

> **Cost warning:** Every additional agent session uses its own context window and API tokens. Two agents doing a job one agent could handle means you pay roughly double for no benefit. Always justify the split before committing to it.
