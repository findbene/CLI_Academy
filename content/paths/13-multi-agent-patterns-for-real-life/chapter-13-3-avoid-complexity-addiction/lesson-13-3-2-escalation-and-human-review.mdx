# Add a review gate between agents

## What you will build
A `review-gate-pipeline/` directory with Researcher, Writer, and Reviewer contracts, an orchestration script with `MAX_REVISIONS`, and output files including `review-verdict.md`.

## Why this matters
Multi-agent workflows are the current frontier of production AI systems. Understanding how to wire agents together, manage handoffs, and avoid complexity spirals is the knowledge gap between hobby projects and production deployments. This path gives you the vocabulary and the architecture patterns used at scale.

## Before you start
A two-agent pipeline can produce output that looks polished but contains factual errors, invented claims, or drift from the source material. In this lesson you will add a third agent — a **Reviewer** — that checks the Writer's article against the Researcher's findings. If the review fails, the pipeline loops back to the Writer with specific feedback. You will cap the loop to prevent runaway iterations.

By the end you will have a three-stage pipeline (Researcher → Writer → Reviewer) with a gated review loop and a hard iteration limit.

### Step 3 — Create the Reviewer contract

This is the new agent. Its job is to compare the article against the findings and produce a pass/fail verdict.

**macOS / Linux:**
```bash
cat > reviewer-CLAUDE.md << 'EOF'
# Reviewer Agent Contract

## Role
You are a factual accuracy reviewer. Compare `article-draft.md` against `research-findings.md`.

## Inputs
- `research-findings.md` (source of truth)
- `article-draft.md` (document under review)

## Outputs
- A file called `review-verdict.md` containing:
  - A single word on line 1: PASS or FAIL
  - If FAIL: a numbered list of specific issues (e.g., "Claim X in paragraph 2 has no source in findings")
- If FAIL, also write `review-feedback.md` with actionable corrections for the Writer.

## Boundaries
- Do NOT modify `research-findings.md` or `article-draft.md`.
- Do NOT rewrite the article yourself.
- Only create `review-verdict.md` and (if FAIL) `review-feedback.md`.

## Stop Condition
Stop after writing your output files.
EOF
```

**Windows PowerShell:**
```powershell
@"
# Reviewer Agent Contract

## Role
You are a factual accuracy reviewer. Compare ``article-draft.md`` against ``research-findings.md``.

## Inputs
- ``research-findings.md`` (source of truth)

## Outputs
- A file called ``review-verdict.md`` containing:
  - A single word on line 1: PASS or FAIL
  - If FAIL: a numbered list of specific issues

## Boundaries

"@ | Set-Content reviewer-CLAUDE.md
```

### Step 4 — Write the orchestration script with review loop (macOS / Linux)

```bash
cat > run-review-pipeline.sh << 'SCRIPT'
#!/usr/bin/env bash
set -euo pipefail

MAX_REVISIONS=2
revision=0

echo "=== Running Researcher ==="
claude --print "Follow researcher-CLAUDE.md exactly."
[ -s research-findings.md ] || { echo "FAIL: no findings"; exit 1; }

echo "=== Running Writer (initial draft) ==="
claude --print "Follow writer-CLAUDE.md exactly."
[ -s article-draft.md ] || { echo "FAIL: no article"; exit 1; }

while [ $revision -lt $MAX_REVISIONS ]; do
  echo "=== Running Reviewer (attempt $((revision + 1))) ==="
  rm -f review-verdict.md review-feedback.md
  claude --print "Follow reviewer-CLAUDE.md exactly."

  if [ ! -s review-verdict.md ]; then
    echo "FAIL: Reviewer produced no verdict"; exit 1
  fi

  verdict=$(head -1 review-verdict.md | tr -d '[:space:]')
  if [ "$verdict" = "PASS" ]; then
    echo "Review PASSED. Pipeline complete."
    exit 0
  fi

  echo "Review FAILED. Sending feedback to Writer..."
  revision=$((revision + 1))
  if [ $revision -lt $MAX_REVISIONS ]; then
    claude --print "Follow writer-CLAUDE.md exactly. Read review-feedback.md and fix the issues."
  fi
done

echo "WARNING: Reached max revisions ($MAX_REVISIONS). Article may still have issues."
echo "Final verdict: $(head -1 review-verdict.md)"
SCRIPT

chmod +x run-review-pipeline.sh
```

### Step 5 — Write the orchestration script (Windows PowerShell)

```powershell
@"
`$ErrorActionPreference = "Stop"
`$MAX_REVISIONS = 2
`$revision = 0

Write-Host "=== Running Researcher ==="
claude --print "Follow researcher-CLAUDE.md exactly."
if (-not (Test-Path research-findings.md) -or (Get-Item research-findings.md).Length -eq 0) {
    Write-Host "FAIL: no findings"; exit 1
}

Write-Host "=== Running Writer (initial draft) ==="
claude --print "Follow writer-CLAUDE.md exactly."
if (-not (Test-Path article-draft.md) -or (Get-Item article-draft.md).Length -eq 0) {
    Write-Host "FAIL: no article"; exit 1
}

while (`$revision -lt `$MAX_REVISIONS) {
    Write-Host "=== Running Reviewer (attempt `$(`$revision + 1)) ==="
    Remove-Item review-verdict.md, review-feedback.md -ErrorAction SilentlyContinue
    claude --print "Follow reviewer-CLAUDE.md exactly."

    if (-not (Test-Path review-verdict.md) -or (Get-Item review-verdict.md).Length -eq 0) {
        Write-Host "FAIL: Reviewer produced no verdict"; exit 1
    }

    `$verdict = (Get-Content review-verdict.md -First 1).Trim()
    if (`$verdict -eq "PASS") {
        Write-Host "Review PASSED. Pipeline complete."
        exit 0
    }

    Write-Host "Review FAILED. Sending feedback to Writer..."
    `$revision++
    if (`$revision -lt `$MAX_REVISIONS) {
        claude --print "Follow writer-CLAUDE.md exactly. Read review-feedback.md and fix the issues."
    }
}

Write-Host "WARNING: Reached max revisions (`$MAX_REVISIONS). Article may still have issues."
Write-Host "Final verdict: `$((Get-Content review-verdict.md -First 1))"
"@ | Set-Content run-review-pipeline.ps1
```

### Step 6 — Run and observe

**macOS / Linux:**
```bash
./run-review-pipeline.sh
```

**Windows PowerShell:**
```powershell
.\run-review-pipeline.ps1
```

Watch the output. You will see one of two outcomes:
- **PASS on first review** — the Writer nailed it. Pipeline exits cleanly.
- **FAIL → revision → re-review** — the loop fires, and the Writer gets a second chance.

After `MAX_REVISIONS`, the pipeline stops regardless. This cap is critical.

### What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| Review loop runs forever | No iteration cap or cap set too high | Always define `MAX_REVISIONS`. Start with 2. |
| Reviewer always says PASS even with errors | Reviewer contract too lenient | Add specific criteria: "Flag any claim with no matching bullet in findings" |
| Reviewer always says FAIL | Criteria too strict or Writer misunderstands feedback | Loosen review criteria or make feedback more specific |
| Writer ignores `review-feedback.md` | Writer contract doesn't mention reading feedback | Add "If review-feedback.md exists, incorporate its feedback" to the Writer contract |
| Runaway API costs from loop iterations | Large context × multiple revisions | Cap `MAX_REVISIONS` at 2–3, use small topics for testing |

> **Cost and safety warning:** A review loop multiplies your token spend. With `MAX_REVISIONS=2`, you could run the Writer up to 3 times and the Reviewer up to 2 times — that is 5 agent sessions for one pipeline run. **Never set `MAX_REVISIONS` to an uncapped value.** An unbounded loop can burn through your API budget in minutes.
