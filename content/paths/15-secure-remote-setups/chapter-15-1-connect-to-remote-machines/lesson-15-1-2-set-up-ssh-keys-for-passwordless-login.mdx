# Set up SSH keys for passwordless login

## Objective

By the end of this lesson you will be able to generate an Ed25519 SSH key pair, copy the public key to a remote server, and log in without a password. You will understand why key-based authentication is safer than passwords and how to protect your private key.

## Walkthrough

Password-based SSH login is convenient for a first connection, but it is vulnerable to brute-force attacks. SSH keys use public-key cryptography: you keep a private key on your machine and place the matching public key on every server you need to access. The private key never leaves your computer.

### Step 1 — Generate an Ed25519 key pair

<StepMeta purpose="Create the modern key pair you will use for safer SSH authentication than passwords alone." action="Run `ssh-keygen -t ed25519 -C \"you@example.com\"`, accept the default path, and set a passphrase on the private key." expectedResult="You end up with `id_ed25519` and `id_ed25519.pub` in your `.ssh` directory and understand which file must never be shared." whyItMatters="Key-based SSH is both safer and more automation-friendly, but only if you protect the private key correctly from the beginning." nextStep="Copy the public key to the remote server so the server can recognize your private key on the next login." askTutor="Explain the difference between the private and public key files and tell me why adding a passphrase is worth the extra step." />

Ed25519 is the modern default — fast, short, and secure.

**macOS / Linux:**

```bash
ssh-keygen -t ed25519 -C "you@example.com"
```

**Windows (PowerShell):**

```powershell
ssh-keygen -t ed25519 -C "you@example.com"
```

The command is identical. Accept the default file location (`~/.ssh/id_ed25519`). When prompted for a passphrase, **set one** — it encrypts the private key on disk so a stolen file alone is not enough to impersonate you.

You now have two files:

| File | Purpose | Share it? |
|---|---|---|
| `~/.ssh/id_ed25519` | Private key — proves your identity | **Never.** Not on Slack, not in Git, not in screenshots. |
| `~/.ssh/id_ed25519.pub` | Public key — goes on servers | Yes — this is safe to share. |

### Step 2 — Copy the public key to the remote server

<StepMeta purpose="Authorize your machine on the server without ever transferring the private key off your computer." action="Use `ssh-copy-id` on macOS or Linux, or the PowerShell pipeline on Windows, to append the public key to `authorized_keys`." expectedResult="The server's `~/.ssh/authorized_keys` file contains your public key and is ready for key-based login." whyItMatters="Only the public key should be uploaded; the whole security model depends on the private key remaining local and protected." nextStep="Test the login path to confirm the server now accepts your key instead of asking for a password every time." askTutor="Help me verify that I copied only the public key and explain what should exist on the server after this step completes." />

**macOS / Linux:**

```bash
ssh-copy-id your-user@your-server.example.com
```

This appends your public key to `~/.ssh/authorized_keys` on the server.

**Windows (PowerShell):**

Windows does not ship `ssh-copy-id`. Use this one-liner instead:

```powershell
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh your-user@your-server.example.com "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

You will need to enter your password one last time.

### Step 3 — Test the key-based login

<StepMeta purpose="Verify that the new SSH key path actually works before you build further workflow on top of it." action="Run `ssh user@host` again and observe whether the server lets you in without a password prompt, aside from an optional key passphrase prompt." expectedResult="You confirm that the key is accepted and that passwordless login is now working as intended." whyItMatters="Authentication changes are only real once tested, and this step prevents later confusion about whether failures come from keys, aliases, or the SSH agent." nextStep="If repeated passphrase prompts are annoying, start the SSH agent so your unlocked key can be cached safely for the session." askTutor="Interpret this login attempt for me and tell me whether the prompt I am seeing is a server password prompt or just the private-key passphrase." />

```bash
ssh your-user@your-server.example.com
```

If the key is loaded you will land on the remote shell with no password prompt. If you set a passphrase on the key, your system's SSH agent will ask for the passphrase once per session.

### Step 4 — Start the SSH agent (if passphrase is not cached)

<StepMeta purpose="Reduce repeated passphrase prompts without giving up the protection of an encrypted private key." action="Start the SSH agent on your platform and add `id_ed25519` so the unlocked key is cached for the current session." expectedResult="Your shell or operating system remembers the unlocked key and future SSH connections reuse it without constant passphrase entry." whyItMatters="The SSH agent makes secure key usage practical day to day, which means you are less tempted to remove the passphrase entirely." nextStep="Lock down the key file permissions so SSH and the operating system treat the private key as restricted material." askTutor="Help me troubleshoot the SSH agent if the key does not stay loaded or if I still get prompted for the passphrase on every connection." />

**macOS / Linux:**

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
```

**Windows (PowerShell — requires admin once):**

```powershell
# Ensure the ssh-agent service is running (run once, elevated)
Set-Service ssh-agent -StartupType Automatic
Start-Service ssh-agent

# Add the key
ssh-add $env:USERPROFILE\.ssh\id_ed25519
```

After this the agent caches your unlocked key for the session so you are not prompted repeatedly.

### Step 5 — Lock down permissions on your key files

<StepMeta purpose="Make sure the private key is only readable by the correct user so SSH will trust it and other accounts cannot." action="Apply the `.ssh` directory and key permission commands for your operating system, then verify SSH no longer complains about open permissions." expectedResult="The private key has restrictive permissions, the public key is readable as needed, and SSH accepts the file without warning." whyItMatters="Open permissions on a private key are both a security problem and a common reason SSH refuses to use the key at all." nextStep="Use the next lesson to simplify repeated server access with SSH config aliases so key-based login also becomes easier to type and remember." askTutor="Review my key-file permissions and tell me whether they are strict enough for SSH and safe enough for a multi-user machine." />

SSH refuses to use a private key with overly open permissions.

**macOS / Linux:**

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

**Windows (PowerShell):**

Windows relies on NTFS ACLs. The simplest fix:

```powershell
icacls $env:USERPROFILE\.ssh\id_ed25519 /inheritance:r /grant:r "$($env:USERNAME):R"
```

This removes inherited permissions and grants only your user read access.

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `Permission denied (publickey)` after copying the key | Wrong permissions on `~/.ssh` or `authorized_keys` on the server | SSH into the server with a password and run `chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys` |
| `WARNING: UNPROTECTED PRIVATE KEY FILE!` | Private key file is readable by others | Fix permissions: `chmod 600 ~/.ssh/id_ed25519` (Linux/Mac) or the `icacls` command above (Windows) |
| `ssh-copy-id: command not found` on Windows | Windows does not include this utility | Use the `type ... | ssh ...` one-liner from Step 2 |
| Agent asks for passphrase every terminal | SSH agent is not running or not persisting | Re-run the agent setup from Step 4; on macOS add `UseKeychain yes` to `~/.ssh/config` |
| Accidentally committed private key to Git | The key is now compromised | Immediately generate a new key pair, remove the old public key from all servers, and revoke the commit with `git filter-repo` |
| Server still prompts for password | Public key not in `authorized_keys`, or server's `PubkeyAuthentication` is set to `no` | Verify the file exists on the server; check `/etc/ssh/sshd_config` for `PubkeyAuthentication yes` |
