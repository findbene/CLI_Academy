# Use SSH config for named connections

## Objective

By the end of this lesson you will be able to create an SSH config file that lets you connect to servers by short nicknames instead of typing full addresses. You will also know how to use `ProxyJump` to reach servers behind a bastion host.

## Walkthrough

Typing `ssh -i ~/.ssh/id_ed25519 -p 2222 deploy@staging-backend-03.us-east.internal.example.com` every time is error-prone. The SSH config file lets you define aliases so you can just type `ssh staging`.

### Step 1 — Create or open the SSH config file

<StepMeta purpose="Prepare the single config file that will hold reusable aliases and defaults for all future SSH connections." action="Create `~/.ssh/config` if needed, ensure the `.ssh` directory exists, and set the config file permissions appropriately for your platform." expectedResult="You have a valid SSH config file ready to accept host aliases and global defaults." whyItMatters="A correct config file location and permission setup is what turns repeated long SSH commands into short, predictable aliases." nextStep="Add a basic host alias so you can connect by nickname instead of typing the full host, user, port, and key each time." askTutor="Check whether my SSH config file is in the right place and has the right permissions before I start adding host blocks." />

**macOS / Linux:**

```bash
mkdir -p ~/.ssh
touch ~/.ssh/config
chmod 600 ~/.ssh/config
```

**Windows (PowerShell):**

```powershell
if (-not (Test-Path $env:USERPROFILE\.ssh)) { New-Item -ItemType Directory $env:USERPROFILE\.ssh }
if (-not (Test-Path $env:USERPROFILE\.ssh\config)) { New-Item -ItemType File $env:USERPROFILE\.ssh\config }
```

Open the file in your preferred editor.

### Step 2 — Add a basic host alias

<StepMeta purpose="Replace long repetitive SSH commands with a short connection nickname that captures the real connection details." action="Add the `Host staging` block, save the config, and test it with `ssh staging`." expectedResult="SSH resolves the alias to the correct hostname, user, port, and identity file automatically." whyItMatters="Aliases reduce mistakes, speed up repeated connections, and make later automation or file-copy commands much easier to read and reuse." nextStep="Add global defaults next so every SSH connection inherits safe, practical behavior like keepalives and single-key use." askTutor="Review this host alias and tell me whether the hostname, user, port, and identity settings are enough for a reliable named connection." />

Add this block to `~/.ssh/config`:

```
Host staging
    HostName staging-backend-03.us-east.internal.example.com
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519
```

Now connect with:

```bash
ssh staging
```

SSH reads the config, fills in the hostname, user, port, and key automatically.

### Step 3 — Add a wildcard default

<StepMeta purpose="Define global SSH behavior once so every host benefits from the same connection hygiene without repetitive flags." action="Add the `Host *` block with agent integration, single-identity use, and keepalive settings, then read back what each directive does." expectedResult="All future SSH connections inherit the defaults unless a host block overrides them explicitly." whyItMatters="Global defaults are a low-effort way to prevent common authentication and idle-timeout issues across every remote setup you use." nextStep="Use `ProxyJump` next so you can reach internal systems through a bastion host with one command." askTutor="Explain these `Host *` defaults in plain language and tell me which ones are most useful for a beginner who keeps getting disconnected." />

Put this at the bottom of the config file so it applies to every connection:

```
Host *
    AddKeysToAgent yes
    IdentitiesOnly yes
    ServerAliveInterval 60
    ServerAliveCountMax 3
```

| Directive | What it does |
|---|---|
| `AddKeysToAgent yes` | Automatically adds keys to the SSH agent after first use |
| `IdentitiesOnly yes` | Prevents SSH from trying every key in `~/.ssh` — uses only the specified `IdentityFile` |
| `ServerAliveInterval 60` | Sends a keepalive packet every 60 seconds to prevent idle disconnects |
| `ServerAliveCountMax 3` | Closes the connection after 3 missed keepalives (3 minutes of silence) |

### Step 4 — ProxyJump through a bastion host

<StepMeta purpose="Learn the standard production pattern for reaching internal machines that are not directly exposed to the internet." action="Add the `bastion` and `internal-db` host blocks with `ProxyJump`, then connect using `ssh internal-db`." expectedResult="SSH automatically hops through the bastion host and reaches the internal machine without a separate manual login step." whyItMatters="Many real remote environments hide sensitive systems behind a jump box, so understanding `ProxyJump` is essential for moving beyond simple single-host labs." nextStep="Reuse your aliases for file transfer so SSH config improves `scp` and `rsync` workflows too, not just interactive shells." askTutor="Review this `ProxyJump` config and tell me how the connection flows through the bastion host when I run a single `ssh internal-db` command." />

Many production environments hide servers behind a bastion (jump box). You SSH into the bastion first, then from there into the target.

```
Host bastion
    HostName bastion.example.com
    User ops
    IdentityFile ~/.ssh/id_ed25519

Host internal-db
    HostName 10.0.4.12
    User dbadmin
    IdentityFile ~/.ssh/id_ed25519
    ProxyJump bastion
```

Now a single command tunnels through automatically:

```bash
ssh internal-db
```

SSH connects to `bastion.example.com` first, then hops to `10.0.4.12` — you never touch the bastion shell.

### Step 5 — Copy files using your aliases

<StepMeta purpose="Prove that the SSH config aliases are useful across the wider remote workflow, not only for interactive login." action="Use your alias in `scp` or `rsync` commands and verify the file transfer works without retyping the full connection details." expectedResult="You can transfer files to the remote host using the alias exactly the same way you connect with `ssh`." whyItMatters="The real productivity gain from SSH config comes when the same short names work everywhere in your remote tooling stack." nextStep="Carry these aliases into cloud-shell and remote-Claude lessons so the remote workflow stays readable as it grows more complex." askTutor="Help me convert a full `scp` or `rsync` command into one that uses my new SSH alias correctly." />

Your SSH config aliases work with `scp` and `rsync` too:

**macOS / Linux:**

```bash
scp ./report.csv staging:/home/deploy/reports/
rsync -avz ./project/ staging:/home/deploy/project/
```

**Windows (PowerShell):**

```powershell
scp .\report.csv staging:/home/deploy/reports/
# rsync is not built into Windows — use WSL or install via scoop/chocolatey
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `Bad configuration option` error | Typo in a directive name, or unsupported option on older OpenSSH | Check spelling; run `ssh -V` and look up supported options for your version |
| `ssh: Could not resolve hostname staging` | Config file not saved to the right path, or bad indentation | Verify the file is at `~/.ssh/config` (or `%USERPROFILE%\.ssh\config` on Windows) |
| ProxyJump fails with `Permission denied` | Your key is not authorized on the bastion host | Copy your public key to the bastion separately: `ssh-copy-id ops@bastion.example.com` |
| `Too many authentication failures` | SSH is trying every key in `~/.ssh` before reaching the right one | Add `IdentitiesOnly yes` in the `Host *` block |
| Config works on Mac but not Windows | Windows OpenSSH does not support `UseKeychain` | Remove `UseKeychain yes` — it is macOS-only |
| Keepalive not preventing disconnects | Corporate proxy or firewall strips keepalive packets | Lower `ServerAliveInterval` to `15` |

Add a host alias to `~/.ssh/config`, then run `ssh -G alias-name` (which prints the resolved configuration without connecting). Confirm that `hostname`, `user`, and `port` match what you defined. Then connect with `ssh alias-name` and verify you reach the remote shell.
