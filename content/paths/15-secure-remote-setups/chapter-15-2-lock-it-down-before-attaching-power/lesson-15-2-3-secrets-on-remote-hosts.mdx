# Secrets on remote hosts

## Objective
Understand the critical importance of secure secret management on remote hosts and formulate a preliminary strategy for storing and rotating these sensitive pieces of information safely.

## Walkthrough
Handling secrets—like API keys, database credentials, or private keys—is one of the most critical security tasks you'll perform on remote hosts. A misstep can have serious consequences, so let's approach this with extreme caution and establish safe practices.

1.  **Understand the Risks:** Never store secrets directly in plain text within your code repository, configuration files that are widely accessible, or unencrypted in user home directories. These are common pitfalls that create severe vulnerabilities.

2.  **Choose a Safe Storage Location:**
    *   **Dedicated Secret Managers (Recommended):** The safest approach is to use a specialized secrets management service (e.g., HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, Google Secret Manager). These tools are designed to securely store, access, and rotate secrets, often integrating directly with applications and providing audit trails.
    *   **Strictly Controlled Files (Minimum Baseline):** If a dedicated manager isn't immediately feasible, store secrets in files with the absolute strictest possible permissions (`chmod 600`), encrypted at rest, and accessible *only* by the specific service account or user that needs them. These files should *never* be committed to version control.
    *   **Environment Variables (Ephemeral Use):** For runtime, secrets can be passed as environment variables to a running process. This is safer than hardcoding but remember they are only ephemeral and not persistent storage.

3.  **Implement a Rotation Strategy:**
    *   **Automate with Secret Managers:** Dedicated secret managers can automatically rotate credentials at predefined intervals without human intervention, greatly reducing risk.
    *   **Regular, Scheduled Rotation:** For manually managed secrets, establish a clear schedule (e.g., quarterly, monthly) for rotation. Treat this as a mandatory operational task.
    *   **Least Privilege:** Ensure the process or user performing the rotation has only the minimum necessary permissions to do so, and nothing more.

4.  **Your Task:** Based on these principles, open a text editor (e.g., `nano` or `vim` on your remote host if you're comfortable, or a local editor) and draft a concise "Remote Secrets Strategy Note." This note should outline:
    *   Where secrets *will* be stored on your host (e.g., "In `~/.my_app/secrets.enc` with `600` permissions, encrypted via GPG" or "Retrieved from AWS Secrets Manager").
    *   How often and by what mechanism they will be rotated (e.g., "Manually rotated quarterly via `ssh` and GPG re-encryption" or "Automated via Vault's dynamic secrets").

Save this note somewhere safe, perhaps in your project documentation, as a foundational security guideline.
