# Use tmux or screen for persistent sessions

## Objective

By the end of this lesson you will be able to start a tmux session on a remote server, run Claude Code inside it, detach from the session, close your SSH connection, reconnect later, and reattach to find Claude Code still running exactly where you left off.

## Walkthrough

When you close your laptop lid or your WiFi drops, your SSH connection dies — and every process attached to it dies too, including Claude Code. Terminal multiplexers like tmux and screen solve this by running your shell inside a persistent session on the server. You detach when you leave and reattach when you come back.

### Step 1 — Install tmux on the server

<StepMeta purpose="Add the session manager that will keep long-running remote shells alive after the SSH connection drops." action="SSH into the server, install `tmux` using the appropriate package manager, and verify the version with `tmux -V`." expectedResult="The server has tmux installed and ready to create named persistent sessions." whyItMatters="Without a multiplexer, every SSH disconnect kills the attached Claude process, which makes remote agent work fragile and frustrating." nextStep="Create a named tmux session so you can start remote work inside a context that survives disconnects." askTutor="Help me confirm tmux installed correctly and explain why it is the preferred fix for dropped SSH sessions in this workflow." />

SSH into your server first, then:

**Ubuntu / Debian:**

```bash
sudo apt update && sudo apt install -y tmux
```

**Amazon Linux / CentOS / RHEL:**

```bash
sudo yum install -y tmux
```

**Verify:**

```bash
tmux -V
# tmux 3.x
```

### Step 2 — Start a named tmux session

<StepMeta purpose="Create the persistent container for your future Claude session instead of running directly in the ephemeral SSH shell." action="Run `tmux new -s work` and note the status bar that indicates you are now inside a named tmux session." expectedResult="A tmux session named `work` is active and everything you launch next will stay attached to that server-side session." whyItMatters="A named session gives you a stable anchor for reconnecting later, which is the whole reason tmux is valuable for remote development." nextStep="Start Claude Code inside this session so the agent process benefits from tmux persistence too." askTutor="Explain what changes when I am inside tmux instead of a normal SSH shell and how the named session helps when I reconnect later." />

```bash
tmux new -s work
```

This creates a new session named `work`. Your terminal now has a green status bar at the bottom — that is tmux's indicator.

Everything you type from here runs inside the tmux session. If your SSH connection drops, the session keeps running on the server.

### Step 3 — Run Claude Code inside tmux

<StepMeta purpose="Move the remote agent session into the persistent tmux context so it can outlive your network connection." action="Change into your project directory and launch `claude` from within the tmux session." expectedResult="Claude Code runs normally, but now its shell is anchored to tmux on the server rather than to the transient SSH connection." whyItMatters="Running Claude Code inside tmux is the main operational habit that makes remote usage resilient instead of brittle." nextStep="Detach from tmux cleanly so you can test that the session and Claude process remain alive after you disconnect." askTutor="Check whether I launched Claude Code from inside tmux correctly and explain how I can tell the process will survive after detaching." />

```bash
cd ~/projects/my-app
claude
```

Claude Code is now running inside a persistent session. You can interact with it normally.

### Step 4 — Detach from the session

<StepMeta purpose="Practice leaving the running tmux session without terminating the work inside it." action="Use `Ctrl+b`, then `d`, return to the normal shell, and then disconnect from SSH with `exit`." expectedResult="The tmux session remains on the server while your SSH connection closes normally." whyItMatters="Detaching correctly is the key difference between a resilient remote workflow and accidentally killing the exact process you wanted to preserve." nextStep="SSH back in later and reattach so you can prove the session is still there with Claude Code intact." askTutor="Explain the detach flow to me and help me distinguish between leaving tmux safely and accidentally exiting the process running inside it." />

Press `Ctrl+b`, then press `d`.

That key sequence means: send the tmux prefix (`Ctrl+b`), then the detach command (`d`). You are back at the regular server shell. Claude Code is still running inside the detached tmux session.

You can now safely disconnect from SSH:

```bash
exit
```

### Step 5 — Reattach to the session later

<StepMeta purpose="Verify the persistence promise by reconnecting after a disconnect and recovering the exact working session." action="SSH back into the server, run `tmux ls`, and reattach with `tmux attach -t work`." expectedResult="You return to the same tmux session and find Claude Code where you left it instead of starting from scratch." whyItMatters="Persistence only matters if recovery is simple and reliable, and this reattach flow is the proof that the pattern works." nextStep="Review the small command set you need most often so tmux becomes a practical tool rather than a one-off trick." askTutor="Help me interpret the `tmux ls` output and tell me what to do if the session name is different or the session is unexpectedly missing." />

SSH back into the server:

```bash
ssh your-user@your-server.example.com
```

List active tmux sessions:

```bash
tmux ls
# work: 1 windows (created Wed Apr  9 10:30:00 2026)
```

Reattach:

```bash
tmux attach -t work
```

Claude Code is right where you left it — no work lost.

### Step 6 — Essential tmux commands

<StepMeta purpose="Give yourself a minimal operating manual for tmux so the tool is usable beyond one attach-detach cycle." action="Read through the listed commands for windows, panes, session listing, killing sessions, and scrolling, then note the ones you will use most." expectedResult="You know the small subset of tmux actions needed to manage everyday remote Claude work confidently." whyItMatters="Tools only become durable habits when you know enough commands to recover, navigate, and clean up without searching docs every time." nextStep="Review the GNU Screen alternative so you still have a fallback pattern if tmux is unavailable on an older server." askTutor="Turn these tmux commands into a small cheat sheet for me focused only on the operations I need for remote Claude sessions." />

| Action | Key / Command |
|---|---|
| Detach | `Ctrl+b` then `d` |
| List sessions | `tmux ls` |
| Attach to a session | `tmux attach -t session-name` |
| Create a new window (tab) | `Ctrl+b` then `c` |
| Switch to next window | `Ctrl+b` then `n` |
| Switch to previous window | `Ctrl+b` then `p` |
| Split pane horizontally | `Ctrl+b` then `"` |
| Split pane vertically | `Ctrl+b` then `%` |
| Move between panes | `Ctrl+b` then arrow key |
| Kill current session | `tmux kill-session -t session-name` |
| Scroll up (copy mode) | `Ctrl+b` then `[`, then arrow keys or Page Up; press `q` to exit |

### Step 7 — Alternative: GNU Screen (if tmux is unavailable)

<StepMeta purpose="Provide a compatible fallback for older or constrained servers that do not have tmux available." action="Review the basic `screen` start, detach, list, and reattach commands and compare them to the tmux flow you already learned." expectedResult="You can keep a remote session alive with GNU Screen even if tmux cannot be installed on the target host." whyItMatters="Remote environments vary, and having one fallback multiplexer pattern keeps you productive when the preferred tool is unavailable." nextStep="Use the mini-project next to combine SSH keys, aliases, remote installs, and tmux into one end-to-end remote development environment." askTutor="Compare tmux and screen for me and tell me when it is worth using screen instead of spending time trying to install tmux on an older server." />

Some older servers have `screen` but not `tmux`:

```bash
screen -S work          # Start a named session
# ... do your work ...
# Ctrl+a then d          # Detach
screen -ls              # List sessions
screen -r work          # Reattach
```

tmux is preferred for its better pane management and scripting, but screen works fine for keeping a single session alive.

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `tmux: command not found` | Not installed | Install with `apt install tmux` or `yum install tmux` |
| `no sessions` when running `tmux ls` | Session was killed (server rebooted, or you ran `tmux kill-session`) | Start a new session with `tmux new -s work` |
| Detach shortcut does not work | Your terminal intercepts `Ctrl+b` (some Windows terminals do this) | Try `Ctrl+b` then release both keys, then press `d`; or remap the prefix in `~/.tmux.conf` |
| Cannot scroll up in tmux | tmux captures scroll by default | Enter copy mode: `Ctrl+b` then `[`, scroll with arrow keys, press `q` to exit |
| `sessions should be nested with care` warning | You are trying to start tmux inside an existing tmux session | You are already in tmux — just create a new window with `Ctrl+b` then `c` |
| Claude Code session loses context after reattach | Claude Code itself timed out (API token expired or idle timeout) | Restart Claude Code inside the tmux session — the tmux session is still alive, only the Claude process exited |
| Server rebooted and all tmux sessions are gone | tmux sessions do not survive reboots | This is expected; restart tmux and Claude Code after a reboot |
