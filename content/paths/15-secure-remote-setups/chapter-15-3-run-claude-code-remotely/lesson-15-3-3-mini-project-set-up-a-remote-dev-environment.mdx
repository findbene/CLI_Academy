# Mini-project: Set up a remote dev environment

## Objective

By the end of this mini-project you will have a complete remote development environment: SSH key authentication, a named SSH config alias, Node.js and Claude Code installed on the server, and a tmux session you can detach from, go offline, and reattach to later. This combines everything from Path 15 into one end-to-end workflow.

## Walkthrough

This project ties together lessons 15.1.1 through 15.3.2. You will go from zero to a working remote Claude Code setup. You need a remote server — a VPS from any provider, a cloud VM, or even a GitHub Codespace.

### Step 1 — Generate SSH keys (if you have not already)

<StepMeta purpose="Begin the full remote setup by establishing secure key-based access from the local machine." action="Generate an Ed25519 key pair if you do not already have one, then verify the public key file exists in `~/.ssh`." expectedResult="Your machine has a usable SSH key pair ready to authorize remote access for the rest of the mini-project." whyItMatters="The whole remote dev environment depends on reliable key-based access, and skipping this foundation only delays the first authentication failure." nextStep="Copy the public key to the remote server and verify passwordless login before adding aliases or tooling." askTutor="Help me decide whether an existing SSH key is good enough for this setup or whether I should generate a fresh dedicated one first." />

Skip this if you completed lesson 15.1.2.

**macOS / Linux:**

```bash
ssh-keygen -t ed25519 -C "you@example.com"
# Accept defaults, set a passphrase
```

**Windows (PowerShell):**

```powershell
ssh-keygen -t ed25519 -C "you@example.com"
# Accept defaults, set a passphrase
```

Verify the key exists:

```bash
ls ~/.ssh/id_ed25519.pub
```

### Step 2 — Copy the public key to the server

<StepMeta purpose="Authorize the local machine on the remote host so every later step can happen over key-based SSH instead of repeated password prompts." action="Upload the public key using `ssh-copy-id` or the PowerShell pipeline, then test the setup with the `echo 'key auth works'` one-off command." expectedResult="The server accepts your key and the test command runs without a password prompt." whyItMatters="Validating key auth early removes ambiguity from the rest of the workflow, especially once aliases and tmux sessions are added." nextStep="Create a stable SSH alias so every later remote command can target the server by a short memorable name." askTutor="Review this key-copy and login test flow with me and tell me what to inspect first if the server still asks for a password afterward." />

**macOS / Linux:**

```bash
ssh-copy-id your-user@your-server-ip
```

**Windows (PowerShell):**

```powershell
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh your-user@your-server-ip "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

Test passwordless login:

```bash
ssh your-user@your-server-ip "echo 'key auth works'"
```

### Step 3 — Create an SSH config alias

<StepMeta purpose="Simplify the full remote workflow by encoding the host details and connection defaults under one short alias." action="Add the `devbox` host block plus the global defaults to `~/.ssh/config`, then test the alias with `ssh devbox \"hostname\"`." expectedResult="You can reach the server by alias and confirm the alias resolves to the intended remote machine." whyItMatters="A reliable alias keeps long remote setup sequences readable and reduces typing mistakes during future daily use." nextStep="SSH to `devbox` and install the core packages needed for remote Claude work." askTutor="Help me verify that this SSH alias and the global defaults are correct before I rely on `devbox` for the rest of the setup." />

Open `~/.ssh/config` (create it if it does not exist) and add:

```
Host devbox
    HostName your-server-ip
    User your-user
    IdentityFile ~/.ssh/id_ed25519

Host *
    AddKeysToAgent yes
    IdentitiesOnly yes
    ServerAliveInterval 60
    ServerAliveCountMax 3
```

Test the alias:

```bash
ssh devbox "hostname"
```

### Step 4 — Install Node.js on the server

<StepMeta purpose="Prepare the remote host with the runtime and utilities needed for a real remote Claude development environment." action="SSH into `devbox`, install Node.js plus `tmux` and `git` for your server family, and verify each tool with its version command." expectedResult="The remote server has Node, npm, tmux, and git installed and ready for project work." whyItMatters="A usable remote dev box needs more than just SSH access; it needs the actual build, agent, and persistence tools in place." nextStep="Install Claude Code globally on the server now that the runtime dependency is satisfied." askTutor="Review the remote package install results and tell me whether this server is now prepared for Claude Code and persistent sessions." />

SSH in and install:

```bash
ssh devbox
```

**On the server (Ubuntu / Debian):**

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs tmux git
node -v
npm -v
tmux -V
```

**On the server (Amazon Linux / RHEL):**

```bash
curl -fsSL https://rpm.nodesource.com/setup_20.x | sudo bash -
sudo yum install -y nodejs tmux git
node -v
npm -v
tmux -V
```

### Step 5 — Install Claude Code on the server

<StepMeta purpose="Add the core agent CLI to the remote dev box so the environment becomes an actual Claude workstation." action="Run the global npm install for Claude Code and verify the install with `claude --version`." expectedResult="Claude Code is installed on the remote server and callable from the shell without error." whyItMatters="The remote environment is only complete once the agent can actually run there, not just once the base packages are present." nextStep="Authenticate Claude Code and then secure the token directory before you move on to project work." askTutor="Help me troubleshoot the remote Claude install here and explain how to handle a permission error without making the whole server workflow sloppy." />

```bash
npm install -g @anthropic-ai/claude-code
claude --version
```

### Step 6 — Authenticate Claude Code

<StepMeta purpose="Complete the account-linking step and secure the resulting remote token before using Claude in anger." action="Run `claude`, finish the OAuth flow in a local browser, and then apply restrictive permissions to `~/.claude`." expectedResult="The remote Claude Code install is authenticated and the token directory is locked down to the current user." whyItMatters="A working but poorly protected remote auth token is still a security liability, so setup is not finished until access is secured." nextStep="Clone or transfer a real project onto the server so the environment has useful code to work on." askTutor="Explain the security tradeoffs of authenticating Claude Code on a remote machine and tell me when I should revoke that token later." />

```bash
claude
```

Copy the authentication URL printed in the terminal, paste it in your local browser, and complete the OAuth flow. Once authenticated, Claude Code is ready.

Lock down the token:

```bash
chmod 700 ~/.claude
```

### Step 7 — Clone a project to work on

<StepMeta purpose="Populate the remote environment with a real codebase so the setup becomes a usable development workspace." action="Create `~/projects`, then either `git clone` the repo on the server or transfer a local copy with `scp`." expectedResult="A project exists under `~/projects/my-app` and is ready to open with Claude Code remotely." whyItMatters="Remote development is only meaningful once the code itself lives where the tools and shell are running." nextStep="Start a tmux session and launch Claude Code inside it so the session remains available after disconnects." askTutor="Compare cloning on the server versus copying from my laptop and recommend the cleaner option for this remote dev box setup." />

```bash
mkdir -p ~/projects
cd ~/projects
git clone https://github.com/you/my-app.git
cd my-app
```

Or transfer a local project:

**From your local terminal (macOS / Linux):**

```bash
scp -r ./my-app devbox:~/projects/
```

**From your local terminal (Windows PowerShell):**

```powershell
scp -r .\my-app devbox:~/projects/
```

### Step 8 — Start a tmux session and launch Claude Code

<StepMeta purpose="Anchor the live remote agent session inside a persistent server-side terminal so the work survives disconnects." action="Start `tmux new -s dev`, change into the project directory, and launch `claude` from inside the session." expectedResult="Claude Code is running inside a named tmux session called `dev` on the remote host." whyItMatters="Persistence is what turns a remote box from a fragile shell into a credible development environment you can step away from." nextStep="Detach and reconnect next so you prove the session actually survives the kind of interruptions this setup is meant to tolerate." askTutor="Check that I launched Claude Code from inside tmux correctly and tell me how to verify the session name and process context before I detach." />

```bash
tmux new -s dev
cd ~/projects/my-app
claude
```

You are now running Claude Code inside a persistent tmux session on your remote server.

### Step 9 — Detach, disconnect, and reattach

<StepMeta purpose="Validate the whole persistence promise of the remote workflow under a realistic disconnect-and-return cycle." action="Detach from tmux, exit SSH, then reconnect with `ssh devbox` and `tmux attach -t dev` to recover the live session." expectedResult="You re-enter the same tmux session and find Claude Code exactly where you left it." whyItMatters="This reattach sequence is the core proof that the remote environment remains useful even when your laptop sleeps or the network drops." nextStep="Finish by learning the full cleanup routine so you can tear the environment down safely when the project is over." askTutor="Help me verify that the reattached session is really the same remote Claude session and not a new shell that just looks similar." />

1. **Detach from tmux:** Press `Ctrl+b` then `d`.
2. **Disconnect from SSH:** Type `exit`.
3. **Close your laptop.** Go get coffee.
4. **Come back, reconnect:**

```bash
ssh devbox
tmux attach -t dev
```

Claude Code is exactly where you left it.

### Step 10 — Clean up when you are done for good

<StepMeta purpose="End the project with a deliberate shutdown routine that preserves work and stops unnecessary remote cost or exposure." action="Exit Claude, kill the tmux session, optionally remove the project directory, and if appropriate shut down or destroy the remote server only after pushing your work." expectedResult="The remote environment is cleaned up safely and no abandoned session or billable machine is left running by accident." whyItMatters="Remote convenience carries cleanup responsibilities, and a disciplined teardown prevents both lost work and unnecessary cost or exposure." nextStep="Use the same end-to-end remote workflow as a reusable template whenever you need a fresh remote Claude development environment in the future." askTutor="Review my remote cleanup checklist and tell me what I should always do before terminating a server that has been running Claude Code and project files." />

When you finish a project and no longer need the remote environment:

```bash
# Inside the tmux session, exit Claude Code
/exit

# Kill the tmux session
tmux kill-session -t dev

# Optionally remove the project
rm -rf ~/projects/my-app
```

If you are using a pay-per-hour VPS, **shut down or destroy the server** to stop charges:

```bash
# From your LOCAL terminal, using your provider's CLI
# DigitalOcean example:
doctl compute droplet delete your-droplet-id

# AWS example:
aws ec2 terminate-instances --instance-ids i-0123456789abcdef0
```

**Safety warning:** Destroying a server deletes everything on it. Push all work to Git before terminating.

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| SSH key auth fails on the new server | Key not in `authorized_keys`, or wrong permissions | Re-run Step 2; check server-side with `ls -la ~/.ssh/` |
| `ssh devbox` says host not found | Typo in `~/.ssh/config` or file in wrong location | Run `ssh -G devbox` to see resolved config; fix the file |
| Node.js install script fails | Server has no `curl` or is behind a proxy | Install curl first: `sudo apt install curl` |
| `npm install -g` permission error | Global npm directory owned by root | Use `sudo npm install -g @anthropic-ai/claude-code` |
| tmux detach shortcut does not work | Terminal capturing `Ctrl+b` | Try pressing `Ctrl+b`, release, then `d`; if still failing, use `tmux detach` command |
| Forgot to push changes before destroying server | Work lost with the server | Always `git push` before shutting down — make it a habit |
| Server still running and billing you | Forgot to stop the VPS after the session | Check your provider's dashboard; set up billing alerts |
| Claude Code says "session expired" after reattach | Anthropic API token expired or idle timeout | Run `claude` again inside the tmux session to restart; your tmux session is fine |

<VerificationBlock>
From your local machine, run `ssh devbox "tmux ls"` and confirm your `dev` session is listed. Then run `ssh devbox` followed by `tmux attach -t dev` and verify Claude Code is responsive inside the session. Finally, from inside the tmux session, run `claude --version` and `git status` to confirm both Claude Code and your project are intact.
</VerificationBlock>
