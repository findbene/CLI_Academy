# Use GitHub Codespaces for cloud development

## Objective

By the end of this lesson you will be able to launch a GitHub Codespace from any repository, use the built-in terminal, install and run Claude Code inside it, and understand the free-tier limits so you do not accidentally incur charges.

## Walkthrough

GitHub Codespaces gives you a full Linux development environment running in Microsoft's cloud. You access it through your browser or VS Code — no local setup required. It is one of the fastest ways to get a working CLI environment when you are on a locked-down computer or want a throwaway sandbox.

### Step 1 — Create a Codespace from a repository

<StepMeta purpose="Launch a disposable remote Linux dev environment without doing any local system setup first." action="Open a GitHub repository, use the Code button's Codespaces tab, and create a Codespace on the branch you want." expectedResult="GitHub provisions a cloud VM, clones the repository, and opens a browser-based VS Code session for that repo." whyItMatters="Codespaces are often the fastest way to get a clean remote CLI environment when the local machine is locked down or temporary." nextStep="Open the terminal inside the Codespace and verify what environment you were actually given." askTutor="Explain what GitHub is creating behind the scenes when I launch a Codespace and when it is a better choice than setting up a local shell." />

1. Go to any GitHub repository (your own or a public one).
2. Click the green **Code** button.
3. Switch to the **Codespaces** tab.
4. Click **Create codespace on main** (or whichever branch you need).

GitHub provisions a virtual machine, clones the repo, and opens VS Code in your browser within about 60 seconds.

### Step 2 — Explore the Codespace terminal

<StepMeta purpose="Orient yourself inside the remote Linux environment before you start installing tools or editing code." action="Open the Codespace terminal and run `uname -a`, `whoami`, and `pwd` to inspect the OS, user, and working directory." expectedResult="You can confirm that the Codespace is a Linux environment, see the default user, and locate the cloned repo under `/workspaces/`." whyItMatters="Remote environments are easier to trust and troubleshoot when you know exactly what machine, user, and filesystem layout you are inside." nextStep="Install Claude Code next so the Codespace becomes a usable cloud-based agent environment, not just a remote editor." askTutor="Interpret these terminal outputs for me and tell me what they imply about how a Codespace is isolated from my local machine." />

At the bottom of the VS Code browser UI, open the **Terminal** panel (`` Ctrl+` `` or **View → Terminal**). This is a real Linux bash shell:

```bash
uname -a
# Linux codespaces-xxxxxx 6.x.x ... x86_64 GNU/Linux

whoami
# codespace

pwd
# /workspaces/your-repo-name
```

Your repo files are already cloned into `/workspaces/`. Git is configured with your GitHub identity.

### Step 3 — Install Claude Code in the Codespace

<StepMeta purpose="Turn the Codespace into a real Claude Code workstation running in the cloud." action="Check the Node.js version, install `@anthropic-ai/claude-code` globally with npm, and verify the install with `claude --version`." expectedResult="Claude Code is installed inside the Codespace and ready for authentication and use." whyItMatters="A remote editor becomes much more useful once the agent tooling runs in the same cloud environment as the repository and shell." nextStep="Review the free-tier limits before you leave the Codespace running longer than you intend." askTutor="Help me verify that the Codespace already has the right Node version and troubleshoot the install if `claude --version` fails afterward." />

The Codespace has Node.js preinstalled:

```bash
node -v
# v20.x.x or later

npm install -g @anthropic-ai/claude-code
claude --version
```

Then authenticate:

```bash
claude
# Follow the browser-based OAuth flow to link your Anthropic account
```

After authentication, Claude Code works exactly as it does locally — but the compute runs on GitHub's cloud.

### Step 4 — Understand the free-tier limits

<StepMeta purpose="Make the cloud environment financially predictable before it turns into an accidental billing leak." action="Read the core-hour, storage, and spending-limit guidance, then note how to stop or delete Codespaces when you are done." expectedResult="You know the monthly allowances, the main cost risks, and the exact controls that prevent surprise charges." whyItMatters="Remote convenience loses its value fast if you forget that idle cloud environments can keep consuming billable resources in the background." nextStep="Connect to the Codespace from your local terminal next so you understand both the browser and local-shell access patterns." askTutor="Summarize the Codespaces cost controls for me and tell me what habits matter most if I want to avoid charges completely." />

| Resource | Free-tier allowance (per month) |
|---|---|
| Core hours (2-core machine) | 60 hours |
| Core hours (4-core machine) | 30 hours |
| Storage | 15 GB |

**Key safety rules:**

- **Stop your Codespace when you are done.** A running Codespace burns core hours even if you are not typing. Click the blue bar at the bottom-left → **Stop Codespace**, or run from any terminal:

```bash
# From your LOCAL terminal (not inside the Codespace)
gh codespace stop
```

- **Delete Codespaces you no longer need.** Idle Codespaces still consume storage.
- **Set a spending limit.** Go to **Settings → Billing → Spending limits → Codespaces** and set it to $0 to prevent any charges.

### Step 5 — Connect to a Codespace from your local terminal

<StepMeta purpose="Show that a Codespace is not limited to the browser UI and can also be used from local tooling via GitHub CLI." action="Install `gh` on your local machine if needed, then run `gh codespace ssh` to open a remote shell into the Codespace." expectedResult="You can access the running Codespace from your local terminal over an SSH-style tunnel with full filesystem access." whyItMatters="Multiple access paths make Codespaces practical in real workflows where you may want a browser editor but a local terminal or automation entry point." nextStep="Compare this managed environment to Google Cloud Shell and AWS CloudShell in the next lesson so you can choose the right remote shell for the job." askTutor="Explain the difference between using a Codespace in the browser and connecting to it from my local terminal with `gh codespace ssh`." />

If you prefer your local editor, install the GitHub CLI:

**macOS / Linux:**

```bash
# macOS
brew install gh

# Ubuntu / Debian
sudo apt install gh
```

**Windows (PowerShell):**

```powershell
winget install GitHub.cli
```

Then SSH into your running Codespace:

```bash
gh codespace ssh
```

This sets up an SSH tunnel — you get a remote shell with full access to the Codespace filesystem.

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| Codespace creation fails | You have hit the free-tier limit or your account lacks Codespaces access | Check **Settings → Billing → Codespaces** for usage; free personal accounts on GitHub Free have Codespaces access |
| `npm install -g` fails with `EACCES` | Permission issue in the Codespace (rare) | Use `sudo npm install -g @anthropic-ai/claude-code` |
| Codespace is running but you forgot about it | Core hours draining silently | Set auto-stop timeout: **Settings → Codespaces → Default idle timeout** to 30 minutes |
| Claude authentication flow does not open a browser | Codespace is running headless | Copy the URL printed in the terminal and paste it into your local browser |
| Files disappear after stopping | You deleted the Codespace instead of stopping it | Stopping preserves files; deleting wipes them — commit and push before deleting |
| Unexpected charges on your GitHub bill | Ran over free-tier hours or storage | Set spending limit to $0 and delete unused Codespaces |
