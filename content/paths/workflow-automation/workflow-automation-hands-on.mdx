## What You'll See

By the end of this lesson, you will have built a working daily project health report workflow. It reads your project, checks for common issues, and generates a structured markdown summary — ready to run manually or on a schedule.

## Why This Matters

Reading about patterns is not the same as building with them. This lesson walks you through constructing a real workflow step by step, applying the sequential chain and guard rail patterns from the previous lesson.

## Core Concepts

### The Project: Daily Health Report

Your workflow will:
1. Count files and code lines by language
2. Check for common code smells (TODOs, console.logs, large files)
3. Summarize recent Git activity
4. Generate a structured markdown report
5. Validate the report is complete before saving

### Step 1: Project Setup

Create a directory for your workflow:

```bash
mkdir -p ~/workflows/health-report
cd ~/workflows/health-report
```

Create the main workflow script:

```bash
touch health-report.sh
chmod +x health-report.sh
```

### Step 2: The Skeleton

Start with the script structure. Every workflow script needs:

```bash
#!/bin/bash
set -euo pipefail

# Configuration
PROJECT_DIR="${1:-.}"          # First argument or current directory
OUTPUT_DIR="./reports"
TIMESTAMP=$(date +%F_%H%M)
REPORT="$OUTPUT_DIR/health-$TIMESTAMP.md"

# Setup
mkdir -p "$OUTPUT_DIR"

echo "=== Health Report: $(date) ==="
echo "Project: $PROJECT_DIR"
```

**Why `set -euo pipefail`?** This is the most important line in any workflow script:
- `-e`: Exit immediately if any command fails
- `-u`: Treat unset variables as errors
- `-o pipefail`: Catch failures in piped commands

Without this, your workflow silently continues after errors and produces garbage output.

### Step 3: Gather Static Metrics

Collect facts that do not need AI:

```bash
# File counts by extension
echo "Gathering file metrics..."
FILE_STATS=$(find "$PROJECT_DIR" \
    -not -path '*/node_modules/*' \
    -not -path '*/.git/*' \
    -not -path '*/.next/*' \
    -type f | \
    sed 's/.*\.//' | \
    sort | uniq -c | sort -rn | head -15)

# Lines of code (quick estimate)
TOTAL_LINES=$(find "$PROJECT_DIR" \
    -not -path '*/node_modules/*' \
    -not -path '*/.git/*' \
    -type f \( -name '*.ts' -o -name '*.tsx' -o -name '*.py' -o -name '*.js' \) | \
    xargs wc -l 2>/dev/null | tail -1 | awk '{print $1}')

# Git activity (last 7 days)
GIT_SUMMARY=$(cd "$PROJECT_DIR" && git log --oneline --since="7 days ago" 2>/dev/null | head -20)
GIT_COUNT=$(echo "$GIT_SUMMARY" | grep -c . 2>/dev/null || echo "0")
```

**Principle:** Gather hard facts with shell commands. Use Claude only for analysis and synthesis.

### Step 4: AI-Powered Analysis

Now use Claude to find code smells:

```bash
echo "Running AI analysis..."

# Find potential issues
ISSUES=$(claude --print "You are a code health checker. Analyze the project at $PROJECT_DIR.

Check for:
1. Files with TODO or FIXME comments (list file:line)
2. Files with console.log or print() statements (list file:line)
3. Files over 300 lines (list file:linecount)
4. Functions over 50 lines (list file:function:linecount)

Output as a markdown table with columns: Category, File, Detail.
If no issues found in a category, say 'None found'.
Limit to 20 issues maximum.")
```

### Step 5: Generate the Report

Assemble everything into a structured report:

```bash
echo "Generating report..."

cat > "$REPORT" << EOF
# Project Health Report
**Generated:** $(date)
**Project:** $PROJECT_DIR

## File Overview
\`\`\`
$FILE_STATS
\`\`\`
**Estimated lines of code:** $TOTAL_LINES

## Git Activity (Last 7 Days)
**Commits:** $GIT_COUNT

\`\`\`
$GIT_SUMMARY
\`\`\`

## Code Health Issues
$ISSUES

---
*Generated by health-report.sh*
EOF
```

### Step 6: Add a Guard Rail

Validate the report before declaring success:

```bash
# Guard rail: verify report has all expected sections
SECTIONS=("File Overview" "Git Activity" "Code Health Issues")
for SECTION in "${SECTIONS[@]}"; do
    if ! grep -q "$SECTION" "$REPORT"; then
        echo "ERROR: Report is missing section: $SECTION"
        exit 1
    fi
done

# Guard rail: report should be at least 20 lines
LINE_COUNT=$(wc -l < "$REPORT")
if [ "$LINE_COUNT" -lt 20 ]; then
    echo "ERROR: Report is suspiciously short ($LINE_COUNT lines)"
    exit 1
fi

echo "Report saved: $REPORT ($LINE_COUNT lines)"
```

### The Complete Script

Put it all together:

```bash
#!/bin/bash
set -euo pipefail

PROJECT_DIR="${1:-.}"
OUTPUT_DIR="./reports"
TIMESTAMP=$(date +%F_%H%M)
REPORT="$OUTPUT_DIR/health-$TIMESTAMP.md"

mkdir -p "$OUTPUT_DIR"
echo "=== Health Report: $(date) ==="
echo "Project: $PROJECT_DIR"

# Static metrics
echo "Gathering file metrics..."
FILE_STATS=$(find "$PROJECT_DIR" \
    -not -path '*/node_modules/*' \
    -not -path '*/.git/*' \
    -not -path '*/.next/*' \
    -type f | \
    sed 's/.*\.//' | sort | uniq -c | sort -rn | head -15)

TOTAL_LINES=$(find "$PROJECT_DIR" \
    -not -path '*/node_modules/*' \
    -not -path '*/.git/*' \
    -type f \( -name '*.ts' -o -name '*.tsx' -o -name '*.py' -o -name '*.js' \) | \
    xargs wc -l 2>/dev/null | tail -1 | awk '{print $1}')

GIT_SUMMARY=$(cd "$PROJECT_DIR" && git log --oneline --since="7 days ago" 2>/dev/null | head -20)
GIT_COUNT=$(echo "$GIT_SUMMARY" | grep -c . 2>/dev/null || echo "0")

# AI analysis
echo "Running AI analysis..."
ISSUES=$(claude --print "You are a code health checker. Analyze the project at $PROJECT_DIR.
Check for:
1. Files with TODO or FIXME comments (list file:line)
2. Files with console.log or print() statements (list file:line)
3. Files over 300 lines (list file:linecount)
4. Functions over 50 lines (list file:function:linecount)
Output as a markdown table. Limit to 20 issues maximum.")

# Generate report
echo "Generating report..."
cat > "$REPORT" << EOF
# Project Health Report
**Generated:** $(date)
**Project:** $PROJECT_DIR

## File Overview
\`\`\`
$FILE_STATS
\`\`\`
**Estimated lines of code:** $TOTAL_LINES

## Git Activity (Last 7 Days)
**Commits:** $GIT_COUNT
\`\`\`
$GIT_SUMMARY
\`\`\`

## Code Health Issues
$ISSUES

---
*Generated by health-report.sh*
EOF

# Guard rails
SECTIONS=("File Overview" "Git Activity" "Code Health Issues")
for SECTION in "${SECTIONS[@]}"; do
    if ! grep -q "$SECTION" "$REPORT"; then
        echo "ERROR: Report is missing section: $SECTION"
        exit 1
    fi
done

LINE_COUNT=$(wc -l < "$REPORT")
if [ "$LINE_COUNT" -lt 20 ]; then
    echo "ERROR: Report too short ($LINE_COUNT lines)"
    exit 1
fi

echo "Report saved: $REPORT ($LINE_COUNT lines)"
```

### Running Your Workflow

```bash
# Run against current directory
./health-report.sh

# Run against a specific project
./health-report.sh ~/my-project

# Check the output
cat reports/health-*.md
```

## Check Your Work

- [ ] Your script runs without errors on a real project
- [ ] The report contains all three sections (File Overview, Git Activity, Code Health Issues)
- [ ] The guard rails catch a problem if you deliberately break a section (e.g., remove the issues step)
- [ ] Running the script twice produces two separate timestamped reports

## Common Issues

**"claude: command not found"**
Claude Code must be installed globally: `npm install -g @anthropic-ai/claude-code`. Verify with `which claude`.

**"Permission denied"**
You forgot `chmod +x health-report.sh`. Run it once.

**The AI analysis step hangs**
The `--print` flag is essential for non-interactive use. Without it, Claude enters interactive mode and waits for input. Also ensure your `ANTHROPIC_API_KEY` is set.

**Report has empty sections**
Check that the project directory actually has the expected files. Run `find "$PROJECT_DIR" -type f | head` to verify.

## Key Takeaways

- Real workflows combine shell commands for hard facts and AI analysis for insights.
- `set -euo pipefail` is non-negotiable in automation scripts.
- Guard rails at the end catch silent failures that would otherwise produce bad output.
- Always save outputs with timestamps so you can compare runs.
