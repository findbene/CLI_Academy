## What You'll See

By the end of this lesson, you will know how to connect workflows to Git (auto-commit results), APIs (fetch and push data), databases (read and write), and notifications (Slack, Discord, email) so your workflows do not exist in isolation.

## Why This Matters

A workflow that generates a report and saves it to a file is useful. A workflow that generates a report, commits it to Git, posts a summary to Slack, and updates a dashboard is a system. Integrations turn isolated scripts into connected automation.

## Core Concepts

### Integration 1: Git Automation

The most common integration. Your workflow generates output and commits it:

```bash
#!/bin/bash
set -euo pipefail

REPORT_DIR="reports"
BRANCH="auto/daily-report"

# Generate the report (your workflow steps here)
./generate-report.sh > "$REPORT_DIR/report-$(date +%F).md"

# Git integration
cd "$PROJECT_DIR"
git checkout -B "$BRANCH"
git add "$REPORT_DIR/"
git commit -m "chore(reports): daily health report $(date +%F)

Generated by health-report workflow.
No manual review required for informational reports."

# Push but do NOT auto-merge
git push origin "$BRANCH" --force-with-lease
```

**Safety rules for Git automation:**
- Always use a dedicated branch, never commit directly to `main`
- Use `--force-with-lease` instead of `--force` to avoid overwriting others' work
- Include `[automated]` or `chore:` prefix in commit messages so they are filterable
- Never auto-merge — let a human approve the PR

**Auto-PR creation** (using GitHub CLI):

```bash
# Create a PR if one does not already exist
if ! gh pr list --head "$BRANCH" --state open | grep -q .; then
    gh pr create \
        --head "$BRANCH" \
        --title "Daily Health Report: $(date +%F)" \
        --body "Automated report. Review before merging." \
        --label "automated"
fi
```

### Integration 2: REST APIs

Fetch data from or push data to web services:

```bash
# Fetch data from an API
RESPONSE=$(curl -s -f "https://api.example.com/data" \
    -H "Authorization: Bearer $API_TOKEN")

# Pass API data to Claude for analysis
ANALYSIS=$(echo "$RESPONSE" | claude --print "Analyze this JSON response. Summarize the key metrics and flag any anomalies.")

# Push results back to an API
curl -s -f -X POST "https://api.example.com/reports" \
    -H "Authorization: Bearer $API_TOKEN" \
    -H "Content-Type: application/json" \
    -d "{\"date\": \"$(date +%F)\", \"summary\": $(echo "$ANALYSIS" | jq -Rs .)}"
```

**Safety rules for API integration:**
- Always use `-f` with curl so it fails on HTTP errors (4xx, 5xx)
- Store tokens in environment variables, never in scripts
- Add timeouts: `curl --connect-timeout 10 --max-time 30`
- Validate API responses before passing them to Claude (check HTTP status, response size, expected fields)

### Integration 3: Databases

Read data for analysis or write workflow results:

```bash
# Read from PostgreSQL
DATA=$(psql "$DATABASE_URL" -t -A -c "
    SELECT json_agg(row_to_json(t))
    FROM (SELECT id, name, status, updated_at
          FROM tasks
          WHERE status = 'pending'
          ORDER BY updated_at DESC
          LIMIT 50) t
")

# Analyze with Claude
SUMMARY=$(echo "$DATA" | claude --print "Analyze these pending tasks. Group by category, identify any that are overdue (older than 7 days), and suggest priorities.")

# Write results back
psql "$DATABASE_URL" -c "
    INSERT INTO reports (type, content, generated_at)
    VALUES ('task_summary', '$(echo "$SUMMARY" | sed "s/'/''/g")', NOW())
"
```

**Safety rules for database integration:**
- Use read-only database credentials for workflows that only read
- Parameterize queries — never interpolate raw AI output into SQL
- Set `statement_timeout` to prevent runaway queries
- Wrap writes in transactions with explicit rollback on failure

**Better pattern for writes** (parameterized):

```bash
psql "$DATABASE_URL" -c "
    INSERT INTO reports (type, content, generated_at)
    VALUES (\$1, \$2, NOW())
" --set=1="task_summary" --set=2="$SUMMARY"
```

### Integration 4: Notifications

Alert humans when workflows complete, fail, or need attention:

**Slack webhook:**
```bash
notify_slack() {
    local MESSAGE="$1"
    local COLOR="${2:-#36a64f}"  # Green by default
    
    curl -s -X POST "$SLACK_WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "{
            \"attachments\": [{
                \"color\": \"$COLOR\",
                \"text\": \"$MESSAGE\",
                \"footer\": \"Workflow Automation\",
                \"ts\": $(date +%s)
            }]
        }"
}

# Usage
notify_slack "Daily report generated: 3 issues found"
notify_slack "WORKFLOW FAILED: health-report.sh exited with error" "#ff0000"
```

**Discord webhook:**
```bash
notify_discord() {
    local MESSAGE="$1"
    curl -s -X POST "$DISCORD_WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "{\"content\": \"$MESSAGE\"}"
}
```

**Email (via mailx):**
```bash
notify_email() {
    local SUBJECT="$1"
    local BODY="$2"
    echo "$BODY" | mailx -s "$SUBJECT" "$NOTIFICATION_EMAIL"
}
```

**ntfy (self-hosted push notifications):**
```bash
notify_ntfy() {
    local MESSAGE="$1"
    local PRIORITY="${2:-default}"
    curl -s -d "$MESSAGE" \
        -H "Priority: $PRIORITY" \
        "https://ntfy.sh/$NTFY_TOPIC"
}
```

### Integration Pattern: Trap on Exit

Ensure notifications fire even when workflows fail:

```bash
#!/bin/bash
set -euo pipefail

# This runs on ANY exit (success or failure)
cleanup() {
    local EXIT_CODE=$?
    if [ $EXIT_CODE -eq 0 ]; then
        notify_slack "Workflow completed successfully"
    else
        notify_slack "WORKFLOW FAILED (exit code: $EXIT_CODE)" "#ff0000"
    fi
}
trap cleanup EXIT

# ... your workflow steps ...
```

### Putting It Together: A Connected Workflow

```bash
#!/bin/bash
set -euo pipefail

trap 'notify_slack "Health report FAILED" "#ff0000"' ERR

# 1. Read data from API
METRICS=$(curl -sf "$METRICS_API/summary")

# 2. Analyze with Claude
REPORT=$(echo "$METRICS" | claude --print "Generate a health report from this data.")

# 3. Save and commit
echo "$REPORT" > reports/health-$(date +%F).md
git add reports/ && git commit -m "chore: daily health report"
git push origin auto/reports --force-with-lease

# 4. Notify
notify_slack "Daily health report ready: $(git log -1 --format='%h')"
```

## Common Issues

**"gh: command not found"**
Install the GitHub CLI: `sudo apt install gh` or `brew install gh`. Authenticate with `gh auth login`.

**"Slack webhook returns 403"**
The webhook URL may have expired or been regenerated. Create a new one in your Slack workspace settings under Incoming Webhooks.

**"Database connection refused"**
Check that `DATABASE_URL` is set and the database accepts connections from your workflow's IP. Use `pg_isready` to test connectivity.

**"Git push rejected"**
The remote branch has diverged. `--force-with-lease` correctly prevents data loss. Investigate what changed on the remote before forcing.

## Key Takeaways

- Git integration should always use dedicated branches, never push directly to main.
- API calls need timeouts, error handling, and token management.
- Database writes must use parameterized queries — never interpolate AI output into SQL.
- Always set up `trap` on exit so failures trigger notifications.
