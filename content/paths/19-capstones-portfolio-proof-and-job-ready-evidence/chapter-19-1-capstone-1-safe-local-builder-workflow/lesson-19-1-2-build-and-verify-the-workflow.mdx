# Build and verify the workflow

## Objective
Build the bounded automation workflow for real, run it against a real repository, capture evidence, and verify that it produces the promised deliverable. By the end of this lesson, you will have a working daily git digest, a run log, and a test that proves the expected output file exists and contains the sections you scoped.

## Walkthrough

### Step 1: Add a prompt file and a real output test

<StepMeta purpose="Create the reusable run input and the structural test that will prove the capstone output meets the scoped contract." action="Write `run-prompt.md` and the platform-specific output-check script so the expected headings and file path are enforced automatically." expectedResult="The capstone has both a reusable run prompt and a concrete test for the promised digest structure." whyItMatters="Capstones become much more credible when they include a repeatable execution prompt and a real test instead of just a one-time successful output." nextStep="Capture the raw repo evidence next so the agent's summary can be traced directly back to observable git inputs." askTutor="Review this run prompt and output test with me and tell me whether the acceptance check is strong enough to prove the workflow delivered what `SCOPE.md` promised." />
Create a reusable run prompt and a basic test that checks the output file for the sections your scope promised.

macOS/Linux:
```bash
cat > run-prompt.md <<'EOF'
Read SCOPE.md and CLAUDE.md. Inspect only this repository. Review the current working tree and the most recent 10 commits. Write output/daily-git-digest.md and logs/run-001.md. Do not commit, push, delete files, or inspect secrets.
EOF

cat > tests/check-output.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

test -f output/daily-git-digest.md
grep -q '^# Daily Git Digest' output/daily-git-digest.md
grep -q '^## Commits Reviewed' output/daily-git-digest.md
grep -q '^## Working Tree Status' output/daily-git-digest.md
grep -q '^## Recommended Next Action' output/daily-git-digest.md

echo 'capstone-1 output check passed'
EOF

chmod +x tests/check-output.sh
```

Windows PowerShell:
```powershell
@"
Read SCOPE.md and CLAUDE.md. Inspect only this repository. Review the current working tree and the most recent 10 commits. Write output/daily-git-digest.md and logs/run-001.md. Do not commit, push, delete files, or inspect secrets.
"@ | Set-Content run-prompt.md

@"
if (-not (Test-Path output/daily-git-digest.md)) { throw 'Missing output/daily-git-digest.md' }

$content = Get-Content output/daily-git-digest.md -Raw
if ($content -notmatch '^# Daily Git Digest') { throw 'Missing title' }
if ($content -notmatch '^## Commits Reviewed') { throw 'Missing commits section' }
if ($content -notmatch '^## Working Tree Status') { throw 'Missing working tree section' }
if ($content -notmatch '^## Recommended Next Action') { throw 'Missing next action section' }

Write-Host 'capstone-1 output check passed'
"@ | Set-Content tests/check-output.ps1
```

### Step 2: Capture the raw repo evidence before the agent summarizes it

<StepMeta purpose="Preserve the exact git signals the capstone will summarize so later claims can be checked against real evidence." action="Write `git status`, recent `git log`, and `git diff --stat` output into the logs folder before running Claude Code." expectedResult="The capstone now has raw evidence files that describe the repo state independently of the generated digest." whyItMatters="Evidence-first design is what makes portfolio artifacts credible and reduces the risk of polished summaries that overstate or invent repo reality." nextStep="Run the bounded workflow next using the evidence files and the prompt contract you just created." askTutor="Help me inspect these raw git evidence files and tell me whether they are enough to support a trustworthy daily digest without additional repo access." />
Your portfolio is stronger if you can show what the workflow looked at before it wrote the digest.

macOS/Linux:
```bash
git status --short > logs/git-status.txt
git log --oneline -10 > logs/git-log.txt
git diff --stat > logs/git-diff-stat.txt
```

Windows PowerShell:
```powershell
git status --short | Set-Content logs/git-status.txt
git log --oneline -10 | Set-Content logs/git-log.txt
git diff --stat | Set-Content logs/git-diff-stat.txt
```

If `git diff --stat` is empty, that is still useful evidence. It means your working tree has no unstaged content changes.

### Step 3: Run the bounded workflow in Claude Code

<StepMeta purpose="Execute the capstone under the exact bounded inputs and outputs you designed rather than improvising at run time." action="Use the provided prompt so Claude Code reads only the prompt, scope, contract, and evidence files, then writes the digest and run log." expectedResult="You get `output/daily-git-digest.md` plus `logs/run-001.md` built from the captured repo evidence." whyItMatters="The capstone proves judgment when the agent is constrained to real evidence and a fixed output surface, not when it has broad freedom to explore the repo." nextStep="Run the automated test and do one human credibility review next so both structure and truthfulness are checked." askTutor="Review this bounded workflow prompt and tell me whether it adequately ties the digest to the captured evidence instead of letting the agent freestyle." />
Open Claude Code in this capstone folder and use the prompt file as your run contract.

Use this prompt:

```text
Read run-prompt.md, SCOPE.md, CLAUDE.md, and the files in logs/. Produce output/daily-git-digest.md and logs/run-001.md. Quote only evidence you can verify from the repo inputs. Stop after those two files.
```

After the run completes, inspect both artifacts.

macOS/Linux:
```bash
cat output/daily-git-digest.md
cat logs/run-001.md
```

Windows PowerShell:
```powershell
Get-Content output/daily-git-digest.md
Get-Content logs/run-001.md
```

### Step 4: Run the test and verify the deliverable matches the scope

<StepMeta purpose="Close the capstone run with both automated structure checks and a human credibility pass against the raw evidence." action="Run the platform-specific test script, then compare the digest to the git evidence files to ensure the content is factual rather than merely well formatted." expectedResult="You have a passing output test and a manual confirmation that the digest's claims match the repo evidence." whyItMatters="Portfolio artifacts need both machine-verifiable structure and human-verifiable truth if they are going to support real claims of skill and judgment." nextStep="Package the capstone into a case study in the next lesson so the build evidence becomes something another person can review and understand quickly." askTutor="Help me review this capstone verification step and tell me whether the digest is not just formatted correctly but also substantively grounded in the raw git inputs." />
Run the test on your platform.

macOS/Linux:
```bash
./tests/check-output.sh
```

Windows PowerShell:
```powershell
powershell -ExecutionPolicy Bypass -File .\tests\check-output.ps1
```

Then do one human check: make sure the digest includes facts, not invented explanations. A passing test proves the file structure; your review proves the content is credible.

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The digest sounds polished but not true | The agent wrote from memory instead of the captured git evidence | Force the prompt to read logs/git-status.txt, logs/git-log.txt, and logs/git-diff-stat.txt |
| The workflow writes more than the planned files | The run prompt does not restate the stop condition clearly | Limit the run to one digest and one log in both run-prompt.md and CLAUDE.md |
| The agent starts recommending commits or pushes | Safety rules focus on style instead of repo risk | Explicitly forbid commit, push, merge, reset, and delete actions |
| The test passes but the digest is still weak | The test only checks headings | Add a human review that compares the digest against the raw git evidence files |
| The workflow inspects a repo with live credentials | You ran the capstone in the wrong folder | Move to a safe local repo and stop if secrets or deployment files are in scope |
