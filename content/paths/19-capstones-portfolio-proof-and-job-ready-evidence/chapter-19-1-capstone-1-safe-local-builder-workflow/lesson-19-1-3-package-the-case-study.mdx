# Package the case study

## What you will build
A portfolio-ready capstone package containing README.md, CASE-STUDY.md, the built digest in output/, and the evidence files that support the claims.

## Why this matters
The portfolio capstone is the artifact that converts learning into professional evidence. Employers and clients cannot assess what they cannot see. A concrete, documented portfolio project answers the question 'show me something you built' — the question that closes every technical hiring conversation.

## Before you start
Turn the automation workflow into portfolio proof. In this lesson, you will package the capstone as a case study, add a README usage section another person could follow, and connect the narrative directly to the evidence files you generated in the build step.


## Walkthrough

### Step 1: Gather the evidence you will cite in the case study
Before writing, make sure the evidence files are present and readable.

macOS/Linux:
```bash
ls logs output tests
cat logs/run-001.md
cat output/daily-git-digest.md
```

Windows PowerShell:
```powershell
Get-ChildItem logs, output, tests
Get-Content logs/run-001.md
Get-Content output/daily-git-digest.md
```

Your case study should point to evidence you actually have, not evidence you wish you had.

### Step 2: Write CASE-STUDY.md as a real portfolio artifact
Use the capstone evidence to write the case study in a hiring-manager-friendly format.

macOS/Linux:
```bash
cat > CASE-STUDY.md <<'EOF'
# Case Study: Daily Git Digest Assistant

## Problem
Teams lose time reconstructing what changed in a repository from scattered commit history and working tree state.

## Approach
I built a bounded Claude Code workflow that reviews one local repository, reads recent git evidence, and writes one markdown digest without making repo changes.

## Result
The workflow produced output/daily-git-digest.md, recorded its actions in logs/run-001.md, and passed the output structure check in tests/check-output.sh or tests/check-output.ps1.

## Evidence
- Raw inputs: logs/git-status.txt, logs/git-log.txt, logs/git-diff-stat.txt
- Agent run log: logs/run-001.md
- Final artifact: output/daily-git-digest.md

## Lessons Learned
- A small scope produces stronger evidence than a broad agent prompt.
- Repo safety rules need to ban irreversible actions explicitly.
- Tests should confirm both file creation and section-level completeness.
EOF
```

Windows PowerShell:
```powershell
@"
```

### Step 3: Add a README.md usage section another operator could follow
Your README should explain how to run the workflow without relying on private context from your chat session.

macOS/Linux:
```bash
cat > README.md <<'EOF'
# Daily Git Digest Assistant

## What this workflow does
Generates one markdown digest of recent repository activity using local git evidence.

## How to run it
1. Confirm you are in the target repository.
2. Review SCOPE.md and CLAUDE.md.
3. Capture fresh git evidence into logs/.
4. Run the bounded Claude Code prompt from run-prompt.md.
5. Verify output/daily-git-digest.md with the test in tests/.

## Safety rules
- No commit, push, merge, reset, or delete actions
- No reading secrets or deployment credentials
- Stop after one digest and one run log

## Expected artifacts
- output/daily-git-digest.md
- logs/run-001.md
- logs/git-status.txt
- logs/git-log.txt
- logs/git-diff-stat.txt
EOF
```


### Step 4: Review the portfolio pack as if someone else will judge it
Open the three public-facing files and read them in order: README.md, CASE-STUDY.md, output/daily-git-digest.md. If a stranger cannot understand the workflow, your packaging is not done.

macOS/Linux:
```bash
cat README.md
cat CASE-STUDY.md
cat output/daily-git-digest.md
```

Windows PowerShell:
```powershell
Get-Content README.md
Get-Content CASE-STUDY.md
Get-Content output/daily-git-digest.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The case study sounds impressive but vague | You did not cite the real run artifacts | Reference the exact files in logs/, output/, and tests/ |
| The README depends on hidden chat context | The usage section explains intent but not execution | Add explicit run order and expected artifacts |
| The portfolio pack hides safety decisions | You only described output quality | Add the no-commit, no-secret, stop-after-one-run controls |
| A reviewer cannot tell whether the workflow actually worked | Result section has no evidence trail | Mention the final output file, run log, and test check by name |
| The case study overclaims business impact | You guessed benefits instead of reporting what the run proved | State actual repo observations and actual workflow outcomes only |
