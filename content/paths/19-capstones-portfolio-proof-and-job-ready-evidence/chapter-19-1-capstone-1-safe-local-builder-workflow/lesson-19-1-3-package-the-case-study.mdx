# Package the case study

## Objective
Turn the automation workflow into portfolio proof. In this lesson, you will package the capstone as a case study, add a README usage section another person could follow, and connect the narrative directly to the evidence files you generated in the build step.

## Walkthrough

### Step 1: Gather the evidence you will cite in the case study

<StepMeta purpose="Collect the exact run artifacts before writing so the case study can only make claims supported by evidence." action="List the logs, output, and test files, then read the run log and final digest to refresh the real evidence you will cite." expectedResult="You know exactly which artifacts are available to support the capstone narrative." whyItMatters="Evidence gathering first prevents the case study from drifting into generic portfolio language or claims that the workflow did not actually prove." nextStep="Write `CASE-STUDY.md` next using only the evidence files you just confirmed exist." askTutor="Review the evidence set with me and tell me whether I have enough real artifacts to write a credible case study or if a key proof file is still missing." />
Before writing, make sure the evidence files are present and readable.

macOS/Linux:
```bash
ls logs output tests
cat logs/run-001.md
cat output/daily-git-digest.md
```

Windows PowerShell:
```powershell
Get-ChildItem logs, output, tests
Get-Content logs/run-001.md
Get-Content output/daily-git-digest.md
```

Your case study should point to evidence you actually have, not evidence you wish you had.

### Step 2: Write CASE-STUDY.md as a real portfolio artifact

<StepMeta purpose="Turn the bounded workflow into a hiring-manager-readable story that maps directly to artifacts and outcomes." action="Create `CASE-STUDY.md` using the problem, approach, result, evidence, and lessons structure, then edit the placeholders to reflect the actual repo, commit count, and observations from your run." expectedResult="You have a case study that explains what you built, what it produced, and what evidence proves it worked." whyItMatters="Portfolio proof is strongest when the story is concrete, bounded, and explicitly backed by named files rather than broad claims about AI capability." nextStep="Write the operator-facing `README.md` usage section next so someone else could run the workflow without your private chat context." askTutor="Audit this case study draft and tell me whether any sentence overclaims what the capstone actually proved or whether the evidence mapping is solid." />
Use the capstone evidence to write the case study in a hiring-manager-friendly format.

macOS/Linux:
```bash
cat > CASE-STUDY.md <<'EOF'
# Case Study: Daily Git Digest Assistant

## Problem
Teams lose time reconstructing what changed in a repository from scattered commit history and working tree state.

## Approach
I built a bounded Claude Code workflow that reviews one local repository, reads recent git evidence, and writes one markdown digest without making repo changes.

## Result
The workflow produced output/daily-git-digest.md, recorded its actions in logs/run-001.md, and passed the output structure check in tests/check-output.sh or tests/check-output.ps1.

## Evidence
- Raw inputs: logs/git-status.txt, logs/git-log.txt, logs/git-diff-stat.txt
- Agent run log: logs/run-001.md
- Final artifact: output/daily-git-digest.md

## Lessons Learned
- A small scope produces stronger evidence than a broad agent prompt.
- Repo safety rules need to ban irreversible actions explicitly.
- Tests should confirm both file creation and section-level completeness.
EOF
```

### Step 3: Add a README.md usage section another operator could follow

<StepMeta purpose="Package the capstone as something another operator could execute and understand without hidden context." action="Write `README.md` with run order, safety rules, and expected artifacts so the workflow can be reproduced from the repo alone." expectedResult="The capstone now includes operator documentation that explains how to run and verify it." whyItMatters="Reusable workflows are more valuable than impressive one-offs, and good usage docs are what make the difference." nextStep="Review the portfolio pack next in the same order an outside reviewer would encounter it." askTutor="Review this README usage section and tell me whether it would let another operator run the workflow safely without needing explanations from me." />
Your README should explain how to run the workflow without relying on private context from your chat session.

macOS/Linux:
```bash
cat > README.md <<'EOF'
# Daily Git Digest Assistant

## What this workflow does
Generates one markdown digest of recent repository activity using local git evidence.

## How to run it
1. Confirm you are in the target repository.
2. Review SCOPE.md and CLAUDE.md.
3. Capture fresh git evidence into logs/.
4. Run the bounded Claude Code prompt from run-prompt.md.
5. Verify output/daily-git-digest.md with the test in tests/.

## Safety rules
- No commit, push, merge, reset, or delete actions
- No reading secrets or deployment credentials
- Stop after one digest and one run log

## Expected artifacts
- output/daily-git-digest.md
- logs/run-001.md
- logs/git-status.txt
- logs/git-log.txt
- logs/git-diff-stat.txt
EOF
```

### Step 4: Review the portfolio pack as if someone else will judge it

<StepMeta purpose="Perform the final external-reader check so the capstone package works as a public-facing proof artifact." action="Read `README.md`, `CASE-STUDY.md`, and the final digest in order and judge whether a stranger could understand the workflow, safety controls, and evidence trail." expectedResult="You have a complete, reviewable portfolio pack for Capstone 1." whyItMatters="The last mile of portfolio work is communication quality, because a strong build still fails as proof if an outside reviewer cannot follow what happened." nextStep="Carry the same evidence-first packaging pattern into Capstone 2, where the workflow centers on knowledge extraction instead of git summarization." askTutor="Help me review this portfolio pack as an outsider would and tell me which part still depends too much on my private knowledge instead of the files themselves." />
Open the three public-facing files and read them in order: README.md, CASE-STUDY.md, output/daily-git-digest.md. If a stranger cannot understand the workflow, your packaging is not done.

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The case study sounds impressive but vague | You did not cite the real run artifacts | Reference the exact files in logs/, output/, and tests/ |
| The README depends on hidden chat context | The usage section explains intent but not execution | Add explicit run order and expected artifacts |
| The portfolio pack hides safety decisions | You only described output quality | Add the no-commit, no-secret, stop-after-one-run controls |
| A reviewer cannot tell whether the workflow actually worked | Result section has no evidence trail | Mention the final output file, run log, and test check by name |
| The case study overclaims business impact | You guessed benefits instead of reporting what the run proved | State actual repo observations and actual workflow outcomes only |
