# Plan the workflow and scope

## Objective
By the end of this lesson, you will have created a foundational plan for your capstone project. This plan will clearly define its user, the problem it solves, its initial boundaries, and how you will determine its success. This crucial step will give you a clear roadmap and help you approach your project with confidence, ensuring you focus on essential features.

## Walkthrough

It's natural to feel excited about building your capstone, but a solid plan is your best friend! It helps you avoid getting overwhelmed and ensures you build something truly valuable. Let's create a clear scope for your project together.

1.  **Prepare Your Workspace Safely:**
    First, let's create a file for your plan. Navigate to your projects directory, or use `mkdir` to make a new one (e.g., `mkdir my_capstone` then `cd my_capstone`). Now, create an empty file named `capstone_plan.md` using the `touch` command:
    ```bash
    touch capstone_plan.md
    ```
    *Why this is safe:* The `touch` command simply creates an empty file if it doesn't exist, or updates its modification timestamp if it does. It doesn't delete, overwrite, or change any existing content without specific flags, making it very safe for file creation.

2.  **Open Your Plan File:**
    Now, open `capstone_plan.md` with a simple text editor like `nano`.
    ```bash
    nano capstone_plan.md
    ```
    *Why this is safe:* `nano` is a beginner-friendly terminal text editor. It only modifies the file you open and won't affect other system files. It's an excellent tool for writing plain text and code. If you prefer, you can use `code capstone_plan.md` if you have VS Code set up.

3.  **Draft Your Capstone Plan:**
    Inside `capstone_plan.md`, write down the following sections. Don't worry about perfection; focus on clarity and being concise. Remember, the AI Tutor will help you "pressure-test" this later, so think about what's *truly essential*.

    *   **User:** Who is this project for? (e.g., "A busy student," "A new command-line user," "Myself to organize notes.")
    *   **Problem:** What specific problem does your project solve for the user? (e.g., "They forget deadlines," "They struggle with file management," "I need a quick way to look up syntax.")
    *   **Boundaries (What it *won't* do initially):** This is crucial. List features you are intentionally *excluding* at first to keep the scope small. (e.g., "It won't save data permanently," "It won't connect to the internet," "It won't have a graphical interface.") *Cutting non-essential features now helps you succeed.*
    *   **Acceptance Criteria:** How will you know your project is *done* and *successful*? These should be measurable actions. (e.g., "The user can input a task," "The user can see their remaining tasks," "The program displays a confirmation message.")

4.  **Save and Exit:**
    Once you're done writing, save the file. In `nano`, press `Ctrl+O` (write out), then `Enter` to confirm the filename, and finally `Ctrl+X` to exit.

You now have a clear, focused plan. This is a huge step towards building a successful capstone project without feeling overwhelmed!
