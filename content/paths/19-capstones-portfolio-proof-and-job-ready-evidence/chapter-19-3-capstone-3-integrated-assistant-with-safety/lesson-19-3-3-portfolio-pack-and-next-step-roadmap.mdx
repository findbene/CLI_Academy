# Portfolio pack and next-step roadmap

## Objective
Assemble the graduation deliverable for Path 19. In this lesson, you will produce portfolio.md with summaries of all three capstones, CV-ready bullets, and a next-step roadmap that shows what work you are ready to own next.

## Walkthrough

### Step 1: Gather the three capstone stories and their evidence
Pull together the artifacts that prove each capstone was real.

macOS/Linux:
```bash
ls ../capstone-1-automation
ls ../capstone-2-knowledge
ls .
```

Windows PowerShell:
```powershell
Get-ChildItem ..\capstone-1-automation
Get-ChildItem ..\capstone-2-knowledge
Get-ChildItem .
```

You should be able to point to:
- automation workflow evidence from Capstone 1
- knowledge workflow evidence from Capstone 2
- integrated pilot evidence from Capstone 3

### Step 2: Write portfolio.md with the three capstone summaries
The summaries should explain the problem, the workflow, the evidence, and the judgment call that kept each project safe.

macOS/Linux:
```bash
cat > portfolio.md <<'EOF'
# Portfolio Pack: CLI Academy Capstones

## Capstone 1: Daily Git Digest Assistant
Designed and built a bounded Claude Code workflow that reviews one local repository, summarizes recent git activity, and produces one digest plus a run log without changing repo state.

## Capstone 2: Codebase TODO Extraction Workflow
Designed a knowledge workflow that turns a raw TODO-style code scan into a structured markdown report with path, line, context, and review priority.

## Capstone 3: Curriculum Change Brief Pilot
Built and piloted an integrated workflow that combines git evidence, real lesson files, and Claude Code summarization to generate a curriculum change brief and document a tested failure mode.

## CV-Ready Bullet Points
- Designed, built, and verified a bounded Claude Code automation workflow that summarized real repository activity into a reusable markdown digest with evidence logs and output checks.
- Built a knowledge-extraction workflow that transformed raw codebase findings into a structured review artifact while handling empty datasets and remote rate-limit scenarios explicitly.
- Piloted an integrated git plus content plus Claude Code workflow, tested a no-op failure mode, and documented an iteration that improved accuracy and safety.

## Next-Step Roadmap
- Build a second integrated workflow that writes a release brief from merged pull requests.
- Add automated regression checks for output quality, not just file existence.
- Expand from single-user operation to teammate handoff with clearer operator docs and review gates.

## Problems I Am Ready To Take On
- repo summaries and engineering handoff briefs
- codebase knowledge extraction and cleanup reporting
- bounded documentation assistants for content-heavy repositories
EOF
```

### Step 3: Tighten the portfolio language until every claim maps to evidence
Use Claude Code as an editor, not a hype machine.

Use this prompt:

```text
Read portfolio.md and the evidence files from all three capstones. Tighten any claim that goes beyond the evidence. Keep the portfolio concrete, credible, and ready for a hiring manager.
```

Then review the result.

macOS/Linux:
```bash
cat portfolio.md
```

Windows PowerShell:
```powershell
Get-Content portfolio.md
```

### Step 4: Do the final graduation check
Read the bullets out loud. If they sound like generic AI output, rewrite them until they point to actual artifacts and actual safety decisions.

macOS/Linux:
```bash
ls ../capstone-1-automation ../capstone-2-knowledge .
```

Windows PowerShell:
```powershell
Get-ChildItem ..\capstone-1-automation, ..\capstone-2-knowledge, .
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The portfolio reads like three unrelated projects | You listed outputs but not the through-line | Emphasize bounded workflows, evidence, and safety decisions in each summary |
| The CV bullets sound generic | They are not tied to named artifacts | Mention the digest, report, brief, logs, and tested failure mode directly |
| The roadmap is vague | It describes ambition instead of next builds | List the exact next workflow, test improvement, and handoff improvement |
| The portfolio overclaims production readiness | You skipped the limits of the pilots | Keep the language at designed, built, piloted, and verified |
| The final pack hides judgment | You only summarized outputs | Call out the no-commit, no-secret, no-invented-summary controls explicitly |
