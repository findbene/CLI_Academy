# Build, pilot, and collect evidence

## Objective
Run the integrated workflow on real repo data, capture the evidence that fed the brief, test one failure mode, and record at least one iteration based on what you learned. By the end of this lesson, you will have a real pilot artifact plus proof that you checked how the workflow behaves when the input is incomplete.

## Walkthrough

### Step 1: Capture the real integration inputs

<StepMeta purpose="Create the evidence seam between git and Claude Code by freezing the exact repo state the pilot is allowed to summarize." action="Write `git status`, changed content files, and content diff stats into the `portfolio-evidence` folder before any assistant summarization begins." expectedResult="You have the three evidence files the integrated pilot will consume as its only source of repo change information." whyItMatters="Integrated capstones are much easier to audit when the non-agent tools produce explicit handoff files instead of leaving the agent to inspect the repo ad hoc." nextStep="Run the pilot next so Claude Code turns those frozen inputs into the curriculum change brief and run log." askTutor="Review these integration inputs and tell me whether they are sufficient and properly bounded for a curriculum-change brief pilot." />
Use git to gather the evidence files the agent will summarize.

macOS/Linux:
```bash
git status --short > portfolio-evidence/git-status.txt
git diff --name-only -- ../content/paths > portfolio-evidence/content-files.txt
git diff --stat -- ../content/paths > portfolio-evidence/content-diff-stat.txt
```

Windows PowerShell:
```powershell
git status --short | Set-Content portfolio-evidence/git-status.txt
git diff --name-only -- ..\content\paths | Set-Content portfolio-evidence/content-files.txt
git diff --stat -- ..\content\paths | Set-Content portfolio-evidence/content-diff-stat.txt
```

These files are the integrated seam: git generated them, and Claude Code will consume them.

### Step 2: Run the pilot and create the curriculum change brief

<StepMeta purpose="Execute the integrated workflow for real using only the approved evidence seam and changed lesson files." action="Use the provided prompt so Claude Code reads the integration plan, contract, git evidence files, and only the lesson files named by `content-files.txt`, then writes the brief and run log." expectedResult="You produce `pilot-output/curriculum-change-brief.md` plus `portfolio-evidence/run-001.md` from real repo evidence." whyItMatters="This pilot is what proves the integrated workflow can combine multiple tools safely without drifting into unrelated content or undocumented repo state." nextStep="Test the empty-file-list failure mode next so the workflow proves it can refuse to invent a summary when no qualifying content files exist." askTutor="Audit this integrated pilot prompt and tell me whether it is strict enough to prevent the agent from reading beyond the changed-file evidence." />
Use Claude Code to turn the evidence into a readable artifact.

Use this prompt:

```text
Read INTEGRATION-PLAN.md, CLAUDE.md, portfolio-evidence/git-status.txt, portfolio-evidence/content-files.txt, and portfolio-evidence/content-diff-stat.txt. Read only the lesson files named in content-files.txt. Write pilot-output/curriculum-change-brief.md and portfolio-evidence/run-001.md. If there are no qualifying content files, write a no-op brief and stop.
```

After the run, inspect the output.

macOS/Linux:
```bash
cat pilot-output/curriculum-change-brief.md
cat portfolio-evidence/run-001.md
```

Windows PowerShell:
```powershell
Get-Content pilot-output/curriculum-change-brief.md
Get-Content portfolio-evidence/run-001.md
```

### Step 3: Test a failure mode on purpose

<StepMeta purpose="Prove that the integrated workflow can stop cleanly instead of fabricating work when a key input seam is empty." action="Create `empty-content-files.txt`, run the failure-mode prompt, and inspect the resulting `failure-test.md` for a controlled no-op response." expectedResult="You have a documented failure-path artifact showing the workflow does not invent content changes when the changed-file list is empty." whyItMatters="A tested no-op path is one of the clearest signs of good judgment in integrated workflows, because it prevents confident summaries of nonexistent evidence." nextStep="Record one iteration next based on the pilot or failure-test results so the capstone shows improvement, not just execution." askTutor="Review this failure-mode test and tell me whether the no-op behavior is clear enough or whether the workflow could still confuse an empty input with a successful summary request." />
Create an empty file list and confirm the workflow refuses to invent a summary.

macOS/Linux:
```bash
: > portfolio-evidence/empty-content-files.txt
```

Windows PowerShell:
```powershell
Set-Content portfolio-evidence/empty-content-files.txt ''
```

Now run this prompt:

```text
Read CLAUDE.md and portfolio-evidence/empty-content-files.txt. Write portfolio-evidence/failure-test.md showing how the workflow behaves when there are no qualifying content files. Do not invent any changes.
```

Inspect the failure result.

macOS/Linux:
```bash
cat portfolio-evidence/failure-test.md
```

Windows PowerShell:
```powershell
Get-Content portfolio-evidence/failure-test.md
```

### Step 4: Record one iteration based on the pilot or failure test

<StepMeta purpose="Show that the integrated capstone improved in response to evidence instead of treating the first run as final." action="Write `portfolio-evidence/iteration-001.md` describing what the pilot or failure test exposed, the contract change you made, and why it improves safety or accuracy." expectedResult="The pilot package now includes a concrete iteration record tied to observed workflow behavior." whyItMatters="Iteration evidence demonstrates maturity: you are not just building assistants, you are refining them based on what the evidence says." nextStep="Use the final lesson to pull all three capstones into one portfolio pack with evidence-backed summaries and a next-step roadmap." askTutor="Help me review this iteration note and tell me whether it clearly connects the observed pilot behavior to a meaningful safety or accuracy improvement." />
If the first brief was too broad or too vague, tighten the contract and document the change.

macOS/Linux:
```bash
cat > portfolio-evidence/iteration-001.md <<'EOF'
# Iteration 001

## Trigger
The pilot or failure test showed the workflow needed a clearer stop rule or tighter content scope.

## Change made
Updated the contract so the workflow stops on an empty content file list and summarizes only lesson files named by the git evidence.

## Why it matters
This prevents invented summaries and keeps the workflow tied to real repo state.
EOF
```

Windows PowerShell:
```powershell
@"
# Iteration 001

## Trigger
The pilot or failure test showed the workflow needed a clearer stop rule or tighter content scope.

## Change made
Updated the contract so the workflow stops on an empty content file list and summarizes only lesson files named by the git evidence.

## Why it matters
This prevents invented summaries and keeps the workflow tied to real repo state.
"@ | Set-Content portfolio-evidence/iteration-001.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The brief summarizes files that never changed | The workflow ignored content-files.txt | Require the prompt to read only files named by the git evidence |
| The pilot mutates the repo instead of summarizing it | The contract did not ban edits during the pilot | Add a no-edit rule for the build phase |
| The failure test still produces a fake summary | No-op behavior is not explicit enough | Force the workflow to write a controlled no qualifying content changes note |
| The pilot hides how the brief was produced | The run log is missing or vague | Record the evidence files read, the files summarized, and the output written |
| The workflow could drift into secret or production data | Safety rules do not stop on credentials or deployed systems | Add hard stop language for secrets, tokens, and production access |
