# Plan the integrated workflow

## Objective
Plan a real workflow that combines multiple tools instead of pretending one agent does everything alone. In this lesson, you will define an integrated workflow that uses Claude Code plus git plus a real content folder to generate a curriculum change brief, and you will document the integration seams, stop rules, and failure test before building it.

## Walkthrough

### Step 1: Create the capstone workspace and define the integrated outcome
This capstone is strongest when each tool has an obvious role.

macOS/Linux:
```bash
mkdir -p capstone-3-integrated/{pilot-output,portfolio-evidence}
cd capstone-3-integrated
pwd
```

Windows PowerShell:
```powershell
New-Item -ItemType Directory -Path capstone-3-integrated, capstone-3-integrated/pilot-output, capstone-3-integrated/portfolio-evidence -Force | Out-Null
Set-Location capstone-3-integrated
Get-Location
```

Recommended integrated workflow for this chapter:
- Tool 1: git provides changed-file evidence
- Tool 2: a real content folder provides the source material
- Tool 3: Claude Code turns the evidence into a readable curriculum-change brief

### Step 2: Write INTEGRATION-PLAN.md with clear seams between the tools
Document where one tool stops and the next begins.

macOS/Linux:
```bash
cat > INTEGRATION-PLAN.md <<'EOF'
# Capstone 3 Integration Plan

## Workflow
Curriculum Change Brief

## Goal
Turn real git and content evidence into one brief that explains what changed in curriculum files and what should happen next.

## Tool seams
- git: produces changed-file lists and diff summaries
- content folder: provides the lesson files referenced by the change list
- Claude Code: reads the evidence and writes the brief

## Inputs
- git status output
- git diff --name-only for content files
- git diff --stat for content files
- selected lesson files from content/paths

## Outputs
- pilot-output/curriculum-change-brief.md
- portfolio-evidence/run-001.md
- portfolio-evidence/failure-test.md
- portfolio-evidence/iteration-001.md

## Stop rules
- do not commit, push, merge, or delete files
- do not summarize files outside the approved content scope
- stop if no qualifying content files exist
- stop if the prompt asks for deployed secrets, tokens, or production access

## Failure test
Run one dry failure mode where the changed-file list is empty and confirm the workflow writes a controlled no-op result instead of inventing changes.
EOF
```

### Step 3: Add the integrated agent contract
The contract should state exactly what Claude Code is allowed to read and write.

macOS/Linux:
```bash
cat > CLAUDE.md <<'EOF'
# Capstone 3 Agent Contract

Allowed actions:
- read git evidence files from portfolio-evidence/
- read approved lesson files under content/paths/
- write pilot-output/curriculum-change-brief.md
- write portfolio-evidence/run-001.md, portfolio-evidence/failure-test.md, and portfolio-evidence/iteration-001.md

Forbidden actions:
- edit lesson files during the pilot
- commit, push, merge, reset, or delete files
- inspect secret-bearing files or deployed credentials
- describe changes that are not present in the git evidence

No-op behavior:
- if the content file list is empty, write a short no qualifying content changes note and stop
EOF
```

### Step 4: Review the integration plan before you pilot it
Use Claude Code to challenge the seams and failure behavior now, before the build step.

Use this prompt:

```text
Read INTEGRATION-PLAN.md and CLAUDE.md. Check whether the tool seams are clear, whether the stop rules are strong enough, and whether the failure test would catch an invented summary. Write the review to portfolio-evidence/plan-review.md.
```

Inspect the review.

macOS/Linux:
```bash
cat portfolio-evidence/plan-review.md
```

Windows PowerShell:
```powershell
Get-Content portfolio-evidence/plan-review.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The workflow is really just one agent with too much freedom | Tool seams were never defined | Assign git, content inspection, and summarization distinct roles in INTEGRATION-PLAN.md |
| The agent summarizes unrelated repo changes | Approved scope does not explicitly narrow to content files | Restrict inputs to content change evidence and named lesson files |
| The pilot could mutate the repo | Contract does not forbid edit and commit actions | Ban edits, commits, pushes, resets, and deletes directly in CLAUDE.md |
| The workflow would invent changes when no files match | No no-op behavior exists | Add a stop rule for an empty content file list |
| The pilot could drift toward production data | Safety rules do not mention secrets or deployed systems | Add an explicit stop on tokens, credentials, and production access |
