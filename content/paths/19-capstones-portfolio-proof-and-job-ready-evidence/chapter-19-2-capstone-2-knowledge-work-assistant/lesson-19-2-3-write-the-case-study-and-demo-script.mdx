# Write the case study and demo script

## Objective
Package the knowledge workflow so another person can understand the problem, see the evidence, and watch a short demo without extra explanation from you. In this lesson, you will create a case study and a 3-minute demo script based on the real TODO report you generated.

## Walkthrough

### Step 1: Collect the evidence you will cite
Your case study should only claim what the artifacts prove.

macOS/Linux:
```bash
cat source-snapshots/raw-findings.txt
cat todo-report.md
cat logs/run-001.md
cat logs/edge-cases.md
```

Windows PowerShell:
```powershell
Get-Content source-snapshots/raw-findings.txt
Get-Content todo-report.md
Get-Content logs/run-001.md
Get-Content logs/edge-cases.md
```

### Step 2: Write CASE-STUDY.md around the real workflow
Use a problem -> approach -> result -> lessons structure and tie every claim to the artifacts you already have.

macOS/Linux:
```bash
cat > CASE-STUDY.md <<'EOF'
# Case Study: Codebase TODO Extraction Workflow

## Problem
Important cleanup work often hides inside scattered TODO, FIXME, and XXX markers that are easy to miss in normal feature work.

## Approach
I designed a bounded Claude Code workflow that reads one captured dataset of raw TODO-style findings, groups the results by folder, and produces one review-friendly markdown report.

## Result
The workflow transformed source-snapshots/raw-findings.txt into todo-report.md, logged the run in logs/run-001.md, and recorded edge-case handling in logs/edge-cases.md.

## Evidence
- Raw dataset: source-snapshots/raw-findings.txt
- Structured report: todo-report.md
- Run log: logs/run-001.md
- Edge-case handling: logs/edge-cases.md

## Lessons Learned
- Capturing the dataset before summarization reduces hallucinated findings.
- Empty-result behavior must be designed up front.
- A schema with path, line number, and context makes the report reusable for real cleanup work.
EOF
```

### Step 3: Write a 3-minute demo script with timestamps
Keep the script short enough that you could actually deliver it in an interview or portfolio video.

macOS/Linux:
```bash
cat > demo-script.md <<'EOF'
# Demo Script: Codebase TODO Extraction Workflow

## 0:00-0:30
I start with a common maintenance problem: TODO markers are everywhere, but they are hard to review in bulk.

## 0:30-1:15
I show the raw dataset in source-snapshots/raw-findings.txt so the audience can see the unprocessed evidence.

## 1:15-2:15
I open todo-report.md and explain how the workflow groups findings by folder, highlights the highest-priority review items, and preserves path and line number detail.

## 2:15-2:45
I show logs/run-001.md and logs/edge-cases.md to prove the workflow behavior was logged and that empty-result or rate-limit scenarios were considered.

## 2:45-3:00
I close by explaining how this workflow turns a noisy code scan into a report a team can actually use during cleanup planning.
EOF
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The demo script sounds generic | You described a workflow pattern instead of your actual run | Mention the real files and real artifact names |
| The case study overstates impact | You inferred outcomes not shown in the evidence | Limit claims to what the dataset, report, and logs prove |
| The script runs longer than 3 minutes | You are trying to show every finding | Focus on raw dataset, final report, and one edge-case note |
| The audience cannot follow the workflow | The presentation jumps between files randomly | Present raw dataset first, report second, logs third |
| The package hides failure handling | Edge cases are not visible in the story | Show logs/edge-cases.md explicitly in the demo |
