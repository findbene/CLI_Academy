# Design the knowledge workflow

## Objective
Design a real knowledge-extraction workflow that produces a structured artifact another person can inspect and reuse. In this lesson, you will define a schema, pick a real source folder, and write the design docs for a TODO extraction workflow that turns a real codebase scan into a clean todo-report.md.

## Walkthrough

### Step 2: Write the design doc with schema, sources, and output format

<StepMeta purpose="Make the knowledge workflow inspectable and reviewable before the agent sees any raw findings." action="Create `WORKFLOW-DESIGN.md` with workflow goal, approved sources, finding schema, output format, and review rules exactly as described." expectedResult="You have a design document that defines what each finding must include and how the final report should be structured." whyItMatters="A strong schema is what turns a noisy grep result into a reusable knowledge artifact instead of an unstructured dump." nextStep="Add the agent contract and edge-case expectations next so the capstone has explicit behavior for empty datasets and remote rate limits." askTutor="Audit this workflow design and tell me whether the schema is specific enough to support consistent reports across multiple runs." />
Make the workflow inspectable before you run it.

macOS/Linux:
```bash
cat > WORKFLOW-DESIGN.md <<'EOF'
# Capstone 2 Workflow Design

## Workflow
Codebase TODO Extraction

## Goal
Summarize unresolved TODO-style markers from a real codebase into one report that is easier to review than raw grep output.

## Sources
- One local repository
- Approved folders only: apps, content, packages, tooling, tests
- Search markers: TODO, FIXME, XXX

## Schema
Each finding in todo-report.md must include:
- file path
- line number
- marker type
- short context snippet
- suggested owner or next action
- priority guess: high, medium, or low

## Output format
todo-report.md must contain:
- report title
- source folders scanned
- total findings count
- grouped findings by folder
- top five items to review first

## Review rules
- Facts first, guesses second
- Do not invent owners if the code gives no clue
- If no findings exist, write a no-findings report instead of faking issues
EOF
```

### Step 3: Add the agent contract and edge-case expectations

<StepMeta purpose="Constrain the capstone's read and write surface and define how it behaves when the dataset is empty or remote limits interfere." action="Write `CLAUDE.md` so it limits scanning to approved folders and artifacts, bans edits and secret inspection, and defines no-findings and rate-limit stop behavior." expectedResult="The capstone now has a contract that keeps the workflow tied to approved sources and explicit edge-case behavior." whyItMatters="Knowledge extraction becomes untrustworthy when it scans too broadly, invents findings on clean datasets, or retries remote sources without limits." nextStep="Run a design review next so Claude Code can challenge the schema and boundaries before the dataset is captured." askTutor="Compare the design and contract for me and tell me whether the source boundaries and edge-case rules are tight enough for a trustworthy knowledge workflow." />
Document how the workflow behaves when the dataset is empty or when you switch from a local scan to a remote source that may rate-limit you.

macOS/Linux:
```bash
cat > CLAUDE.md <<'EOF'
# Capstone 2 Agent Contract

Allowed actions:
- inspect approved folders only
- read raw search results from source-snapshots/
- write todo-report.md
- write logs/run-001.md and logs/edge-cases.md

Forbidden actions:
- edit source files
- commit, push, or delete files
- inspect secrets or environment files
- scan folders outside the approved source list

Edge-case handling:
- if no matches exist, write a no-findings report and say the scan was clean
- if a remote source rate-limits you, log the limit and stop instead of retrying forever
EOF
```

### What can go wrong

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| The report will be impossible to compare across runs | The schema is missing fields like line number or marker type | Add the missing fields before the first run |
| The agent scans too much of the machine | Source folders are not explicitly bounded | Limit the scan to named folders in WORKFLOW-DESIGN.md and CLAUDE.md |
| The workflow would expose secrets during inspection | Secret-bearing files are not excluded | Ban .env and credential files directly in the contract |
| The agent will retry forever on remote data | Rate-limit behavior is not defined | Add a hard stop that logs the rate limit and exits |
| A clean repo would force fake findings | No empty-result behavior exists | Require a no-findings report when the dataset is empty |
