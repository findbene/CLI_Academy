## Why This Matters

Autonomous agents fail silently. A skill can be broken for hours before you notice the morning briefing didn't arrive. Knowing how to read logs and trace failures saves you from assuming problems that aren't there — and finding the actual issues faster.

## Core Concepts

### Reading Logs

The first tool for any OpenClaw problem:

```bash
# Follow live logs
docker compose logs -f openclaw

# Last 100 lines
docker compose logs --tail=100 openclaw

# Filter to a specific component
docker compose logs openclaw 2>&1 | grep "HEARTBEAT\|ERROR\|WARN"
```

Log severity levels used by OpenClaw:
- `[INFO]` — Normal operation
- `[WARN]` — Degraded but running (e.g., rate limit approaching)
- `[ERROR]` — Something failed (e.g., API call rejected)
- `[FATAL]` — Gateway stopped working (e.g., missing required env var)

## Symptom Index

### WhatsApp: "Session expired" or QR code required again

**Cause:** WhatsApp sessions expire if the linked device is inactive for ~14 days, or if you log out all linked devices on your phone.

**Fix:**
1. Web UI → Channels → WhatsApp → Disconnect
2. Re-scan the QR code
3. Sessions can also expire if the Gateway container was stopped for an extended period

**Prevention:** Keep the container running with `restart: unless-stopped` (covered in the VPS lesson).

---

### Telegram: Bot not responding

**Cause (most common):** Wrong token or token was regenerated by BotFather.

**Diagnose:**
```bash
# Test the token directly
curl "https://api.telegram.org/bot<YOUR_TOKEN>/getMe"
```

If the response is `{"ok":false,"error_code":401}` the token is invalid.

**Fix:** Get a new token from @BotFather → `/mybots` → select your bot → API Token → Regenerate. Update `.env`, restart the Gateway.

---

### API Error: "Insufficient funds" or "Invalid API key"

**Diagnose:**
```bash
docker compose logs openclaw 2>&1 | grep "API\|key\|auth\|invalid"
```

**Fix options:**
- `ANTHROPIC_API_KEY invalid` — Regenerate at console.anthropic.com
- `Insufficient funds` — Top up balance or reduce daily cap
- Key pasted with whitespace — Check `.env` for leading/trailing spaces

**Quick key format test:**
```bash
# Anthropic keys start with sk-ant-api03-
# OpenAI keys start with sk-proj- or sk-
# Look for accidental newlines:
cat -A .env | grep API_KEY
# Should show $ at end of line, not ^M$ (Windows line endings)
```

---

### Rate Limit Errors

**Symptom:** Skills run but intermittently return errors containing `429` or `rate_limit_exceeded`.

**Diagnosis:**
```bash
docker compose logs openclaw 2>&1 | grep "429\|rate_limit\|throttl"
```

**Fix:**
```env
# Throttle skill execution to avoid burst rate limits
OPENCLAW_SKILL_DELAY_MS=500    # Wait 500ms between skill API calls
OPENCLAW_MAX_CONCURRENT_SKILLS=2  # Run at most 2 skills at once
```

Anthropic's tier-1 rate limit is typically 60 requests/minute for Haiku. The heartbeat loop with many skills can hit this if not throttled.

### Heartbeat Not Running

**Symptom:** No proactive messages from the agent. Daily briefing didn't arrive.

**Diagnose:**
```bash
docker compose logs openclaw 2>&1 | grep "HEARTBEAT"
```

Expected healthy log:
```
[HEARTBEAT] 2026-04-07 08:00:00 — Cycle started
[HEARTBEAT] 2026-04-07 08:00:02 — 3 skills triggered
[HEARTBEAT] 2026-04-07 08:00:15 — Cycle complete (13s)
```

If you see none: the heartbeat is disabled or the container isn't running.

**Check container status:**
```bash
docker compose ps
```

**Re-enable heartbeat:**
```env
OPENCLAW_HEARTBEAT_ENABLED=true
OPENCLAW_HEARTBEAT_INTERVAL_MINUTES=30
```

### Skill Failing Silently

**Symptom:** You know a skill should have run but it produced no output and no error.

**Diagnose:**
```bash
# Check skill execution history in Web UI
# Or from CLI:
openclaw skill-log --skill daily-briefing --last 10
```

**Common silent failure causes:**
- Skill's required env variables not set (e.g., CALENDAR_API_KEY missing)
- Skill output channel (WhatsApp/Telegram) disconnected
- Skill YAML syntax error — validate with:

```bash
openclaw validate-skill ~/.openclaw/skills/daily-briefing.skill.md
```

---

### Docker Container Restart Loop

**Symptom:** `docker compose ps` shows container in status `Restarting (1)`.

**Diagnose:**
```bash
docker compose logs --tail=50 openclaw
```

Look for the last line before restart — it's usually a configuration error like a missing required env variable or a port conflict.

**Check for port conflicts:**
```bash
# Is port 3301 already in use?
ss -tlnp | grep 3301
# or on macOS
lsof -i :3301
```

Fix: Change `OPENCLAW_PORT` in `.env` if another process owns 3301.

---

### Web UI Unreachable

**Symptom:** `http://localhost:3301` returns "connection refused".

**Check container is running:**
```bash
docker compose ps
# Status should be "Up X minutes", not "Exited"
```

**Check port binding:**
```bash
docker compose port openclaw 3301
```

Expected: `0.0.0.0:3301`

If you're on WSL2 and accessing from Windows: ensure the WSL2 integration in Docker Desktop is enabled and try `http://127.0.0.1:3301`.

---

### Full Reset (Nuclear Option)

If the Gateway is in a broken state you can't diagnose:

```bash
# Stop everything
docker compose down

# Remove persistent volumes (WARNING: this clears saved sessions and learned memory)
docker compose down -v

# Restart fresh
docker compose up -d
```

You'll need to re-scan the WhatsApp QR code and re-link Telegram. Your skill files and `.env` are untouched — only runtime sessions and cached memory are cleared.

## Useful OpenClaw CLI Commands

```bash
# Gateway status
openclaw status

# Test model routing
openclaw test-model --provider anthropic
openclaw test-model --provider ollama

# Validate a skill file
openclaw validate-skill path/to/skill.md

# Force a heartbeat cycle right now
openclaw heartbeat --run-now

# List active channels
openclaw channels list

# View skill execution history
openclaw skill-log --last 20
```
