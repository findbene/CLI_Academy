# Rename and move without breaking things

## Objective

Use `mv` / `Move-Item` and `cp` / `Copy-Item` to rename files, move them between folders, copy them for safety, and undo a bad move. By the end you will have reorganized `tiny-agent-project/` without losing any content.

## Walkthrough

### Step 1 — Check your starting point

<StepMeta purpose="Make sure you are manipulating the correct project before renaming or moving anything." action="Move into tiny-agent-project and list its current files and folders." expectedResult="You see the expected project root with README.md, src, docs, and tests before any reorganization starts." whyItMatters="File moves are safe only when you are certain which project root you are operating inside." nextStep="Once the starting structure looks right, rename one file so you can distinguish renaming from moving." askTutor="I'm on Step 1 of Rename and move without breaking things. Help me verify that this listing really is the correct tiny-agent-project root before I start moving files." />

Make sure you are inside `tiny-agent-project/` and review what you have.

**macOS / Linux**

```bash
cd ~/tiny-agent-project
ls -la
```

**Windows PowerShell**

```powershell
cd ~/tiny-agent-project
Get-ChildItem
```

You should see `README.md`, plus the `src/`, `docs/`, and `tests/` folders from the previous lesson.

### Step 2 — Rename a file

<StepMeta purpose="Practice changing a file's name without changing its content or location." action="Rename README.md to INSTRUCTIONS.md and list the project root again." expectedResult="README.md disappears and INSTRUCTIONS.md appears in the same folder." whyItMatters="Renaming is one of the simplest refactors, and it trains you to verify results immediately after the change." nextStep="After renaming successfully, create a backup copy of a source file before attempting a move." askTutor="I'm on Step 2 of Rename and move without breaking things. Help me confirm this command performs only a rename and does not move the file somewhere else." />

On macOS/Linux, `mv` handles both renaming and moving. On Windows, `Rename-Item` is the clearest rename command. Let's rename `README.md` to `INSTRUCTIONS.md`.

**macOS / Linux**

```bash
mv README.md INSTRUCTIONS.md
ls
```

**Windows PowerShell**

```powershell
Rename-Item README.md INSTRUCTIONS.md
Get-ChildItem -Name
```

Confirm `README.md` is gone and `INSTRUCTIONS.md` appears in its place.

### Step 3 — Copy a file as a backup before moving it

<StepMeta purpose="Build the habit of creating a recovery point before making higher-risk file changes." action="Copy src/main.py to src/main.py.bak and list the src folder contents." expectedResult="Both the original file and the .bak copy exist side by side in src." whyItMatters="Backups turn file moves from stressful one-way actions into reversible experiments." nextStep="With a backup in place, move one file into a different folder and inspect both locations." askTutor="I'm on Step 3 of Rename and move without breaking things. Help me verify that I created a backup copy rather than overwriting the original file." />

Before you move anything important, make a copy. If the move goes wrong you still have the original.

**macOS / Linux**

```bash
cp src/main.py src/main.py.bak
ls src/
```

**Windows PowerShell**

```powershell
Copy-Item src\main.py src\main.py.bak
Get-ChildItem src\ -Name
```

You now have both `main.py` and `main.py.bak` inside `src/`.

### Step 4 — Move a file into a different folder

<StepMeta purpose="Practice moving a file to a new location while tracking both the source and destination." action="Move docs/plan.md to the project root and inspect the root folder plus docs after the move." expectedResult="plan.md appears at the root and the docs folder becomes empty." whyItMatters="Successful file moves depend on checking both where the file arrived and where it left from." nextStep="Once you have confirmed the move, reverse it to prove the operation is easy to undo." askTutor="I'm on Step 4 of Rename and move without breaking things. Help me verify that plan.md moved to the root and explain how to confirm it is no longer inside docs." />

Move the documentation plan file to the project root.

**macOS / Linux**

```bash
mv docs/plan.md ./plan.md
ls
ls docs/
```

**Windows PowerShell**

```powershell
Move-Item docs\plan.md .\plan.md
Get-ChildItem -Name
Get-ChildItem docs\ -Name
```

`plan.md` now lives at the project root. The `docs/` folder is empty.

### Step 5 — Undo the move

<StepMeta purpose="Show that a deliberate file move can be reversed cleanly when you change your mind." action="Move plan.md back into docs and list the docs folder to confirm the restore." expectedResult="docs/plan.md exists again and the project layout returns to the earlier shape." whyItMatters="Being able to undo file moves reduces fear and makes reorganization work feel controlled instead of risky." nextStep="After practicing a safe move and reversal, run the overwrite-risk demo to understand the real danger of careless moves." askTutor="I'm on Step 5 of Rename and move without breaking things. Help me confirm that I truly restored the previous structure and did not create a duplicate or stray file." />

Changed your mind? Move it back.

**macOS / Linux**

```bash
mv plan.md docs/plan.md
ls docs/
```

**Windows PowerShell**

```powershell
Move-Item plan.md docs\plan.md
Get-ChildItem docs\ -Name
```

`plan.md` is safely back inside `docs/`. This is the terminal equivalent of Ctrl+Z.

### Step 6 — See the danger: overwrite risk

<StepMeta purpose="Experience the overwrite hazard in a disposable way so you respect it in real projects." action="Create alpha.txt and beta.txt with different contents, then move beta over alpha and inspect the result." expectedResult="alpha.txt now contains beta's content, proving the original alpha content was overwritten." whyItMatters="The safest move habits come from understanding that some commands replace files silently if you are not careful." nextStep="After the demo, clean up the temporary files and verify the real project structure one last time." askTutor="I'm on Step 6 of Rename and move without breaking things. Explain why beta.txt overwrote alpha.txt here and tell me the safest habits to avoid doing that by accident in a real repo." />

The `mv` command will silently overwrite an existing file if the destination name already exists. Try this safe demonstration.

**macOS / Linux**

```bash
echo "file one" > alpha.txt
echo "file two" > beta.txt
cat alpha.txt
cat beta.txt
mv beta.txt alpha.txt
cat alpha.txt
```

**Windows PowerShell**

```powershell
Set-Content alpha.txt "file one"
Set-Content beta.txt "file two"
Get-Content alpha.txt
Get-Content beta.txt
Move-Item beta.txt alpha.txt -Force
Get-Content alpha.txt
```

After the move, `alpha.txt` contains "file two." The original "file one" content is gone. This is why Step 3 (backup copy) matters.

Clean up the demo files:

**macOS / Linux**

```bash
rm alpha.txt
```

**Windows PowerShell**

```powershell
Remove-Item alpha.txt
```

### What can go wrong

| Problem | Cause | Fix |
|---|---|---|
| `mv: overwrite 'file'?` prompt appears | Some systems alias `mv` to `mv -i` (interactive mode) | Type `y` and Enter to confirm, or use `\mv` to bypass the alias |
| `Move-Item` asks for confirmation | PowerShell `$ConfirmPreference` may be set low | Add `-Force` to the command to skip the prompt |
| File disappears after `mv` and you cannot find it | You accidentally moved it inside a subfolder or used a wrong path | Run `find . -name "filename"` (macOS/Linux) or `Get-ChildItem -Recurse -Filter "filename"` (Windows) to locate it |
| `Rename-Item` fails with "file already exists" | A file with the new name already exists | List the folder first to check, then pick a different name or remove/rename the conflicting file |
| `cp` copies a folder name but not its contents | You need `-r` (recursive) to copy folders: `cp -r src/ src-backup/` | Use `cp -r` (macOS/Linux) or `Copy-Item -Recurse` (Windows) for directories |
