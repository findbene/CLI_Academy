# Setting API Rate Limits: Daily Budgets and Hard Stops

## What you will build

A Python module called `rate_limiter.py` that tracks Anthropic API calls in a local JSON file, raises a `RateLimitExceeded` exception when a configurable daily limit is hit, and resets the counter automatically at midnight. The module exposes a single public function — `check_and_increment()` — that every agent call passes through before touching the API.

## Why this matters

An agent without a spending cap is a liability, not a tool. When a prompt triggers an unexpected loop — a tool result that re-prompts the model, a retry cascade from a transient error, or a poorly scoped task that spawns dozens of sub-calls — costs accumulate silently until your credit card gets declined or your Anthropic account is suspended.

A hard stop in your code enforces a limit the console alone cannot guarantee. The Anthropic console lets you set monthly spend limits and email alerts, but those are asynchronous guardrails. They notify you after money is spent. A local rate limiter stops the agent before the next call exits your process.

Every professional agentic deployment has two layers: a console budget set high enough to catch catastrophic failures, and a per-process daily counter that stops normal runaway loops before they escalate.

## Before you start

- Python 3.9 or later installed (`python3 --version`)
- The `anthropic` package installed in your environment (`pip install anthropic`)
- A working Anthropic API key in your environment as `ANTHROPIC_API_KEY`
- Familiarity with Python exceptions and file I/O (Path 5 level)
- An Anthropic account where you can view usage at console.anthropic.com

## The walkthrough

### Step 1 — Understand the data model

Your rate limiter needs to track two things: the total call count for today, and the date that count belongs to. When the limiter starts, it checks whether the stored date matches today. If it does not match, it resets the count to zero before proceeding. This gives you a midnight-reset daily counter without needing a scheduled job.

The data lives in a single JSON file — no database required:

```json
{"date": "2026-04-13", "count": 47}
```

### Step 3 — Write the rate limiter module

**macOS / Linux:**
```bash
cat > rate_limiter.py << 'PYEOF'
"""
rate_limiter.py — Daily API call limiter for production agents.

Usage:
    from rate_limiter import check_and_increment, RateLimitExceeded

    try:
        check_and_increment()
    except RateLimitExceeded as e:
        print(f"Daily limit reached: {e}")
        raise SystemExit(1)

    # safe to call Anthropic API here
"""

import json
import os
from datetime import date
from pathlib import Path

# Adjust this constant for your production budget.
# At ~$0.003 per 1K input tokens (claude-3-haiku), 500 calls ≈ $1.50/day at average usage.
DEFAULT_DAILY_LIMIT = int(os.environ.get("AGENT_DAILY_CALL_LIMIT", "500"))

# Store the counter next to this module by default, or override via env var.
COUNTER_FILE = Path(os.environ.get("AGENT_COUNTER_FILE", Path(__file__).parent / ".rate_counter.json"))


class RateLimitExceeded(Exception):
    """Raised when the daily API call limit has been reached."""
    pass


def _load_counter() -> dict:
    """Load the current counter from disk. Returns a fresh counter if missing or corrupt."""
    if not COUNTER_FILE.exists():
        return {"date": "", "count": 0}
    try:
        with COUNTER_FILE.open("r", encoding="utf-8") as f:
            data = json.load(f)
        # Validate structure
        if not isinstance(data.get("date"), str) or not isinstance(data.get("count"), int):
            return {"date": "", "count": 0}
        return data
    except (json.JSONDecodeError, OSError):
        return {"date": "", "count": 0}


def _save_counter(data: dict) -> None:
    """Write the counter to disk atomically using a temp file."""
    tmp = COUNTER_FILE.with_suffix(".tmp")
    try:
        with tmp.open("w", encoding="utf-8") as f:
            json.dump(data, f)
        tmp.replace(COUNTER_FILE)
    except OSError as exc:
        raise RuntimeError(f"Could not save rate counter to {COUNTER_FILE}: {exc}") from exc


def check_and_increment(limit: int = DEFAULT_DAILY_LIMIT) -> int:
    """
    Check whether today's call count is below the daily limit.
    If below: increment the counter, persist it, and return the new count.
    If at or above: raise RateLimitExceeded without incrementing.

    Args:
        limit: Maximum API calls allowed today. Defaults to DEFAULT_DAILY_LIMIT.

    Returns:
        The new call count for today after incrementing.

    Raises:
        RateLimitExceeded: If today's count has already reached the limit.
    """
    today = date.today().isoformat()
    data = _load_counter()

    # Midnight reset: if the stored date is not today, start fresh.
    if data["date"] != today:
        data = {"date": today, "count": 0}

    if data["count"] >= limit:
        raise RateLimitExceeded(
            f"Daily API call limit of {limit} reached for {today}. "
            f"Current count: {data['count']}. Resets at midnight."
        )

    data["count"] += 1
    _save_counter(data)
    return data["count"]


def get_today_count() -> int:
    """Return today's call count without incrementing. Returns 0 if no calls made today."""
    today = date.today().isoformat()
    data = _load_counter()
    if data["date"] != today:
        return 0
    return data["count"]


def reset_for_testing() -> None:
    """
    Delete the counter file. Use ONLY in tests — never in production code.
    """
    if COUNTER_FILE.exists():
        COUNTER_FILE.unlink()
PYEOF
echo "rate_limiter.py written"
```

### Step 4 — Write a quick smoke test

This test exercises the three behaviors you care about: normal increment, limit enforcement, and midnight reset.

**macOS / Linux:**
```bash
cat > test_rate_limiter.py << 'PYEOF'
"""Smoke tests for rate_limiter.py"""
import os
import json
from datetime import date, timedelta
from pathlib import Path

# Point counter file at a temp location for testing
os.environ["AGENT_COUNTER_FILE"] = "/tmp/test_rate_counter.json"

from rate_limiter import check_and_increment, get_today_count, reset_for_testing, RateLimitExceeded

def test_increment_and_count():
    reset_for_testing()
    count = check_and_increment(limit=10)
    assert count == 1, f"Expected 1, got {count}"
    assert get_today_count() == 1
    print("PASS: increment_and_count")

def test_limit_enforced():
    reset_for_testing()
    for _ in range(3):
        check_and_increment(limit=3)
    try:
        check_and_increment(limit=3)
        assert False, "Should have raised RateLimitExceeded"
    except RateLimitExceeded as e:
        assert "3" in str(e)
        print("PASS: limit_enforced")

def test_midnight_reset():
    reset_for_testing()
    # Manually write yesterday's date into the counter file
    counter_path = Path(os.environ["AGENT_COUNTER_FILE"])
    yesterday = (date.today() - timedelta(days=1)).isoformat()
    counter_path.write_text(json.dumps({"date": yesterday, "count": 999}))
    # Should reset and allow a call
    count = check_and_increment(limit=10)
    assert count == 1, f"Expected midnight reset to 1, got {count}"
    print("PASS: midnight_reset")

if __name__ == "__main__":
    test_increment_and_count()
    test_limit_enforced()
    test_midnight_reset()
    print("\nAll tests passed.")
PYEOF
python3 test_rate_limiter.py
```

### Step 5 — Set your console spending limit

No code guards you from a compromised API key or a second process you forgot about. Set a hard monthly limit in the Anthropic console as well:

1. Go to [console.anthropic.com](https://console.anthropic.com) and sign in.
2. Navigate to **Settings → Billing → Usage limits**.
3. Set a **Monthly spend limit** — start conservatively (e.g., $20/month for development).
4. Enable **Email notifications** at 75% and 100% of that limit.

The console limit is your backstop. Your `rate_limiter.py` is your first line of defense.

## What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `ModuleNotFoundError: No module named 'rate_limiter'` when running tests | Test file is in a different directory than `rate_limiter.py` | Run both files from the same directory, or add the module path with `sys.path.insert(0, '/path/to/module')` at the top of the test file |
| Counter file grows stale and never resets | Clock on the machine is wrong or the `date` field was manually edited | Check `date.today().isoformat()` output in a Python shell; correct the system clock if it is wrong |
| `PermissionError` saving the counter file | Running in a read-only directory or the `.tmp` file from a crashed run is locked | Set `AGENT_COUNTER_FILE` env var to a writable path such as `/tmp/rate_counter.json` (macOS/Linux) or `$env:TEMP\rate_counter.json` (Windows) |
| Tests pass but production calls still exceed the limit | Two processes share a counter file and both read before either writes (race condition) | For multi-process deployments, replace the file-based counter with a Redis `INCR` with an `EXPIREAT` set to the next midnight timestamp |
| `RateLimitExceeded` fires at count 0 on first run | Counter file from a previous test with a high count still exists | Call `reset_for_testing()` at the start of each test, or delete `.rate_counter.json` manually |
