# Learn the stop, undo, recover loop

## Objective

You will practice three emergency skills on your own machine: stopping a runaway command with `Ctrl+C`, undoing a file change with `git restore`, and recovering a moved file with `mv`. After this lesson, you will have muscle memory for the three most common "oh no" moments in terminal work.

## Walkthrough

### Step 2 — Practice STOP: Kill a running command

<StepMeta purpose="Build reflexes for the fastest safe way to stop a runaway terminal command." action="Run ping, watch it continue, then stop it with Ctrl+C." expectedResult="The endless output stops and your normal prompt returns." whyItMatters="Knowing how to stop a command quickly prevents panic and limits damage when something runs longer than expected." nextStep="After you have the stop reflex, practice restoring a broken file with Git." askTutor="I'm on Step 2 of Learn the stop, undo, recover loop. Explain what Ctrl+C actually does here and tell me what I should expect to see when the command stops cleanly." />

We will start a harmless long-running command, then kill it.

**macOS / Linux:**
```bash
ping localhost
```

**Windows PowerShell:**
```powershell
ping localhost -t
```

Your terminal will start printing lines nonstop. This is normal — `ping` runs forever until you stop it.

**Now press `Ctrl+C`.**

The command stops immediately. The terminal prints a summary and gives you your prompt back. `Ctrl+C` is your emergency brake. It works on every operating system, in every terminal, for almost every command.

> **Important:** `Ctrl+C` does not delete anything. It simply tells the running program to quit. It is always safe to press.

### Step 3 — Practice UNDO: Revert a file with Git

<StepMeta purpose="Learn how to get back to the last known good version of a tracked file." action="Create a test file, commit it, break it on purpose, then restore it with Git." expectedResult="The file returns to its original safe contents after the restore command." whyItMatters="Undoing a bad file change is one of the most common recovery tasks in real AI-assisted repo work." nextStep="Once file restore makes sense, practice recovering a file that was moved to the wrong folder." askTutor="I'm on Step 3 of Learn the stop, undo, recover loop. Help me understand the difference between the broken file state and the restored file state, and tell me the safest restore command to use now." />

First, create a test file and commit it so Git has a "known good" copy.

**macOS / Linux:**
```bash
echo "This file is safe." > test-undo.txt
git add test-undo.txt
git commit -m "Add test file for undo practice"
```

**Windows PowerShell:**
```powershell
"This file is safe." | Set-Content test-undo.txt
git add test-undo.txt
git commit -m "Add test file for undo practice"
```

Now simulate a mistake — overwrite the file with bad content:

**macOS / Linux:**
```bash
echo "OOPS I BROKE EVERYTHING" > test-undo.txt
cat test-undo.txt
```

**Windows PowerShell:**
```powershell
"OOPS I BROKE EVERYTHING" | Set-Content test-undo.txt
Get-Content test-undo.txt
```

You will see the broken text. Now restore the last committed version:

```bash
git restore test-undo.txt
```

(This command is the same on macOS, Linux, and Windows PowerShell.)

Read the file again:

**macOS / Linux:**
```bash
cat test-undo.txt
```

**Windows PowerShell:**
```powershell
Get-Content test-undo.txt
```

It should say `This file is safe.` again. The damage is completely reversed.

### Step 4 — Practice RECOVER: Rescue a moved file

<StepMeta purpose="Show yourself that a moved file is usually recoverable, not lost forever." action="Move the test file out of the folder, confirm it disappeared here, then move it back." expectedResult="The file leaves the projects folder temporarily, then returns with its original contents intact." whyItMatters="Beginners often confuse moved files with deleted files, and that confusion creates unnecessary panic." nextStep="After the move-and-return drill, write down the stop, undo, and recover rules in your own quick-reference playbook." askTutor="I'm on Step 4 of Learn the stop, undo, recover loop. Help me trace where my file went after I moved it and tell me how to bring it back safely." />

Accidentally moving or renaming a file is common. Let's practice finding it again.

**macOS / Linux:**
```bash
mv test-undo.txt ../notes/lost-file.txt
ls test-undo.txt
```

The `ls` command will say "No such file or directory" — the file is gone from this folder. But it is not deleted. Find it:

```bash
ls ../notes/lost-file.txt
```

Move it back:

```bash
mv ../notes/lost-file.txt test-undo.txt
cat test-undo.txt
```

**Windows PowerShell:**
```powershell
Move-Item test-undo.txt ../notes/lost-file.txt
Test-Path test-undo.txt
```

`Test-Path` returns `False` — the file is not here. Find it:

```powershell
Test-Path ../notes/lost-file.txt
```

Returns `True`. Move it back:

```powershell
Move-Item ../notes/lost-file.txt test-undo.txt
Get-Content test-undo.txt
```

Your file is home again.

### Step 5 — Create your undo-recover playbook

<StepMeta purpose="Turn the recovery drills into a short reference you can use when you are stressed." action="Create undo-recover.md with the stop, undo, recover, and nuclear-option notes." expectedResult="A playbook file exists in notes with the key commands and when to use them." whyItMatters="You will not remember every recovery command under pressure, so a short playbook keeps safe defaults close at hand." nextStep="Keep the playbook in notes and revisit it whenever a future lesson introduces a new recovery pattern." askTutor="I'm on Step 5 of Learn the stop, undo, recover loop. Help me review my playbook wording and make sure each command description matches what the command actually does." />

Save what you learned as a quick-reference card.

**macOS / Linux:**
```bash
cat <<'EOF' > ../notes/undo-recover.md
# Stop, Undo, Recover Playbook

## STOP — Ctrl+C
Kill any running terminal command instantly. Safe to press anytime.

## UNDO — git restore filename
Restores a file to its last committed state. Only works on files tracked by Git.

## RECOVER — mv / Move-Item
If a file was moved, not deleted. Check nearby folders. Use mv (macOS/Linux) or Move-Item (Windows) to put it back.

## Nuclear option — git restore .
Reverts ALL uncommitted changes in the current folder. Use when everything is broken and you want to go back to your last commit.
EOF
```

**Windows PowerShell:**
```powershell
@"
# Stop, Undo, Recover Playbook

## STOP — Ctrl+C
Kill any running terminal command instantly. Safe to press anytime.

## UNDO — git restore filename
Restores a file to its last committed state. Only works on files tracked by Git.

## RECOVER — mv / Move-Item
If a file was moved, not deleted. Check nearby folders. Use mv (macOS/Linux) or Move-Item (Windows) to put it back.

## Nuclear option — git restore .
Reverts ALL uncommitted changes in the current folder. Use when everything is broken and you want to go back to your last commit.
"@ | Set-Content ../notes/undo-recover.md
```

### What can go wrong

| Problem | Cause | Fix |
|---|---|---|
| `Ctrl+C` did nothing | Your terminal window was not focused when you pressed it | Click inside the terminal window first, then press `Ctrl+C` again |
| `error: pathspec 'test-undo.txt' did not match any file(s)` | The file was never committed with `git add` and `git commit` | Go back to Step 3 and run the `git add` / `git commit` commands first |
| `fatal: not a git repository` | You are in a folder that was not initialized with `git init` | Run `cd ~/cli-academy` then `git init`, then return to `projects/` and retry |
| `mv: cannot stat 'test-undo.txt': No such file or directory` | You already moved the file in a previous attempt | Check `../notes/` for the file — it may already be there as `lost-file.txt` |
| `git restore` says the path is unknown | The file was never committed, so Git has no version to restore | Go back and run `git add` plus `git commit` for the file before testing restore |
