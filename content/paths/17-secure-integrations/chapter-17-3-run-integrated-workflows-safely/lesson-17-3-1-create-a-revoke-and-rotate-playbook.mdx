# Create a revoke-and-rotate playbook

## What you will build
A revoke-rotate-playbook.md with real commands for revoking via gh auth logout, verifying revocation, rotating with minimum scopes via gh auth login --scopes, and updating dependent systems.

## Why this matters
Integrating Claude with external APIs and services is where most production security incidents happen. A misconfigured webhook, an exposed key, or an over-permissioned OAuth scope can expose entire systems. This path teaches the integration patterns that hold up under a security review.

## Before you start
Token compromise is not a hypothetical. GitHub tokens leak through git history, CI/CD logs, screenshots, and accidental `echo` calls. When that happens, you have one goal: **stop the bleeding fast**. A revoke-and-rotate playbook is a pre-written, tested sequence of commands you can run in under five minutes without having to think under pressure.

In this lesson you will learn the real commands for revoking a GitHub token, verify revocation worked, generate a fresh token with only the scopes you need, and write a playbook document you can execute without hesitation in an emergency.


## Walkthrough

### Step 1 — Understand the three triggers for revocation

| Trigger | Action |
|---|---|
| Token appeared in git history, a log file, or a screenshot | Revoke immediately — assume it is compromised |
| Team member who held the token leaves the organization | Revoke and rotate within 24 hours |
| Scheduled maintenance (every 90 days recommended) | Rotate proactively — no urgency but plan downtime |
| CI/CD provider had a breach | Revoke all tokens that CI accessed |

### Step 2 — Check your current active GitHub sessions

Before revoking, know what you are revoking:

**macOS / Linux:**
```bash
# See which token is active and what it can do
gh auth status

# Print the raw token so you can find it in github.com/settings/tokens
# WARNING: this prints the token value — do not run this in shared terminal sessions
gh auth token
```

**Windows PowerShell:**
```powershell
# See which token is active and what it can do
gh auth status

# Print the raw token value (see warning above)
gh auth token
```

Copy the token prefix (first 8 characters like `gho_XXXX`) so you can identify it on github.com/settings/tokens.

### Step 3 — Revoke the current token

There are two revocation paths:

**Path A: Revoke via gh CLI (fastest)**

```bash
# macOS / Linux / Windows PowerShell — same command
gh auth logout
```

This removes the stored token from your local keychain. The token itself may still be valid on GitHub until you also delete it at github.com/settings/tokens.

**Path B: Revoke directly on GitHub (most thorough)**

1. Go to: **github.com → Settings → Developer settings → Personal access tokens**
2. Find the token by name or the prefix you copied in Step 2
3. Click **Delete** (classic PAT) or **Revoke** (fine-grained PAT)
4. Confirm the modal

After deletion, verify the token no longer works:

**macOS / Linux:**
```bash
# After gh auth logout, this should return an auth error
gh api /user
# Expected: "To get started with GitHub CLI, please run:  gh auth login"
```

**Windows PowerShell:**
```powershell
gh api /user
# Expected: error message indicating no auth token
```

### Step 4 — Rotate: create a fresh token with minimum scopes

**macOS / Linux:**
```bash
# Login fresh — this creates a new token via the OAuth device flow
# Specify only the scopes your workflow needs (comma-separated, no spaces)
gh auth login --scopes repo,read:org

# Confirm the new token is active and scoped correctly
gh auth status
```

**Windows PowerShell:**
```powershell
gh auth login --scopes repo,read:org
gh auth status
```

If you need a raw PAT (for CI/CD secrets) instead of the OAuth flow:
1. Go to **github.com → Settings → Developer settings → Fine-grained personal access tokens → Generate new token**
2. Set expiry to 30–90 days (never choose "No expiration")
3. Select only the repositories and permissions the workflow needs
4. Copy the token and store it immediately in your CI/CD secrets vault — not in a file, not in Slack

### Step 5 — Write the playbook document

**macOS / Linux:**
```bash
cd ~/projects/integration-safety

cat > revoke-rotate-playbook.md << 'EOF'
# Revoke-and-Rotate Playbook: GitHub

**Use this when:** Token leaked / team member departed / scheduled 90-day rotation

## Step 1 — Identify the token (< 1 min)
```
gh auth status
gh auth token   # WARNING: prints raw token — use only in a private terminal
```
Copy the token prefix to identify it on github.com/settings/tokens.

## Step 2 — Revoke locally (< 30 sec)
```
gh auth logout
```

## Step 3 — Revoke on GitHub (< 2 min)
1. Open: github.com/settings/tokens
2. Find and delete/revoke the token
3. Verify: `gh api /user` returns an auth error

## Step 4 — Rotate with minimum scopes (< 2 min)
```
gh auth login --scopes repo,read:org
gh auth status
```

## Step 5 — Update dependent systems (varies)
- Update any CI/CD secrets that used the old token
- Update any `.env` files on servers that stored the old token
- Notify the team that credentials have rotated

## Recovery time target: < 5 minutes for Steps 1–4
EOF

cat revoke-rotate-playbook.md
```

**Windows PowerShell:**
```powershell
Set-Location "$HOME\projects\integration-safety"

$playbook = @'

**Use this when:** Token leaked / team member departed / scheduled 90-day rotation

## Step 1 — Identify the token (< 1 min)
  gh auth status
  gh auth token   # WARNING: prints raw token — use only in a private terminal
Copy the token prefix to identify it on github.com/settings/tokens.

1. Open: github.com/settings/tokens
2. Find and delete/revoke the token
3. Verify: gh api /user returns an auth error

## Step 4 — Rotate with minimum scopes (< 2 min)
  gh auth login --scopes repo,read:org
  gh auth status

- Update any CI/CD secrets that used the old token
- Update any .env files on servers that stored the old token

## Recovery time target: < 5 minutes for Steps 1-4
'@

$playbook | Out-File -FilePath "revoke-rotate-playbook.md" -Encoding utf8
Get-Content revoke-rotate-playbook.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `gh auth logout` succeeds but `gh api /user` still works | A second token for the same account is stored | Check `gh auth status --show-token`; look for multiple hosts |
| Token deleted on GitHub but still works in curl | GitHub token invalidation can take up to 60 seconds to propagate | Wait 60 seconds and retry |
| `gh auth login --scopes` prompts for browser but browser does not open | Headless/SSH environment | Use `gh auth login --with-token` and paste a pre-generated PAT |
| CI/CD pipeline breaks after rotation | Pipeline was using the old token stored as a secret | Update the secret in your CI/CD provider's vault and re-run the pipeline |
| Playbook not found during an incident | Stored only locally | Commit `revoke-rotate-playbook.md` to a private runbooks repo or a password-manager note |

> **Safety warning — `gh auth token` is dangerous in shared contexts:** Running `gh auth token` in a screen-share session, CI log, or recorded terminal session exposes your raw OAuth token. That token is as powerful as your GitHub password with the granted scopes. Only run it in an isolated private terminal, treat the output as a secret, and delete it from your terminal scrollback immediately.
