# Mini project: safe integrated assistant

## Objective

This mini-project ties together every lesson in Path 17. You will build a complete secure integration end-to-end: map the surface, choose minimum scopes, connect with real credentials, validate read-only, execute one bounded write, audit the risk, and write the rotation playbook. The deliverable is a folder of interconnected documents that proves you can connect external services responsibly.

The integration target is GitHub — the same service you have been working with throughout this path.

### Step 4 — Validate read-only access

<StepMeta purpose="Prove the integrated assistant can read the target GitHub data safely before any persistent change enters the flow." action="Set the target repository variables, run the issues GET request, and write `read-only-results.md` with the endpoint and timestamp." expectedResult="You have both successful read-only API output and a project file documenting that read-only validation passed." whyItMatters="The safe order is still read first, then bounded write, even inside a larger end-to-end integration project." nextStep="Perform one reversible write next so the assistant package includes proof that it can change state and roll it back deliberately." askTutor="Help me evaluate whether this read-only validation is strong enough for the mini-project or whether another safe check should happen before any write." />

Set your target repo and confirm read access works:

**macOS / Linux:**
```bash
export GH_OWNER="your-github-username"
export GH_REPO="your-test-repo"

# Read current open issues
gh api /repos/$GH_OWNER/$GH_REPO/issues \
    --jq '[.[] | {number: .number, title: .title}]' | head -30

echo "Read-only validation: PASSED" > read-only-results.md
echo "Endpoint tested: GET /repos/$GH_OWNER/$GH_REPO/issues" >> read-only-results.md
echo "Date: $(date)" >> read-only-results.md
cat read-only-results.md
```

**Windows PowerShell:**

### Step 5 — Execute one bounded write and roll it back

<StepMeta purpose="Show that the mini-project can perform exactly one external mutation and reverse it cleanly as part of the documented workflow." action="Create the labeled test issue, read it back, close it, and write `bounded-write-record.md` with the issue number and rollback proof." expectedResult="The project now contains evidence of a successful POST write followed by a verified PATCH rollback to closed state." whyItMatters="A real secure integration must prove both controlled write capability and disciplined cleanup of external state." nextStep="Complete the risk audit and rotation playbook next so the project covers operational safety, not just successful API usage." askTutor="Review this mini-project bounded write and tell me whether the rollback proof is strong enough to count as a safe reversible mutation." />

**macOS / Linux:**
```bash
# Create the test issue
RESPONSE=$(gh api /repos/$GH_OWNER/$GH_REPO/issues \
    -X POST \
    --field title="[safe-assistant-test] Mini-project 17.3.3" \
    --field body="Test issue created in CLI Academy lesson 17.3.3. Safe to close immediately.")

ISSUE_NUMBER=$(echo "$RESPONSE" | jq -r '.number')
ISSUE_URL=$(echo "$RESPONSE" | jq -r '.html_url')
echo "Created issue #$ISSUE_NUMBER: $ISSUE_URL"

# Read back to confirm
gh api /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER --jq '{number:.number,state:.state,title:.title}'

# Rollback: close it
gh api /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER -X PATCH --field state="closed"
gh api /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER --jq '.state'
# Expected: "closed"

cat > bounded-write-record.md << EOF
# Bounded Write Record — Mini Project
Date: $(date)
Issue created: #$ISSUE_NUMBER ($ISSUE_URL)
Write: POST /repos/$GH_OWNER/$GH_REPO/issues
Rollback: PATCH / state=closed
Verified closed: yes
EOF
cat bounded-write-record.md
```

**Windows PowerShell:**
```powershell
# Create the test issue
$resp = gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues `
    -X POST `
    --field title="[safe-assistant-test] Mini-project 17.3.3" `
    --field body="Test issue created in CLI Academy lesson 17.3.3. Safe to close immediately." |
    ConvertFrom-Json

$issueNumber = $resp.number
Write-Host "Created issue #$issueNumber: $($resp.html_url)"

# Read back to confirm
gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber |
    ConvertFrom-Json | Select-Object number, state, title | Format-List

# Rollback: close it
gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber -X PATCH --field state="closed"
gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber |
    ConvertFrom-Json | Select-Object state
# Expected: state = closed

@"
# Bounded Write Record — Mini Project
Date: $(Get-Date -Format 'yyyy-MM-dd HH:mm')
Issue created: #$issueNumber ($($resp.html_url))
Write: POST /repos/$env:GH_OWNER/$env:GH_REPO/issues
Rollback: PATCH / state=closed
Verified closed: yes
"@ | Out-File -FilePath "bounded-write-record.md" -Encoding utf8
Get-Content bounded-write-record.md
```

### Step 6 — Complete the risk audit and rotation playbook

<StepMeta purpose="Add the operational safety layer that keeps the integrated assistant manageable after the happy-path API calls are done." action="Create `risk-audit.md` and `revoke-rotate-playbook.md`, listing key risks, mitigations, trigger conditions, and the command sequence for rapid credential response." expectedResult="The mini-project now includes both a standing risk review and an incident-response playbook." whyItMatters="Secure integrations fail in production when they have no living risk model and no fast response path for leaked or overpowered tokens." nextStep="Finish by writing the README and verifying that every required file exists with content." askTutor="Check these mini-project safety documents and tell me whether they are practical enough to use during a real incident or pre-production review." />

**macOS / Linux:**
```bash
cat > risk-audit.md << 'EOF'
# Risk Audit — Safe Integrated Assistant

| # | Risk                                         | Rating     | Mitigation                                          |
|---|----------------------------------------------|------------|-----------------------------------------------------|
| 1 | `repo` scope grants write to all repos       | 🟠 High    | Rotate to fine-grained PAT for single repo          |
| 2 | Token not rotated on schedule                | 🟡 Medium  | Set 90-day calendar reminder for gh auth login      |
| 3 | Issues created by this workflow create noise | 🟢 Low     | Use a bot account or label prefix for filtering     |
| 4 | API rate limit not monitored                 | 🟡 Medium  | Add `gh api /rate_limit` check before workflow runs |

## Actions before production use
- [ ] Create fine-grained PAT scoped to one repo with issues:write only
- [ ] Add rate limit pre-check
- [ ] Schedule 90-day rotation reminder
EOF
cat risk-audit.md

cat > revoke-rotate-playbook.md << 'EOF'
# Revoke-and-Rotate Playbook — Safe Integrated Assistant

## Trigger conditions
- Token value appears in any log, paste, commit, or screenshot
- Team member with access to this token leaves
- 90-day scheduled rotation

## Playbook steps (target: < 5 min for steps 1–4)

1. `gh auth status`                          — identify active token
2. `gh auth logout`                          — remove from local keychain
3. Visit github.com/settings/tokens → Delete — revoke on GitHub
4. `gh api /user` → confirm auth error       — verify revocation
5. `gh auth login --scopes repo,read:org`    — rotate with minimum scopes
6. `gh auth status`                          — verify new token active
7. Update CI/CD secrets vault with new token — restore dependent workflows
EOF
cat revoke-rotate-playbook.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| Step 5 POST returns 403 | No write access to the target repo | Switch to a repo you own; check `gh auth status` for `repo` scope |
| Issue created on a team repo | Wrong `GH_OWNER`/`GH_REPO` set | Close the issue manually and apologize; reset variables carefully |
| Files have 0 lines in the verification loop | Used `touch` without content | Re-run the `cat >` or PowerShell heredoc blocks to fill each file |
| Risk audit has only "Low" findings | Under-rating real risks | Re-read the scope decisions; `repo` on a multi-repo account is always at least Medium |
| Playbook not tested end-to-end | Skipped validation | Run `gh auth logout` and `gh auth login` in a test flow before trusting the playbook |

> **Safety warning — integration debt compounds.** The gap between a well-documented integration and an undocumented one starts small and grows every time someone adds a new scope, copies a token into a new place, or forgets to rotate on schedule. This folder is not a one-time exercise — bookmark it, keep it current, and treat it as a living document.
