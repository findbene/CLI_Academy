# Choose minimum viable scopes

## Objective

Granting an integration the `repo` scope when it only needs to read issues is like giving someone your car keys when they asked to borrow a pen. The **principle of least privilege** means every token or OAuth grant covers only what the workflow actually needs — nothing more.

In this lesson you will inspect the scopes on a real GitHub token using the `gh` CLI, identify over-permissioned scopes on your current auth, and create a `scope-decisions.md` file that maps each required action to its true minimum scope.

## Walkthrough

### Step 1 — Check what scopes your current GitHub token carries

`gh auth status` prints the OAuth scopes that your active GitHub token was granted. Run it now:

**macOS / Linux:**
```bash
gh auth status
```

**Windows PowerShell:**
```powershell
gh auth status
```

Expected output (example):
```
github.com
    ✓ Logged in to github.com as yourname (keyring)
    ✓ Git operations for github.com configured to use https protocol.
    ✓ Token: gho_••••••••••••
    ✓ Token scopes: 'gist', 'read:org', 'repo', 'workflow'
```

Write down the `Token scopes` line — you will compare it against the minimum required in the next step.

If `gh` is not installed:
- **macOS:** `brew install gh`
- **Windows:** `winget install GitHub.cli` (then restart your terminal)

### Step 2 — Map required actions to minimum scopes

| Action needed | Minimum scope | Over-broad alternative to avoid |
|---|---|---|
| Read public repo data | *(no scope — public API)* | `repo` |
| Read issues on a private repo | `repo` | `admin:repo` |
| Read org membership | `read:org` | `admin:org` |
| Read Actions workflow runs | `actions:read` | `workflow` |
| Read gists | `gist` | — |
| Push commits to a private repo | `repo` | — |
| Create a pull request | `repo` | — |

> **GitHub scope caveat:** GitHub's scope system is coarser than most providers. There is no `repo:read` — the full `repo` scope is the smallest available for private repository access. This makes scope hygiene especially important: holding `repo` means you also have write access. Document this limitation explicitly in your decision record.

### Step 3 — Create your scope decision record

**macOS / Linux:**
```bash
cd ~/projects/integration-safety

cat > scope-decisions.md << 'EOF'
# Scope Decision Record

## Workflow: Read issues from a private GitHub repo

| Action                  | Minimum scope          | Justification                                         |
|-------------------------|------------------------|-------------------------------------------------------|
| Authenticate with API   | (built into gh CLI)    | gh manages token storage securely                     |
| List repo issues        | `repo`                 | Smallest scope GitHub offers for private repo access  |
| Read org membership     | `read:org`             | Required if filtering by org team                     |

## Over-permissioned scopes found in current token

Review `gh auth status` output and list any scopes this workflow does NOT need:
- `workflow` — only needed for CI/CD pipeline management
- `gist` — only needed for Gist CRUD operations
- `delete_repo` — CRITICAL: remove unless you explicitly delete repos programmatically

## Recommended action
Rotate the token scoped to only `repo` and `read:org` for this workflow.
Command: gh auth login --scopes repo,read:org
EOF

cat scope-decisions.md
```

**Windows PowerShell:**
```powershell
Set-Location "$HOME\projects\integration-safety"
```

### Step 4 — Verify the scope decision record exists

**macOS / Linux:**
```bash
ls -la scope-decisions.md
grep "Minimum scope" scope-decisions.md
```

**Windows PowerShell:**
```powershell
Get-Item scope-decisions.md
Select-String -Path scope-decisions.md -Pattern "Minimum scope"
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `gh auth status` shows no scopes | Token created via automation without scope selection | Run `gh auth login` interactively and choose scopes manually |
| `gh: command not found` | GitHub CLI not installed | `brew install gh` (Mac) / `winget install GitHub.cli` (Windows) |
| Decision record shows `admin` everywhere | Reviewing defaults without challenging them | Ask per row: what is the minimum single action this step requires? |
| `gh auth login --scopes` does not shrink existing scopes | `gh` creates a new token; old one still active | Revoke the old token at github.com/settings/tokens after fresh login |

> **Safety warning — token scope drift:** Scopes accumulate quietly. A token created months ago may carry `delete_repo` or `admin:org` that nobody remembers granting. Run `gh auth status` at the start of every new project and remove any scope you cannot justify in one sentence.
