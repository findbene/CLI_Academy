# Enable one bounded write action

## Objective

A **bounded write** has three properties: it does exactly one thing, it targets a specific resource (not a category of resources), and there is a clear way to undo it. In this lesson you will create a real GitHub issue using the API — a live write action against GitHub's servers — and then document and execute the rollback (closing the issue).

This is the first time in Path 17 that you will make a change that persists outside your local machine. That makes verifying and reversing it essential skills.

## Walkthrough

### Step 1 — Choose your target repository

<StepMeta purpose="Constrain the first live write to a repository you control so the blast radius stays small and reversible." action="Set `GH_OWNER` and `GH_REPO`, then verify the target repository exists and that your permissions include push access before attempting the write." expectedResult="You have an explicitly named target repo and evidence that the token can write there." whyItMatters="Most integration mistakes happen before the write itself, when the wrong target is chosen and the API dutifully changes the wrong resource." nextStep="Create the bounded issue next using a single POST request with a clearly labeled title and body." askTutor="Check my chosen target repository and tell me whether it is safe for this bounded write test or whether I should pick a less risky repo." />

Use a repo you own or a personal test repo. Do **not** use a team or org repo for this exercise — creating test issues on shared repositories creates noise for your teammates.

Set your repo as a shell variable to avoid retyping it:

**macOS / Linux:**
```bash
# Replace with your actual GitHub username and a repo you own
export GH_OWNER="your-github-username"
export GH_REPO="your-test-repo"

# Verify the repo exists and you have write access
gh api /repos/$GH_OWNER/$GH_REPO --jq '{name: .name, permissions: .permissions}'
```

**Windows PowerShell:**
```powershell
# Replace with your actual GitHub username and a repo you own
$env:GH_OWNER = "your-github-username"
$env:GH_REPO  = "your-test-repo"

# Verify the repo exists and you have write access
gh api /repos/$env:GH_OWNER/$env:GH_REPO | ConvertFrom-Json |
    Select-Object name, @{Name="push";Expression={$_.permissions.push}}
```

Look for `"push": true` in the output. If you see `"push": false` you do not have write access and the next step will fail with a 403.

### Step 2 — Create one bounded issue (the write action)

<StepMeta purpose="Perform one tightly scoped persistent write with a clear identifier so it can be reviewed and rolled back safely." action="Run the POST request to create the labeled test issue, capture the JSON response, and extract the issue number for follow-up checks and rollback." expectedResult="A single test issue is created in the chosen repo and you have its number and URL captured for control." whyItMatters="A bounded write is only safe when it is singular, targeted, and immediately traceable to a rollback path." nextStep="Read the issue back next so you confirm the write produced exactly the state you intended." askTutor="Review this bounded write step and tell me whether the issue creation flow is scoped tightly enough and documented clearly enough to stay reversible." />

**macOS / Linux:**
```bash
# Create a single, clearly labelled test issue
RESPONSE=$(gh api /repos/$GH_OWNER/$GH_REPO/issues \
    -X POST \
    --field title="[cli-academy-test] Lesson 17.2.3 bounded write" \
    --field body="This issue was created by lesson 17.2.3 of CLI Academy to test a bounded POST write action. Safe to close immediately.")

# Extract the issue number for cleanup
ISSUE_NUMBER=$(echo "$RESPONSE" | jq -r '.number')
echo "Issue #$ISSUE_NUMBER created at: $(echo "$RESPONSE" | jq -r '.html_url')"
```

**Windows PowerShell:**
```powershell
# Create a single, clearly labelled test issue
$responseJson = gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues `
    -X POST `
    --field title="[cli-academy-test] Lesson 17.2.3 bounded write" `
    --field body="This issue was created by lesson 17.2.3 of CLI Academy to test a bounded POST write action. Safe to close immediately."

$response = $responseJson | ConvertFrom-Json
$issueNumber = $response.number
Write-Host "Issue #$issueNumber created at: $($response.html_url)"
```

Open the URL printed in your terminal and verify the issue appears on GitHub. That is a live external write — you created something on GitHub's servers using the API.

### Step 3 — Verify the issue was created (read-back check)

<StepMeta purpose="Confirm that the write produced the exact resource and state you intended before taking any further action." action="Call the issue endpoint for the specific issue number you just created and inspect the returned number, title, state, and timestamp." expectedResult="The read-back shows the exact issue is present and currently open with the expected title." whyItMatters="Read-after-write checks catch mismatched IDs, wrong targets, or unexpected state changes before they compound into larger mistakes." nextStep="Execute the rollback next by closing the issue and verifying that the external state returns to the intended final condition." askTutor="Help me inspect this read-back response and tell me whether it proves the write happened exactly where and how I intended." />

Always read back after a write to confirm what you intended to create actually exists:

**macOS / Linux:**
```bash
# Read back the specific issue you just created
gh api /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER \
    --jq '{number: .number, title: .title, state: .state, created_at: .created_at}'
```

**Windows PowerShell:**
```powershell
# Read back the specific issue you just created
gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber |
    ConvertFrom-Json |
    Select-Object number, title, state, created_at |
    Format-List
```

Expected: `state: "open"`, title matches what you set. If the state is wrong or the title changed, stop and investigate before proceeding.

### Step 4 — Execute the rollback (close the issue)

<StepMeta purpose="Practice the reversal path immediately so the first write is paired with a tested undo operation." action="Run the PATCH request that closes the issue, then read the issue state again until you confirm it is closed." expectedResult="The test issue is rolled back cleanly and the verified final state is closed." whyItMatters="Safe integrations depend on known rollback mechanics, not just successful writes, because every external change should be reversible when possible." nextStep="Document both the write and rollback next so the pattern becomes a repeatable operational record." askTutor="Review my rollback step and tell me whether it is strong enough to count as a real bounded-write reversal instead of just an informal cleanup." />

Closing the issue is the rollback for this write action. For a real workflow, "rollback" means you know the exact API call that reverses the action before you commit to the write.

**macOS / Linux:**
```bash
# Close (rollback) the test issue
gh api /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER \
    -X PATCH \
    --field state="closed"

# Confirm it is closed
gh api /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER --jq '.state'
# Expected: "closed"
```

**Windows PowerShell:**
```powershell
# Close (rollback) the test issue
gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber `
    -X PATCH `
    --field state="closed"

# Confirm it is closed
gh api /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber |
    ConvertFrom-Json | Select-Object state
# Expected: state = closed
```

### Step 5 — Document the write action and its rollback

<StepMeta purpose="Capture the full lifecycle of the bounded write so future reviews can see what changed, why, and how it was undone." action="Create `bounded-write-record.md` with the POST endpoint, captured issue number, rollback PATCH endpoint, and final closed-state verification." expectedResult="You have a durable write record showing the action, rollback path, and resulting scope implications." whyItMatters="Documented bounded writes are easier to audit, reproduce, and improve than ad hoc API mutations done from memory." nextStep="Use the next lessons to build the incident-response and risk-review side of secure integrations now that you have experienced a real reversible write." askTutor="Review my bounded write record and tell me whether it proves both the original write and the rollback with enough detail for a future audit." />

**macOS / Linux:**
```bash
cd ~/projects/integration-safety

cat > bounded-write-record.md << EOF
# Bounded Write Record

## Write action
- Date: $(date +%Y-%m-%d)
- Service: GitHub Issues API
- HTTP method: POST
- Endpoint: /repos/$GH_OWNER/$GH_REPO/issues
- Scope required: repo
- Issue number created: #$ISSUE_NUMBER

## Rollback action
- HTTP method: PATCH
- Endpoint: /repos/$GH_OWNER/$GH_REPO/issues/$ISSUE_NUMBER
- Field changed: state → closed
- Rollback verified: yes — confirmed state = "closed" via read-back

## Scope impact
- This write used the full `repo` scope.
- An alternative fine-grained PAT with `issues:write` on this repo only would further restrict blast radius.
EOF

cat bounded-write-record.md
```

**Windows PowerShell:**
```powershell
Set-Location "$HOME\projects\integration-safety"

$record = @"
# Bounded Write Record

## Write action
- Date: $(Get-Date -Format 'yyyy-MM-dd')
- Service: GitHub Issues API
- HTTP method: POST
- Endpoint: /repos/$env:GH_OWNER/$env:GH_REPO/issues
- Scope required: repo
- Issue number created: #$issueNumber

## Rollback action
- HTTP method: PATCH
- Endpoint: /repos/$env:GH_OWNER/$env:GH_REPO/issues/$issueNumber
- Field changed: state → closed
- Rollback verified: yes — confirmed state = 'closed' via read-back

## Scope impact
- This write used the full ``repo`` scope.
- An alternative fine-grained PAT with ``issues:write`` on this repo only would further restrict blast radius.
"@

$record | Out-File -FilePath "bounded-write-record.md" -Encoding utf8
Get-Content bounded-write-record.md
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `403 Forbidden` on the POST call | Token does not have write access to the repo | Verify `"push": true` in Step 1; if false, choose a repo you own |
| `422 Unprocessable Entity` | Missing required field in the POST body | Both `title` and `body` fields must be provided |
| Issue created but with wrong title | Shell variable expansion in the field value | Quote the `--field` value carefully; avoid `$` inside double-quoted strings in bash |
| Read-back after POST returns 404 | Wrong issue number captured | Print the full response JSON first: `echo "$RESPONSE"` and extract `.number` manually |
| PATCH close returns `200` but state still `open` | GitHub caches issue state briefly | Wait 2–3 seconds and re-run the read-back |
| Test issue left open in a team repo | Wrong repo was targeted | Close it manually, then apologize to teammates |

> **Safety warning — write actions have real consequences:** A POST to `/repos/ORG/REPO/issues` on a team repo sends email notifications to every watcher. A PATCH to close an issue belonging to someone else may break their workflow. Always verify `GH_OWNER` and `GH_REPO` point to a repo you own before running any write command.
