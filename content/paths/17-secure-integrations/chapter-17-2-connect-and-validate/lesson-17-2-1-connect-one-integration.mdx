# Connect one integration

## Objective

Connecting to an external API for the first time involves two things that beginners often get backwards: they focus on *getting it to work* and forget to document *where the credentials live and what they can do*. In this lesson you will authenticate to GitHub's API using the `gh` CLI, make a real API call to confirm the connection works, store your credential reference safely in a `.env` file, and write connection notes that document the credential type, scope, and storage location.

## Walkthrough

### Step 1 — Authenticate with the GitHub API using gh

<StepMeta purpose="Establish a safe first connection path that uses GitHub CLI-managed auth instead of raw token handling." action="Run `gh auth login`, complete the device-flow prompts, and make sure the login finishes through the browser-based approval flow." expectedResult="GitHub CLI stores an active OAuth token in the system keychain without requiring you to paste the raw token into files." whyItMatters="Secure connection setup is about both successful auth and avoiding token exposure through shell history, plaintext files, or copy-paste mistakes." nextStep="Make a real API call next so you know the connection works before documenting anything about its scope or storage location." askTutor="Walk me through the safest way to complete `gh auth login` and tell me how to avoid leaking a token during the initial connection setup." />

The `gh` CLI handles OAuth token storage securely in your system keychain. You do not paste tokens into files or environment variables manually at this stage — `gh` manages that for you.

**macOS / Linux:**
```bash
gh auth login
```

**Windows PowerShell:**
```powershell
gh auth login
```

When prompted:
1. Select **GitHub.com** (not Enterprise, unless that is your setup)
2. Select **HTTPS** as the protocol
3. Select **Login with a web browser**
4. Copy the one-time code shown in your terminal, press Enter to open the browser
5. Paste the code at github.com/login/device and click Authorize

You should see: `✓ Authentication complete. Press Enter to continue...`

### Step 2 — Verify the connection with a real API call

<StepMeta purpose="Confirm the integration is genuinely live by reading a real API response through the stored GitHub auth." action="Run `gh api user` and inspect the returned profile JSON to verify the token is being used successfully behind the scenes." expectedResult="You receive a live GitHub user profile response, proving the CLI auth is active and usable for API calls." whyItMatters="Connection setup is only trustworthy once a real request succeeds, not just because a login flow claimed to finish." nextStep="Inspect the auth status and accessible repos next so you understand what the token can currently see." askTutor="Help me interpret the `gh api user` output and tell me what it proves about the state of my GitHub connection and token storage." />

Fetch your GitHub user profile to confirm the token works:

**macOS / Linux:**
```bash
gh api user
```

**Windows PowerShell:**
```powershell
gh api user
```

Expected output (truncated):
```json
{
    "login": "yourname",
    "id": 123456,
    "name": "Your Name",
    "public_repos": 12,
    "private_repos": 3,
    "plan": {
        "name": "pro"
    }
}
```

This is a live API response from GitHub. The `gh api` command uses your stored OAuth token automatically — you never handle the raw token string.

### Step 3 — Check what the token can see

<StepMeta purpose="Measure the current visibility and scope of the connected token before any workflow starts relying on it." action="Run `gh auth status` and the repo-listing command, then note both the granted scopes and the set of repositories visible to the token." expectedResult="You understand both the scope grant and the reach of the current token across GitHub resources." whyItMatters="A token can be technically connected but still dangerously overpowered, and visibility checks expose that blast radius early." nextStep="Document the credential type and storage model next using a safe `.env.example` reference file that contains no secrets." askTutor="Review my auth-status and repo-visibility output and tell me whether the token's current reach looks broader than this workflow should tolerate." />

**macOS / Linux:**
```bash
# Show your active auth status and scopes
gh auth status

# List the repos your token can access
gh api /user/repos --jq '.[].full_name' | head -10
```

**Windows PowerShell:**
```powershell
# Show your active auth status and scopes
gh auth status

# List the repos your token can access
gh api /user/repos | ConvertFrom-Json | Select-Object -First 10 -ExpandProperty full_name
```

### Step 4 — Document the connection in a .env reference file

<StepMeta purpose="Capture where the credential lives and how it is managed without ever writing the actual secret into the repo." action="Create `.env.example` as a reference file, documenting credential type, storage location, scope source, refresh path, and revocation method only." expectedResult="You have a safe credential reference file that explains the integration setup without containing any token value." whyItMatters="Good integration docs tell future you where auth lives and how to rotate it while still keeping secret material out of source control." nextStep="Make sure `.env` is ignored next so a future real environment file cannot be committed by accident." askTutor="Review my `.env.example` and tell me whether it documents the connection clearly without crossing the line into secret exposure." />

Never paste raw token values into files. Instead, document *where* the credential lives and what it grants. Create a `.env.example` file that serves as a safe reference:

**macOS / Linux:**
```bash
cd ~/projects/integration-safety

cat > .env.example << 'EOF'
# Integration connection reference (NOT a real .env — no secrets here)
# -----------------------------------------------------------------
# Service: GitHub
# Credential type: OAuth token managed by gh CLI
# Storage location: System keychain (managed by gh auth login)
# Scopes granted: see `gh auth status`
# How to refresh: gh auth login (follow prompts)
# How to revoke: gh auth logout / github.com/settings/tokens
#
# If your workflow needs a raw token (e.g., CI/CD environment variable):
# GITHUB_TOKEN=<generate at github.com/settings/tokens — use fine-grained PAT>
# Store in: your CI/CD secrets vault, NOT in this file
EOF

cat .env.example
```

**Windows PowerShell:**
```powershell
Set-Location "$HOME\projects\integration-safety"

$envExample = @'
# Integration connection reference (NOT a real .env — no secrets here)
# -----------------------------------------------------------------
# Service: GitHub
# Credential type: OAuth token managed by gh CLI
# Storage location: System keychain (managed by gh auth login)
# Scopes granted: see `gh auth status`
# How to refresh: gh auth login (follow prompts)
# How to revoke: gh auth logout / github.com/settings/tokens
#
# If your workflow needs a raw token (e.g., CI/CD environment variable):
# GITHUB_TOKEN=<generate at github.com/settings/tokens — use fine-grained PAT>
# Store in: your CI/CD secrets vault, NOT in this file
'@

$envExample | Out-File -FilePath ".env.example" -Encoding utf8
Get-Content .env.example
```

### Step 5 — Add .env to .gitignore immediately

<StepMeta purpose="Put a hard guardrail in place before any real `.env` file exists with sensitive values inside it." action="Ensure `.env` appears in `.gitignore` and print the matching entry so you can verify the ignore rule is active." expectedResult="The repo now has an explicit ignore rule protecting future `.env` files from accidental commits." whyItMatters="Credential leaks often happen before people think they are dealing with production secrets, so early ignore rules are a low-cost, high-value safeguard." nextStep="Use the next lesson to validate the integration with read-only GET calls before you attempt any write operation." askTutor="Check whether my `.gitignore` change is sufficient for this workflow and tell me if any other secret-bearing files should be protected at the same time." />

If you ever create a real `.env` file with secrets, it must never be committed to git:

**macOS / Linux:**
```bash
# Check if .gitignore already excludes .env
grep -q "^\.env" .gitignore 2>/dev/null || echo ".env" >> .gitignore
echo ".env is in .gitignore: $(grep '\.env' .gitignore)"
```

**Windows PowerShell:**
```powershell
$gitignorePath = ".gitignore"
$hasEnv = Test-Path $gitignorePath -and (Select-String -Path $gitignorePath -Pattern "^\.env" -Quiet)
if (-not $hasEnv) { Add-Content -Path $gitignorePath -Value ".env" }
Write-Host ".env entry:" (Select-String -Path $gitignorePath -Pattern "\.env")
```

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `gh auth login` opens browser but code says "expired" | You waited too long after copying the one-time code | Run `gh auth login` again; codes expire in ~15 minutes |
| `gh api user` returns `401 Unauthorized` | Auth did not complete successfully | Run `gh auth status`; if no token shown, run `gh auth login` again |
| `gh api /user/repos` returns empty array `[]` | Token has no `repo` scope | Run `gh auth login --scopes repo` for a fresh token with that scope |
| Raw token value appears in terminal history | You ran `export GITHUB_TOKEN=ghp_...` manually | Clear history with `history -d` (bash) or `Clear-History` (PowerShell); rotate the token immediately |
| `.env` was committed to git | `.gitignore` was missing the entry | Run `git rm --cached .env`, add `.env` to `.gitignore`, commit, and rotate every secret in the file |

> **Safety warning — token leaks through history:** Running `gh api user -H "Authorization: token ghp_YOURTOKEN"` instead of using `gh auth login` puts your raw token into shell history, process lists, and potentially server logs. Always use `gh auth login` so the token stays in the keychain, not in plaintext.
