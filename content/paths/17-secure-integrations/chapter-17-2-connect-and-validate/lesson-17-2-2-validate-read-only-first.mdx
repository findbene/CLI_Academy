# Validate read-only first

## Objective

The safest order when connecting to any API is: **read first, write never until proven safe**. Reading from a live API confirms your credentials work, shows you the data shape you will work with, and builds confidence — all without any risk of creating, modifying, or deleting anything. Write access is a privilege you earn after the read-only baseline is solid.

In this lesson you will make three real read-only API calls against GitHub using the `gh` CLI, inspect the responses, and confirm that your integration cannot accidentally mutate data at this stage.

## Walkthrough

### Step 1 — List your accessible repositories (read-only)

<StepMeta purpose="Begin validation with a harmless GET call that proves your token can read data without changing anything." action="Run the repo-listing command and inspect the returned `owner/repo` names from the first set of accessible repositories." expectedResult="You get a real list of repositories the token can read, confirming basic API access through a read-only request." whyItMatters="Read-only validation is the safest way to establish trust in a new integration before any persistent change is even possible." nextStep="Use a second GET call against issues data next so you can confirm the token can read specific workflow-relevant resources." askTutor="Interpret this repo-list output for me and tell me what it proves about the integration while staying fully read-only." />

**macOS / Linux:**
```bash
# List up to 10 repos your token can see
gh api /user/repos --jq '.[].full_name' | head -10
```

**Windows PowerShell:**
```powershell
# List up to 10 repos your token can see
gh api /user/repos | ConvertFrom-Json | Select-Object -First 10 -ExpandProperty full_name
```

You should see a list of `owner/repo-name` strings. This is a GET request — it reads data and returns it. Nothing on GitHub changes.

### Step 2 — Read issues from a specific repository

<StepMeta purpose="Validate that the token can read the exact type of repository data your later workflow depends on." action="Run the issues endpoint against a real repository you can access and inspect either the first five issues or the empty array response." expectedResult="You confirm that the issues endpoint is readable and returns structured data for a real repository." whyItMatters="Validating the exact endpoint shape early reduces surprises later when you build automation that depends on issue fields and states." nextStep="Read repository metadata next so you capture another non-mutating endpoint and compare the data shape." askTutor="Help me evaluate this issues read and tell me whether the result looks like a successful endpoint call, a permissions problem, or just a repository with no issues." />

Replace `OWNER` and `REPO` with a real repository you have access to (your own works fine):

**macOS / Linux:**
```bash
# Read the 5 most recent open issues
gh api /repos/OWNER/REPO/issues \
    --jq '.[] | {number: .number, title: .title, state: .state}' \
    | head -20
```

**Windows PowerShell:**
```powershell
# Read the 5 most recent open issues
$issues = gh api /repos/OWNER/REPO/issues | ConvertFrom-Json
$issues | Select-Object -First 5 number, title, state | Format-Table -AutoSize
```

If the repo has no issues you will get an empty array `[]` — that is fine. The call succeeded and returned data.

### Step 3 — Read repository metadata

<StepMeta purpose="Round out the read-only baseline by inspecting a second endpoint that returns repository configuration details." action="Run the repo metadata request and check fields such as name, visibility, default branch, and stars in the response." expectedResult="You receive a structured metadata response for the target repository without making any change to it." whyItMatters="Multiple successful GET calls build confidence that the integration is healthy while still keeping the risk profile at zero writes." nextStep="Confirm the HTTP method next so you can prove the validation phase stayed read-only at the transport level as well." askTutor="Review this repository metadata response and tell me what it says about the repository while confirming that the request remained read-only." />

**macOS / Linux:**
```bash
# Read repo details: default branch, visibility, stars
gh api /repos/OWNER/REPO \
    --jq '{name: .name, visibility: .visibility, default_branch: .default_branch, stars: .stargazers_count}'
```

**Windows PowerShell:**
```powershell
# Read repo details: default branch, visibility, stars
gh api /repos/OWNER/REPO | ConvertFrom-Json |
    Select-Object name, visibility, default_branch, stargazers_count |
    Format-List
```

### Step 4 — Confirm read-only by checking the HTTP method used

<StepMeta purpose="Verify not just that the calls worked, but that they truly used a safe non-mutating HTTP method." action="Run the verbose `gh api /user` command and inspect the response trace to confirm the request pattern stayed on GET with a 200 response." expectedResult="You can show evidence that the validation phase used GET rather than POST or PATCH." whyItMatters="Method-level confirmation prevents accidental escalation from a harmless read into an unintended write caused by copied flags or modified commands." nextStep="Write the read-only validation record next so the successful baseline is preserved before any bounded write is attempted." askTutor="Help me read this verbose output and tell me whether it proves the request path stayed read-only or whether anything unexpected happened." />

All three calls above use HTTP GET. The `gh api` command defaults to GET, which is always read-only. You can confirm this with the `--verbose` flag:

**macOS / Linux:**
```bash
gh api /user --verbose 2>&1 | grep "< HTTP"
# Should show: < HTTP/2 200
```

**Windows PowerShell:**
```powershell
gh api /user --verbose 2>&1 | Select-String "< HTTP"
# Should show: < HTTP/2 200
```

`200 OK` with a GET method = read-only success. Any `4xx` means the call was rejected (check credentials). `5xx` means a GitHub server error — retry later.

### What can go wrong

| Symptom | Likely cause | Fix |
|---|---|---|
| `404 Not Found` on `/repos/OWNER/REPO/issues` | Repository doesn't exist or you mistyped OWNER/REPO | Run `gh api /user/repos --jq '.[].full_name'` to find valid names to use |
| `401 Unauthorized` on every call | Token expired or was revoked | Run `gh auth login` to refresh |
| `403 Forbidden` on a private repo | Token lacks `repo` scope | Run `gh auth status` to check scopes; re-login with `gh auth login --scopes repo` |
| Empty array `[]` on issues endpoint | Repo has no open issues | That is correct behavior — the call succeeded |
| `--verbose` shows POST instead of GET | You accidentally added `-X POST` to the command | Remove `-X POST`; GET is the default and is always read-only |

> **Safety warning — GET is not always harmless:** The GitHub API `GET /repos/OWNER/REPO/traffic/views` requires push access to the repo, not just read. Some APIs log or rate-limit read calls in ways that affect other users. Always check API rate limits with `gh api /rate_limit` before scripting many calls in a loop.
