# Reading Error Messages with Claude

## What you will build

Three documented error analyses in `LEARNING_LOG.md` — one for each of three common Python error types (`NameError`, `FileNotFoundError`, and a port conflict). Each entry records: the error message, a Claude explanation using the "what, where, why" framework, and the fix you applied.

## Why this matters

Most beginners read an error message, panic, and immediately start changing things at random. This is the worst possible approach. Error messages are not insults — they are precise technical statements. They tell you exactly what went wrong, exactly where it went wrong, and often why.

Claude is exceptionally good at decomposing error messages because it has seen thousands of them. The pattern is simple: paste the full error into Claude, tell it what you were trying to do, and ask it to explain using the "what, where, why" framework. Once you understand the error, the fix is usually obvious.

The three error types in this lesson cover a large fraction of real-world bugs: `NameError` (using something that doesn't exist), `FileNotFoundError` (looking for something that isn't there), and port conflicts (infrastructure collision). These are not toy examples — they are the actual errors that slow people down on real projects.

## Before you start

- Python 3 installed.
- The project directory from Lesson 23.1.1, or a new one.
- `LEARNING_LOG.md` already exists, or you will create it.
- Claude Code running in your project directory.

## The walkthrough

### Step 1 — The "what, where, why" framework

Before touching Claude, learn to read an error message yourself using this decomposition:

- **What**: The error type and message. `NameError: name 'process_order' is not defined` — What happened? A name lookup failed.
- **Where**: The file path and line number in the traceback. `File "app.py", line 47` — Where did it fail?
- **Why**: The root cause. Why was the name not defined? Usually because of a typo, a missing import, or a function defined after it is called.

Claude's explanation should always answer all three. If an explanation only covers "what" without "why", ask a follow-up: "Why does Python not find this name?"

**Script 3: Port conflict simulation**

This one does not crash with a Python exception — it produces an OS-level error that looks different. We will simulate what you see when two processes try to bind the same port.

**macOS / Linux:**
```bash
cat > port_conflict_demo.py << 'EOF'
# port_conflict_demo.py — demonstrates an address-already-in-use error

import socket

PORT = 9876

# Start a "server" that holds the port
server1 = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server1.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 0)
server1.bind(('localhost', PORT))
server1.listen(1)
print(f"Server 1 bound to port {PORT}")

# Try to start a second "server" on the same port — this will fail
server2 = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server2.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 0)
server2.bind(('localhost', PORT))  # OSError: [Errno 98] Address already in use
print(f"Server 2 also bound to port {PORT}")

server1.close()
server2.close()
EOF
```

### Step 5 — Document the three analyses in LEARNING_LOG.md

**macOS / Linux:**
```bash
cat >> LEARNING_LOG.md << 'EOF'

## Error Message Analyses (Lesson 23.1.2)

### Error 1: NameError

**Error:**
```
NameError: name 'proccess_order' is not defined. Did you mean: 'process_order'?
```

**What**: Python could not find a name called `proccess_order` in the current scope.
**Where**: Line 6 of name_error_demo.py, inside the `process_orders` function.
**Why**: The function name was misspelled (`proccess` with a double 'c') at the call site. The correctly spelled `process_order` function exists but was not matched because Python names are case- and spelling-sensitive.

**Fix**: Correct the typo on line 6 — change `proccess_order` to `process_order`.

---

### Error 2: FileNotFoundError

**Error:**
```
FileNotFoundError: [Errno 2] No such file or directory: 'config/app_settings.json'
```

**What**: The `open()` call tried to open a file that does not exist at the specified path.
**Where**: Line 6 of file_not_found_demo.py, inside `load_config`.
**Why**: The file `config/app_settings.json` does not exist in the working directory where the script was run. Either the file was never created, or the script was run from the wrong directory.

**Fix**: Create the config file (`mkdir -p config && echo '{}' > config/app_settings.json`) or add a check with `os.path.exists()` before opening.

---

### Error 3: Port Conflict (OSError)

**Error:**
```
OSError: [Errno 98] Address already in use
```

**What**: The OS refused to bind a second socket to port 9876 because that port is already in use.
**Where**: Line 17 of port_conflict_demo.py, at the `server2.bind()` call.
**Why**: Only one process can hold a TCP port at a time. Server 1 was already bound and listening on port 9876 when Server 2 tried to claim the same address.

**Fix**: Use a different port number for Server 2, or shut down Server 1 first with `server1.close()`. In real applications: check what process is holding the port with `lsof -i :9876` (macOS/Linux) or `netstat -ano | findstr :9876` (Windows).

EOF
```

## What can go wrong

| Problem | Cause | Fix |
|---------|-------|-----|
| `NameError` says "not defined" but the function clearly exists | Python's suggestion "Did you mean X?" is a hint that X is the correct spelling | Look for typos; Python 3.10+ gives suggestions, but earlier versions just say "not defined" |
| `FileNotFoundError` even though the file exists | Script is running from a different directory than expected | Run `import os; print(os.getcwd())` to confirm the working directory, then use an absolute path |
| Port conflict error on a different port number | Port 9876 was already taken on your machine | Change `PORT = 9876` to any unused port; run `lsof -i :9876` to check |
| Claude gives a one-line answer without "what, where, why" | The prompt did not request the framework explicitly | Add "structure your answer using what, where, why" to the prompt |
| Claude's explanation is correct but the fix does not work | There is a second bug the error was masking | After applying the fix, run the script again — a new error is a new bug, not a failure of the first fix |
